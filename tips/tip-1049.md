# TIP-1049: Admin Access Keys

## Abstract

This TIP adds an `admin` boolean flag to the `AuthorizedKey` storage struct and the `KeyRestrictions` configuration. When `admin` is `true`, the access key may authorize and revoke other access keys on behalf of the account, and is eligible for TIP-1020 signature verification. Non-admin access keys retain current behavior: they cannot manage keys and are invisible to TIP-1020.

## Motivation

Access keys today are strictly subordinate: they can sign transactions within their spending limits and call scopes, but they cannot authorize new keys or participate in onchain signature verification. This creates two practical problems:

1. **Cross-device key management.** A user with a passkey on their phone cannot authorize a second passkey on their laptop without a transaction from the root EOA key. In practice, multi-device setups require the root key to be available for every new device onboarding, which is inconvenient and fragile.

2. **Onchain signature verification.** TIP-1020 allows contracts to verify Tempo signatures, but there is no way for a contract to verify that a signature was produced by an authorized access key of a given account. Contracts that want to gate actions on "this account approved this message" (e.g. ERC-1271-style flows, gasless approvals, cross-chain authorizations) cannot use access keys for this purpose.

Adding an `admin` flag solves both problems with a single, conservative mechanism: admin keys can manage other keys and can be verified via TIP-1020, while non-admin keys remain restricted.

## Assumptions

- The root EOA key is always implicitly admin. The `admin` flag only applies to access keys registered in the AccountKeychain precompile.
- Admin keys inherit all existing access key restrictions (expiry, spending limits, call scopes). The `admin` flag is additive ŌĆö it grants key-management and verification capabilities but does not bypass other restrictions.
- An admin key cannot grant the `admin` flag to keys it authorizes unless the admin key itself was authorized with that capability. This prevents privilege escalation: only the root key (or another admin key) can create admin keys.

## Interface Changes

### `KeyRestrictions`

Add an `admin` field to the `KeyRestrictions` struct:

```solidity
struct KeyRestrictions {
    uint64 expiry;
    bool enforceLimits;
    TokenLimit[] limits;
    bool allowAnyCalls;
    CallScope[] allowedCalls;
    bool admin;               // NEW
}
```

### `KeyInfo`

Add an `isAdmin` field to the `KeyInfo` return struct:

```solidity
struct KeyInfo {
    SignatureType signatureType;
    address keyId;
    uint64 expiry;
    bool enforceLimits;
    bool isRevoked;
    bool isAdmin;             // NEW
}
```

### `KeyAuthorized` Event

Add `isAdmin` to the `KeyAuthorized` event:

```solidity
event KeyAuthorized(
    address indexed account,
    address indexed publicKey,
    uint8 signatureType,
    uint64 expiry,
    bool isAdmin                // NEW
);
```

## Behavioral Changes

### Key Authorization by Access Keys

Currently, the handler rejects any transaction where an access key attempts to authorize a different key (`AccessKeyCannotAuthorizeOtherKeys`). With this TIP:

- If the signing access key has `is_admin = true`, the transaction MAY include a `KeyAuthorization` for a different `key_id`. The `KeyAuthorization` must still be signed by the root account key.
- If the signing access key has `is_admin = false`, the existing restriction remains: the key can only authorize itself (same-tx auth+use).

### Key Revocation by Access Keys

Currently, only the root key can call `revokeKey`. With this TIP:

- Admin access keys MAY call `revokeKey` to revoke other access keys on the same account.
- Non-admin access keys MUST NOT call `revokeKey`.

The `ensure_admin_caller` check in the AccountKeychain precompile is updated to also accept admin access keys (identified via the `transaction_key` transient storage).

### TIP-1020 Signature Verification

Currently, TIP-1020's `recover` and `verify` functions reject keychain-type signatures (`0x03`, `0x04` prefixes) with `InvalidFormat`. With this TIP:

- TIP-1020 verification for keychain signatures is accepted when the recovered key has `is_admin = true` in the AccountKeychain for the given account.
- The verification flow becomes:
  1. Decode the keychain signature to extract the inner signature and key ID.
  2. Look up the key in AccountKeychain for the claimed account.
  3. Verify the key exists, is not revoked, is not expired, and has `is_admin = true`.
  4. Verify the inner signature against the key ID using standard TIP-1020 logic.
- If the key is not admin, verification MUST revert.

### Admin Key Privilege Propagation

An admin access key MAY authorize new keys with `admin = true`, enabling delegation chains. This is intentional: it allows multi-device setups where any trusted device can onboard new devices without the root key.

Revocation of an admin key does NOT automatically revoke keys that it authorized. Each key is independently tracked; cascading revocation is out of scope for this TIP. Accounts that need cascading revocation should revoke downstream keys explicitly.

## Transaction Encoding

The `KeyAuthorization` RLP encoding gains an optional trailing `admin` field:

```
RLP: [chain_id, key_type, key_id, expiry?, limits?, allowed_calls?, admin?]
```

- `None` (omitted or `0x80`) = non-admin key (default)
- `Some(true)` / `0x01` = admin key

This is backward compatible: existing key authorizations without the field default to non-admin.

# Invariants

1. **Admin-only key management.** A non-admin access key MUST NOT authorize or revoke other keys. Transactions that violate this MUST be rejected with `AccessKeyCannotAuthorizeOtherKeys`.

2. **Root key equivalence for admin operations.** Admin access keys MUST be subject to the same validation as root key operations for `authorizeKey` and `revokeKey`: the `KeyAuthorization` must be signed by the root account, and all existing checks (expiry, revocation, signature type) apply.

3. **No implicit privilege escalation.** The `admin` flag does not bypass spending limits, call scopes, or expiry. An admin key with a 1-hour expiry expires after 1 hour regardless of admin status.

4. **TIP-1020 admin gate.** Keychain signature verification via TIP-1020 MUST succeed only for keys with `is_admin = true`. Non-admin keys MUST cause verification to revert.

5. **Backward compatibility.** Existing access keys (authorized before this TIP activates) default to `is_admin = false`. Their behavior is unchanged.

6. **No cascading revocation.** Revoking an admin key MUST NOT automatically revoke keys that were authorized while that admin key was active. Each key's revocation status is independent.
