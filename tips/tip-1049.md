# TIP-1049: Admin Access Keys

## Abstract

This TIP adds an `admin` boolean flag to the `AuthorizedKey` storage struct and the `KeyRestrictions` configuration. When `admin` is `true`, the access key may authorize and revoke other access keys on behalf of the account, including the root key. `revokeKey` is extended so the root key itself can be permanently revoked. The TIP also adds helper views to the `AccountKeychain` precompile so external contracts can resolve a key id back to its root account and inspect its admin status, without modifying TIP-1020 itself.

## Motivation

Access keys today are strictly subordinate: they can sign transactions within their spending limits and call scopes, but they cannot authorize new keys or participate in onchain key-management flows. This creates two practical problems:

1. **Cross-device key management.** A user with a passkey on their phone cannot authorize a second passkey on their laptop without a transaction from the root EOA key. In practice, multi-device setups require the root key to be available for every new device onboarding, which is inconvenient and fragile.

2. **Onchain account ↔ key association.** Contracts that want to gate actions on "this key belongs to this account" have no canonical way to resolve a public key back to the account that authorized it, or to check whether the key has admin capabilities.

Adding an `admin` flag plus a small set of read helpers in the `AccountKeychain` precompile solves both problems without entangling TIP-1020. TIP-1020 remains a pure signature-verification primitive; contracts that want admin-key semantics call the keychain explicitly.

## Assumptions

- The root EOA key is implicitly admin until it is explicitly revoked via `revokeKey`. The `admin` flag only applies to access keys registered in the AccountKeychain precompile.
- Admin keys MAY be authorized with spending limits, call scopes, and expiry just like non-admin keys. However, callers should be aware that an admin key can effectively bypass its own spending-limit and call-scope restrictions by authorizing a new, unrestricted key. Expiry remains a meaningful constraint because the admin key cannot extend its own expiry.
- An admin key cannot grant the `admin` flag to keys it authorizes unless the admin key itself was authorized with that capability. This prevents privilege escalation: only the root key (or another admin key) can create admin keys.
- Root-key revocation is permanent and irreversible. An account whose root key is revoked can only be operated through admin access keys; if no usable admin key exists at the time of revocation (or all admin keys later expire/are revoked), the account becomes permanently inaccessible. This footgun is intentional and consistent with existing access-key revocation semantics.
- TIP-1020 remains unchanged. It continues to act as a pure signature-recovery / verification primitive. Contracts that need to verify "an admin access key of account X signed this" call `AccountKeychain.verifySignature`, which composes TIP-1020 verification with an admin-status check.

# Specification

## Storage Layout

The `AuthorizedKey` packed struct gains a new `is_admin` boolean at byte 11:

```
Byte 0:     signature_type (u8)
Bytes 1–8:  expiry (u64, little-endian)
Byte 9:     enforce_limits (bool)
Byte 10:    is_revoked (bool)
Byte 11:    is_admin (bool)        ← NEW
```

This adds one byte to the existing packed slot. No new storage slots are introduced for the admin flag.

### Root key revocation slot

Root-key revocation reuses the existing `keys[account][keyId]` mapping by treating `keyId == account` (the EOA address itself) as the canonical slot for the root key's `AuthorizedKey`. When the root key is revoked, this slot is written with `is_revoked = true` and all other fields cleared, identical to how an access key is marked revoked. No additional storage layout is introduced.

The slot at `keys[account][account]` is reserved for root-key state under this TIP and MUST NOT be used as a regular access-key slot. Authorization of an access key with `keyId == account` MUST be rejected with `InvalidKeyId`.

## Interface Changes

### `KeyRestrictions`

Add an `admin` field to the `KeyRestrictions` struct:

```solidity
struct KeyRestrictions {
    uint64 expiry;
    bool enforceLimits;
    TokenLimit[] limits;
    bool allowAnyCalls;
    CallScope[] allowedCalls;
    bool admin;               // NEW
}
```

### `KeyInfo`

Add an `isAdmin` field to the `KeyInfo` return struct:

```solidity
struct KeyInfo {
    SignatureType signatureType;
    address keyId;
    uint64 expiry;
    bool enforceLimits;
    bool isRevoked;
    bool isAdmin;             // NEW
}
```

### `KeyAuthorized` Event

Add `isAdmin` to the `KeyAuthorized` event:

```solidity
event KeyAuthorized(
    address indexed account,
    address indexed publicKey,
    uint8 signatureType,
    uint64 expiry,
    bool isAdmin                // NEW
);
```

### New AccountKeychain Helpers

Three read-only helpers are added to the AccountKeychain precompile so external contracts can reason about key ↔ account associations and verify admin-key signatures:

```solidity
/// Returns true if `keyId` is currently an authorized, non-revoked,
/// non-expired key on `account` with `is_admin = true`.
function isAdminKey(address account, address keyId) external view returns (bool);

/// Returns the admin status of `keyId` for `account`, plus its full KeyInfo.
/// Reverts if the key is unknown.
function adminInfo(address account, address keyId)
    external
    view
    returns (bool isAdmin, KeyInfo memory info);

/// Verifies that `signature` over `digest` was produced by an admin access key
/// of `account`. Internally calls TIP-1020 `verify` against the recovered
/// `keyId` and then checks `isAdminKey(account, keyId)`. Returns true on
/// success, false on signature failure or non-admin key.
function verifySignature(
    address account,
    bytes32 digest,
    bytes calldata signature
) external view returns (bool);
```

`verifySignature` is the canonical entry point for ERC-1271-style contracts that want to accept admin access keys. It is a thin composition of TIP-1020 verification with an admin-status check; TIP-1020 itself is not modified.

## Behavioral Changes

### Key Authorization by Access Keys

Currently, the handler rejects any transaction where an access key attempts to authorize a different key (`AccessKeyCannotAuthorizeOtherKeys`). With this TIP:

- If the signing access key has `is_admin = true`, the transaction MAY include a `KeyAuthorization` for a different `key_id`. The `KeyAuthorization` must still be signed by the root account key.
- If the signing access key has `is_admin = false`, the existing restriction remains: the key can only authorize itself (same-tx auth+use).

### Key Revocation by Access Keys

Currently, only the root key can call `revokeKey`, and only access keys can be revoked. With this TIP:

- Admin access keys MAY call `revokeKey` to revoke other access keys on the same account.
- Non-admin access keys MUST NOT call `revokeKey`.
- `revokeKey(keyId)` accepts `keyId == account` (the EOA address itself) as a request to revoke the **root key**. This MAY be called by either the root key (self-revocation) or an admin access key. Once revoked, the root key cannot sign transactions for the account and cannot be re-authorized.
- `revokeKey(keyId)` for any other registered access key works as before, with the addition that admin access keys (not just the root key) may be the caller. Admin and non-admin access keys are revocable by the same call path; the only thing the `is_admin` flag of the *target* key affects is the resulting capability set lost.

The `ensure_admin_caller` check in the AccountKeychain precompile is updated to also accept admin access keys (identified via the `transaction_key` transient storage). For root-key revocation, the precompile additionally checks that `keyId == account` and writes the revocation marker into `keys[account][account]`.

### Effects of Root Key Revocation

After the root key is revoked:

- Transaction validation MUST reject any transaction whose recovered signer is the root EOA but whose account has `keys[account][account].is_revoked == true`. The error is `RootKeyRevoked`.
- All admin access keys remain fully functional. They retain authorization, revocation, and signing capabilities.
- `AccountKeychain.verifySignature(account, digest, sig)` MUST return `false` for signatures recovered to the root key after revocation. Only non-revoked admin access keys produce a `true` result.
- Re-authorization of the root key is impossible. The revocation marker at `keys[account][account]` is permanent, mirroring access-key revocation semantics.

If no admin access key exists at the moment of root revocation, the account becomes permanently inaccessible. The precompile does not enforce a minimum admin-key count; callers are responsible for ensuring an admin key is in place before revoking root.

### TIP-1020 Compatibility

TIP-1020 is **not modified** by this TIP. Contracts that wish to accept admin access keys call `AccountKeychain.verifySignature(account, digest, signature)`, which internally invokes TIP-1020 verification and then checks that the recovered key id is an admin access key of `account`.

### Admin Key Privilege Propagation

An admin access key MAY authorize new keys with `admin = true`, enabling delegation chains. This is intentional: it allows multi-device setups where any trusted device can onboard new devices without the root key.

Revocation of an admin key does NOT automatically revoke keys that it authorized. Each key is independently tracked; cascading revocation is out of scope for this TIP. Accounts that need cascading revocation should revoke downstream keys explicitly.

## Transaction Encoding

The `KeyAuthorization` RLP encoding gains an optional trailing `admin` field:

```
RLP: [chain_id, key_type, key_id, expiry?, limits?, allowed_calls?, admin?]
```

- `None` (omitted or `0x80`) = non-admin key (default)
- `Some(true)` / `0x01` = admin key

This is backward compatible: existing key authorizations without the field default to non-admin.

# Invariants

1. **Admin-only key management.** A non-admin access key MUST NOT authorize or revoke other keys. Transactions that violate this MUST be rejected with `AccessKeyCannotAuthorizeOtherKeys`.

2. **Root key equivalence for admin operations.** Admin access keys MUST be subject to the same validation as root key operations for `authorizeKey` and `revokeKey`: the `KeyAuthorization` must be signed by the root account, and all existing checks (expiry, revocation, signature type) apply.

3. **TIP-1020 unchanged.** TIP-1020's `recover` and `verify` MUST behave identically to pre-TIP-1049 semantics. Admin-key signature verification is exposed via `AccountKeychain.verifySignature`, which is the only path that combines TIP-1020 verification with an admin-status check.

4. **Backward compatibility.** Existing access keys (authorized before this TIP activates) default to `is_admin = false`. Their behavior is unchanged.

5. **No cascading revocation.** Revoking an admin key MUST NOT automatically revoke keys that were authorized while that admin key was active. Each key's revocation status is independent.

6. **Root key revocation is permanent.** Once `keys[account][account].is_revoked == true`, the precompile MUST reject any subsequent attempt to authorize a key with `keyId == account` and MUST reject all transactions signed by the root EOA for that account with `RootKeyRevoked`.

7. **Admin keys survive root revocation.** Revoking the root key MUST NOT revoke or modify any access key. Admin access keys retain full authorize/revoke capability, and non-admin access keys retain their normal signing capability subject to limits/scopes/expiry.

8. **Reserved root slot.** `keys[account][account]` MUST NOT be authorized as a regular access key. Authorization attempts where `keyId == account` MUST be rejected with `InvalidKeyId`.
