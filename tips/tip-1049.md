# TIP-1049: Admin Access Keys

## Abstract

This TIP adds an `admin` boolean flag to the `AuthorizedKey` storage struct and the `KeyRestrictions` configuration. When `admin` is `true`, the access key may authorize and revoke other access keys on behalf of the account. The TIP also adds helper views to the `AccountKeychain` precompile so external contracts can resolve a key id back to its root account and inspect its admin status, without modifying TIP-1020 itself.

## Motivation

Access keys today are strictly subordinate: they can sign transactions within their spending limits and call scopes, but they cannot authorize new keys or participate in onchain key-management flows. This creates two practical problems:

1. **Cross-device key management.** A user with a passkey on their phone cannot authorize a second passkey on their laptop without a transaction from the root EOA key. In practice, multi-device setups require the root key to be available for every new device onboarding, which is inconvenient and fragile.

2. **Onchain account ↔ key association.** Contracts that want to gate actions on "this key belongs to this account" have no canonical way to resolve a public key back to the account that authorized it, or to check whether the key has admin capabilities.

Adding an `admin` flag plus a small set of read helpers in the `AccountKeychain` precompile solves both problems without entangling TIP-1020. TIP-1020 remains a pure signature-verification primitive; contracts that want admin-key semantics call the keychain explicitly.

## Assumptions

- The root EOA key is implicitly admin. The `admin` flag only applies to access keys registered in the AccountKeychain precompile.
- Admin keys MAY be authorized with spending limits, call scopes, and expiry just like non-admin keys. However, callers should be aware that an admin key can effectively bypass its own spending-limit and call-scope restrictions by authorizing a new, unrestricted key. Expiry remains a meaningful constraint because the admin key cannot extend its own expiry.
- TIP-1020 remains unchanged. It continues to act as a pure signature-recovery / verification primitive. Contracts that need to verify "an admin access key of account X signed this" call `AccountKeychain.verifySignature`, which composes TIP-1020 verification with an admin-status check.

# Specification

## Storage Layout

The `AuthorizedKey` packed struct gains a new `is_admin` boolean at byte 11:

```
Byte 0:     signature_type (u8)
Bytes 1–8:  expiry (u64, little-endian)
Byte 9:     enforce_limits (bool)
Byte 10:    is_revoked (bool)
Byte 11:    is_admin (bool)        ← NEW
```

This adds one byte to the existing packed slot. No new storage slots are introduced for the admin flag.

## Interface Changes

### `KeyRestrictions`

Add an `admin` field to the `KeyRestrictions` struct:

```solidity
struct KeyRestrictions {
    uint64 expiry;
    bool enforceLimits;
    TokenLimit[] limits;
    bool allowAnyCalls;
    CallScope[] allowedCalls;
    bool admin;               // NEW
}
```

### `KeyInfo`

Add an `isAdmin` field to the `KeyInfo` return struct:

```solidity
struct KeyInfo {
    SignatureType signatureType;
    address keyId;
    uint64 expiry;
    bool enforceLimits;
    bool isRevoked;
    bool isAdmin;             // NEW
}
```

### `KeyAuthorized` Event

Add `isAdmin` to the `KeyAuthorized` event:

```solidity
event KeyAuthorized(
    address indexed account,
    address indexed publicKey,
    uint8 signatureType,
    uint64 expiry,
    bool isAdmin                // NEW
);
```

### New AccountKeychain Helpers

Three read-only helpers are added to the AccountKeychain precompile so external contracts can reason about key ↔ account associations and verify admin-key signatures:

```solidity
/// Returns true if `keyId` is currently an authorized, non-revoked,
/// non-expired key on `account` with `is_admin = true`.
function isAdminKey(address account, address keyId) external view returns (bool);

/// Returns the admin status of `keyId` for `account`, plus its full KeyInfo.
/// Reverts if the key is unknown.
function adminInfo(address account, address keyId)
    external
    view
    returns (bool isAdmin, KeyInfo memory info);

/// Verifies that `signature` over `digest` was produced by an admin access key
/// of `account`. Internally calls TIP-1020 `verify` against the recovered
/// `keyId` and then checks `isAdminKey(account, keyId)`. Returns true on
/// success, false on signature failure or non-admin key.
function verifySignature(
    address account,
    bytes32 digest,
    bytes calldata signature
) external view returns (bool);
```

`verifySignature` is the canonical entry point for ERC-1271-style contracts that want to accept admin access keys. It is a thin composition of TIP-1020 verification with an admin-status check; TIP-1020 itself is not modified.

## Behavioral Changes

### Key Authorization by Access Keys

Currently, the handler rejects any transaction where an access key attempts to authorize a different key (`AccessKeyCannotAuthorizeOtherKeys`). With this TIP:

- If the signing access key has `is_admin = true`, the transaction MAY include a `KeyAuthorization` for a different `key_id`. The `KeyAuthorization` must still be signed by the root account key.
- If the signing access key has `is_admin = false`, the existing restriction remains: the key can only authorize itself (same-tx auth+use).

### Key Revocation by Access Keys

Currently, only the root key can call `revokeKey`. With this TIP:

- Admin access keys MAY call `revokeKey` to revoke other access keys on the same account.
- Non-admin access keys MUST NOT call `revokeKey`.
- `revokeKey` MUST NOT be used to revoke the root key. Revocation targets are restricted to registered access keys; any attempt to revoke the root EOA key MUST be rejected.
- Admin and non-admin access keys are revocable by the same call path; the only thing the `is_admin` flag of the *target* key affects is the resulting capability set lost.

The `ensure_admin_caller` check in the AccountKeychain precompile is updated to also accept admin access keys (identified via the `transaction_key` transient storage).

### TIP-1020 Compatibility

TIP-1020 is **not modified** by this TIP. Contracts that wish to accept admin access keys call `AccountKeychain.verifySignature(account, digest, signature)`, which internally invokes TIP-1020 verification and then checks that the recovered key id is an admin access key of `account`.

### Admin Key Privilege Propagation

An admin access key MAY authorize new keys with `admin = true`, enabling delegation chains. This is intentional: it allows multi-device setups where any trusted device can onboard new devices without requiring the root key to be *interactively available* — the admin key signs the transaction that carries the `KeyAuthorization`, but the `KeyAuthorization` itself must still be signed by the root account key (which may have been pre-signed or delegated via a custodial flow).

Revocation of an admin key does NOT automatically revoke keys that it authorized. Each key is independently tracked; cascading revocation is out of scope for this TIP. Accounts that need cascading revocation should revoke downstream keys explicitly.

## Transaction Encoding

The `KeyAuthorization` RLP encoding is bumped to V2 to carry the `admin` field unambiguously. The version is indicated by a new leading byte:

```
V1 (existing): [chain_id, key_type, key_id, expiry?, limits?, allowed_calls?]
V2 (this TIP):  [0x02, chain_id, key_type, key_id, expiry?, limits?, allowed_calls?, admin]
```

- `admin` is encoded as `u8`: `0x00` = non-admin, `0x01` = admin. `Option<bool>` is NOT used because `bool::false` decodes from `0x80` (the RLP `None` sentinel), which is ambiguous under `#[rlp(trailing(canonical))]`.
- V1 authorizations remain valid and default to `admin = false`.
- V2 authorizations MUST include the `admin` field explicitly (not trailing-optional).

# Invariants

1. **Admin-only key management.** A non-admin access key MUST NOT authorize or revoke other keys. Transactions that violate this MUST be rejected with `AccessKeyCannotAuthorizeOtherKeys`.

2. **Root key equivalence for admin operations.** Admin access keys MUST be subject to the same validation as root key operations for `authorizeKey` and `revokeKey`: the `KeyAuthorization` must be signed by the root account, and all existing checks (expiry, revocation, signature type) apply. The T2 `tx_origin == msg_sender` guard MUST be enforced for admin key callers.

3. **TIP-1020 unchanged.** TIP-1020's `recover` and `verify` MUST behave identically to pre-TIP-1049 semantics. Admin-key signature verification is exposed via `AccountKeychain.verifySignature`, which is the only path that combines TIP-1020 verification with an admin-status check.

4. **Backward compatibility.** Existing access keys (authorized before this TIP activates) default to `is_admin = false`. Their behavior is unchanged.

5. **No cascading revocation.** Revoking an admin key MUST NOT automatically revoke keys that were authorized while that admin key was active. Each key's revocation status is independent.

6. **Root key is not revocable.** `revokeKey` MUST reject any attempt to revoke the root EOA key. The root key remains the ultimate authority for the account under this TIP.
