# TIP-1049: Admin Access Keys

## Abstract

This TIP adds an `admin` boolean flag to the `AuthorizedKey` storage struct and the `KeyRestrictions` configuration. When `admin` is `true`, the access key may authorize and revoke other access keys on behalf of the account. The TIP also adds helper views to the `AccountKeychain` precompile so external contracts can resolve a key id back to its root account and inspect its admin status, without modifying TIP-1020 itself.

## Motivation

Access keys today are strictly subordinate: they can sign transactions within their spending limits and call scopes, but they cannot authorize new keys or participate in onchain key-management flows. This creates two practical problems:

1. **Cross-device key management.** A user with a passkey on their phone cannot authorize a second passkey on their laptop without a transaction from the root EOA key. In practice, multi-device setups require the root key to be available for every new device onboarding, which is inconvenient and fragile.

2. **Onchain account ↔ key association.** Contracts that want to gate actions on "this key belongs to this account" have no canonical way to resolve a public key back to the account that authorized it, or to check whether the key has admin capabilities.

Adding an `admin` flag plus a small set of read helpers in the `AccountKeychain` precompile solves both problems without entangling TIP-1020. TIP-1020 remains a pure signature-verification primitive; contracts that want admin-key semantics call the keychain explicitly.

## Assumptions

- The root EOA key is implicitly admin. The `admin` flag only applies to access keys registered in the AccountKeychain precompile.
- Admin keys MUST NOT carry spending limits, call scopes, or expiry. Authorizations that combine `admin = true` with non-default `enforceLimits`, `allowedCalls`, or `expiry` MUST be rejected.
- TIP-1020 (`SignatureVerifier`) is extended with a stateful `verifyStateful` method that returns true only for signatures produced by the account's root key or an admin access key. Non-admin access keys MUST NOT satisfy `verifyStateful`.

# Specification

## Storage Layout

The `AuthorizedKey` packed struct gains a new `is_admin` boolean at byte 11:

```
Byte 0:     signature_type (u8)
Bytes 1–8:  expiry (u64, little-endian)
Byte 9:     enforce_limits (bool)
Byte 10:    is_revoked (bool)
Byte 11:    is_admin (bool)        ← NEW
```

This adds one byte to the existing packed slot. No new storage slots are introduced for the admin flag.

## Interface Changes

### `authorizeAdminKey`

`KeyRestrictions` is **unchanged** to preserve the ABI selector of the existing `authorizeKey` function. Admin key authorization is exposed via a new precompile method that takes only the signed authorization payload:

```solidity
function authorizeAdminKey(bytes calldata keyAuthorization) external;
```

The `keyAuthorization` payload is the V2 `KeyAuthorization` envelope (see [Transaction Encoding](#transaction-encoding)) signed by **either the root account or admin access key** of the account. No `KeyRestrictions` argument is accepted: admin keys carry no limits, scopes, or expiry (Invariant 7). All other existing `authorizeKey` validation (T2 `tx_origin == msg_sender`, signature type, etc.) applies identically; the precompile additionally MUST reject any V2 envelope whose `admin` bit is `0`.

### `AdminKeyAuthorized` Event

The existing `KeyAuthorized` event is **unchanged** to preserve backward compatibility for indexers and decoders. A new event is added and emitted **alongside** `KeyAuthorized` whenever `admin = true`:

```solidity
event AdminKeyAuthorized(
    address indexed account,
    address indexed publicKey
);
```

### `SignatureVerifier.verifyStateful`

TIP-1020's `SignatureVerifier` precompile is extended with a single stateful method:

```solidity
/// Returns true if `signature` over the account-bound digest
///   keccak256(0x04 || digest || account)
/// was produced by the root key or an admin key of `account`. Returns false
/// for non-admin access keys, unknown keys, and invalid signatures.
function verifyStateful(
    address account,
    bytes32 digest,
    bytes calldata signature
) external view returns (bool);
```

`verifyStateful` MUST internally recompute the account-bound digest `keccak256(0x04 || digest || account)` before recovery, mirroring the V2 keychain signature binding. This prevents cross-account replay when the same admin key is registered on multiple accounts. Callers sign over the raw `digest`; the binding is applied inside the precompile and cannot be opted out of.

The existing stateless `verify(signer, digest, sig)` is unchanged.

### `AccountKeychain.isAdminKey`

A single read-only helper is added to the AccountKeychain precompile:

```solidity
/// Returns true if `keyId` is currently an authorized, non-revoked,
/// non-expired key on `account` with `is_admin = true`. Returns true
/// when `keyId == account` (root key).
function isAdminKey(address account, address keyId) external view returns (bool);
```

Contracts that need to inspect admin status outside a signature-verification flow (e.g. permissioned hooks) call this directly.

## Behavioral Changes

### Admin-Gated Keychain Mutators

Today, the AccountKeychain precompile gates the following mutators on the root key via `ensure_admin_caller`:

- `authorizeKey`
- `revokeKey`
- `updateSpendingLimit`
- `setAllowedCalls`
- `removeAllowedCalls`

With this TIP, `ensure_admin_caller` is widened to also accept admin access keys. Admin access keys MAY call any of the five mutators above (including on itself).

The five mutators MUST revert with `AccountKeychainError::UnauthorizedCaller` when invoked by a non-admin access key. The single exception is `authorizeKey` in the existing same-tx auth+use flow, where a non-admin key MAY authorize itself; transactions outside that flow are still rejected at the handler level with `AccessKeyCannotAuthorizeOtherKeys`.

`revokeKey` MUST NOT be used to revoke the root key. Revocation targets are restricted to registered access keys; any attempt to revoke the root EOA key MUST be rejected.

### TIP-1020 Compatibility

TIP-1020's existing stateless `verify(signer, digest, sig)` is unchanged. This TIP only **adds** a new stateful `verifyStateful(account, digest, sig)` method to the same `SignatureVerifier` precompile. Existing callers of `verify` are unaffected.

### Admin Key Privilege Propagation

An admin access key MAY authorize new keys (admin or non-admin) by signing the `KeyAuthorization` itself, enabling delegation chains. This is intentional: it allows multi-device setups where any trusted device can onboard new devices without the root key being available at all.

Revocation of an admin key does NOT automatically revoke keys that it authorized. Each key is independently tracked; cascading revocation is out of scope for this TIP. Accounts that need cascading revocation should revoke downstream keys explicitly.

## Transaction Encoding

The `KeyAuthorization` encoding is bumped to V2 to carry the `admin` field. V2 uses a typed envelope mirroring [EIP-2718](https://eips.ethereum.org/EIPS/eip-2718): a single leading version byte followed by the RLP list.

```
V1 (legacy):    rlp([chain_id, key_type, key_id, expiry?, limits?, allowed_calls?])
V2 (this TIP):  0x02 || rlp([chain_id, key_type, key_id, expiry?, limits?, allowed_calls?, admin])
```

Dispatch is a one-byte peek:
- First byte `≥ 0xc0` → V1 (RLP list header range).
- First byte `0x02` → V2 typed envelope.

This is disjoint by RLP construction (V1 cannot start with `0x02`) and unaffected by chain_id width, eliminating the single-byte-chain_id ambiguity of an in-list version field.

- `admin` is encoded as `u8`: `0x00` = non-admin, `0x01` = admin. `Option<bool>` is NOT used because `bool::false` decodes from `0x80` (the RLP `None` sentinel), which is ambiguous under `#[rlp(trailing(canonical))]`.
- V1 authorizations remain valid and default to `admin = false`.
- V2 authorizations MUST include the `admin` field explicitly (not trailing-optional).
- Future versions reserve typed-envelope bytes `0x03..0x7f`.

# Invariants

1. **Admin-only keychain mutation.** `authorizeKey` (outside the same-tx auth+use flow), `revokeKey`, `updateSpendingLimit`, `setAllowedCalls`, and `removeAllowedCalls` MUST revert with `AccountKeychainError::UnauthorizedCaller` when invoked by a non-admin access key.

2. **Root-equivalence for admin keys.** Admin access keys MUST be accepted by `ensure_admin_caller` for all five keychain mutators (`authorizeKey`, `revokeKey`, `updateSpendingLimit`, `setAllowedCalls`, `removeAllowedCalls`). A `KeyAuthorization` MUST be accepted when signed by the root account OR admin access key of the account; all other existing checks (signature type, etc.) apply. The T2 `tx_origin == msg_sender` guard MUST be enforced for admin key callers.

3. **TIP-1020 stateless methods unchanged.** TIP-1020's `recover` and `verify` MUST behave identically to pre-TIP-1049 semantics. The new stateful `verifyStateful` is additive; it is the only path that combines signature recovery with account/admin-status checks.

4. **Backward compatibility.** Existing access keys (authorized before this TIP activates) default to `is_admin = false`. Their behavior is unchanged.

5. **No cascading revocation.** Revoking an admin key MUST NOT automatically revoke keys that were authorized while that admin key was active. Each key's revocation status is independent.

6. **Root key is not revocable.** `revokeKey` MUST reject any attempt to revoke the root EOA key. The root key remains the ultimate authority for the account under this TIP.

7. **Admin keys carry no restrictions.** Admin authorizations MUST go through `authorizeAdminKey`, which structurally cannot accept `KeyRestrictions`. `authorizeKey` MUST reject any V2 `KeyAuthorization` whose `admin` bit is `1`. Admin keys are always unconditional and non-expiring.

8. **Account-bound stateful verification.** `verifyStateful(account, digest, sig)` MUST recover the signer from `keccak256(0x04 || digest || account)`, never from `digest` directly. Implementations MUST NOT expose a variant that omits the binding.
