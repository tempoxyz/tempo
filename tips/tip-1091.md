# TIP-1091: Enshrined ZoneFactory

## Abstract

This TIP enshrines `ZoneFactory` as a Tempo precompile. Zone creation becomes a
native protocol operation, and each new zone gets a deterministic `ZonePortal`
account at a reserved vanity address.

The full zone protocol remains specified in the Zones repository. The Solidity
files included here are reference artifacts for the factory and portal behavior
that this TIP changes.

## Motivation

Zones should have one canonical registry and predictable portal addresses. A
native factory removes dependence on a separately deployed factory contract and
lets Tempo reserve recognizable portal accounts for zones.

`ZonePortal` is expected to become native later. Until precompiles can perform
the external EVM calls needed by the portal flow, each portal is an EVM account
installed by the protocol. The account is a portal proxy: it contains only
minimal proxy/caller bytecode that points at one central `ZonePortal` logic
contract.

## Specification

Tempo adds a `ZoneFactory` precompile at:

```solidity
address constant ZONE_FACTORY_ADDRESS = 0x5AF2000000000000000000000000000000000000;
```

The precompile exposes `IZoneFactory`. Its registry behavior MUST match the
canonical Zones implementation: monotonically increasing zone IDs, `zones`,
`zoneCount`, `isZonePortal`, `isValidVerifier`, `verifier`, `messenger`, and
`ZoneCreated`.

Each zone portal is installed at:

```text
ZONE_PORTAL_PREFIX = 0x5AD000000000000000000000
portal = bytes20(ZONE_PORTAL_PREFIX || uint64(zoneId))
```

The suffix is the zone ID encoded big-endian into the low 8 bytes. `zoneId == 0`
is reserved, so `zoneId == 1` maps to
`0x5AD0000000000000000000000000000000000001`.

The central portal logic contract's runtime code is stored at:

```solidity
address constant ZONE_PORTAL_LOGIC_ADDRESS = 0x5AD1000000000000000000000000000000000000;
```

Each portal proxy is etched with this delegatecall proxy runtime:

```text
portal_proxy_runtime =
  hex"363d3d373d3d3d363d73"
  || bytes20(ZONE_PORTAL_LOGIC_ADDRESS)
  || hex"5af43d82803e903d91602b57fd5bf3"
```

`createZone` MUST NOT deploy the portal with EVM `CREATE` or `CREATE2`, and
MUST NOT rely on nonce-based address prediction. Instead, the native factory:

1. validates the request as the canonical factory would;
2. assigns the next zone ID;
3. computes the portal vanity address;
4. verifies that account has no existing code, storage, or EIP-7702 delegation;
5. etches `portal_proxy_runtime` at that address;
6. calls the portal proxy's factory-only initializer with the `createZone` parameters;
7. records the zone metadata, marks the portal proxy in `isZonePortal`, and emits
   `ZoneCreated`.

The portal proxy MUST NOT contain a full copy of `ZonePortal` runtime code. It
MUST route to one protocol-managed portal logic contract, while each portal
proxy keeps independent observable state.

Only `ZONE_FACTORY_ADDRESS` MAY call a portal proxy initializer, and
initialization MUST succeed at most once. The initializer MUST be callable only
by `DELEGATECALL`: a direct call to the portal logic contract at
`ZONE_PORTAL_LOGIC_ADDRESS` MUST revert. The factory MUST initialize a newly
etched portal proxy in the same `createZone` operation before the portal proxy
becomes externally usable. `ZONE_FACTORY_ADDRESS` MUST remain constant across
portal logic contract deployments and upgrades. Every portal logic contract
implementing `ZonePortal` MUST embed both
`ZONE_FACTORY_ADDRESS` as its initializer authority and
`ZONE_PORTAL_LOGIC_ADDRESS` as its canonical portal logic contract address; neither
value may be derived from the temporary address used to construct the runtime.

The initialized portal MUST use the shared messenger returned by
`ZoneFactory.messenger()`. Withdrawal callback behavior MUST match the canonical
`ZonePortal.sol` reference.

User withdrawals MUST expose only a nonzero `uint64 fallbackNonce` on Tempo.
The zone outbox assigns this nonce monotonically and stores its mapping to the
private zone fallback recipient; a withdrawal bounce-back carries the nonce back
to the zone, where the inbox resolves and consumes the mapping. This keeps the
recipient out of L1-visible data while preserving deterministic execution.

The activation hardfork MUST install at `ZONE_PORTAL_LOGIC_ADDRESS` the runtime
code produced by executing `ZonePortal` creation code with constructor arguments
`(ZONE_FACTORY_ADDRESS, ZONE_PORTAL_LOGIC_ADDRESS)`. It MAY obtain that runtime
by deploying a temporary template and copying its deployed runtime code with
`EXTCODECOPY`, or by an equivalent state-transition mechanism. The hardfork MUST
NOT install creation code as account code; copying runtime code does not execute
the constructor or copy its storage. A later hardfork MAY replace the runtime at
that same address, but the replacement runtime MUST embed the same initializer
authority and canonical portal logic contract address. Because every portal
proxy delegates to the fixed portal logic contract address, the replacement
upgrades all existing portal proxies without changing their proxy bytecode or
per-portal-proxy storage. Replacement portal logic contracts MUST preserve the
existing storage layout or the hardfork MUST migrate affected portal-proxy state.

Once native precompiles can perform the external EVM calls required by the
portal flow, a later hardfork MAY register the `ZONE_PORTAL_PREFIX` address
family for native dispatch. The native implementation MUST operate on the
called portal account's existing storage and preserve the `IZonePortal` ABI,
events, and authorization behavior. The migration MUST NOT change portal
addresses or require per-zone state migration solely because execution becomes
native; the existing proxy runtime MAY remain as observable account code while
native dispatch takes precedence.

## Out Of Scope

This TIP does not enshrine `ZonePortal`, `ZoneMessenger`, `Verifier`, or
`SwapAndDepositRouter`. It also does not specify migration for existing zones or
the final proof system.
