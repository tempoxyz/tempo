# TIP-1091: Enshrined ZoneFactory

## Abstract

This TIP enshrines `ZoneFactory` as a Tempo precompile so zone creation is a
first-class protocol operation. The zone contract behavior is specified by the
Solidity reference implementation checked in with this TIP; this document only
defines how the native Tempo deployment model differs from that Solidity
deployment model.

## Motivation

Zones need stable, protocol-owned creation semantics. A native factory gives
Tempo one canonical zone registry, removes dependence on a separately deployed
factory contract, and lets the protocol assign recognizable zone portal and
messenger addresses.

`ZonePortal` is intended to become native later. Until precompiles can perform
the external EVM calls required by the portal and messenger flows, those
accounts are installed as EVM bytecode accounts at protocol-assigned vanity
addresses. To avoid duplicating full bytecode at every zone address, each
portal and messenger address uses a minimal proxy caller that executes one
central logic implementation for that contract.

## Assumptions

- The Solidity files under `tips/verify/src/zones` are the canonical zone
  contract specification for this TIP.
- The initial implementation applies to newly created zones. Migration of
  already-created zones is out of scope unless a later TIP specifies it.
- Central portal and messenger logic upgrades happen only through Tempo
  protocol upgrades, not through an externally owned admin key.
- Portal addresses use the reserved `0x20D0` address prefix, messenger
  addresses use the reserved `0x20D1` address prefix, and both are installed
  only by the enshrined factory.

# Specification

## Canonical Reference

The canonical zone contract behavior is defined by the Solidity reference
implementation:

1. [`tips/verify/src/zones/tempo/ZoneFactory.sol`](verify/src/zones/tempo/ZoneFactory.sol)
2. [`tips/verify/src/zones/tempo/ZonePortal.sol`](verify/src/zones/tempo/ZonePortal.sol)
3. [`tips/verify/src/zones/tempo/ZoneMessenger.sol`](verify/src/zones/tempo/ZoneMessenger.sol)
4. [`tips/verify/src/zones/tempo/Verifier.sol`](verify/src/zones/tempo/Verifier.sol)
5. [`tips/verify/src/zones/tempo/SwapAndDepositRouter.sol`](verify/src/zones/tempo/SwapAndDepositRouter.sol)

Supporting interfaces and libraries live under
[`tips/verify/src/zones/interfaces`](verify/src/zones/interfaces) and
[`tips/verify/src/zones/libraries`](verify/src/zones/libraries).

Implementations MUST preserve the external ABI, events, errors, and observable
state transitions of the Solidity reference unless this TIP explicitly says
otherwise.

## ZoneFactory Precompile

Tempo adds a `ZoneFactory` precompile at:

```solidity
address constant ZONE_FACTORY_ADDRESS = 0x20F2000000000000000000000000000000000000;
```

The precompile exposes the `IZoneFactory` interface from
[`tips/verify/src/zones/interfaces/IZone.sol`](verify/src/zones/interfaces/IZone.sol).
It MUST maintain the same zone registry semantics as `ZoneFactory.sol`,
including monotonically increasing zone IDs, `zones(zoneId)`, `zoneCount()`,
`isZonePortal(address)`, `isZoneMessenger(address)`, `isValidVerifier(address)`,
`verifier()`, and the `ZoneCreated` event.

The native `createZone` implementation differs from `ZoneFactory.sol` in one
way: it does not deploy the portal and messenger with EVM `CREATE`. Instead it
allocates protocol-assigned addresses and initializes the corresponding accounts
directly as described below.

## Portal Account Installation

Each created zone gets a `ZonePortal` account at a deterministic
protocol-assigned vanity address:

```text
ZONE_PORTAL_PREFIX = 0x20D000000000000000000000
portal = bytes20(ZONE_PORTAL_PREFIX || uint64(zoneId))
```

The suffix is the zone ID encoded big-endian into the low 8 bytes. `zoneId == 0`
is reserved; `zoneId == 1` maps to
`0x20D0000000000000000000000000000000000001`.

The `0x20D0` prefix mirrors TIP-20's 12-byte prefix plus 8-byte suffix address
shape, but is disjoint from TIP-20's `0x20C0` token prefix and `0x20FC`
factory address.

The portal account MUST NOT contain a full copy of the `ZonePortal` runtime
bytecode. It MUST contain minimal proxy/caller bytecode that executes one
central `ZonePortal` logic implementation while keeping each portal's observable
state independent.

The native factory MUST initialize each portal so that it behaves as if the
`ZonePortal.sol` constructor ran with the `createZone` parameters. Because the
reference contract uses Solidity immutables for values such as `zoneId`,
`messenger`, `verifier`, and `genesisTempoBlockNumber`, the native/proxy
implementation MAY represent those values differently internally, but the
public getters and all downstream behavior MUST match the reference contract.

The central portal logic implementation is protocol-managed. Replacing it is a
network-upgrade action and MUST preserve the behavior promised by this TIP, or
be specified by a later TIP.

## Messenger Account Installation

Each created zone gets a `ZoneMessenger` account at a deterministic
protocol-assigned vanity address:

```text
ZONE_MESSENGER_PREFIX = 0x20D100000000000000000000
messenger = bytes20(ZONE_MESSENGER_PREFIX || uint64(zoneId))
```

The suffix is the zone ID encoded big-endian into the low 8 bytes. `zoneId == 0`
is reserved; `zoneId == 1` maps to
`0x20D1000000000000000000000000000000000001`.

The messenger account MUST NOT contain a full copy of the `ZoneMessenger`
runtime bytecode. It MUST contain minimal proxy/caller bytecode that executes
one central `ZoneMessenger` logic implementation while preserving each zone's
portal binding and withdrawal callback origin semantics.

The native factory MUST initialize each messenger so that it behaves as if the
`ZoneMessenger.sol` constructor ran with that zone's portal address. The
`portal()` getter, `OnlyPortal` check, token transfer, callback gas forwarding,
and callback selector validation MUST match the reference contract.

The central messenger logic implementation is protocol-managed. Replacing it is
a network-upgrade action and MUST preserve the behavior promised by this TIP,
or be specified by a later TIP.

## Verifier And Router Contracts

`Verifier` remains an ordinary EVM contract account. The enshrined factory's
`verifier()` and `isValidVerifier(address)` views refer to ordinary deployed
verifier contracts, not native precompiles or protocol-etched proxy accounts.

`SwapAndDepositRouter` remains an ordinary L1 contract. The protocol does not
install, proxy, or enshrine router bytecode.

## Out Of Scope

This TIP does not:

- Enshrine `ZonePortal` as a precompile.
- Enshrine `ZoneMessenger` as a precompile.
- Enshrine `Verifier`; it remains an ordinary L1 contract in the reference
  implementation.
- Enshrine `SwapAndDepositRouter`; it remains an ordinary L1 contract in the
  reference implementation.
- Specify migration for existing zones.
- Specify the final proof system beyond the `Verifier` interface used by the
  reference implementation.

# Tooling

Zone creation tooling should call `ZONE_FACTORY_ADDRESS`. Generated zone
metadata should continue to record the zone ID, portal, messenger, admin,
sequencer, verifier, and anchor block in the same shape used today.

# Observability

No new events are required. `ZoneCreated`, portal deposit events, batch events,
withdrawal events, and router callback behavior are inherited from the
Solidity reference implementation.

# Invariants

- `createZone` assigns each successful zone a unique, monotonically increasing
  zone ID.
- Every created portal address is registered by `isZonePortal`.
- Every created messenger address is registered by `isZoneMessenger`.
- No non-protocol `CREATE`, `CREATE2`, or EIP-7702 delegation may install code
  or delegation at `ZONE_FACTORY_ADDRESS` or any address with the
  `ZONE_PORTAL_PREFIX` or `ZONE_MESSENGER_PREFIX`.
- The portal address for a zone ID is a pure function of that zone ID.
- The messenger address for a zone ID is a pure function of that zone ID.
- Portal proxy accounts have independent observable portal state.
- Messenger proxy accounts preserve independent per-zone portal bindings.
- Portal behavior matches `ZonePortal.sol` even if immutable constructor values
  are represented differently internally.
- Messenger behavior matches `ZoneMessenger.sol` even if immutable constructor
  values are represented differently internally.
- A central portal or messenger logic upgrade cannot happen outside a Tempo
  protocol upgrade.
- Existing TIP-20 and TIP-403 policy checks remain enforced exactly as in the
  reference implementation.
