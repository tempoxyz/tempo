# TIP-1092: TIP-20 Policy IDs in TIP-403

## Abstract

This TIP makes TIP-403 the canonical source of truth for the active transfer policy of each TIP-20 token. New TIP-20 tokens register their transfer policy in TIP-403 at creation time, and existing TIP-20 tokens are migrated by a normal transaction before activation.

TIP-20 keeps its existing external policy APIs, but authorization checks resolve the token's policy ID from TIP-403 instead of relying on token-local policy storage.

## Motivation

TIP-403 policy contents are already registry-owned, but a TIP-20 token's selected `transferPolicyId` is currently stored in the TIP-20 token. This makes the policy contents provable from TIP-403 while leaving the token-to-policy binding outside the registry state that zones and provable-contract flows want to rely on.

Moving the binding into TIP-403 lets a verifier prove both parts of the transfer-policy decision from one registry path: which policy a token uses, and what that policy permits.

The alternatives considered were:

- Mirroring the policy ID in both TIP-20 and TIP-403 forever.
- Lazily writing the policy ID only when a token is enabled for a zone.
- Automatically batch-migrating policy IDs at hardfork activation.

This TIP chooses a registry-owned binding plus an explicit migration transaction because it is simple, avoids ZonePortal coupling, and avoids a hardfork-time system transaction.

## Assumptions

- The TIP-403 registry is included in the provable state required by zone and provable-contract flows.
- Existing TIP-20 tokens can be enumerated before activation on each network.
- The number of existing TIP-20 tokens is small enough that a normal migration transaction or short script can register all current bindings.
- Every existing TIP-20 token has a valid local transfer policy ID before migration.

If migration cannot be completed before the target activation, this TIP should not activate on that network.

## Threat Model

- Token admins keep the same authority they have today to change a token's transfer policy through TIP-20.
- A migration caller may trigger copying an existing token-local policy ID into TIP-403, but must not be able to choose a different policy ID.
- TIP-403 must reject direct attempts to set another token's policy binding.
- Provers and zone operators rely on TIP-403 registry state for the token-to-policy binding after activation.

# Specification

## Registry State

TIP-403 MUST maintain a registry-owned transfer-policy binding for TIP-20 tokens:

```text
token_transfer_policy: mapping(address token => uint256 packed)

bits 0..63: policy_id
bit 64: is_set
bits 65..255: reserved
```

The `is_set` bit is required because policy ID `0` is a valid built-in reject-all policy. A packed value of zero means unset. A packed value with `is_set == 1` and `policy_id == 0` means the token is explicitly registered to policy `0`.

For a registered token, `policy_id` is the token's active transfer policy ID. The policy ID MUST refer to an existing TIP-403 policy, including the built-in policies.

## TIP-403 Interface

TIP-403 MUST expose token-keyed policy lookup and authorization helpers. Exact names may be adjusted during implementation, but the behavior must be equivalent to:

```solidity
function tokenTransferPolicyId(address token) external view returns (uint64 policyId);

function isAuthorizedSenderForToken(address token, address user) external view returns (bool);
function isAuthorizedRecipientForToken(address token, address user) external view returns (bool);
function isAuthorizedMintRecipientForToken(address token, address user) external view returns (bool);
```

The token-keyed authorization helpers resolve `policyId = tokenTransferPolicyId(token)` and then apply the existing TIP-403 policy authorization rules, including TIP-1015 compound sender, recipient, and mint-recipient roles.

If `token` has no registered policy ID, the lookup and token-keyed authorization helpers MUST revert.

The existing policy-ID-keyed TIP-403 APIs remain available for callers that already know a policy ID.

## TIP-20 Behavior

After activation, TIP-20 transfer-policy behavior changes as follows:

- `transferPolicyId()` returns the policy ID registered for the token in TIP-403.
- `changeTransferPolicyId(uint64 newPolicyId)` validates that `newPolicyId` exists in TIP-403 and updates the TIP-403 token binding.
- TIP-20 transfer, transferFrom, mint, burn-blocked, fee, and internal authorization paths use the TIP-403 token-keyed helpers instead of reading a token-local policy ID and passing it back to TIP-403.

The existing `TransferPolicyUpdate(address indexed updater, uint64 indexed newPolicyId)` event remains the TIP-20 event emitted when a token's active transfer policy changes.

TIP-20 implementations MAY keep the legacy local policy ID slot for compatibility or migration bookkeeping, but after activation that slot is not canonical and MUST NOT be used for authorization.

## New TIP-20 Creation

The TIP-20 creation path MUST initialize the TIP-403 token binding atomically with token initialization.

A new TIP-20 token MUST NOT be initialized in a state where its active transfer policy exists only in token-local storage.

## Existing TIP-20 Migration

Before activation, TIP-20 MUST expose a migration path equivalent to:

```solidity
function migrateTransferPolicyIdToRegistry() external returns (uint64 policyId);
```

The migration function:

1. Reads the token's current local `transferPolicyId`.
2. Verifies that the policy ID exists in TIP-403.
3. Writes that policy ID into the TIP-403 token binding for `address(this)`.
4. Is callable by any account, because the caller cannot choose the policy ID being written.
5. Is idempotent when TIP-403 already stores the same policy ID for the token.
6. Reverts if TIP-403 already stores a different policy ID for the token.

Network operators SHOULD run an offchain script before activation that enumerates existing TIP-20 tokens, calls this migration function for each token, and verifies that TIP-403 contains the expected binding.

This TIP does not require an automatic hardfork-time batch migration or system transaction.

# Tooling

A migration script is required for each network before activation. The script should:

1. Enumerate deployed TIP-20 tokens.
2. Call `migrateTransferPolicyIdToRegistry()` for every unmigrated token.
3. Verify `TIP403.tokenTransferPolicyId(token) == TIP20(token).transferPolicyId()`.
4. Fail if any token is missing a registry binding.

SDKs and indexers that read `transferPolicyId()` from TIP-20 can keep using that API. Low-level tools that read TIP-20 storage directly must switch to the TIP-403 token binding after activation.

# Observability

The existing TIP-20 `TransferPolicyUpdate` event remains required for active policy changes.

TIP-403 SHOULD emit a registry-level event whenever a token binding is created or changed:

```solidity
event TokenTransferPolicyUpdated(address indexed token, uint64 indexed policyId);
```

This event answers which token-to-policy bindings were registered during migration and which binding changed during later policy updates.

# Invariants

- Every initialized TIP-20 token has exactly one canonical TIP-403 transfer policy binding after activation.
- Policy ID `0` remains usable as a token's active policy ID, so unset registry state must be represented separately from policy ID `0`.
- TIP-20 authorization after activation is based only on the TIP-403 token binding and TIP-403 policy state.
- `changeTransferPolicyId` cannot set a token binding to a policy ID that does not exist.
- Migration cannot change a token's policy to a caller-selected value; it can only copy the token's existing local policy ID.
- Migration is idempotent when the registry already contains the same policy ID.
- A token with no registered TIP-403 policy binding cannot pass token-keyed authorization checks after activation.
- Existing policy-ID-keyed TIP-403 APIs preserve their current behavior.

The test suite must cover new token creation, existing token migration, idempotent migration, migration mismatch rejection, policy ID `0`, transfer authorization, mint authorization, burn-blocked authorization, `changeTransferPolicyId`, and transaction-pool/prewarming behavior for registry-backed policy lookup.
