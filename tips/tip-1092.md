# TIP-1092: TIP-20 Policy IDs in TIP-403

## Abstract

This TIP adds a TIP-403 registry binding for the active transfer policy of each TIP-20 token. New TIP-20 tokens register their transfer policy in TIP-403 at creation time, and existing TIP-20 tokens can be manually migrated when their token-to-policy binding needs to be provable.

TIP-20 keeps its existing external policy APIs. Authorization resolves the policy ID from TIP-403 when a registry binding exists, and falls back to the token-local policy ID for unmigrated existing tokens.

## Motivation

TIP-403 policy contents are already registry-owned, but a TIP-20 token's selected `transferPolicyId` is currently stored in the TIP-20 token. This makes the policy contents provable from TIP-403 while leaving the token-to-policy binding outside the registry state that zones and provable-contract flows want to rely on.

Moving a binding into TIP-403 lets a verifier prove both parts of the transfer-policy decision from one registry path: which policy a token uses, and what that policy permits.

As of July 10, 2026, chain scans found:

| Network | Block scanned | TIP-20 tokens |
|---------|--------------:|--------------:|
| Presto mainnet | 29,189,230 | 4,355 |
| Moderato testnet | 25,830,853 | 15,291,992 |

The testnet count makes full automatic migration impractical. A complete testnet migration would create one new TIP-403 registry slot per historical token, requiring trillions of gas. Zones only need registry-backed policy bindings for tokens that are successfully enabled on a running zone, so this TIP uses manual migration for those tokens and preserves fallback execution behavior for the rest.

The alternatives considered were:

- Requiring every existing TIP-20 token to be migrated before activation.
- Automatically batch-migrating policy IDs at hardfork activation.
- Treating an unset TIP-403 binding as an execution failure for all existing tokens.

This TIP chooses a registry-owned binding for new and zone-relevant tokens, plus a fallback read from TIP-20 local storage for unmigrated existing tokens. This avoids a hardfork-time system transaction and avoids migrating millions of historical testnet tokens that are not enabled on running zones.

## Assumptions

- The TIP-403 registry is included in the provable state required by zone and provable-contract flows.
- Existing TIP-20 tokens that are not migrated keep a valid token-local transfer policy ID.
- Zone enablement can require or perform migration for the token being enabled.
- Zone and provable-contract flows only rely on registry-backed bindings, not fallback local reads.

If an existing token is used only for normal TIP-20 execution and is not enabled on a running zone, it does not need to be migrated before activation.

## Threat Model

- Token admins keep the same authority they have today to change a token's transfer policy through TIP-20.
- A migration caller may trigger copying an existing token-local policy ID into TIP-403, but must not be able to choose a different policy ID.
- TIP-403 must reject direct attempts to set another token's policy binding.
- Provers and zone operators rely only on TIP-403 registry state for the token-to-policy binding. The fallback local read is an execution-compatibility path and is not a proof source.

# Specification

## Registry State

TIP-403 MUST maintain a registry-owned transfer-policy binding for TIP-20 tokens:

```text
token_transfer_policy: mapping(address token => uint256 packed)

bits 0..63: policy_id
bit 64: is_set
bits 65..255: reserved
```

The `is_set` bit is required because policy ID `0` is a valid built-in reject-all policy. A packed value of zero means unset. A packed value with `is_set == 1` and `policy_id == 0` means the token is explicitly registered to policy `0`.

For a registered token, `policy_id` is the token's active transfer policy ID. The policy ID MUST refer to an existing TIP-403 policy, including the built-in policies.

## TIP-403 Interface

TIP-403 MUST expose token-keyed policy lookup and authorization helpers. Exact names may be adjusted during implementation, but the behavior must be equivalent to:

```solidity
function hasTokenTransferPolicy(address token) external view returns (bool);
function tokenTransferPolicyId(address token) external view returns (uint64 policyId);

function isAuthorizedSenderForToken(address token, address user) external view returns (bool);
function isAuthorizedRecipientForToken(address token, address user) external view returns (bool);
function isAuthorizedMintRecipientForToken(address token, address user) external view returns (bool);
```

`hasTokenTransferPolicy(token)` returns whether TIP-403 stores a registry binding for `token`.

`tokenTransferPolicyId(token)` returns the registry binding when one is set. If no registry binding is set, it falls back to reading the legacy token-local TIP-20 policy ID storage directly. The fallback MUST NOT call the effective TIP-20 `transferPolicyId()` public API, because that API may itself consult TIP-403.

The token-keyed authorization helpers resolve `policyId = tokenTransferPolicyId(token)` and then apply the existing TIP-403 policy authorization rules, including TIP-1015 compound sender, recipient, and mint-recipient roles.

If `token` is not a deployed TIP-20 token, the lookup and token-keyed authorization helpers MUST revert.

Zone and provable-contract flows MUST NOT rely on the fallback behavior. A token can be enabled on a running zone only after TIP-403 has a registry binding for that token.

The existing policy-ID-keyed TIP-403 APIs remain available for callers that already know a policy ID.

## TIP-20 Behavior

After activation, TIP-20 transfer-policy behavior changes as follows:

- `transferPolicyId()` returns the effective policy ID: the TIP-403 registry binding if one exists, otherwise the token-local policy ID.
- `changeTransferPolicyId(uint64 newPolicyId)` validates that `newPolicyId` exists in TIP-403. If the token has a TIP-403 registry binding, it updates that binding. Otherwise, it updates the token-local policy ID.
- TIP-20 transfer, transferFrom, mint, burn-blocked, fee, and internal authorization paths use the effective policy ID.

The existing `TransferPolicyUpdate(address indexed updater, uint64 indexed newPolicyId)` event remains the TIP-20 event emitted when a token's active transfer policy changes.

TIP-20 implementations MUST keep the legacy local policy ID slot for unmigrated existing tokens. Once a token has a TIP-403 registry binding, that binding is canonical for provable flows.

## New TIP-20 Creation

The TIP-20 creation path MUST initialize the TIP-403 token binding atomically with token initialization.

A new TIP-20 token MUST NOT be initialized in a state where its active transfer policy exists only in token-local storage.

## Existing TIP-20 Migration

TIP-20 MUST expose a migration path equivalent to:

```solidity
function migrateTransferPolicyIdToRegistry() external returns (uint64 policyId);
```

The migration function:

1. Reads the token's legacy local policy ID storage.
2. Verifies that the policy ID exists in TIP-403.
3. Writes that policy ID into the TIP-403 token binding for `address(this)`.
4. Is callable by any account, because the caller cannot choose the policy ID being written.
5. Is idempotent when TIP-403 already stores the same policy ID for the token.
6. Reverts if TIP-403 already stores a different policy ID for the token.

Operators SHOULD manually migrate existing tokens that are enabled on, or about to be enabled on, a running zone. A zone enablement path MAY call this migration function itself before accepting the token.

This TIP does not require an automatic hardfork-time batch migration or system transaction.

# Tooling

A migration script should support targeted migration. The script should:

1. Accept a list of TIP-20 tokens to migrate, such as tokens enabled on running zones.
2. Call `migrateTransferPolicyIdToRegistry()` for every listed unmigrated token.
3. Verify `TIP403.hasTokenTransferPolicy(token) == true`.
4. Verify `TIP403.tokenTransferPolicyId(token) == TIP20(token).transferPolicyId()`.

SDKs and indexers that read `transferPolicyId()` from TIP-20 can keep using that API. Low-level tools that need provable token-to-policy bindings must use TIP-403 and must check that `hasTokenTransferPolicy(token)` is true.

# Observability

The existing TIP-20 `TransferPolicyUpdate` event remains required for active policy changes.

TIP-403 SHOULD emit a registry-level event whenever a token binding is created or changed:

```solidity
event TokenTransferPolicyUpdated(address indexed token, uint64 indexed policyId);
```

This event answers which token-to-policy bindings were registered during migration and which binding changed during later policy updates.

# Invariants

- Every new TIP-20 token has exactly one TIP-403 transfer policy binding at initialization.
- Existing unmigrated TIP-20 tokens continue to authorize against their token-local transfer policy ID.
- Zone and provable-contract flows require a TIP-403 registry binding and must not use the fallback local policy ID as proof.
- Policy ID `0` remains usable as a token's active policy ID, so unset registry state must be represented separately from policy ID `0`.
- `changeTransferPolicyId` cannot set a token binding to a policy ID that does not exist.
- Migration cannot change a token's policy to a caller-selected value; it can only copy the token's existing local policy ID.
- Migration is idempotent when the registry already contains the same policy ID.
- A token with no registered TIP-403 policy binding can use fallback execution, but cannot be treated as registry-proven.
- Existing policy-ID-keyed TIP-403 APIs preserve their current behavior.

The test suite must cover new token creation, existing token migration, fallback reads for unmigrated tokens, zone enablement requiring a registry binding, idempotent migration, migration mismatch rejection, policy ID `0`, transfer authorization, mint authorization, burn-blocked authorization, and `changeTransferPolicyId`.
