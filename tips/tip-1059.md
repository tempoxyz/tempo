# TIP-1059: Discounted Pure Payment Transfers

## Abstract

This TIP makes the first 50 million gas of simple TIP-20 transfers in each block cheaper than other transactions. Eligible transfers are charged 2 billion attodollars per gas instead of the normal 20 billion attodollars per gas.

The goal is to make ordinary payments feel cheap while keeping the normal price for transactions that may create state, run arbitrary contracts, or consume the general lane.

## Motivation

Tempo's current fixed base fee is deliberately high enough to make worst-case spam and state growth expensive. That protects the chain, but it also means the most common payment path pays the same gas price as more expensive operations.

Simple TIP-20 transfers to existing accounts do not have the same long-term cost profile as state creation. They are the path we most want to be cheap, predictable, and easy to explain. This TIP introduces a narrow discount for that path without lowering the price of everything else.

Users and wallets do not need to rely on the discount for reliability. A transfer may still authorize the normal gas price. If it lands inside the discounted per-block budget, the chain charges the discounted price and refunds the difference.

## Assumptions

- A TIP-20 transfer with `gas_limit <= 250,000` is not expected to create new account state. Transfers that create new account state are expected to require more gas.
- The discount is a payment UX improvement, not a replacement for the normal base fee as the chain's main spam-resistance mechanism.
- Wallets and partners may continue to set `max_fee_per_gas` to the normal base fee for reliable inclusion.
- The discount is applied by protocol rules, not by trusting a block builder to voluntarily charge less.

# Specification

## Parameters

At the T6 hardfork, introduce the following parameters:

| Parameter | Value | Meaning |
| --- | ---: | --- |
| Normal base fee | 20,000,000,000 | Existing gas price for ordinary transactions |
| Discounted transfer gas price | 2,000,000,000 | Gas price charged to eligible discounted transfers |
| Discounted transfer budget | 50,000,000 gas per block | Maximum gas limit that can receive the discount in one block |
| Maximum discounted transaction gas limit | 250,000 gas | Per-transaction eligibility cap |

All gas prices are denominated in attodollars per gas.

## Discount eligibility

A transaction is eligible for the discount only if all of the following are true:

1. It is a TIP-20 payment-lane transaction.
2. Every user call in the transaction is one of:
   - `transfer(address to, uint256 amount)`
   - `transferWithMemo(address to, uint256 amount, bytes32 memo)`
3. The transaction has `gas_limit <= 250,000`.
4. The transaction does not create a contract.
5. The transaction does not include an EIP-7702 authorization list.

For account-abstraction transactions, every call in the batch must satisfy the TIP-20 transfer requirement. A mixed batch is not eligible for the discount.

## Applying the discount

Each block has a `discounted_transfer_gas_left` counter initialized to 50,000,000 gas.

Transactions are processed in block order. For each transaction:

1. If the transaction is not discount-eligible, it uses the normal base fee.
2. If the transaction is discount-eligible and `gas_limit <= discounted_transfer_gas_left`, it uses the discounted transfer gas price.
3. If the transaction is discount-eligible but `gas_limit > discounted_transfer_gas_left`, it uses the normal base fee.

When a transaction receives the discount, `discounted_transfer_gas_left` is reduced by that transaction's `gas_limit`.

The discount is all-or-nothing for a transaction. A transaction is never partly charged at the discounted gas price and partly charged at the normal base fee.

## Fee caps and refunds

A discounted transaction is valid if its `max_fee_per_gas` is at least the discounted transfer gas price. A non-discounted transaction remains valid only if its `max_fee_per_gas` is at least the normal base fee.

A transaction with `max_fee_per_gas` below the normal base fee is valid only while it receives the discount. If it appears after the discounted transfer budget is exhausted, the block is invalid.

A transaction with `max_fee_per_gas` at or above the normal base fee remains valid whether or not it receives the discount. If it receives the discount, the charged gas price is the discounted transfer gas price and the unused authorized amount is refunded through the existing gas refund path.

Wallets SHOULD continue to use the normal base fee by default unless they are intentionally submitting discount-only transactions. This gives users reliable inclusion while still letting them benefit from the discount when capacity is available.

## Block building

Block builders SHOULD prefer eligible low-fee transfers only while discounted transfer gas remains. Once the discounted budget is exhausted, builders SHOULD skip discount-only transactions whose `max_fee_per_gas` is below the normal base fee.

The payload fee total used for builder comparison MUST use the same gas price that will be charged by execution.

## RPC and estimation

Fee estimation for ordinary transactions is unchanged.

Fee estimation for eligible pure TIP-20 transfers MAY show the discounted gas price, but RPCs and wallets SHOULD make clear that authorizing the normal base fee gives more reliable inclusion. Partner RPCs that control ordering may choose to submit discount-only transfers when they expect discounted capacity to be available.

# Alternatives Considered

## Lower the base fee for all transactions

This is the simplest change, but it makes storage creation and arbitrary contract execution cheaper too. That weakens the main reason Tempo has a fixed base fee today: protecting the chain from cheap worst-case spam and permanent state growth.

## Lower the payment-lane base fee for all payment-lane transactions

This is broader than the proposal in this TIP. The current payment-lane classifier includes more than simple transfers. Discounting the whole lane risks making approvals, mints, burns, or future payment-lane operations cheaper even when they do not have the same cost profile as an ordinary transfer.

## Use a capped EIP-1559 fee for the payment lane

A dynamic fee could make payments cheap most of the time while rising under load. It is also more complex: the protocol would need a second fee market, wallet behavior would need to change, and the interaction between payment-lane fees and normal fees would need careful design.

## Separate storage pricing from execution pricing

This is the clean long-term model. State creation could stay expensive while execution gets cheaper. It is also a larger protocol change than needed for the immediate goal of making ordinary payments cheaper.

# Invariants

- The normal base fee remains unchanged for transactions that do not receive the discount.
- No more than 50,000,000 gas limit per block receives the discounted transfer gas price.
- A discounted transaction consumes discount budget by `gas_limit`, not by post-execution gas used.
- The discount never applies to contract creation.
- The discount never applies to a transaction with `gas_limit > 250,000`.
- The discount never applies to a mixed account-abstraction batch that includes anything other than eligible TIP-20 transfers.
- A transaction that authorizes the normal base fee remains valid whether or not discounted budget is available.
