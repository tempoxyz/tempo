# TIP-1059: Discounted Pure Payment Transfers

## Abstract

This TIP makes the first 50 million gas of simple TIP-20 payment operations in each block cheaper than other transactions. Eligible operations are charged 2 billion attodollars per gas instead of the normal 20 billion attodollars per gas.

The goal is to make ordinary payments cheap while keeping the normal price for transactions that may create state, run arbitrary contracts, or consume the general lane.

## Motivation

Tempo's current fixed base fee is deliberately high enough to make worst-case spam and state growth expensive. That protects the chain, but it also means that transfers that do not create new state can be relatively expensive even in periods of low load. 

Simple TIP-20 transfers to existing accounts and burns do not have the same long-term cost profile as state creation. They are the payment and redemption paths we most want to be cheap, predictable, and easy to explain. This TIP introduces a narrow discount for those paths without lowering the price of everything else.

Users and wallets do not need to rely on the discount for reliability. A payment operation may still authorize the normal gas price. If it is actually cheap to execute and lands inside the discounted per-block budget, the chain charges the discounted price and refunds the difference.

## Assumptions

- A TIP-20 payment operation that actually uses no more than 250,000 gas is treated as a cheap payment operation for this TIP.
- The discount is a payment UX improvement, not a replacement for the normal base fee as the chain's main spam-resistance mechanism.
- Wallets and partners may continue to set `max_fee_per_gas` to the normal base fee for reliable inclusion.

# Specification

## Parameters

At the T6 hardfork, introduce the following parameters:

| Parameter | Value | Meaning |
| --- | ---: | --- |
| Normal base fee | 20,000,000,000 | Existing gas price for ordinary transactions |
| Discounted operation gas price | 2,000,000,000 | Gas price charged to eligible discounted operations |
| Discounted operation budget | 50,000,000 gas per block | Maximum discounted capacity in one block |
| Maximum discounted operation gas used | 250,000 gas | Per-transaction execution cap for the discount |

All gas prices are denominated in attodollars per gas.

## Eligible operations

A transaction is syntactically eligible for the discount only if all of the following are true:

1. It is a TIP-20 payment-lane transaction.
2. Every user call in the transaction is one of:
   - `transfer(address to, uint256 amount)`
   - `transferWithMemo(address to, uint256 amount, bytes32 memo)`
   - `transferFrom(address from, address to, uint256 amount)`
   - `transferFromWithMemo(address from, address to, uint256 amount, bytes32 memo)`
   - `burn(uint256 amount)`
   - `burnWithMemo(uint256 amount, bytes32 memo)`

For batch Tempo transactions, every call in the batch must satisfy the TIP-20 operation requirement. A mixed batch is not eligible for the discount.

`transferFrom` is included because it is also a payment movement. The discount does not make allowance creation cheaper: approvals are not eligible.

`burn` is included because it is the token-side operation for redemption and off-ramp flows. It does not create recipient state, and the same gas cap and per-block budget limit its spam surface.

## Applying the discount

Each block has a `discounted_operation_gas_left` counter initialized to 50,000,000 gas.

Transactions are processed in block order. There are two cases.

### Normal-fee-authorized transactions

A transaction with `max_fee_per_gas` at or above the normal base fee is valid whether or not discounted capacity is available.

After execution, such a transaction receives the discount if all of the following are true:

1. The transaction is syntactically eligible.
2. The transaction used no more than 250,000 gas.
3. `gas_used <= discounted_operation_gas_left`.

When a normal-fee-authorized transaction receives the discount, it is charged the discounted operation gas price and `discounted_operation_gas_left` is reduced by `gas_used`. If any condition is not met, the transaction is charged the normal base fee.

This lets wallets use conservative gas limits. If a user authorizes the normal base fee and the transaction turns out to be a cheap payment operation, the chain charges the discounted price and refunds the difference. If it creates state or otherwise uses too much gas, it simply pays the normal price.

### Discount-only transactions

A transaction with `max_fee_per_gas` below the normal base fee is valid only if it can be included as a discount-only transaction.

Before execution, a discount-only transaction must satisfy all of the following:

1. The transaction is syntactically eligible.
2. `max_fee_per_gas` is at least the discounted operation gas price.
3. `gas_limit <= 250,000`.
4. `gas_limit <= discounted_operation_gas_left`.

If these pre-execution checks pass, the transaction is charged the discounted operation gas price and `discounted_operation_gas_left` is reduced by `gas_limit`. If any check fails, the block is invalid.

The separate discount-only rule keeps low-fee transactions easy to validate before execution. Normal wallets SHOULD NOT rely on discount-only transactions for reliable inclusion.

The discount is all-or-nothing for a transaction. A transaction is never partly charged at the discounted gas price and partly charged at the normal base fee.

## Fee caps and refunds

A transaction that receives the discount is charged the discounted operation gas price. A transaction that does not receive the discount is charged the normal base fee.

A transaction with `max_fee_per_gas` at or above the normal base fee remains valid whether or not it receives the discount. If it receives the discount, the unused authorized amount is refunded through the existing gas refund path.

Wallets SHOULD continue to use the normal base fee by default unless they are intentionally submitting discount-only transactions. This gives users reliable inclusion while still letting them benefit from the discount when capacity is available.

## Block building

Block builders SHOULD prefer eligible low-fee operations only while discounted operation gas remains. Once the discounted budget is exhausted, builders SHOULD skip discount-only transactions whose `max_fee_per_gas` is below the normal base fee.

The payload fee total used for builder comparison MUST use the same gas price that will be charged by execution.

## RPC and estimation

Fee estimation for ordinary transactions is unchanged.

Fee estimation for eligible pure TIP-20 payment operations MAY show the discounted gas price, but RPCs and wallets SHOULD make clear that authorizing the normal base fee gives more reliable inclusion. Partner RPCs that control ordering may choose to submit discount-only operations when they expect discounted capacity to be available.

# Alternatives Considered

## Lower the base fee for all transactions

This is the simplest change, but it makes storage creation and arbitrary contract execution cheaper too. That weakens the protections the chain has from cheap worst-case spam and permanent state growth.

## Lower the payment-lane base fee for all payment-lane transactions

This is broader than the proposal in this TIP. The current payment-lane classifier includes more than simple payment and redemption operations. Discounting the whole lane risks making approvals, mints, channel operations, or future payment-lane operations cheaper even when they do not have the same cost profile as an ordinary payment.

## Use a capped EIP-1559 style fee controller

A dynamic fee could make payments cheap most of the time while rising under load. It is also more complex: the protocol would need a fee market, wallet behavior would need to change, and a lot of permanent state could be added cheaply to the chain during periods of low prices. 

## Separate storage pricing from execution pricing

This is the clean long-term model. State creation could stay expensive while execution gets cheaper. It is also a larger protocol change than needed for the immediate goal of making ordinary payments cheaper.

# Invariants

- The normal base fee remains unchanged for transactions that do not receive the discount.
- No more than 50,000,000 gas per block receives the discounted operation gas price.
- A normal-fee-authorized discounted transaction consumes discount budget by `gas_used`.
- A discount-only transaction consumes discount budget by `gas_limit`.
- The discount never applies to contract creation.
- The discount never applies to a normal-fee-authorized transaction that uses more than 250,000 gas.
- The discount never applies to a discount-only transaction with `gas_limit > 250,000`.
- The discount never applies to a mixed Tempo Transaction batch that includes anything other than eligible TIP-20 transfers or burns.
- A transaction that authorizes the normal base fee remains valid whether or not discounted budget is available.
