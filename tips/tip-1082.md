# TIP-1081: Provable Contract Trie

## Abstract

This TIP adds a second state commitment to the block header, `proof_root`, computed as a standard Ethereum-style Merkle Patricia Trie over a small, protocol-selected subset of L1 state. It is a companion to TIP-1078, which makes `state_root = LtHash(full L1 state)` the canonical execution commitment and removes the global MPT. `state_root` keeps execution commitment fast and incremental; `proof_root` preserves Merkle proofs, but only for accounts on a hardcoded, hardfork-governed whitelist. Provable accounts are always introduced empty at the hardfork that whitelists them, so the trie only ever accrues state going forward and no historical state is ever imported.

## Motivation

TIP-1078 replaces the global MPT with an additive lattice hash. LtHash gives a full-state commitment that updates per write without sorting or recomputing a global tree, which is the property we need at target throughput. What it does not give us is Merkle proofs: there is no efficient way to prove that a specific account or storage slot has a specific value against an LtHash commitment.

We still need proofs for one concrete use case: mirroring a bounded set of L1 state into Zones. A Zone (or a bridge) must be able to verify "this key has this value on L1" against a commitment in the L1 header. Today that is served by the global MPT — exactly the structure TIP-1078 removes for performance reasons.

The observation that makes this tractable is that the state Zones need is small, protocol-owned, and known ahead of time: Zone inbox/outbox state, bridge precompile state, and protocol-owned mirrors of facts we choose to expose. The vast majority of L1 state — TIP-20 balances, DEX order books, general contract storage — never needs to be proven and should not pay for trie maintenance. We therefore maintain a normal MPT, but only over a whitelist. The trie is sparse, cheap to maintain, and structurally identical to the one we compute today — it is the current state-root calculation filtered down to the whitelisted accounts.

This TIP further constrains the whitelist: **an account may only be added if it is empty at the activation block, in practice because it is a new contract or precompile introduced at that same hardfork.** Existing accounts with state are never inserted. This removes the need for a snapshot-and-backfill activation: there is no pre-existing storage to import, so adding an account is a single-step hardfork change rather than a two-phase one, and the class of activation-divergence bugs that backfill would introduce does not exist.

The consequence is that facts about pre-existing state cannot be proven by inserting the account that holds them. To make such facts provable, introduce a new empty mirror/inbox/outbox precompile at a hardfork and write the facts into it from activation onward. For example, AccountKeychain is already live with state and so cannot itself be whitelisted; proving keychain-related facts means a new precompile records those facts going forward. This is the same pattern the protocol would use to expose bridging facts without paying to represent all of TIP-20 storage, and it keeps proof support focused on the Zone-facing mirror interface Tempo deliberately chooses to expose.

This is a prerequisite-dependent change: it assumes TIP-1078 is active. It is specified independently of TIP-1078 and adds a new commitment rather than modifying `state_root`.

**Alternatives considered.**

- *User opt-in trie.* An earlier proposal let accounts opt into the trie at creation (default to a lattice hash; pay extra to be trie-backed). Rejected as fragile: it leaves migration unsolved and "anyone can opt in" gives weaker guarantees than a protocol-chosen set.
- *Adding existing accounts via snapshot-and-backfill.* The prior draft of this TIP allowed whitelisting an account with pre-existing state via a two-phase fork (backfill from a snapshot block `B`, activate at `A > B`). This TIP drops that capability in favor of the empty-at-insertion rule, trading the ability to retroactively prove existing state for a simpler, single-step activation with no backfill determinism requirement.
- *Custom MPT keyspace.* A custom `(domain, account, key)` namespace with versioned leaves was considered and dropped in favor of reusing the existing Ethereum-style account/storage trie semantics, so consumers verify a normal proof and the implementation is the existing state-root task plus a filter.
- *Keep the global MPT alongside LtHash.* Reintroduces the cost TIP-1078 exists to remove.
- *Hybrid provable/non-provable state in one trie.* Judged unclean and a long-term tech-debt source. Two cleanly separated roots is preferred.

**Deliberately out of scope.** This descopes generalized L1 state proofs and generic light clients. Proofs exist to serve the Zone-facing mirror interface Tempo chooses to expose, not to prove arbitrary L1 storage. The header field is named `proof_root` rather than `zones_proof_root`: Zones are the first consumer, but bridging and other future consumers are anticipated, so the name is kept generic. Whether Zones themselves need a state commitment is a separate question that depends on the TEE design and is not addressed here.

## Assumptions

- **TIP-1078 is active.** `state_root` is the LtHash of full L1 state and the global MPT is no longer maintained. If this TIP activated without TIP-1078, both a global MPT and the sparse MPT would be maintained, defeating the performance goal. The activation fork for this TIP MUST be at or after TIP-1078's.
- **Every whitelisted account is empty at the block that adds it.** Membership activation coincides with the account's introduction: it is a new contract/precompile created at that hardfork with zero storage. If a non-empty account were added, representing it would require importing existing storage (backfill), which this design explicitly does not do; a node that included pre-existing storage would compute a different `proof_root` than one that did not, and fork. Implementations MUST reject an upgrade that adds an account with non-empty storage.
- **`proof_root` is non-executing.** It never affects state transition or `state_root`. An implementation that let `proof_root` influence execution would diverge from spec and from other nodes.
- **Whitelist members are bounded, protocol-owned contracts.** The cost model assumes each provable account has bounded storage. A mirror precompile that records facts without cleanup (the analogue of the AccountKeychain growth concern, TIP-1073) can grow unbounded and violate `MAX_PROVABLE_ACCOUNT_STORAGE_SLOTS`, making per-block maintenance expensive.
- **Header encoding uses the trailing fork-activated optional pattern** established by TIP-1031, so pre-activation header hashes are preserved.

## Threat Model

- **Proposers / validators** are trusted to compute `proof_root` honestly; a wrong value is a hard validity failure and the block is rejected, so a single dishonest proposer cannot corrupt the commitment without forking off. Because newly whitelisted accounts are empty at activation, all honest nodes derive the same activation root from known constants with no scan of existing state; the only divergence risk is a buggy node that treats a non-empty account as eligible, which it MUST reject.
- **Authors of whitelisted precompiles** can, in principle, write chosen data into a provable precompile's storage to make arbitrary data provable. This abuse is in scope and accepted: demand is expected to be low, consequences are bounded, and it does not reintroduce a global trie.
- **Storage-explosion DoS** against a whitelisted account (adversarial or accidental growth) is in scope; mitigated by the per-account storage bound and by restricting the whitelist to bounded protocol-owned contracts.
- **Proof consumers (Zones, bridges, external verifiers)** are untrusted and verify standard MPT proofs against `proof_root`; soundness rests only on the header commitment.
- Out of scope: proving arbitrary, pre-existing, or non-whitelisted L1 state (TIP-20 balances, general contract storage); generic L1 light clients.

# Specification

This section assumes TIP-1078 is active. `proof_root` is defined and maintained independently of `state_root`.

## Header field

When activated, a new field `proof_root` is appended to `TempoHeader`, following the trailing fork-activated field pattern of TIP-1031. It is encoded as `Option<B256>`:

- When the providing fork is not active, the field is `None` and MUST be omitted entirely from the RLP stream, preserving existing block hashes for pre-activation headers.
- When active, the field MUST be `Some` and set to the root of the Provable Contract Trie defined below.

```rust
// Appended after any prior trailing fork-activated fields (e.g. TIP-1031 `Context`).
// `None` pre-activation; `Some(root)` from the activation block onward.
pub proof_root: Option<B256>,
```

`proof_root` is consensus-critical: because it is in the header, it is transitively committed to by any notarization or finalization certificate over the block digest.

## The Provable Contract Trie

`proof_root` commits to a standard Ethereum-style secure Merkle Patricia Trie, identical in format and semantics to the pre-TIP-1078 global state trie, restricted so that only accounts in the active whitelist `PROVABLE_ACCOUNTS` are inserted.

- The account trie is keyed by `keccak256(address)`; each leaf is the RLP-encoded account object `RLP([nonce, balance, storageRoot, codeHash])`.
- Each provable account's `storageRoot` is the root of its storage trie, keyed by `keccak256(slot)`, with leaves `RLP(value)`, exactly as today.
- A whitelisted account is represented by its account object from its activation block onward, with empty storage at activation; its storage trie grows only as it is written. Because the account is empty at insertion, the activation-time contribution is a known constant requiring no database scan.
- Accounts not in `PROVABLE_ACCOUNTS` are absent. The trie contains no placeholder leaves for them.
- If `PROVABLE_ACCOUNTS` is empty, `proof_root` is the canonical empty-trie root `keccak256(RLP(""))`.

The keyspace is intentionally identical to the legacy EVM state layout rather than a custom namespace, so proof verification on the consumer side is indistinguishable from verifying a standard Ethereum account/storage proof.

## Whitelist

`PROVABLE_ACCOUNTS` is a hardcoded set of account addresses and a hardfork parameter. It is not user-settable; an account becomes provable only through a network upgrade.

- Membership is keyed by **address**. The full account object (nonce, balance, codeHash, storageRoot) is included so standard account proofs work unchanged.
- **An account is eligible only if it has zero storage at the activation block that adds it.** In practice, the account is a new contract or precompile introduced at that same hardfork, so its creation and whitelist membership co-activate. Implementations MUST verify emptiness at activation and MUST reject an upgrade definition that adds a non-empty account.
- Members SHOULD be bounded, protocol-owned precompiles, e.g. Zone inbox/outbox precompiles, bridge precompiles, and protocol-owned mirrors introduced empty at the fork. User contracts MUST NOT be members.
- Each whitelisted account is subject to `MAX_PROVABLE_ACCOUNT_STORAGE_SLOTS`.

Address-keying is chosen for simplicity. Its consequence — that a future code change at a whitelisted address keeps that address provable — is acceptable because every member is a protocol-owned precompile. If finer control is later required, membership MAY be tightened to `(address, codeHash)`.

## Parameters

```
MAX_PROVABLE_ACCOUNT_STORAGE_SLOTS = TBD   // benchmark-driven per-account cap
```

A concrete value is pending the PoC benchmarks for proof-root maintenance cost under target TPS. No backfill-window parameter is required, because accounts are empty at insertion and nothing is imported.

## Block production and the state-root task

Proposers maintain `proof_root` as a filtered side-output of the existing state-commitment task:

1. Execution writes full L1 state as normal. Every account/storage write feeds the LtHash task that produces `state_root` (TIP-1078).
2. After execution, the same writes are filtered to those touching members of the active `PROVABLE_ACCOUNTS`.
3. Filtered writes — and only those — update the sparse MPT. Untouched provable state carries forward unchanged.
4. The proposer MUST set `header.proof_root` to the resulting root.

Maintaining the trie is therefore the current state-root computation restricted to the whitelist. Accounts outside the whitelist never enter the trie regardless of how heavily they are written.

## Block verification

1. If the providing fork is not active, `proof_root` MUST be `None`.
2. If active, `proof_root` MUST be `Some` and MUST equal the root recomputed by the verifier over the active whitelist after applying the block's writes.
3. Verification otherwise proceeds as-is.

A mismatch is a hard validity failure; the block MUST be rejected.

## Genesis and first activation

The genesis block MUST have `proof_root = None`. At the activation block of this TIP, the initial `PROVABLE_ACCOUNTS` members are accounts introduced empty at that fork. The initial `proof_root` is therefore the trie over those empty account objects (or the empty-trie root if the initial set is empty), derivable from known constants with no scan of existing state.

## Adding a provable account

Adding an account to `PROVABLE_ACCOUNTS` is a single-step hardfork change. Let `A` be the activation block.

- The account MUST be empty at `A`. In practice it is a new contract/precompile created at the fork that activates at `A`, so it has no pre-existing storage.
- At `A`, the account is inserted into the trie as its empty account object. Because there is no pre-existing storage, there is nothing to import or backfill, no snapshot block, and no warmup window.
- From `A` onward, writes to the account update the sparse MPT and proofs against it are valid immediately.
- Adding an account MUST NOT alter any `proof_root` for a block before `A`; historical roots are fixed by the whitelist active at the time.

To make facts about pre-existing state provable, do not attempt to insert the account that holds them. Introduce a new empty mirror/inbox/outbox precompile at a hardfork and write the relevant facts into it from activation onward.

## Edge cases

1. **Empty-at-insertion enforcement**: at `A`, nodes MUST confirm each newly added account has zero storage; an account with pre-existing storage makes the upgrade invalid.
2. **Reorgs**: trie state MUST roll back correctly across ordinary reorgs, including a reorg spanning an activation block `A` (the added account reverts to absent before `A`).
3. **Redeploy / destruct**: because membership is address-keyed, the behavior when a whitelisted address self-destructs or is redeployed MUST be explicit. Members are protocol-owned precompiles, so this is constrained.
4. **Storage explosion**: a whitelisted account whose storage grows unexpectedly makes maintenance expensive; enforce `MAX_PROVABLE_ACCOUNT_STORAGE_SLOTS` or admit only bounded protocol-owned contracts. A mirror precompile that records facts without cleanup is the live concern (cf. TIP-1073 for the analogous keychain cleanup).
5. **Partial inclusion**: proving only some storage keys of an otherwise-normal contract is not supported and is where migration pain starts. Prefer mirror precompiles that write exactly the facts to be proven.
6. **Proof serving / retention**: nodes serving historical proofs MUST retain trie data for the requested range, or the header root exists but proofs are unavailable.
7. **Whitelist versioning**: proof consumers MUST know which whitelist applies to a given block, or they may verify against the wrong logical keyspace.

# Tooling

- **Header**: `proof_root` is exposed in block/header responses. SDKs and header types add the trailing optional field; encoders/decoders MUST follow the omit-when-`None` rule so pre-activation header hashes are unchanged.
- **Proof RPC**: an `eth_getProof`-style endpoint MUST verify against `proof_root` and succeed only for whitelisted accounts. Responses MUST surface the four-way absence distinction (not whitelisted / whitelisted but empty / storage key absent / storage key set to zero). The endpoint MUST indicate, or let the caller determine, the whitelist version applicable to the queried block.
- **Zone / bridge verifiers**: proof-verification libraries MUST reference `proof_root`, not `state_root` (which is now an LtHash and not Merkle-provable), and MUST be whitelist-version aware.
- **Test fixtures**: add header fixtures with `proof_root: Some(..)` and `proof_root: None`; add an activation fixture where a new empty account is whitelisted at `A` and a golden `proof_root` is asserted immediately before and after `A`.
- **Recovery/debug**: operators need a way to inspect and recompute the sparse trie for a given block and whitelist version to diagnose a `proof_root` mismatch.

# Observability

These are node-level metrics and alerts (not on-chain events); each MUST be emitted with the listed fields.

- `proof_root_update_duration` (block number, slots touched, duration) — answers: is sparse-trie maintenance negligible relative to execution under target TPS?
- `provable_account_trie_size` (address, slot count) — answers: is any whitelisted account (notably a fact-recording mirror) growing toward the storage bound?
- `proof_root_activation_agreement` (activation block `A`, added addresses, local root, expected/peer root, match bool) — answers: did all nodes derive the same `proof_root` at the whitelist change? Because added accounts are empty, this should be trivially satisfied; a mismatch MUST alert as a potential fork or a non-empty-account bug.

If a deployment determines existing block/state telemetry already covers a given signal, it MAY map these onto existing metrics rather than adding new ones.

# Invariants

1. **Independence**: `proof_root` is computed independently of `state_root`; `state_root` remains `LtHash(full state)` and `proof_root` never affects execution.
2. **Sparsity**: only members of the active `PROVABLE_ACCOUNTS` appear in the trie; no write to a non-whitelisted account can change `proof_root`.
3. **Empty at insertion**: every account is empty at the block that adds it to `PROVABLE_ACCOUNTS`; an upgrade adding a non-empty account is invalid.
4. **Historical immutability**: for any block `N`, `proof_root` at `N` is fixed by the whitelist active at `N`; later whitelist changes MUST NOT alter it.
5. **Activation determinism**: at activation block `A`, every upgraded node derives the same `proof_root` from known constants, with no import of pre-existing state.
6. **Encoding backward compatibility**: `proof_root` is a trailing fork-activated field; when `None` it MUST be omitted from RLP so pre-activation header hashes are unchanged.
7. **Commitment**: `proof_root` is transitively committed to by any certificate over the block digest.
8. **Proof soundness**: a valid inclusion/absence proof against `header.proof_root` reflects the exact L1 state of the whitelisted account at that block.

**Critical cases the test suite MUST cover:**

- RLP round-trip for `TempoHeader` with `proof_root: Some(..)` and `proof_root: None`, and that a pre-activation header's hash is unchanged after the upgrade.
- `proof_root` equals the empty-trie root when the whitelist is empty.
- A write to a non-whitelisted account does not change `proof_root`; a write to a whitelisted account does.
- Inclusion and all four absence cases produce verifiable, distinguishable proofs against `proof_root`.
- Adding a new empty account at `A`: `proof_root` immediately before `A` excludes it and is unchanged from the prior whitelist; immediately after `A` it includes the empty account object, derived without any state import; nodes agree on both roots.
- An upgrade definition that adds an account with non-empty storage at `A` is rejected.
- A reorg spanning `A` reverts the added account to absent before `A`.
- Historical `proof_root` values are unchanged after a later whitelist addition.
