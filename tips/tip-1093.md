# TIP-1093: General Purpose Compound Policy Roles

## Abstract

This TIP makes TIP-403 compound policies general purpose by extending them from the fixed TIP-20 roles of sender, recipient, and mint recipient to reusable, application-defined role sets. Each immutable, ordered role set defines a schema, and a compound policy binds every role index to an existing simple or built-in TIP-403 policy.

At T9, a set-aware creation overload is added, `compoundPolicyData` adopts a generalized return ABI, and role authorization resolves the compound's stored role set directly. Historical compounds normalize to the built-in TIP-20 role-set ID `0` without migration, preserving their storage, authorization behavior, and normal-path storage-read counts.

## Motivation

TIP-403 is a shared authorization registry, but TIP-1015 compounds are not general purpose: they expose only three TIP-20-specific roles. Other consumers may need application-defined address roles, such as deposit source, withdrawal recipient, callback target, or refund recipient.

Adding protocol-specific fields to the existing compound structure does not scale. Having each application store and dispatch several independent policy IDs works, but duplicates schema and wiring logic across consumers. Role sets provide one immutable compound handle and a reusable description of what each child policy means.

The general-purpose role-set model improves policy reuse and configuration safety. It does not remove application-side operation classification, reduce the number of underlying simple policies, or make TIP-403 cheaper than application-local authorization. Applications with private, non-reused permission state may remain better served by local storage.

## Assumptions

- TIP-1015 is active before this TIP.
- `PolicyData.policyType` and `PolicyData.admin` retain their existing byte offsets. The remaining bytes in that storage word are zero for historical policies.
- Existing compound records store their three child policy IDs in the low 192 bits of `PolicyRecord` offset `1`; the high 64 bits are zero.
- Simple-policy types remain immutable after creation.
- Referenced child policies remain independently mutable by their admins. Applications requiring fixed membership must separately use policies whose administration has been renounced or otherwise constrained.
- Applications know the expected role-set ID and role index for each operation. A configurable consumer that does not validate the role set before storing a policy ID may assign the wrong meaning to an index.
- TIP-403 protocol-address exemptions are transfer-specific behavior. They continue to apply to the canonical TIP-20 role set but do not apply to custom role sets.

If historical metadata bytes are nonzero, or if an existing compound's fourth packed position is nonzero, activation MUST be blocked until the conflicting state is resolved.

## Threat Model

- Role-set and compound creators are untrusted. They may supply duplicate roles, malformed schemas, nonexistent child policies, compound children, or misleading role ordering; creation MUST reject invalid structures.
- Applications are responsible for validating and pinning the expected role-set ID when accepting configuration. The registry exposes the stored set but cannot determine which schema an application intended to use.
- Child-policy admins are trusted to manage membership according to the consuming application's requirements. Compromise of a child admin changes authorization for every compound that references that child.
- Policy subjects are untrusted and may attempt to exploit out-of-range indexes, built-in policy IDs, protocol-address exemptions, or malformed historical state.
- Indexers and SDKs may receive duplicate or reordered role definitions from untrusted sources. They MUST verify event data against the onchain role-set commitment.

# Specification

## Terminology

| Term | Meaning |
|---|---|
| Simple policy | A TIP-403 whitelist or blacklist, including built-in reject-all policy `0` and allow-all policy `1`. |
| Role key | A namespaced `bytes32` identifier describing one address-authorization role. |
| Role set | An immutable ordered list of unique role keys. |
| Role-set ID | A `uint64` identifier for a role set. |
| Role index | A zero-based position in a role set. |
| Role-set compound | A compound policy containing one simple child policy per role index. |
| Canonical TIP-20 role set | Built-in role-set ID `0`, ordered as sender, recipient, and mint recipient. |

A custom role-set compound is a policy bundle. It has no implicit AND, OR, threshold, or other aggregate authorization semantics.

## Limits

```solidity
uint8 constant MAX_ROLE_COUNT = 32;
uint64 constant TIP20_ROLE_SET_ID = 0;
```

A role set MUST contain between `1` and `MAX_ROLE_COUNT` roles. The initial bound of 32 limits creation calldata, duplicate detection, child validation, event size, and worst-case storage writes. A later protocol upgrade may raise the bound by appending additional compound-policy words without changing existing offsets.

## Role Keys

Applications SHOULD derive role keys as:

```solidity
bytes32 roleKey = keccak256("<namespace>.<role-name>.<version>");
```

For example:

```solidity
keccak256("tempo.zone.deposit-source.v1")
keccak256("tempo.zone.withdrawal-recipient.v1")
keccak256("tempo.zone.callback-target.v1")
```

A role key MUST be nonzero. A role set MUST NOT contain duplicate role keys. Ordering is part of the schema: the same keys in a different order define a different role set.

Role keys are descriptive identifiers only. Authorization uses the role-set ID and role index; the registry does not interpret role names.

## Role-Set Commitment

The domain separator is:

```solidity
bytes32 constant ROLE_SET_DOMAIN =
    keccak256("TEMPO_TIP403_ROLE_SET_V1");
```

The full commitment is:

```solidity
bytes32 fullHash = keccak256(
    abi.encode(ROLE_SET_DOMAIN, roles)
);
```

The registry stores the complete `bytes32` commitment. The role count is packed with the reverse lookup:

```solidity
struct RoleSetLookup {
    uint64 roleSetId;
    uint8 roleCount;
}
```

The ordered role list is emitted but is not stored onchain. Keeping the full commitment avoids truncated-hash semantics; role-set registration is not an authorization hot path.

## Canonical TIP-20 Role Set

Role-set ID `0` is synthesized by the registry and consumes no role-set storage:

| Index | Role key derivation | Role key |
|---:|---|---|
| 0 | `keccak256("tempo.tip20.sender.v1")` | `0xdc5f291c03c80dec78fdde62a71ef4dd5775db3c61b9f0ece27672bd7d806094` |
| 1 | `keccak256("tempo.tip20.recipient.v1")` | `0xd69817f5faca037ea7af194f5f7a5e4b2ececc8aa270e3185be7679542371080` |
| 2 | `keccak256("tempo.tip20.mint-recipient.v1")` | `0x5dd70e5f0ea97d83b9de9ea6e2be546e7fa3da37479058a93f2f5ef1acf3c3f1` |

Using the commitment algorithm above, the canonical full hash is:

```text
0x75257f2ee557ff53d5dd41db45c54686781d6da5dae399b604997926dbbd11eb
```

`roleSetExists(0)` MUST return `true`. `roleSetData(0)` MUST return the full canonical commitment and role count `3`.

Role-set IDs and policy IDs are independent namespaces.

## Role-Set Registration

Custom role-set IDs start at `1`. The registry appends the following state after all existing TIP-403 top-level fields:

```solidity
uint64 internal roleSetIdCounter; // effective initial value is 1
mapping(uint64 roleSetId => bytes32 fullHash) internal roleSetHashes;
mapping(bytes32 fullHash => RoleSetLookup lookup) internal roleSetsByHash;
```

`createRoleSet` MUST:

1. require `1 <= roles.length <= MAX_ROLE_COUNT`;
2. reject zero role keys;
3. reject duplicate role keys;
4. preserve the supplied order;
5. compute the domain-separated full commitment;
6. reject the canonical TIP-20 schema and any custom schema whose full commitment is already registered;
7. allocate the next nonzero role-set ID;
8. store the full hash by role-set ID;
9. store `RoleSetLookup(roleSetId, uint8(roles.length))` by full hash;
10. emit `RoleSetCreated` with the full ordered role list.

Role sets have no admin and are immutable. Duplicate registration is rejected so one ordered schema has one role-set ID per network. Each custom schema occupies one full-hash slot and one packed reverse-lookup slot.

For a custom ID, `roleSetExists` returns true exactly when `1 <= roleSetId < effectiveRoleSetIdCounter`. `roleSetData` reads `roleSetHashes[roleSetId]` and resolves its count from `roleSetsByHash[fullHash]`. `roleSetIdByHash` synthesizes `(true, 0)` for the canonical hash; for custom hashes it returns `exists = lookup.roleSetId != 0`.

## Compound Metadata

The existing base policy word is extended without moving existing fields:

```solidity
struct PolicyData {
    uint8 policyType;  // existing byte offset 0
    address admin;     // existing byte offsets 1..20
    uint64 roleSetId;  // new byte offsets 21..28
    uint8 roleCount;   // new byte offset 29
    // byte offsets 30..31 remain reserved
}
```

For a compound policy:

- `roleSetId == 0` means the canonical TIP-20 role set;
- canonical compounds store `roleCount == 0` and have effective role count `3`;
- `roleSetId > 0` means a custom role set and MUST store its nonzero role count;
- simple policies MUST store zero in both fields.

The custom role count is duplicated from the role-set record so authorization does not read role-set storage on the hot path. Role sets are immutable, so the values cannot diverge after validated creation.

## Packed Child Policies

All child policy IDs use one uniform fixed array:

```solidity
struct PolicyRecord {
    PolicyData base;                 // offset 0
    bytes32[8] compoundPolicyData;   // offsets 1..8
}
```

`compoundPolicyData[0]` occupies the existing compound slot at offset `1`. Its low three `uint64` positions preserve the sender, recipient, and mint-recipient child IDs, while its high `uint64` position stores role `3`. Each later word stores another four roles:

```text
compoundPolicyData[0] -> roles  0..3, offset 1
compoundPolicyData[1] -> roles  4..7, offset 2
...
compoundPolicyData[7] -> roles 28..31, offset 8
```

Extending the mapping value does not move existing offsets and does not physically allocate unwritten fixed-array slots. Creation MUST write only the `ceil(roleCount / 4)` child words required by the role count.

Child IDs are packed in role order. For a role index, `wordIndex = roleIndex >> 2`, `position = roleIndex & 3`, and the child occupies bits `(position * 64)..((position + 1) * 64 - 1)` of `compoundPolicyData[wordIndex]`.

For `N` roles, a compound occupies one base metadata slot plus `ceil(N / 4)` populated child-policy slots.

## Set-Aware Compound APIs

Starting at T9, compound creation is overloaded. The existing overload implies canonical TIP-20 role-set ID `0`; the new overload accepts and validates a role-set ID. The existing `createCompoundPolicy` selector retains its ABI and behavior.

The new `createCompoundPolicy(roleSetId, childPolicyIds)` overload MUST:

1. resolve the supplied role set, including built-in role-set ID `0`;
2. require `childPolicyIds.length == roleCount`;
3. require every child policy to exist;
4. require every child policy to be simple or built-in;
5. reject compound children;
6. allocate the next TIP-403 policy ID;
7. store `PolicyType.COMPOUND` and zero admin;
8. store zero role metadata for role-set ID `0`;
9. otherwise store the custom role-set ID and count;
10. pack and write only the required child-policy words;
11. emit `RoleSetCompoundPolicyCreated`.

The existing `createCompoundPolicy(sender, recipient, mintRecipient)` overload is equivalent to calling the set-aware overload with role-set ID `0` and the three supplied children. It retains its existing selector and emits only `CompoundPolicyCreated`.

When the set-aware overload is called explicitly with role-set ID `0`, it MUST emit both `RoleSetCompoundPolicyCreated` and `CompoundPolicyCreated` so generic and legacy indexers discover the policy. Canonical compounds created through either overload MUST have identical storage and authorization behavior.

Child references are immutable. Membership and administration of referenced simple policies retain their existing behavior.

## Interface

The TIP-403 registry adds:

```solidity
interface ITIP403RoleSets {
    /// @notice Emitted when a new ordered role schema is registered.
    /// @param roleSetId The allocated nonzero role-set ID.
    /// @param rolesHash The full domain-separated commitment to `roles`.
    /// @param roles The complete ordered role-key list.
    event RoleSetCreated(
        uint64 indexed roleSetId,
        bytes32 indexed rolesHash,
        bytes32[] roles
    );

    /// @notice Emitted when a compound policy is created from a role set.
    /// @param policyId The allocated TIP-403 policy ID.
    /// @param creator The account that created the compound.
    /// @param roleSetId The role-set ID defining child ordering.
    /// @param childPolicyIds The ordered simple child policy IDs.
    event RoleSetCompoundPolicyCreated(
        uint64 indexed policyId,
        address indexed creator,
        uint64 indexed roleSetId,
        uint64[] childPolicyIds
    );

    /// @notice Thrown when a role-set ID does not exist.
    error RoleSetNotFound(uint64 roleSetId);

    /// @notice Thrown when a role count is zero or exceeds the protocol maximum.
    error RoleCountOutOfRange(uint256 actual, uint256 maximum);

    /// @notice Thrown when a role key is zero.
    error InvalidRoleKey(uint8 roleIndex);

    /// @notice Thrown when the same role key appears more than once.
    error DuplicateRole(bytes32 roleKey, uint8 firstIndex, uint8 duplicateIndex);

    /// @notice Thrown when an identical ordered schema is already registered.
    error RoleSetAlreadyExists(bytes32 rolesHash, uint64 roleSetId);

    /// @notice Thrown when the child count differs from the role-set count.
    error ChildPolicyCountMismatch(uint256 actual, uint256 expected);

    /// @notice Thrown when a role index is outside the compound's role set.
    error InvalidRoleIndex(uint8 roleIndex, uint8 roleCount);

    /// @notice Thrown when a caller expects a different role set than the compound uses.
    error IncompatibleRoleSet(uint64 expected, uint64 actual);

    /// @notice Returns the next custom role-set ID, with an effective initial value of 1.
    function roleSetIdCounter() external view returns (uint64 nextRoleSetId);

    /// @notice Registers an immutable ordered role schema.
    /// @param roles Unique, nonzero, namespaced role keys in canonical application order.
    /// @return roleSetId The allocated nonzero role-set ID.
    /// @dev Reverts when the count is outside `1..MAX_ROLE_COUNT`, a key is zero or
    /// duplicated, the schema already exists, or the role-set counter overflows.
    function createRoleSet(bytes32[] calldata roles)
        external
        returns (uint64 roleSetId);

    /// @notice Returns whether a role-set ID exists.
    /// @param roleSetId The role-set ID to inspect. ID 0 always exists.
    function roleSetExists(uint64 roleSetId) external view returns (bool);

    /// @notice Returns an immutable role set's full commitment and count.
    /// @param roleSetId The role-set ID to inspect.
    /// @return rolesHash The full domain-separated role-set commitment.
    /// @return roleCount The number of ordered roles.
    /// @dev Reverts with `RoleSetNotFound` if the role set does not exist.
    function roleSetData(uint64 roleSetId)
        external
        view
        returns (bytes32 rolesHash, uint8 roleCount);

    /// @notice Resolves a full role-set commitment to its unique network-local ID.
    /// @param rolesHash The full domain-separated role-set commitment.
    /// @return exists Whether the schema is registered.
    /// @return roleSetId The role-set ID, including 0 for the canonical TIP-20 schema.
    function roleSetIdByHash(bytes32 rolesHash)
        external
        view
        returns (bool exists, uint64 roleSetId);

    /// @notice Creates a canonical TIP-20 compound policy using role-set ID 0.
    /// @param senderPolicyId Existing simple policy for sender authorization.
    /// @param recipientPolicyId Existing simple policy for recipient authorization.
    /// @param mintRecipientPolicyId Existing simple policy for mint-recipient authorization.
    /// @return policyId The allocated TIP-403 policy ID.
    function createCompoundPolicy(
        uint64 senderPolicyId,
        uint64 recipientPolicyId,
        uint64 mintRecipientPolicyId
    ) external returns (uint64 policyId);

    /// @notice Creates an immutable compound policy for an ordered role set.
    /// @param roleSetId The existing role set defining child order.
    /// @param childPolicyIds One existing simple or built-in policy ID per role.
    /// @return policyId The allocated TIP-403 policy ID.
    /// @dev Reverts for an unknown role set, wrong child count, nonexistent child,
    /// compound child, invalid child type, or policy-counter overflow.
    function createCompoundPolicy(
        uint64 roleSetId,
        uint64[] calldata childPolicyIds
    ) external returns (uint64 policyId);

    /// @notice Returns a compound's role-set metadata and all ordered children.
    /// @param policyId The compound policy ID.
    /// @return roleSetId The compound's effective role-set ID.
    /// @return roleCount The number of roles and child policies.
    /// @return childPolicyIds One simple or built-in policy ID per role, in set order.
    /// @dev Reverts with the existing policy errors for nonexistent or non-compound IDs.
    function compoundPolicyData(uint64 policyId)
        external
        view
        returns (
            uint64 roleSetId,
            uint8 roleCount,
            uint64[] memory childPolicyIds
        );

    /// @notice Checks an account against one role of a compound policy.
    /// @param policyId The compound policy ID.
    /// @param roleIndex The zero-based index in the compound's stored role set.
    /// @param account The address subject to authorization.
    /// @return authorized Whether the selected simple child authorizes `account`.
    /// @dev Applications MUST validate the compound's role set when accepting
    /// configuration. Custom role sets use strict evaluation without TIP-20-specific
    /// protocol-address exemptions; canonical role-set ID 0 retains those exemptions.
    function isAuthorizedForRole(
        uint64 policyId,
        uint8 roleIndex,
        address account
    ) external view returns (bool authorized);
}
```

Existing `PolicyNotFound`, `PolicyNotSimple`, `InvalidPolicyType`, and `IncompatiblePolicyType` errors remain in use where stated.

## Child Lookup

The logical extraction algorithm is:

```solidity
function _compoundPolicyForRole(
    uint64 policyId,
    uint8 roleIndex
) internal view returns (uint64 childPolicyId) {
    uint8 wordIndex = roleIndex >> 2;
    uint8 position = roleIndex & 3;

    bytes32 packed =
        _policyRecords[policyId].compoundPolicyData[wordIndex];

    childPolicyId = uint64(
        uint256(packed) >> (uint256(position) * 64)
    );
}
```

Implementations may use equivalent direct storage handlers but MUST preserve the offsets and bit ordering above.

## Compound Inspection and Authorization

`compoundPolicyData(policyId)` MUST return the compound's effective role-set ID, effective role count, and exactly one child policy ID per role in index order. For historical and canonical TIP-20 compounds it returns role-set ID `0`, role count `3`, and the sender, recipient, and mint-recipient child IDs in that order.

`isAuthorizedForRole` MUST:

1. require `policyId` to exist and be `COMPOUND`;
2. normalize zero role metadata to role-set ID `0` and count `3`;
3. require `roleIndex < roleCount`;
4. read the one packed child word containing `roleIndex`;
5. extract the selected `uint64` child policy ID;
6. evaluate the child as a simple policy.

For custom role sets, step 6 is strict whitelist, blacklist, reject-all, or allow-all evaluation and MUST NOT apply TIP-20 protocol-address exemptions. This prevents transfer-specific exceptions from becoming implicit authorization for unrelated applications.

For canonical role-set ID `0`, existing TIP-20 protocol-address exemptions MUST apply so role authorization matches canonical TIP-20 directional behavior.

Built-in child policies `0` and `1` retain their existing no-storage short circuits. Applications MUST call `compoundPolicyData` and validate the returned role-set ID and commitment before storing or otherwise accepting a configurable compound policy.

## Example

An application can register a route schema once:

```solidity
bytes32[] memory roles = new bytes32[](3);
roles[0] = keccak256("tempo.zone.deposit-source.v1");
roles[1] = keccak256("tempo.zone.withdrawal-recipient.v1");
roles[2] = keccak256("tempo.zone.callback-target.v1");

uint64 roleSetId = registry.createRoleSet(roles);
```

It then binds one simple policy to each role:

```solidity
uint64[] memory children = new uint64[](3);
children[0] = depositSourcePolicyId;
children[1] = withdrawalRecipientPolicyId;
children[2] = callbackTargetPolicyId;

uint64 policyId =
    registry.createCompoundPolicy(roleSetId, children);
```

A callback check pins both schema and index:

```solidity
registry.isAuthorizedForRole(policyId, 2, callbackTarget);
```

The application still classifies the callback operation and decides how authorization failure affects its control flow.

## Legacy Aggregate and Directional Authorization

Arbitrary role sets have no aggregate authorization result. Applications MUST query the role relevant to the operation.

The existing creation and authorization APIs retain their current behavior for simple policies and canonical compounds:

```solidity
createCompoundPolicy(senderPolicyId, recipientPolicyId, mintRecipientPolicyId)
isAuthorized(policyId, account)
isAuthorizedSender(policyId, account)
isAuthorizedRecipient(policyId, account)
isAuthorizedMintRecipient(policyId, account)
```

At T9, `compoundPolicyData(policyId)` intentionally changes to the generalized return ABI defined above.

For canonical compounds:

```text
isAuthorized = sender authorization AND recipient authorization
```

The existing sender-failure short circuit and all existing protocol-address exemptions remain unchanged.

For custom role-set compounds, the TIP-20-specific authorization APIs MUST revert with `IncompatibleRoleSet(0, actualRoleSetId)` after existing protocol-address exemptions have been considered. Callers inspect custom compounds through `compoundPolicyData` and authorize through `isAuthorizedForRole`.

Keeping protocol-address exemptions first preserves the existing no-storage short circuit. Custom compounds cannot be installed as TIP-20 transfer policies, so this precedence does not affect valid TIP-20 execution.

## TIP-20 and Backwards Compatibility

TIP-20 transfer policies MUST be built-in, simple, or compound with effective role-set ID `0`. All token-policy creation and update paths MUST reject custom role-set compounds. The transaction pool and policy caches MUST verify role-set ID `0` before interpreting compound indexes as TIP-20 roles and fail closed on malformed token state.

Historical compounds normalize to role-set ID `0` and count `3`; their child positions, event history, and authorization behavior remain unchanged. No policy migration or ID renumbering is required, and `PolicyType.COMPOUND` is retained.

The `compoundPolicyData(uint64)` selector intentionally changes return ABI at T9. Existing contracts and SDKs that decode its legacy fixed tuple MUST upgrade because stale callers would misdecode the dynamic result. TIP-20 execution is unaffected because it does not depend on this getter.

## Activation

All new selectors and custom role-set interpretation activate at T9. Before activation, new selectors are unavailable, spare metadata bytes are ignored, and `compoundPolicyData(uint64)` returns the legacy fixed tuple. Starting at T9, that selector returns the generalized `(roleSetId, roleCount, childPolicyIds)` ABI.

At activation:

- existing compound records normalize to role-set ID `0`, count `3`;
- existing simple policies remain unchanged;
- custom role-set counter has effective initial value `1`;
- no historical policy, token, or membership state is rewritten.

Implementations MUST gate new selectors and behavior at the activation boundary and MUST include pre- and post-activation state-transition tests.

## Gas and Storage Behavior

Legacy TIP-20 authorization MUST NOT read:

- custom role-set records;
- role-set commitments;
- reverse role-set mappings;
- compound-policy words beyond the first;
- role keys.

Reading `roleSetId` from the already-loaded base policy word does not add an SLOAD. Existing canonical sender, recipient, mint-recipient, aggregate, built-in, and protocol-address short circuits MUST retain their existing storage-operation counts.

A non-built-in custom role authorization normally reads:

1. compound metadata;
2. one packed child word;
3. child policy metadata;
4. child membership.

Role-set size does not change single-role authorization cost. Every role reads one indexed `compoundPolicyData` word. `compoundPolicyData` returns the full vector and therefore reads `ceil(roleCount / 4)` child words.

Role-set creation MUST charge for calldata, commitment hashing, duplicate detection, event data, and storage writes according to the active precompile gas schedule. Duplicate detection MUST be bounded by `MAX_ROLE_COUNT`.

# Tooling

TIP-403 SDKs and command-line tools MUST update `compoundPolicyData(uint64)` decoding at the T9 boundary and add support for:

- deriving namespaced role keys;
- computing role-set commitments;
- resolving or creating role sets;
- creating role-set compounds;
- displaying a compound's role set and ordered children;
- validating an application's expected role-set commitment and indexes;
- checking TIP-20 compatibility before assigning a policy to a token.

Indexers MUST reconstruct full role schemas from `RoleSetCreated`, verify the full commitment, and expose the network-local role-set ID. Role-set IDs other than `0` are not portable across networks; deployment tooling MUST resolve them by commitment rather than assume a numeric ID.

Raw-storage tools, the Rust precompile implementation, Solidity reference layout, transaction-pool policy cache, and generated bindings MUST agree on metadata offsets, child-word offsets, and packed endianness.

# Observability

`RoleSetCreated` answers which ordered schema was registered under a role-set ID. The event includes the full commitment and role list so indexers can reconstruct and verify the schema without onchain role-name storage.

`RoleSetCompoundPolicyCreated` answers who created a compound, which schema it uses, and which ordered child policies it references. Canonical compounds created through the set-aware overload also emit `CompoundPolicyCreated` for legacy indexer compatibility.

Simple-policy membership and admin changes continue to use the existing TIP-403 events. No compound-level membership event is added because membership belongs to the child policy.

Operational monitoring SHOULD flag:

- attempts to bind custom role-set compounds to TIP-20 tokens;
- application configuration whose expected role-set commitment does not match the configured ID;
- child-policy admin changes affecting widely reused compounds.

# Security Considerations

## Role-Index Confusion

A role index has meaning only within its role set. Applications MUST validate the role-set ID returned by `compoundPolicyData(policyId)` when accepting a configurable compound policy and SHOULD also verify the role-set commitment. Authorization then uses the validated policy ID and role index without repeating the role-set ID on every call.

## Mutable Child Policies

Compound wiring is immutable, but child membership and child admins remain mutable. Reusing one child policy intentionally couples every consuming compound to its updates. This TIP does not freeze policies or transfer their administration.

## Protocol-Address Exemptions

TIP-20 protocol-address exemptions are preserved only for role-set ID `0`. Custom role sets use strict simple-policy evaluation so a transfer-specific exemption cannot silently authorize an unrelated application role.

## No Recursive Compounds

Every child must be simple or built-in. This prevents cycles, recursive evaluation, and input-dependent authorization depth.

## Event-Backed Schema Discovery

Role names are emitted rather than stored. Tooling MUST verify the event's full ordered role list against the full onchain commitment.

# Invariants

- Every compound has exactly one existing simple or built-in child per role. Child references are immutable and compounds cannot reference compounds.
- Role-set ID `0` is always the immutable canonical TIP-20 schema. TIP-20 only accepts the canonical role-set ID `0`. Its aggregate and directional authorization semantics remain unchanged.
- Every role set contains between `1` and `MAX_ROLE_COUNT` unique, nonzero role keys. Custom role sets have no aggregate authorization semantics and do not inherit TIP-20 protocol-address exemptions.
- For every valid role index `i`, authorization evaluates `childPolicyIds[i]`. Lookup reads exactly one packed child word, which stores 4 consecutive child policy IDs.
