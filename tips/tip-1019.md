# TIP-1019: Subblocks v2 - Builder Delegation & Bundle Proposals

## Abstract

This TIP introduces major changes to the subblock system:

1. **Builder Delegation**: Validators delegate subblock production to a builder identified by an address. Multiple validators can delegate to the same builder, who accumulates their combined gas budget. The builder address is also the fee recipient for subblock fees.

2. **Builder P2P Keys**: Each physical builder machine has its own ed25519 key registered under the builder address in the SubblockRegistry (max 4 per builder). These keys serve dual purpose: commonware p2p authentication and subblock signing.

3. **Bundle Proposal Model**: The builder signs a SubBlock containing **all proposed transactions** (of any format). The proposer processes the bundle in order — executing valid transactions, **dropping** precheck-invalid ones, and optionally **truncating** overflow beyond the gas budget. Dropped and unprocessed transactions are placed in the SubBlockMetadata system transaction so validators can reconstruct the original signed SubBlock and verify the builder's signature.

4. **Any Transaction Format**: Subblock transactions are not restricted to the `0x5b` nonce key format. Standard Ethereum transactions (EIP-1559, etc.) and Tempo AA transactions are allowed. The `0x5b` nonce key remains available as an **optional** mechanism for nonce isolation.

5. **Metadata-Based Fee Routing**: Fees are routed to the builder based on the transaction's position within a metadata entry's executed range, not by extracting a builder address from the nonce key.

6. **Relaxed Signing Requirements**: Keychain (access key) signatures are allowed in subblocks. The drop mechanism handles the key revocation race condition — if a key is revoked in the main block, the proposer drops the transaction with `AccessKeyRevoked`.

7. **Overflow**: Builders may propose more transactions than their gas budget. The proposer is guaranteed to include valid transactions up to the builder's gas budget (the "floor") and may include additional "overflow" transactions on a best-effort basis.

8. **Expiring Nonces in Subblocks**: Subblock transactions using the `0x5b` nonce key can use expiring nonces by setting the nonce remainder to all 1s (`0xFFFF...`).

## Motivation

The current subblock design has several limitations:

1. **Validator coupling**: The validator's consensus key must also sign subblocks, meaning validators cannot delegate subblock production to separate infrastructure without sharing their consensus key.

2. **One-to-one only**: Each validator must run their own subblock builder; there is no way to delegate to a shared builder.

3. **Restricted transaction format**: Only transactions with a special `0x5b` nonce key format can be included in subblocks, preventing standard Ethereum transactions from being routed through builders.

4. **Root-key-only signing**: Keychain (access key) signatures are rejected due to a key revocation race condition, excluding the most common Tempo signing method.

5. **Rigid sizing**: Subblocks must exactly fit within the builder's gas budget. There is no mechanism for builders to propose extra transactions that the proposer could include opportunistically.

6. **No expiring nonces**: Subblock transactions require sequential nonces within their nonce key, preventing use of expiring nonces (TIP-1009).

7. **No high availability**: A single builder machine failure takes down subblock production for that validator. There is no mechanism for multiple machines to operate under a shared identity. The latter is because the CommonwareP2P network is authenticated, and each machine joining the network must have a unique identity.

This TIP addresses these issues by:

- Allowing validators to delegate to builders identified by address
- Supporting many-to-one delegation where multiple validators share a builder
- Each builder machine has its own ed25519 key for both p2p authentication and subblock signing — no shared key material (KMS/HSM) required
- Allowing any transaction format in subblocks with metadata-based fee routing
- Using a drop mechanism to handle precheck-invalid transactions (including revoked keychain keys)
- Supporting overflow beyond the gas budget with proposer best-effort inclusion
- Enabling expiring nonces within subblock transactions

# Specification

## Definitions

- **Parent post-state**: The world state after executing the parent block — equivalently, the state at the very start of the current block, before any of its transactions execute.
- **Post-main-block state** (`S_post_main`): The world state after executing all main block transactions but before executing any subblock transactions. This is the state used for subblock precheck validation.
- **Builder address**: A 20-byte address that identifies the builder and serves as the fee recipient. Must be capable of calling `registerP2pKey`, `removeP2pKey`, and `FeeManager.setValidatorToken`.
- **Validator in the set**: An address that is an active validator in `ValidatorConfig`. (Currently: non-zero `publicKey`; after ValidatorConfig V2: `deactivatedAtHeight == 0`.) Note: `ValidatorConfig` records the *intended* validator set. The *actual* active consensus committee is determined by the DKG outcome stored in boundary block headers. During rare DKG failures these may temporarily diverge; they reconverge on the next successful DKG. Gas weight uses `ValidatorConfig` as the practical source of truth since it is readable per-block from precompile state.
- **Gas weight**: The number of validators in the set currently delegating to a given builder. Computed on the fly from `delegatedBuilder` mappings and the current validator set — not stored on-chain.
- **Gas floor**: The builder's gas budget, computed as `per_validator_gas × gas_weight(builder)`. The proposer is obligated to execute valid transactions up to this limit.
- **Precheck**: A set of cheap, deterministic validation checks (no EVM execution) that determine whether a transaction can be executed. Precheck failure is the only valid reason for a proposer to drop a transaction from a bundle.
- **Drop**: A transaction in a signed bundle that the proposer removes because it fails precheck. The raw transaction bytes and drop reason are recorded in the SubBlockMetadata system transaction.
- **Overflow**: Transactions in a signed bundle beyond the builder's gas floor. The proposer may execute these on a best-effort basis or leave them unprocessed.
- **Nonce key** (`nonceKey`): The full `U256` routing key embedded in a transaction. For subblock nonce isolation, the format `0x5b | builder_address | nonce_remainder` is available but not required.

## 1. Subblock Registry

A new Subblock Registry precompile stores validator delegations and builder p2p key registrations.

### 1.1 Storage

```solidity
// Validator → builder address they delegate to (zero = not delegated)
mapping(address => address) public delegatedBuilder;

// Builder address → registered ed25519 p2p keys (max 4)
mapping(address => bytes32[]) public p2pKeys;
```

**Gas weight** is not stored on-chain. Since there are few builders (only those with at least one delegation from a validator in the set), nodes keep all builder configs in memory and compute gas weight on the fly by counting `delegatedBuilder` entries from validators currently in the set.

**Fee token preference** is not stored in SubblockRegistry. Builders set their preferred fee token via the existing `FeeManager.setValidatorToken(token)`, which is permissionless (any address can call it). During subblock execution, `block.beneficiary` is temporarily set to the builder address, so `FeeManager.validatorTokens[builder]` is consulted for fee routing — the same mechanism used for proposer fees.

### 1.2 Interface

```solidity
interface ISubblockRegistry {
    // ========== Delegation ==========

    /// @notice Delegates the caller's subblock production to a builder.
    /// @param builder The builder's address (also the fee recipient).
    /// @dev Sets delegatedBuilder[msg.sender] = builder.
    /// @dev Anyone can call, but only delegations from validators in the set
    ///      contribute to gas weight.
    /// @dev Requires p2pKeys[builder].length > 0 (builder has at least one registered key).
    function delegateToBuilder(address builder) external;

    /// @notice Clears the caller's delegation.
    /// @dev Sets delegatedBuilder[msg.sender] = address(0).
    function clearDelegation() external;

    // ========== Builder Functions ==========

    /// @notice Registers an ed25519 p2p key under the caller's builder address.
    /// @param p2pPubkey The ed25519 public key for p2p authentication and subblock signing.
    /// @param signature Ed25519 signature from p2pPubkey over
    ///        keccak256(abi.encodePacked("TEMPO_SUBBLOCK_REGISTER_P2P_KEY", chainId, address(this), msg.sender))
    ///        proving the caller controls the corresponding private key.
    /// @dev Max 4 p2p keys per builder. Reverts if limit exceeded.
    /// @dev Reverts if p2pPubkey is already registered to any builder.
    function registerP2pKey(bytes32 p2pPubkey, bytes calldata signature) external;

    /// @notice Removes an ed25519 p2p key from the caller's builder address.
    /// @param p2pPubkey The ed25519 public key to remove.
    /// @dev Reverts if p2pPubkey is not registered to the caller.
    function removeP2pKey(bytes32 p2pPubkey) external;

    // ========== View Functions ==========

    /// @notice Returns the builder a validator has delegated to.
    function builderOf(address validator) external view returns (address);

    /// @notice Returns the p2p keys registered to a builder.
    function p2pKeysOf(address builder) external view returns (bytes32[] memory);
}
```

### 1.3 Behavior

**Registry state snapshot:** All registry lookups (delegations, p2p keys) during subblock validation and execution use the **parent post-state**. Mutations within the current block do not affect that block's subblock validity or fee routing.

**Delegation:**
- Anyone can call `delegateToBuilder()`, but only delegations from validators in the set contribute to gas weight
- When delegating, any existing delegation is cleared first
- Delegation requires the builder to have at least one registered p2p key
- On-chain delegation changes take effect immediately in the next block

**P2P keys:**
- Each builder can register up to 4 ed25519 p2p keys (one per physical machine)
- These keys serve dual purpose: commonware p2p authentication and subblock signing
- `registerP2pKey()` requires an ed25519 signature from the p2p key over a domain-separated message including `chainId`, the SubblockRegistry address, and the caller's address. This prevents key sniping (an attacker registering someone else's key to grief them).
- `registerP2pKey()` reverts if the key is already registered to any builder
- `removeP2pKey()` reverts if the key is not owned by the caller
- A builder MAY remove all their p2p keys even while validators delegate to them. This effectively disables their subblock production until a key is re-registered.

**Gas weight (computed, not stored):**
- Gas weight for a builder is the count of validators **in the set** whose `delegatedBuilder` points to that builder
- Computed on the fly from the current validator set and `delegatedBuilder` mappings
- When a validator leaves the set, their delegation entry becomes inert — it is not counted toward gas weight since the validator is no longer in the set. No cleanup callback is needed.

**Fee token:**
- Builders set their preferred fee token by calling `FeeManager.setValidatorToken(token)` from the builder address
- This is the same mechanism validators use
- If `FeeManager.validatorTokens[builder]` is unset (zero), fees are routed in PathUSD (the FeeManager default). Builders SHOULD set an explicit preference to ensure predictable fee routing.

### 1.4 Precompile Address

```
SUBBLOCK_REGISTRY_ADDRESS = 0x0b10c0000000000000000000000000000000000X  // TBD
```

The precompile address will be assigned before activation. Does not affect wire format or spec semantics.

### 1.5 Relationship to ValidatorConfig

The SubblockRegistry is a **separate precompile** from ValidatorConfig. It reads the current validator set to determine which delegations contribute to gas weight:

```
┌─────────────────────┐  reads   ┌─────────────────────┐
│   ValidatorConfig   │◄─────────│   SubblockRegistry  │
│                     │          │                     │
│ (intended validator │          │ - delegatedBuilder  │
│       set)          │          │ - p2pKeys           │
└─────────────────────┘          └─────────────────────┘
     Intended validator            Builder delegation
     set (per-block)               and p2p key config
```

- ValidatorConfig defines the **intended validator set** — the set of validators that should be active. Gas weight is computed from this set because it is readable per-block from precompile state.
- The actual active consensus committee is determined by the DKG outcome (stored in boundary block headers). On rare DKG failures, these may briefly diverge; they reconverge on the next successful DKG.
- SubblockRegistry tracks **builder delegations and p2p keys** (separate from consensus)
- No callback from ValidatorConfig is needed; gas weight is computed on the fly

### 1.6 P2P Peer Set

Builder p2p keys whose owning builder has gas weight > 0 are included in the peer set alongside validators. The node reads active builder p2p keys from SubblockRegistry and adds them to the commonware p2p oracle, allowing builder machines to join the authenticated network.

**Key distinction:**
- Builder p2p keys (ed25519) are authorized for **subblock delivery only**, not consensus participation
- P2P peer set updates occur **per-block**, on a best-effort basis

## 2. Transaction Format

### 2.1 Any Transaction Format

Subblock transactions are **not restricted** to a specific nonce key format. Any valid Tempo transaction may appear in a subblock, including:
- Standard Ethereum transactions (EIP-1559, EIP-2930, legacy)
- Tempo AA transactions with keychain (access key) signatures
- Transactions using the `0x5b` nonce key for subblock nonce isolation

Fee routing and builder identification are determined by the SubBlockMetadata entry, not by the transaction's nonce key.

### 2.2 Optional `0x5b` Nonce Key for Nonce Isolation

The `0x5b` nonce key format is available as an **optional** mechanism for nonce isolation. When a user wants to prevent nonce conflicts between their main-block and subblock transactions, they can use:

```
32 bytes total:
┌──────────────────┬──────────────────────────────────┬──────────────────────────┐
│ 0x5b (1 byte)    │ builder_address (20 bytes)        │ nonce_remainder (11 bytes)│
├──────────────────┼──────────────────────────────────┼──────────────────────────┤
│ byte 0 (MSB)     │ bytes 1-20                       │ bytes 21-31 (LSB)        │
└──────────────────┴──────────────────────────────────┴──────────────────────────┘
```

- **Byte 0 (MSB)**: `0x5b` prefix identifying a subblock-isolated nonce lane
- **Bytes 1-20**: Builder's 20-byte address
- **Bytes 21-31**: 11-byte nonce remainder (2^88 possible values)

This format gives the sender a separate nonce lane per `(sender, nonceKey)` pair, preventing conflicts with protocol-nonce transactions in the main block.

### 2.3 Nonce Key Construction

```
nonceKey = (0x5b << 248) | (builder_address << 88) | nonce_remainder
```

### 2.4 Nonce Key Helpers

```solidity
function isSubblockIsolatedNonce(uint256 nonceKey) pure returns (bool) {
    return uint8(nonceKey >> 248) == 0x5b;
}

function extractBuilderFromNonceKey(uint256 nonceKey) pure returns (address) {
    return address(uint160(nonceKey >> 88));
}

function extractNonceRemainder(uint256 nonceKey) pure returns (uint88) {
    return uint88(nonceKey);
}
```

### 2.5 V1 vs V2 Nonce Key Discrimination

Both V1 and V2 use the `0x5b` prefix when nonce isolation is used. The version is determined by **fork activation height**, not by the nonce key format:

| Version | Bytes 1-15 | Bytes 16-20 | Bytes 21-31 |
|---------|------------|-------------|-------------|
| V1 | PartialValidatorKey (15 bytes) | nonce high bytes | nonce low bytes |
| V2 | builder_address[0..15] | builder_address[15..20] | nonce_remainder (11 bytes) |

Before activation height: V1 interpretation. After activation height: V2 interpretation.

### 2.6 Nonce Conflict Risk for Protocol-Nonce Transactions

Transactions using protocol nonces (no `0x5b` key) in subblocks risk nonce conflicts with the main block:
- If the main block consumed the same nonce, the proposer drops the transaction with `NonceTooLow`
- The builder bears the risk of wasted bundle space
- Builders are incentivized to prefer `0x5b` nonces or expiring nonces for high-value transactions
- The drop mechanism handles conflicts gracefully — no block invalidity

## 3. Expiring Nonces in Subblocks

### 3.1 Motivation

TIP-1009 introduced expiring nonces for parallel transaction submission without sequential ordering. This TIP extends that support to subblock transactions using the `0x5b` nonce key.

### 3.2 Detection

A subblock transaction uses expiring nonce semantics if:
1. The nonce key has `0x5b` prefix (byte 0)
2. The nonce remainder (bytes 21-31) is all 1s: `0xFFFFFFFFFFFFFFFFFFFFFF` (max uint88, 11 bytes)

```solidity
uint88 constant EXPIRING_NONCE_MARKER = type(uint88).max;  // 0xFFFFFFFFFFFFFFFFFFFFFF

function isSubblockExpiringNonce(uint256 nonceKey) pure returns (bool) {
    return isSubblockIsolatedNonce(nonceKey) && extractNonceRemainder(nonceKey) == EXPIRING_NONCE_MARKER;
}
```

### 3.3 Nonce Key Format (Expiring)

```
32 bytes total:
┌──────────────────┬──────────────────────────────────┬──────────────────────────┐
│ 0x5b (1 byte)    │ builder_address (20 bytes)        │ 0xFFF...FF (11 bytes)    │
├──────────────────┼──────────────────────────────────┼──────────────────────────┤
│ byte 0 (MSB)     │ bytes 1-20                       │ bytes 21-31 = max        │
└──────────────────┴──────────────────────────────────┴──────────────────────────┘
```

### 3.4 Replay Protection

Subblock expiring nonce transactions follow TIP-1009 semantics:
- `validBefore` field is required
- `nonce` field must be `0`
- Replay protection via transaction hash + **same circular buffer** as TIP-1009
- Same validity window (30 seconds max)

The only difference from standard expiring nonces is the nonce key format, which provides nonce isolation. The replay protection shares the same `expiringNonceSeen` and `expiringNonceRing` storage as regular expiring nonces — no separate buffer is needed since the transaction hash is globally unique.

### 3.5 Nonce Key Construction (Expiring)

```
expiringNonceKey = (0x5b << 248) | (builder_address << 88) | EXPIRING_NONCE_MARKER
```

## 4. Signed Subblock Structure

### 4.1 SubBlock

The builder constructs a SubBlock containing **all proposed transactions**, including those that may be dropped or left unprocessed by the proposer:

```rust
struct SubBlock {
    version: SubBlockVersion,    // V2 for this spec
    parent_hash: B256,           // Block hash this subblock builds on
    builder: Address,            // Builder's address (= fee recipient)
    chain_id: u64,               // Chain ID for cross-chain replay protection
    transactions: Vec<Transaction>,  // ALL proposed transactions (any format)
}
```

The `transactions` list is the builder's complete proposal. Transactions may be of any valid Tempo transaction type — they are not required to use the `0x5b` nonce key or root-key-only signatures.

### 4.2 SignedSubBlock

```rust
struct SignedSubBlock {
    subblock: SubBlock,
    signer: B256,      // ed25519 public key that signed this subblock
    signature: Bytes,  // ed25519 signature
}
```

The `signer` is one of the builder's registered p2p keys. Verification:
1. `ed25519_verify(signer, signature_hash, signature)` — valid signature
2. `signer` is in `p2pKeys[subblock.builder]` — signer is registered to this builder

### 4.3 Signature Computation

```
signature_hash = keccak256(0x78 || RLP(subblock))
signature = ed25519_sign(p2p_private_key, signature_hash)
```

The builder signs over the **entire** SubBlock, including all proposed transactions. This commits the builder to the exact ordered set of transactions they proposed.

### 4.4 Signature Verification

```
hash = keccak256(0x78 || RLP(subblock))
valid = ed25519_verify(signed.signer, hash, signed.signature)
       && signed.signer in p2pKeys[signed.subblock.builder]
```

### 4.5 Version

This TIP introduces `SubBlockVersion::V2`. Nodes MUST reject V1 subblocks after activation.

### 4.6 Bundle Size Limits

```
MAX_TX_COUNT_PER_BUNDLE = 1024
```

Builders MUST NOT propose subblocks with more than `MAX_TX_COUNT_PER_BUNDLE` transactions. Proposers MUST reject bundles exceeding this limit.

## 5. Subblock Metadata & Drop Mechanism

### 5.1 SubBlockMetadata

The proposer includes metadata about processed subblocks in a system transaction. This system transaction contains: the builder's signature for verification, the layout of executed transactions in the block body, and the **raw bytes** of all dropped and unprocessed transactions so validators can reconstruct the original signed SubBlock.

```rust
struct SubBlockMetadataEntry {
    version: SubBlockVersion,   // V2
    builder: Address,           // Builder's address (= fee recipient)
    signer: B256,               // ed25519 public key that signed the subblock
    signature: Bytes,           // ed25519 signature over the original SubBlock

    // Original bundle info
    total_proposed: u32,        // Total tx count in the original signed SubBlock
    processed_count: u32,       // How many bundle indices the proposer processed (from 0)

    // Executed transactions (in block body, contiguous)
    tx_start_index: u32,        // Index of first executed tx in the block body
    executed_count: u32,        // Number of executed transactions in the block body

    // Dropped transactions (precheck-invalid, raw bytes in this system tx)
    drops: Vec<DropEntry>,

    // Unprocessed transactions (beyond processed_count, raw bytes in this system tx)
    unprocessed_txs: Vec<Bytes>,
}

struct DropEntry {
    bundle_index: u32,          // Index in the original SubBlock's transactions list
    tx_bytes: Bytes,            // Raw transaction bytes
    reason: DropReason,         // Why the proposer dropped this transaction
}

enum DropReason {
    NonceTooLow,
    NonceTooHigh,
    InsufficientBalance,
    FeeCapBelowBaseFee,
    IntrinsicGasTooLow,
    AccessKeyRevoked,
    InvalidSignature,
    DecodingError,
}

struct SubBlockMetadata {
    entries: Vec<SubBlockMetadataEntry>,
}
```

**Key invariant**: `executed_count + drops.len() == processed_count` — every processed bundle index is either executed or dropped. And `unprocessed_txs.len() == total_proposed - processed_count` — every index beyond the processing point has its raw bytes in the system tx.

The `builder` field is kept explicitly in metadata (rather than derived from `signer`) so that verification does not require a reverse lookup.

### 5.2 SubBlock Reconstruction

Validators reconstruct the original SubBlock from on-chain data to verify the builder's signature:

```
Given a SubBlockMetadataEntry:

1. Create ordered list of all transactions [0..total_proposed):
   - Let exec_cursor = 0  (indexes into block body starting at tx_start_index)
   - Let drop_cursor = 0  (indexes into drops list)
   - For bundle_index in 0..processed_count:
     a. If drop_cursor < drops.len() AND drops[drop_cursor].bundle_index == bundle_index:
        → transactions[bundle_index] = decode(drops[drop_cursor].tx_bytes)
        → drop_cursor += 1
     b. Else:
        → transactions[bundle_index] = block_body[tx_start_index + exec_cursor]
        → exec_cursor += 1
   - For bundle_index in processed_count..total_proposed:
     → transactions[bundle_index] = decode(unprocessed_txs[bundle_index - processed_count])

2. Reconstruct SubBlock:
   - version = entry.version
   - parent_hash = block header's parent hash
   - builder = entry.builder
   - chain_id = chain's ID
   - transactions = ordered list from step 1

3. Verify signature:
   - hash = keccak256(0x78 || RLP(reconstructed_subblock))
   - ed25519_verify(entry.signer, hash, entry.signature)
   - entry.signer is in p2pKeys[entry.builder] at parent post-state
```

### 5.3 Drop Justification (Precheck)

A transaction may be dropped **only** if it fails precheck — stateless or cheap stateful validation that does not require EVM execution. All precheck evaluations use `S_post_main` (post-main-block state) as the starting point, evolved by prior executed subblock transactions.

| Check | Condition | DropReason |
|-------|-----------|------------|
| Decoding | Cannot decode raw bytes into valid transaction | `DecodingError` |
| Signature | Cannot recover valid sender / invalid signature | `InvalidSignature` |
| Intrinsic gas | `tx.gas_limit < intrinsic_gas(tx)` | `IntrinsicGasTooLow` |
| Nonce too low | `tx.nonce < expected_nonce[sender]` | `NonceTooLow` |
| Nonce too high | `tx.nonce > expected_nonce[sender]` | `NonceTooHigh` |
| Fee cap | `tx.max_fee_per_gas < block.base_fee` | `FeeCapBelowBaseFee` |
| Balance | `balance(sender) < value + gas_limit * max_fee_per_gas` | `InsufficientBalance` |
| Access key | Keychain key used but revoked in `S_post_main` | `AccessKeyRevoked` |

**Execution reverts are NOT droppable.** A transaction that passes precheck but reverts during execution MUST be included and charged gas normally. This prevents the proposer from using drops as a censorship mechanism.

### 5.4 Deterministic Processing Order

The proposer processes bundle transactions **in bundle order** (index 0, 1, 2, ...). This order is deterministic and must be replayed identically by validators.

```
expected_nonce: map[sender → nonce], initialized from S_post_main
cumulative_gas = 0

for i in 0..processed_count:
    tx = bundle.transactions[i]

    if Precheck(tx, current_state, expected_nonce) fails:
        record Drop(i, tx_bytes, reason)
        // Do NOT update expected_nonce or state
    else:
        execute(tx)
        update current_state
        expected_nonce[tx.sender] = next_nonce(tx)
        cumulative_gas += tx.gas_used

// Indices processed_count..total_proposed are unprocessed (overflow)
// Their raw bytes are placed in the system tx
```

### 5.5 Nonce Cascade Handling

The deterministic processing order makes nonce cascades straightforward:

- Only **executed** transactions advance `expected_nonce[sender]`
- **Dropped** transactions do NOT modify `expected_nonce`
- Each transaction is prechecked against the `expected_nonce` at the point it is processed in bundle order
- This is fully deterministic: every validator replaying the processing arrives at the same result

Example cascade:
```
Bundle from sender Alice (on-chain nonce = 5 after main block):
  [0] Alice nonce=4  → Drop: NonceTooLow  (expected_nonce stays 5)
  [1] Alice nonce=5  → Execute            (expected_nonce becomes 6)
  [2] Alice nonce=5  → Drop: NonceTooLow  (expected_nonce stays 6)
  [3] Alice nonce=6  → Execute            (expected_nonce becomes 7)
```

### 5.6 Processing Obligation

The proposer MUST process enough of the bundle to fill the builder's gas floor with valid transactions:

- The proposer MUST set `processed_count` high enough that either:
  a. `cumulative_gas_used >= gas_floor` (floor is filled), or
  b. `processed_count == total_proposed` (entire bundle was processed)
- Within the processed range, the proposer MUST execute every transaction that passes precheck (no skipping valid transactions)
- Beyond the gas floor, the proposer MAY stop processing at any point (set `processed_count` and truncate)

Validators verify this obligation: if `cumulative_gas_used < gas_floor` and `processed_count < total_proposed`, the block is invalid.

### 5.7 Metadata Validation

During block execution, the SubBlockMetadata system transaction is processed:

1. For each entry (all lookups at parent post-state):
   a. Verify `gas_weight(entry.builder) > 0`
   b. Verify `entry.signer` is in `p2pKeys[entry.builder]`
   c. Verify `executed_count + drops.len() == processed_count`
   d. Verify `unprocessed_txs.len() == total_proposed - processed_count`
   e. Verify drop entries are ordered by `bundle_index` and within `[0, processed_count)`
   f. Reconstruct the original SubBlock (Section 5.2)
   g. Verify ed25519 signature over the reconstructed SubBlock
   h. Replay deterministic processing (Section 5.4):
      - For each executed tx: verify it passes precheck at the processing point
      - For each drop: verify it fails precheck with the claimed `DropReason`
   i. Verify processing obligation (Section 5.6)

2. Verify metadata entry executed ranges are non-overlapping, ordered, and within block bounds
3. At most one metadata entry per builder address
4. A SubBlockMetadata system transaction MUST be present (and last) whenever any subblock metadata entries exist, including entries with `executed_count == 0`. The trigger is "subblocks were included," not "transactions were executed."
5. Empty subblocks (`executed_count == 0`, all transactions dropped or unprocessed) are valid. They carry a signature proving the builder was active, and including them unlocks the builder's gas budget for the incentive lane (see Section 9.3).

### 5.8 Bandwidth Griefing Mitigation

Dropped and unprocessed transactions consume block space in the system transaction without paying gas. To prevent abuse:

**Hard caps per bundle:**

| Limit | Value | Rationale |
|-------|-------|-----------|
| `MAX_TX_COUNT_PER_BUNDLE` | 1024 | Bounds total transactions (and unprocessed bytes) |
| `MAX_DROPS_PER_BUNDLE` | 32 | Bounds dropped tx data in system tx |
| `MAX_DROP_BYTES_PER_BUNDLE` | 128 KB | Bounds total dropped tx bytes |

If a bundle exceeds `MAX_DROPS_PER_BUNDLE` or `MAX_DROP_BYTES_PER_BUNDLE` during processing, the proposer MAY reject the entire bundle. If the proposer includes it anyway, the block is invalid.

**Proposer discretion:**
- The proposer is never obligated to include a bundle with many drops
- A rational proposer will prefer bundles with fewer drops (less system tx overhead, more fee revenue)
- Builders are incentivized to submit clean bundles because dirty bundles risk rejection

### 5.9 Gas Budget Enforcement

Each builder's gas floor is computed from their accumulated delegations:

```
per_validator_gas = (block_gas_limit / TEMPO_SHARED_GAS_DIVISOR) / num_validators
gas_floor = per_validator_gas × gas_weight(builder)
```

During validation:
1. Compute gas weight for the builder (count of validators in the set with `delegatedBuilder == builder`, at parent post-state)
2. Compute builder's gas floor as `per_validator_gas × gas_weight`
3. Verify the processing obligation (Section 5.6): proposer processed enough to fill the floor
4. Executed transactions up to the gas floor consume the builder's gas budget
5. Executed transactions beyond the gas floor consume overflow capacity (see Section 9)

By including a builder's subblock, the proposer unlocks the builder's **entire** gas budget for the gas incentive lane (see Section 9.3), regardless of how much of it the subblock actually uses.

## 6. Fee Routing

### 6.1 Metadata-Based Fee Attribution

Fee routing is determined by the SubBlockMetadata entry, **not** by the transaction's nonce key. Any transaction within a builder's executed range (`tx_start_index` to `tx_start_index + executed_count`) has its fees routed to that builder's address.

During execution of each subblock transaction:
1. Set `block.beneficiary = entry.builder` (temporarily)
2. Execute the transaction
3. Fees are routed via FeeManager using `validatorTokens[entry.builder]`
4. Restore `block.beneficiary` after the transaction

This means:
- Standard EIP-1559 transactions can be in subblocks and fees route to the builder
- No `0x5b` nonce key required for fee routing
- The same FeeManager architecture applies

### 6.2 Fee Token Resolution

The user's fee token is resolved using standard Tempo resolution:
1. Transaction's explicit `fee_token` field (if set)
2. User's default fee token preference (from user config)
3. Inference from transaction context
4. PathUSD (default)

### 6.3 Fee Token Swap

The builder's preferred fee token is read from `FeeManager.validatorTokens[builder]`. If the user's fee token differs from the builder's preference:
- If sufficient FeeAMM liquidity exists, swap to the builder's preferred token using the same swap mechanism as proposer fee swaps
- If insufficient liquidity, the builder receives the user's fee token directly

The transaction is **never invalidated** due to missing AMM liquidity. If the builder has not set a fee token preference, fees are routed in PathUSD (the FeeManager default).

### 6.4 Overflow Fee Split

Transactions are categorized relative to the builder's gas floor:

| Zone | Gas range | Fee recipient |
|------|-----------|---------------|
| Floor | `0` to `gas_floor` | Builder (via `block.beneficiary = builder`) |
| Overflow | `> gas_floor` | Proposer (via `block.beneficiary = proposer`) |

The gas floor represents the builder's earned gas budget from delegated validators. The builder receives all fees for this capacity. Overflow transactions are a bonus — the proposer includes them voluntarily and keeps the fees as incentive to include overflow.

### 6.5 Fee Failure

If a transaction passes precheck but cannot pay fees at execution time (e.g., balance changed during subblock execution by a prior subblock tx), the transaction becomes a **noop** but the **nonce is still incremented**. The block remains valid.

Note: `InsufficientBalance` drops (Section 5.3) catch the case where the sender clearly cannot afford fees at precheck time against `S_post_main`. Fee failure at execution handles edge cases where balance changes between precheck and execution within the same subblock.

### 6.6 Transaction Failure Semantics

The following failures **invalidate the block** (consensus-critical):
- Malformed transaction (invalid encoding, missing fields)
- Invalid transaction signature (for executed transactions only — drops with `InvalidSignature` are valid)
- Invalid nonce for executed transactions
- SubBlockMetadata validation failures (see Section 5.7)
- Drop justification failures (a drop that doesn't actually fail precheck)
- Processing obligation violations (Section 5.6)

The only per-transaction failure that does **not** invalidate the block:
- **Fee payment failure**: transaction becomes a noop, nonce is still incremented, block remains valid

All other transaction execution outcomes (revert, out-of-gas) follow standard Tempo execution semantics.

## 7. Transaction Routing

### 7.1 Builder Endpoint

Subblock builders MUST expose an RPC endpoint for receiving transactions from users. Users submit transactions to the builder they want to route through. Builders SHOULD authenticate incoming requests (e.g., verify sender signature before full transaction decode) to mitigate DDoS.

Users may submit any transaction format — standard EIP-1559, Tempo AA with keychain signatures, or `0x5b` nonce-key transactions for nonce isolation.

### 7.2 Subblock Delivery via P2P

Builder machines join the authenticated commonware p2p network using their ed25519 p2p keys. They send signed subblocks directly to the next proposer via p2p, the same mechanism used in V1.

The proposer stores received subblocks and selects them for inclusion during block building.

### 7.3 Builder P2P Authorization

Builder p2p keys whose owning builder has gas weight > 0 are added to the per-block peer set. When a builder's gas weight drops to zero (all validators undelegated or removed from the set), their p2p keys are removed from the authorized set.

Builder p2p keys are authorized for **subblock delivery only**, not consensus participation.

### 7.4 High Availability

This design supports high availability without shared key material: multiple physical machines can operate under a single builder address, each with its own registered ed25519 key. Each machine independently builds and signs subblocks; the proposer deduplicates by selecting the best one (see Section 10.3).

## 8. End-to-End Flow

### 8.1 Setup (Once per Builder)

```
1. Builder selects an address to receive fees
2. Builder calls FeeManager.setValidatorToken(preferredToken) to set fee token preference
3. For each physical machine:
   a. Generate an ed25519 keypair for p2p identity and subblock signing
   b. Builder calls SubblockRegistry.registerP2pKey(p2pPubkey, signature) from the builder address
   c. Start commonware p2p node with the machine's ed25519 key
```

### 8.2 Setup (Once per Validator)

```
1. Validator selects a builder to delegate to (by address)
2. Validator calls SubblockRegistry.delegateToBuilder(builderAddress)
```

### 8.3 Transaction Creation (User)

```
1. User selects a builder to route their transaction through
2. User constructs transaction:
   - Standard format (EIP-1559, etc.) with protocol nonce, OR
   - 0x5b nonce key for nonce isolation: 0x5b | builder_address | nonce_remainder, OR
   - 0x5b expiring nonce: 0x5b | builder_address | 0xFFF...FF (max uint88)
3. User signs transaction with any valid key (root EOA or keychain access key)
4. User submits transaction to builder's RPC endpoint
```

### 8.4 Subblock Building (Builder Machine)

```
1. Builder machine collects transactions from users
2. Builder machine validates each transaction (basic checks)
3. Builder machine orders transactions, respecting gas floor budget
   - Transactions up to gas_floor are the "floor" (guaranteed inclusion)
   - Transactions beyond gas_floor are "overflow" (best-effort)
4. Builder machine creates SubBlock with:
   - parent_hash = current chain tip
   - chain_id = chain's ID
   - builder = builder's address
   - transactions = ALL proposed transactions (floor + overflow)
5. Builder machine signs with its own ed25519 key → SignedSubBlock
6. Builder machine sends to next proposer via commonware p2p
```

### 8.5 Block Building (Proposer)

```
1. Proposer receives SignedSubBlocks from builder machines via p2p
2. For each subblock (using parent post-state for identity lookups):
   a. Verify ed25519 signature against signed.signer
   b. Verify signer is in p2pKeys[subblock.builder]
   c. Verify chain_id matches
   d. Verify parent_hash matches current tip
   e. Verify gas_weight(builder) > 0
   f. Verify total_proposed <= MAX_TX_COUNT_PER_BUNDLE
3. Select valid subblocks: at most one subblock per builder address
   (see Section 10.3 for conflict resolution when multiple are received)
4. Execute main block transactions → S_post_main
5. For each selected subblock, process in bundle order:
   a. For each tx index 0..N:
      - Run Precheck against current state
      - If precheck fails: record Drop(index, tx_bytes, reason)
      - If precheck passes and within gas floor: execute
      - If precheck passes and beyond gas floor: execute (overflow) or truncate
   b. Record processed_count (where processing stopped)
   c. Collect unprocessed tx bytes (indices beyond processed_count)
6. Build SubBlockMetadata with entries for each builder:
   - Identity: builder, signer, signature
   - Bundle info: total_proposed, processed_count
   - Execution: tx_start_index, executed_count
   - Drops: list of DropEntry
   - Unprocessed: list of raw tx bytes
7. Append SubBlockMetadata system transaction at end of block (MUST be last)
```

### 8.6 Block Execution (All Nodes)

```
1. Execute main block transactions → S_post_main
2. Execute subblock transactions in block body order:
   a. For each transaction in a builder's executed range:
      - Set block.beneficiary = builder_address (temporarily)
      - Execute the transaction
      - If within gas floor: fees route to builder via FeeManager
      - If overflow: fees route to proposer
      - If fee payment fails: tx becomes noop, nonce still incremented
      - Restore block.beneficiary
3. Process SubBlockMetadata system transaction (last tx in block):
   a. For each entry:
      - Reconstruct original SubBlock from block body + drops + unprocessed
      - Verify ed25519 signature over reconstructed SubBlock
      - Verify signer is in p2pKeys[builder] at parent post-state
      - Replay deterministic precheck for each drop, verify failure with claimed reason
      - Verify processing obligation (gas floor filled or bundle exhausted)
   b. Verify entry ranges are non-overlapping, ordered, within bounds
   c. Verify at most one entry per builder
   d. If any verification fails, block is invalid
```

## 9. Gas Accounting

### 9.1 Per-Validator Gas Budget

Gas budgets are computed per-validator:

```
shared_gas_limit = block_gas_limit / TEMPO_SHARED_GAS_DIVISOR
per_validator_gas = shared_gas_limit / num_validators
```

### 9.2 Builder Gas Floor

A builder's gas floor is the sum of per-validator gas for all validators in the set delegating to them:

```
gas_floor = per_validator_gas × gas_weight(builder)
```

Gas weight is computed on the fly from the current validator set and `delegatedBuilder` mappings. When a validator leaves the set, their delegation is no longer counted — no cleanup is needed.

### 9.3 Gas Incentive Lane

Unused gas from subblocks becomes available for the gas incentive lane:

```
incentive_gas = Σ (gas_floor(builder) - min(subblock_gas_used, gas_floor(builder)))
```

The gas incentive lane is **only available to the proposer if they include subblocks**. This creates a direct incentive for proposers to include subblocks: if a proposer omits subblocks, they forfeit access to the incentive gas pool.

By including a builder's subblock, the proposer unlocks the builder's **entire** gas floor for the incentive lane, not just the portion used by the subblock.

### 9.4 Overflow Gas

Overflow transactions (beyond the gas floor) are **not** deducted from the builder's gas budget and do **not** contribute to the incentive lane calculation. Overflow gas counts against a separate allocation determined by the proposer's remaining block gas capacity.

## 10. Subblock Lifecycle

### 10.1 Subblock Validity

A signed subblock is valid for inclusion if (all lookups at parent post-state):
- `parent_hash` matches the current chain tip
- `chain_id` matches the chain's ID
- The ed25519 signature verifies against `signed.signer`
- `signed.signer` is in `p2pKeys[subblock.builder]` (signer is registered to this builder)
- `gas_weight(builder) > 0` (at least one validator in the set delegates to this builder)
- `subblock.transactions.len() <= MAX_TX_COUNT_PER_BUNDLE`

Note: individual transaction validity is NOT checked at subblock acceptance time. Invalid transactions are handled by the drop mechanism during processing.

### 10.2 Subblock Expiry

Subblocks expire immediately when `parent_hash` becomes stale. Subblocks have a 1-block lifetime.

### 10.3 Conflicting Subblocks

If a builder submits multiple subblocks for the same `parent_hash` (e.g., from different machines), the proposer selects the one with the higher total `gas_limit` sum (static check, not gas used). Ties are broken by selecting the subblock with the **lower `signature_hash`** (lexicographic comparison of `keccak256(0x78 || RLP(subblock))`).

### 10.4 Cross-Builder Independence

Subblocks from different builders are processed independently. However, cross-builder state interactions can occur when one builder's executed transactions change state that another builder's transactions depend on (e.g., balance changes). The proposer determines the order of builder processing.

### 10.5 Deduplication

Nodes deduplicate received subblocks by `(builder, parent_hash)`. Only the highest-`gas_limit`-sum subblock is retained per builder per parent.

## Must Hold

| ID | Invariant |
|----|-----------|
| **S1** | Gas weight correctness: Gas weight for a builder equals the count of validators in the set with `delegatedBuilder == builder`, computed on the fly |

| **S7** | Range validity: Metadata executed ranges are non-overlapping, ordered, and within block bounds |
| **S8** | Fee routing: Fees for floor-zone txs are sent to builder; fees for overflow txs are sent to proposer |
| **S9** | One subblock per builder: At most one metadata entry per builder address per block |
| **S10** | State snapshot: All registry lookups (delegations, p2p keys) use parent post-state; precheck uses `S_post_main` |
| **S11** | P2P key cap: Each builder can register at most 4 p2p keys |
| **S12** | P2P key uniqueness: Each ed25519 p2p pubkey is registered to at most one builder |
| **S13** | Metadata required: A SubBlockMetadata system transaction must be present (and last) whenever any subblock metadata entries exist |
| **S14** | Key ownership proof: `registerP2pKey` requires ed25519 signature proving control of the private key |

| **S16** | Bundle size cap: Each bundle has at most `MAX_TX_COUNT_PER_BUNDLE` transactions |
| **S17** | Deterministic processing: Bundle transactions are processed in index order; validators replaying the processing arrive at identical results |

# Design Rationale

## Bundle Proposal Model

The bundle proposal model treats the builder's subblock as a **proposed ordered list** rather than a hard commitment. The proposer can drop precheck-invalid transactions and truncate overflow, recording these actions in the system transaction for validator verification.

This design was chosen over the "include-or-reject-entire-subblock" model (V1) because:
- It allows any transaction format (not just `0x5b` nonce key)
- It handles race conditions between main block and subblock execution (nonce conflicts, key revocations, balance changes)
- It enables overflow for higher throughput
- It relaxes the root-key-only signing restriction

The key security property: **execution reverts are not droppable**. Only precheck failures (cheap, deterministic, no EVM execution) justify drops. This prevents the proposer from censoring transactions by claiming they would revert.

## Dropped Transactions in the System Transaction

Dropped and unprocessed transaction bytes are placed in the SubBlockMetadata system transaction so that validators can:
1. Reconstruct the original SubBlock (including dropped/unprocessed txs)
2. Verify the builder's ed25519 signature over the complete original SubBlock
3. Replay precheck for each drop to confirm it was justified

This ensures the builder's signature commits to the full set of proposed transactions, and validators can independently verify that the proposer's drops were honest.

## ed25519 for Both P2P and Subblock Signing

Each builder machine has its own ed25519 key that serves dual purpose:
- **P2P authentication**: The key is used for commonware p2p network authentication, allowing the machine to connect to the network and deliver subblocks.
- **Subblock signing**: The same key signs subblocks. The SubBlockMetadata includes the `signer` pubkey, which nodes verify against `p2pKeys[builder]`.

This eliminates the need for shared key material (KMS/HSM) across machines. Each machine operates independently with its own key. The on-chain `p2pKeys` mapping is the source of truth for which keys are authorized to sign on behalf of a builder.

## Builder Address as Fee Recipient

The builder address directly receives subblock fees (within the gas floor zone). This eliminates the need for a separate `feeRecipient` field and the `BuilderConfig` struct. A builder's on-chain state is just: which p2p keys are registered to it, and which validators delegate to it. Fee token preference is stored in the existing FeeManager.

## Permissionless Delegation

`delegateToBuilder` does not restrict callers to validators. Any address can set a delegation, but only delegations from addresses that are currently validators in the set contribute to gas weight. This simplifies the interface — no membership check is needed at delegation time — and supports pre-registration before a validator joins the set.

## Relaxed Signing via Drop Mechanism

The V1/V2 root-key-only requirement existed because of a key revocation race condition:

1. User signs subblock tx with access key, submits to builder
2. Main block tx revokes that access key
3. Subblock tx executes after main block — revoked key used

The bundle proposal model solves this with `AccessKeyRevoked` drops:
- The proposer checks keychain validity against `S_post_main` (post-main-block state)
- If a key was revoked in the main block, the proposer drops the transaction
- Validators verify the drop by checking keychain state at `S_post_main`
- User expectation is preserved: revoking a key in the main block prevents its use in subblocks of the same block

Intra-subblock revocation cannot occur because keychain management functions are not callable from subblock context.

## Overflow Fee Split

The clean boundary at `gas_floor` simplifies incentive alignment:
- **Floor zone**: Builder receives all fees (this is their earned capacity from delegated validators)
- **Overflow zone**: Proposer receives all fees (incentive to include extra transactions)

A percentage split (e.g., 80/20) was considered but adds complexity. The proposer's incentive to include overflow comes entirely from keeping the overflow fees.

Key changes from V1:
1. **Bundle proposal model**: Builder proposes all txs; proposer drops/truncates; system tx carries dropped bytes
2. **Any transaction format**: No `0x5b` nonce key requirement; metadata-based fee routing
3. **Keychain signatures allowed**: Drop mechanism handles key revocation races
4. **Overflow support**: Builders can propose beyond gas floor; proposer includes extras on best-effort basis
5. **Separate builder identity**: Builder address is independent from validator identity
6. **Many-to-one delegation**: Validators can share a builder, accumulating gas budgets
7. **Builder address = fee recipient**: No separate fee recipient configuration
8. **Per-machine signing**: Each machine signs with its own ed25519 key, no shared key needed
9. **Builder p2p keys**: Per-machine ed25519 keys for p2p and signing, registered under a builder (max 4)
10. **Expiring nonce support**: Subblock txs can use expiring nonces with `0x5b` nonce key
11. **chain_id field**: SubBlock includes chain_id for cross-chain replay protection

# Migration

## Activation

This TIP activates at a specific block height. After activation:

- SubBlockVersion::V1 subblocks are rejected
- The V1 nonce key format (`0x5b | partial_validator_key[15] | nonce[16]`) is no longer valid for nonce isolation
- The V2 nonce key format (`0x5b | builder_address[20] | nonce_remainder[11]`) is available for nonce isolation (optional)
- Any transaction format may appear in subblocks
- Validators must delegate to a builder before their subblock gas is usable
- SubBlockMetadata system transaction uses the new format with drops and unprocessed entries

## Migration Steps

1. **Pre-activation**: Builders call `FeeManager.setValidatorToken(token)` for fee token preference
2. **Pre-activation**: Builders call `registerP2pKey(p2pPubkey, signature)` for each physical machine
3. **Pre-activation**: Validators call `delegateToBuilder(builderAddress)` to delegate their gas budget
4. **Activation block**: Protocol switches to V2 subblock validation with bundle proposal model
5. **Post-activation**: V1 subblocks and old nonce key formats are rejected

Validators who want to build their own subblocks should register a p2p key and self-delegate:
```solidity
// From the builder address (can be same as validator address):
FeeManager.setValidatorToken(preferredToken);
SubblockRegistry.registerP2pKey(myP2pPubkey, mySignature);

// From the validator's address:
SubblockRegistry.delegateToBuilder(builderAddress);
```

# Open Questions

## O.1 Builder Ordering Across Bundles

The proposer chooses the order of builders' bundles within the subblock zone. This order affects cross-builder state interactions (e.g., one builder's executed tx changes a balance that another builder's tx depends on).

**Question**: Should builder ordering be deterministic (e.g., sorted by builder address) or proposer-chosen? Proposer-chosen gives MEV extraction opportunities; deterministic ordering removes this but may be suboptimal for throughput.

## O.2 Total Drop Budget Per Block

The current design caps drops per bundle (32 drops, 128 KB). With multiple builders, total drops per block could be `MAX_DROPS_PER_BUNDLE × num_builders`.

**Question**: Should there be a global cap on total drops across all builders per block?

## O.3 Builder Incentive for Clean Bundles

The current design relies on proposer discretion to penalize dirty bundles. A rational proposer rejects bundles with high drop rates.

**Question**: Is proposer discretion sufficient, or do we need on-chain enforcement (e.g., builder bonds, reputation slashing)?

## O.4 Overflow Gas Allocation

Overflow transactions count against a separate allocation outside the builder's gas budget.

**Question**: Should overflow gas count against the proposer's own gas limit, the incentive lane, or a dedicated overflow allocation?
