# TIP-1054: Non-USD User Fee Tokens

## Abstract

This TIP allows users to pay Tempo transaction fees in non-USD `TIP-20` tokens without changing signed gas fields or Tempo's attodollar-denominated gas accounting. Validators continue to receive fees through the existing USD-denominated fee-token path.

A non-USD user fee token becomes usable when the token admin configures a fee oracle that quotes the token in `pathUSD`. For fee settlement, Tempo converts the transaction's existing 6-decimal USD fee amount into FX-token units using a block-wide parent-state oracle snapshot. This version supports only direct `userToken -> validatorToken` FeeAMM pools and keeps validator-side accounting aligned with the existing `FeeManager` model.

## Motivation

Tempo already supports user-selectable fee payment in USD-denominated `TIP-20` tokens. The remaining limitation is that a user who primarily holds a non-USD asset must first trade into a USD fee token before transacting.

This TIP extends the user side only. It preserves the existing gas fields, the existing derivation of 6-decimal USD fee amounts from attodollar gas accounting, and the existing validator fee-token preferences. That keeps the signed transaction surface unchanged and avoids introducing a second fee-cap dimension in token units.

Oracle-based systems commonly expose a latest quote together with a round identifier and update timestamp. This TIP adopts that shape, but fixes both the quote asset and the scale:

- all FX fee oracles quote against `pathUSD`;
- all such quotes use the same fixed `1e18` scale; and
- all fee settlement inside a block uses a single parent-state snapshot.

That removes the largest sources of ambiguity: quote direction, mixed decimals, and same-block races between oracle updates and fee-paying transactions.

## Design summary and terms

This TIP uses the following design model:

1. **Gas accounting is unchanged.** `TIP-1000` and `TIP-1010` continue to define Tempo gas prices in attodollars. This TIP does not change base fee checks, gas fields, or the derivation of 6-decimal USD fee amounts.
2. **`pathUSD` is an oracle quote asset, not the gas unit.** FX fee oracles quote one whole FX token in `pathUSD`; accepted USD fee tokens are treated at par with `pathUSD` for this TIP.
3. **Validator settlement remains USD-only.** `setValidatorToken()` remains restricted to valid USD-denominated fee tokens. This TIP changes only the user's fee token.
4. **FX settlement is direct-route only.** Fee settlement uses only the direct FeeAMM pool `(userToken, validatorToken)`. This TIP does not introduce multi-hop routing or generalized route search.
5. **Users do not sign a token-unit fee cap.** The signed fee bound remains the existing bound implied by the transaction's gas fields. The FX-token debit is derived from the active block's oracle snapshot.
6. **Token-admin-selected oracles are trusted for fee use.** If an oracle is wrong, stale, or malicious, the token should be disabled for fee payment or handled in a follow-up TIP.
7. **All `TIP-20` tokens continue to use fixed 6-decimal units.** The conversion formulas below rely on this existing invariant.

The following terms are used throughout the specification:

- **USD6 amount** means a nominal USD amount expressed in ordinary 6-decimal `TIP-20` units.
- **`gasBalanceSpending(gas, gasPrice)`** means `ceilDiv(gas * gasPrice, GAS_PRICE_SCALE)`, where `gasPrice` is denominated in attodollars per gas and the result is a USD6 amount.
- **`maxFeeUsd6`** means `gasBalanceSpending(tx.gasLimit, tx.effectiveGasPrice(B.basefee))`, excluding any transaction value.
- **`actualFeeUsd6`** means `gasBalanceSpending(gasUsed, tx.effectiveGasPrice(B.basefee))`, excluding any transaction value.
- **USD token** means a valid USD-denominated `TIP-20` fee token under the existing Tempo rules.
- **FX token** means a non-USD `TIP-20` token.
- **0x76 Tempo transaction** means an EIP-2718 typed Tempo transaction with type byte `0x76`, represented by `TempoTransaction` / `AASigned`, whose executable calls are taken from `aa_calls`.

The following constants are used throughout the specification:

```text
ORACLE_SCALE       = 10^18
GAS_PRICE_SCALE    = 10^12
FEE_AMM_SCALE      = 10000
FEE_AMM_M          = 9970   // existing fee-swap multiplier
FEE_AMM_N          = 9985   // existing rebalance multiplier
FEE_ORACLE_GAS     = 200000
MAX_FEE_ORACLE_AGE = 3600   // seconds
```

All token amounts and USD6 amounts are expressed in Tempo's ordinary 6-decimal token units unless a field or formula explicitly says `X18`.

# Specification

## 1. Scope

This TIP applies only to **user-side** payment of transaction fees in non-USD `TIP-20` tokens.

This TIP is intentionally out of scope for:

- non-USD validator fee-token preferences;
- changes to the transaction format;
- signed token-unit fee caps;
- multi-hop or generalized FeeAMM routing;
- oracle-based repricing of USD validator tokens relative to `pathUSD`;
- standardized mempool invalidation, receipt metadata, RPC additions, or wallet UX.

For USD-denominated user fee tokens, existing Tempo semantics remain unchanged except where this TIP explicitly broadens an interface to also admit non-USD tokens.

## 2. Token Configuration and Oracle Interface

### 2.1 `TIP-20` Oracle Configuration

This TIP extends `TIP-20` token configuration with a fee-oracle pointer:

```solidity
interface ITIP20FeeOracleConfig {
    /// @notice Emitted when the token's fee oracle is updated.
    event FeeOracleUpdated(address indexed updater, address indexed oldOracle, address indexed newOracle);

    /// @notice Reverts when `oracle` is nonzero but not a valid fee-oracle contract.
    error InvalidFeeOracle();

    /// @notice Returns the token's configured fee oracle.
    /// @return oracle The configured oracle address, or address(0) if FX fee use is disabled.
    function feeOracle() external view returns (address oracle);

    /// @notice Sets the token's fee oracle.
    /// @dev `address(0)` disables FX fee use for this token.
    ///      Authorization MUST match the token's existing admin-only mutable configuration surface.
    function setFeeOracle(address oracle) external;
}
```

`setFeeOracle(oracle)` MUST:

1. allow only the token's authorized admin to update the field;
2. accept `address(0)` to disable FX fee use for the token;
3. revert with `InvalidFeeOracle` if the token is USD-denominated and `oracle != address(0)`;
4. revert with `InvalidFeeOracle` if `oracle != address(0)` and the target address has no code; and
5. emit `FeeOracleUpdated(updater, oldOracle, newOracle)`.

For USD tokens, `feeOracle()` MUST be `address(0)`.

`feeOracle()` is independent of the existing `quoteToken()` / `nextQuoteToken()` configuration. This TIP does not read the `quoteToken` graph for fee settlement, and it imposes no consistency requirement between `feeOracle()` and `quoteToken()`.

### 2.2 Oracle Contract Interface

A fee oracle MUST implement the following interface:

```solidity
interface IFeeOracle {
    /// @notice Returns the latest quote of one whole token in pathUSD.
    /// @return roundId Oracle round identifier for this quote.
    /// @return pathUsdPerTokenX18 pathUSD per whole token, scaled by 1e18.
    /// @return updatedAt Unix timestamp when this round was produced.
    function latestPathUsdQuote()
        external
        view
        returns (
            uint64 roundId,
            uint192 pathUsdPerTokenX18,
            uint64 updatedAt
        );
}
```

Query semantics and quote validity are defined in §3.

## 3. Effective Fee Snapshot

Let `B` be the block currently being executed, and let `P` be its parent block. The one-block delay is enforced by Tempo block execution clients, not by oracle contracts.

For each FX token used in block `B`, the client MUST derive a single effective fee snapshot from `state(P)`:

1. read the token's `feeOracle()` value from `state(P)`; and
2. if the oracle is nonzero, evaluate `latestPathUsdQuote()` for that oracle against `state(P)` and the parent block context.

The snapshot is cached by FX token. Clients MAY precompute it or fill it lazily on first use, but they MUST evaluate it at most once per FX token per block. All fee settlement, FX-pool mint, and FX-pool rebalance operations in `B` MUST use the cached result, including cached invalid results.

Transactions in `B` MAY update `feeOracle()` or oracle contract storage, but those writes MUST NOT affect the effective fee snapshot for `B`. They can affect fee settlement starting in the next block after `B`.

This rule requires no extra commitment in block `B`: `B` identifies `P` through `parentHash`, and `P` commits to its post-state root and block environment. Blocks do not include oracle prices, round IDs, or token-oracle mappings.

The oracle query is a protocol-internal `STATICCALL` with:

- `from`, `msg.sender`, and `tx.origin` set to `TIP_FEE_MANAGER_ADDRESS`;
- `to` set to the oracle address;
- `value = 0`;
- input equal to the selector for `latestPathUsdQuote()`;
- gas limit equal to `FEE_ORACLE_GAS`; and
- block environment equal to the parent block `P`.

The query is not charged to the user transaction, does not consume block gas, does not modify persistent or transient state, and does not affect the warm/cold access state of transaction execution. Warm/cold access accounting inside the query MUST be initialized as a fresh transaction from `TIP_FEE_MANAGER_ADDRESS` to the oracle with no access list, then discarded.

Oracle code MAY perform additional `STATICCALL`s under ordinary EVM call semantics. Those calls read `state(P)` and share the `FEE_ORACLE_GAS` budget.

A quote is **valid for block `B`** iff all of the following hold:

1. the oracle address is nonzero;
2. the oracle call succeeds and decodes correctly;
3. `roundId != 0`;
4. `pathUsdPerTokenX18 != 0`;
5. `updatedAt != 0`;
6. `updatedAt <= P.timestamp`; and
7. `B.timestamp - updatedAt <= MAX_FEE_ORACLE_AGE`.

If any condition fails, the FX token is not fee-enabled for block `B`. Any operation in this TIP that requires a valid quote MUST revert if the quote is invalid for the current block.

## 4. Fee-Token Resolution and Validity

### 4.1 User Fee Tokens

A user transaction MAY resolve to either:

- a USD token under the existing Tempo rules; or
- an FX token under this TIP.

This section uses three status terms for FX tokens:

- **fee-configured FX token in state `S`**: an FX token with a nonzero configured fee-oracle address in state `S`;
- **fee-enabled FX token for block `B`**: a fee-configured FX token in `state(P)` whose effective oracle quote is valid for block `B`, where `P` is `B`'s parent block; and
- **executable FX fee token for transaction `T` in block `B`**: a fee-enabled FX token that also satisfies the transaction-specific live checks in §4.4.

`setUserToken(token)` MUST accept:

- any valid USD token that is accepted today; and
- any valid `TIP-20` FX token with `feeOracle() != address(0)`.

`setValidatorToken(token)` is unchanged by this TIP and MUST remain USD-only.

`setUserToken(token)` is a live execution-state configuration check. For an FX token, it checks `feeOracle()` at the time the call executes, not the parent-state snapshot. It MUST NOT require the quote for the current block to be valid. Execution-time validity is defined separately in §4.4.

### 4.2 Same-Tx `setUserToken()` Resolution

This TIP preserves the existing immediate-effect rule for direct `setUserToken()` calls.

If all of the following hold:

1. the transaction does not explicitly set a `feeToken`;
2. the transaction is not a 0x76 Tempo transaction;
3. the fee payer equals the transaction caller; and
4. the first and only fee-token-resolution-relevant call is a direct call to `FeeManager.setUserToken(token)`,

then the resolved user fee token for that same transaction MUST be `token`, rather than the user's previously stored preference.

This rule applies equally when `token` is:

- a USD token accepted by existing rules; or
- a fee-configured FX token in live execution state accepted by this TIP.

The transaction MAY still fail later if the resolved FX token is not executable in the current block.

### 4.3 Fee-Token Inference

This TIP preserves automatic fee-token inference only for the narrow existing transfer-like inference surface.

Tempo MUST continue to infer a fee token from a `TIP-20` token contract only for the existing qualifying call surface:

- `transfer(...)`
- `transferWithMemo(...)`
- `distributeReward(...)`

For transactions that are not 0x76 Tempo transactions, the existing direct-call inference rule is unchanged except that the inferred token MAY now be an FX token.

For 0x76 Tempo transactions, this inference rule applies only when:

1. the fee payer equals the transaction caller; and
2. every `aa_calls` entry targets the same `TIP-20` token contract and uses one of the three selectors above.

This TIP does **not** extend FX inference to the Stablecoin DEX input-token path. Stablecoin-DEX fee-token inference remains USD-only in this version.

Inference only resolves a candidate fee token. If the inferred token is an FX token, the transaction is executable only if §4.4 also holds.

### 4.4 Execution-Time Validity

An FX token is executable as a user fee token in block `B` only if all of the following hold:

1. it is a valid `TIP-20` token;
2. it is not paused at fee-collection time;
3. it has `feeOracle() != address(0)` in `state(P)`;
4. its effective quote for block `B` is valid;
5. the fee payer is authorized to transfer that token to `FeeManager` under the existing token-policy rules;
6. the fee payer has enough balance to cover the precharged max token amount; and
7. the direct FeeAMM pool from `(userToken, validatorToken)` has sufficient validator-token liquidity for the transaction's `maxFeeUsd6`.

## 5. Conversion Functions and Rounding

For any valid effective quote `priceX18 = pathUsdPerTokenX18`, define:

```text
ceilDiv(a, b) = 0 if a == 0, otherwise floor((a - 1) / b) + 1

fxTokenAmountForUsd6(usd6Amount, priceX18)
    = ceilDiv(usd6Amount * ORACLE_SCALE, priceX18)

usd6ValueFloor(tokenAmount, priceX18)
    = floor(tokenAmount * priceX18 / ORACLE_SCALE)

usd6ValueCeil(tokenAmount, priceX18)
    = ceilDiv(tokenAmount * priceX18, ORACLE_SCALE)

validatorFeeOut(usd6Amount)
    = floor(usd6Amount * FEE_AMM_M / FEE_AMM_SCALE)

validatorRebalanceIn(usd6Amount)
    = floor(usd6Amount * FEE_AMM_N / FEE_AMM_SCALE) + 1
```

All arithmetic in this TIP's formulas is checked unsigned 256-bit integer arithmetic. Any overflow, underflow, division by zero, or pool reserve result that cannot be represented in the pool's storage type MUST revert. `ceilDiv(a, b)` is only defined for `b != 0`.

Rounding rules are part of the consensus behavior:

- user-token debits round **up**;
- USD6 valuation for user-token withdrawals in `rebalanceSwapFX` rounds **up**;
- validator-token payout in fee settlement rounds **down**; and
- validator-token input in rebalance swaps follows the existing FeeAMM rule `floor(x * N / SCALE) + 1`.

The existing USD6 fee amount derived from the transaction's signed gas fields remains the source of truth. Any extra user-token dust created by upward rounding remains in the pool or fee-manager path exactly as implied by these formulas; validator payout is determined from the USD6 fee amount, not by re-valuing the rounded user-token debit.

## 6. Pre-Tx and Post-Tx Fee Collection

For a transaction that resolves to a USD user fee token, existing semantics are unchanged.

For a transaction that resolves to an FX user fee token, fee collection MUST work as follows.

### 6.1 Pre-Tx

Let:

- `userToken` be the resolved FX fee token;
- `validatorToken` be the block beneficiary's resolved USD fee token under existing Tempo rules;
- `quote` be the effective fee snapshot for `userToken` in block `B`; and
- `maxFeeUsd6` be the transaction's existing maximum fee spending expressed as a USD6 amount.

At `collect_fee_pre_tx` time, the protocol MUST:

1. validate that `userToken` is fee-enabled for block `B`;
2. compute
   `maxUserTokenFee = fxTokenAmountForUsd6(maxFeeUsd6, quote.pathUsdPerTokenX18)`;
3. transfer `maxUserTokenFee` from the fee payer to `FeeManager` using the existing fee-precharge path;
4. compute
   `reservedValidatorOut = validatorFeeOut(maxFeeUsd6)`;
5. check that the direct pool `(userToken, validatorToken)` has at least `reservedValidatorOut` validator-token reserve available; and
6. reserve that validator-token amount for the pending transaction exactly as existing `FeeManager` reservations do.

Because this TIP allows only direct routes and uses a block-wide effective quote, no additional route identifier is required.

### 6.2 Post-Tx

Let `actualFeeUsd6` be the transaction's actual fee spending expressed as a USD6 amount under existing gas accounting.

At `collect_fee_post_tx` time, the protocol MUST:

1. recompute `actualFeeUsd6` using existing gas accounting;
2. compute
   `actualUserTokenSpend = fxTokenAmountForUsd6(actualFeeUsd6, quote.pathUsdPerTokenX18)`;
3. compute
   `refundUserToken = maxUserTokenFee - actualUserTokenSpend`;
4. refund `refundUserToken` to the fee payer through the existing fee-refund path;
5. compute
   `validatorCredit = validatorFeeOut(actualFeeUsd6)`;
6. add `actualUserTokenSpend` to the pool's `reserve_user_token`;
7. subtract `validatorCredit` from the pool's `reserve_validator_token`; and
8. increment the validator's collected-fee balance in `validatorToken` by `validatorCredit`, matching the existing `FeeManager` ledgered-credit model.

If `actualFeeUsd6 == 0`, then `actualUserTokenSpend == 0`, the full precharge is refunded, and no pool swap or validator credit occurs.

Post-tx settlement MUST NOT use any live same-block oracle read. It MUST use the same effective fee snapshot defined for block `B`.

### 6.3 FX Fee Settlement Event

This TIP extends `FeeManager` with the following event:

```solidity
event FXFeeSettled(
    address indexed feePayer,
    address indexed beneficiary,
    address indexed userToken,
    address validatorToken,
    uint64 roundId,
    uint192 pathUsdPerTokenX18,
    uint256 actualFeeUsd6
);
```

`collect_fee_post_tx` MUST emit `FXFeeSettled` for every FX fee-token transaction with nonzero `actualUserTokenSpend` or nonzero `refundUserToken`.

The event records the settlement context that existing USD fee logs do not expose in one place:

- fee payer, beneficiary, FX token, and validator token;
- oracle round and price used for settlement; and
- actual USD6 fee amount.

It intentionally omits fields that are visible from existing logs or derivable from this TIP's formulas:

- actual FX-token spend is visible in the existing `TIP-20` `Transfer(feePayer, FeeManager, actualUserTokenSpend)` log and derivable from `actualFeeUsd6` plus `pathUsdPerTokenX18`;
- refund amount is derivable from the transaction's maximum USD6 fee and the same oracle price; and
- validator credit is derivable from `actualFeeUsd6` and `validatorFeeOut`.

## 7. FX FeeAMM Pool Semantics

This TIP introduces **FX direct pools**. An FX direct pool is a directional FeeAMM pool whose key is:

```text
(userToken, validatorToken)
```

with:

- `userToken` = an FX token; and
- `validatorToken` = a USD validator fee token.

For every FX direct pool operation:

- `userToken` MUST be a valid non-USD `TIP-20` token;
- `validatorToken` MUST be a valid USD fee token;
- `userToken != validatorToken`; and
- the existing FeeAMM validation that both pool tokens are USD-denominated is replaced by the FX direct-pool validation in this section.

Fee-settlement reserve updates are defined in §6.2.

### 7.1 Pool Liquidity Check for Fee Settlement

For FX fee settlement, sufficient liquidity depends only on the USD6 fee amount and the pool's validator-token reserve. The user-token input amount is irrelevant to the liquidity check.

Therefore, for a transaction with max fee `maxFeeUsd6`, the pool is liquid enough iff:

```text
reserve_validator_token >= validatorFeeOut(maxFeeUsd6)
```

### 7.2 Rebalance Swap

The existing `rebalanceSwap(userToken, validatorToken, amountOut, to)` remains unchanged for USD pools and MUST revert for FX direct pools.

This TIP adds the following FX-specific rebalance entry point:

```solidity
function rebalanceSwapFX(
    address userToken,
    address validatorToken,
    uint256 amountOutUserToken,
    uint256 maxAmountInValTok,
    address to
) external returns (uint256 amountInValTok);
```

`rebalanceSwapFX` MUST use the block's effective fee snapshot for `userToken`.

If `userToken` does not have a valid effective quote for the current block, the call MUST revert.

It MUST compute:

```text
fairUsd6       = usd6ValueCeil(amountOutUserToken, priceX18)
amountInValTok = validatorRebalanceIn(fairUsd6)
```

If `amountInValTok > maxAmountInValTok`, the call MUST revert.

Then it MUST:

1. transfer `amountInValTok` of `validatorToken` from the caller into the pool;
2. transfer `amountOutUserToken` of `userToken` from the pool to `to`; and
3. update pool reserves accordingly.

On success, `rebalanceSwapFX` MUST emit the existing `RebalanceSwap` event with `amountIn = amountInValTok` and `amountOut = amountOutUserToken`.

### 7.3 Mint

For an FX direct pool, the first `mint(userToken, validatorToken, amountValidatorToken, to)` requires `userToken` to have a valid effective quote for the current block. The caller deposits only validator-token, `MIN_LIQUIDITY` remains permanently locked, and the bootstrap formula is unchanged. The quote is used only to validate that the pool's FX token is currently fee-enabled; it is not used in the bootstrap formula.

For subsequent mints, let:

- `U = reserve_user_token`;
- `V = reserve_validator_token`;
- `S = totalSupply`;
- `priceX18` be the effective quote for `userToken` in the current block; and
- `amountValidatorToken` be the caller's deposit.

If `userToken` does not have a valid effective quote for the current block, the call MUST revert.

Then the minted liquidity MUST be:

```text
discountedUserReserveUsd6
    = floor(usd6ValueFloor(U, priceX18) * FEE_AMM_N / FEE_AMM_SCALE)

liquidity
    = floor(amountValidatorToken * S / (V + discountedUserReserveUsd6))
```

As in the existing FeeAMM, `liquidity == 0` MUST revert.

### 7.4 Burn

`burn(userToken, validatorToken, liquidity, to)` is unchanged for FX direct pools: it returns the caller's pro-rata share of both reserves and updates supply and reserves exactly as it does today.

No oracle read is required for `burn` itself. `burn` MUST remain available even if `userToken` is no longer fee-configured or has no valid effective quote.

# Invariants

The following invariants must always hold.

1. **Attodollar and USD6 fee accounting remain the source of truth.** This TIP only converts existing `maxFeeUsd6` and `actualFeeUsd6` amounts into FX-token units.
2. **`pathUSD` is only an oracle quote asset.** It is not a new gas-accounting unit.
3. **Validator settlement remains USD-only.** Validators continue to receive fees only through the existing USD fee-token path and `FeeManager` credit model.
4. **FX oracle snapshots are parent-state, delayed, and cached.** Changes to `feeOracle()` or oracle state in block `N` affect fee settlement starting in block `N + 1`, and repeated use of the same FX token in a block observes the same cached result.
5. **No multi-hop routing exists in this TIP.** FX fee settlement uses only the direct pool `(userToken, validatorToken)`.
6. **Actual FX spend never exceeds the precharged maximum.** If `actualFeeUsd6 <= maxFeeUsd6`, then `actualUserTokenSpend <= maxUserTokenFee`.
7. **Validator credit depends only on the USD6 fee amount and the existing FeeAMM spread.** It is not recomputed from the rounded FX-token debit.
8. **FX fee use does not bypass existing pause or policy checks.** Token pause state, transfer authorization, and spending-limit logic continue to apply on the live execution path exactly as they do for USD fee tokens.
9. **FX rebalancing is bounded, and burns are quote-free.** `rebalanceSwapFX` MUST respect the caller's max-input bound; `burn()` MUST remain pro-rata and MUST NOT consult the oracle.
10. **Arithmetic is checked.** Overflow, underflow, division by zero, and reserve values that cannot fit in storage MUST NOT silently wrap.
