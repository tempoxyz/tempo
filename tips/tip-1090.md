# TIP-1090: AWS Nitro Enclave Attestation Verification Precompile

## Abstract

This TIP introduces a precompile that verifies AWS Nitro Enclave attestation documents: it decodes the COSE_Sign1-signed CBOR document, validates the embedded X.509 certificate chain against the pinned AWS Nitro Attestation PKI root, verifies the ECDSA P-384 document signature, and returns the signed fields (PCRs, module ID, timestamp, public key, user data, nonce) to the caller. Contracts use the returned fields to enforce their own enclave identity and freshness policy.

## Motivation

Trusted execution environments let offchain services prove *what code they are running* to onchain contracts. A Nitro Enclave can generate a key pair, request an attestation document that binds the public key to the enclave's measurements (PCRs), and register that key with a contract. The contract can then trust signatures from that key as coming from known, measured code. This enables TEE-backed oracles, payment automation agents, key custody services, and bridge validators without trusting the operator of the hardware.

Verifying an attestation document requires CBOR/COSE decoding, X.509 certificate chain validation, SHA-384 hashing, and multiple ECDSA P-384 signature verifications — none of which the EVM supports natively.

Alternatives considered:

- **Pure Solidity verification** ([base/nitro-validator](https://github.com/base/nitro-validator)): demonstrates full onchain verification is possible, but P-384 verification in Solidity leans on `MODEXP` for hundreds of modular inversions per signature. A full cold verification costs on the order of 267M gas, and even with off-chain "inverse hint" witnesses it requires ~27 KB of calldata per signature, five transactions, and a certificate cache contract. This is workable on a general-purpose L2 but is poor UX and poor economics.
- **ZK proof of attestation**: proving COSE + X.509 + P-384 in a circuit adds proving latency and heavy off-chain tooling for what is fundamentally a few signature verifications.
- **Trusted oracle/committee attesting validity**: reintroduces exactly the trust the attestation is meant to remove.

Since Tempo controls the node, a native precompile makes verification a single call at honest gas cost, consistent with the approach taken for signature verification in [TIP-1020](./tip-1020.md).

## Assumptions

- **AWS Nitro Attestation PKI is the root of trust.** The commercial-partition root certificate (SHA-256 `8cf60e2b2efca96c6a9e71e851d00c1b6991cc09eadbe64a6a1d1b1eb9faff7c`, valid ~30 years, expiring 2049) is pinned in the node per chain spec. If AWS rotates or adds roots, a protocol upgrade is required to add them. If this assumption is violated (root key compromise), attestations are forgeable and applications relying on them are compromised.
- **No revocation checking.** AWS's own validator guidance disables CRL checking, and leaf/intermediate certificates in the Nitro PKI are short-lived (hours to days), which bounds exposure. Applications that need revocation can layer an application-level registry keyed by the returned leaf certificate hash.
- **Block timestamps approximate wall-clock time.** Certificate validity windows are checked against the block timestamp. If validator timestamps skew, expiry enforcement skews with them. Because leaf certificates are valid for only a few hours, attestations must be submitted promptly after generation.
- **The document format is stable.** The precompile implements the validation rules from the [AWS attestation process specification](https://github.com/aws/aws-nitro-enclaves-nsm-api/blob/main/docs/attestation_process.md). Format changes by AWS (e.g., a new `digest` algorithm) require a TIP update and hardfork.
- **Deterministic cryptography.** SHA-384, strict DER parsing, and P-384 verification must produce identical accept/reject results across all node versions; the implementation is consensus-critical.

## Threat Model

- **AWS (Nitro hypervisor and Attestation PKI)** — trusted. Signs attestation documents and issues the certificate chain. Compromise of the root or an intermediate CA allows attestation forgery; out of scope.
- **Enclave operator (EC2 instance owner)** — untrusted. Cannot forge measurements: PCRs are computed by the Nitro hypervisor. Can run *any* enclave image, including malicious ones, and obtain valid attestations for them. The precompile therefore proves *authenticity* (AWS signed this document for an enclave with these measurements), never *trustworthiness*. PCR/module ID policy is the calling contract's responsibility.
- **Transaction submitter / onchain observer** — untrusted. Attestation documents in calldata are public and replayable by anyone. The precompile does not enforce freshness or nonce binding; contracts MUST implement replay protection appropriate to their protocol (nonce challenge, timestamp window, one-time registration).
- **Malicious caller (DoS)** — untrusted. May submit oversized, deeply nested, or pathological CBOR/DER inputs. Mitigated by a hard input size cap, bounded collection sizes, bounded nesting depth, and gas charged before verification work.
- **ECDSA malleability** — P-384 signatures in the Nitro PKI are not guaranteed low-s, and the precompile accepts both forms (matching AWS behavior). Contracts MUST NOT use raw attestation bytes or signatures as uniqueness keys.

## Precompile Address

```
0xA77E570000000000000000000000000000000000
```

The address is provisional until scheduling. The precompile is stateless and MUST reject `DELEGATECALL`, consistent with other Tempo precompiles.

## Interface

```solidity
interface INitroAttestationVerifier {
    /// @notice Input is structurally invalid: not valid CBOR/COSE, exceeds size
    ///         limits, missing mandatory fields, or violates field constraints.
    error InvalidFormat();

    /// @notice The certificate chain is invalid: root mismatch, broken issuer
    ///         chain, expired or not-yet-valid certificate, wrong algorithm or
    ///         key type, or a basic-constraints / key-usage / critical-extension
    ///         violation.
    error InvalidCertificate();

    /// @notice A certificate signature or the COSE document signature failed
    ///         ECDSA P-384 verification.
    error InvalidSignature();

    /// @notice A single PCR entry.
    struct Pcr {
        /// @dev PCR index, in [0, 32).
        uint8 index;
        /// @dev PCR value; length is 32, 48, or 64 bytes.
        bytes value;
    }

    /// @notice The validated, signed contents of an attestation document.
    struct NitroAttestation {
        /// @dev Issuing Nitro hypervisor module ID (non-empty).
        string moduleId;
        /// @dev Document creation time, milliseconds since UNIX epoch (> 0).
        uint64 timestamp;
        /// @dev All locked PCRs, sorted ascending by index, no duplicates.
        Pcr[] pcrs;
        /// @dev Optional DER-encoded public key supplied by the enclave;
        ///      empty bytes if absent.
        bytes publicKey;
        /// @dev Optional user data; empty bytes if absent or zero-length.
        bytes userData;
        /// @dev Optional nonce; empty bytes if absent or zero-length.
        bytes nonce;
        /// @dev SHA-256 hash of the DER-encoded leaf (signing) certificate.
        ///      Enables application-layer revocation registries.
        bytes32 leafCertHash;
    }

    /// @notice Verifies a signed AWS Nitro Enclave attestation document.
    /// @dev Performs full validation: COSE_Sign1 decoding, mandatory-field and
    ///      size checks, certificate chain validation to the pinned AWS Nitro
    ///      root CA (including validity windows against the block timestamp),
    ///      and ECDSA P-384 verification of every certificate signature and the
    ///      document signature.
    ///
    ///      The precompile enforces NO application policy. Callers MUST check
    ///      PCR values, module ID, timestamp freshness, and nonce/user_data
    ///      binding themselves.
    /// @param document The raw signed attestation document as returned by the
    ///        Nitro Security Module: a COSE_Sign1 structure (CBOR), with or
    ///        without the leading CBOR tag 18.
    /// @return attestation The parsed and validated document contents.
    ///         Reverts with InvalidFormat, InvalidCertificate, or
    ///         InvalidSignature otherwise.
    function verifyAttestation(bytes calldata document)
        external
        view
        returns (NitroAttestation memory attestation);
}
```

Absent optional fields are returned as empty bytes. Callers MUST NOT rely on distinguishing an absent `user_data`/`nonce` from a present zero-length one.

## Validation Rules

Validation follows the [AWS Nitro attestation process specification](https://github.com/aws/aws-nitro-enclaves-nsm-api/blob/main/docs/attestation_process.md), sections 3.1–3.2. All checks below MUST pass; any failure MUST revert with the indicated error and consume the gas charged.

### 1. Size limit

`document` MUST be at most **24,576 bytes** (the COSE payload itself is capped at 16,384 bytes by the AWS specification; the limit allows for COSE framing and signature). Violation: `InvalidFormat`.

### 2. COSE_Sign1 decoding

- The input MUST decode as a CBOR `COSE_Sign1` array of exactly 4 elements: `[protected, unprotected, payload, signature]`. A leading CBOR tag 18 is OPTIONAL and MUST be accepted; any other tag MUST be rejected.
- The protected header MUST decode to the map `{1: -35}` (algorithm = ECDSA w/ SHA-384). Any other algorithm MUST be rejected.
- The unprotected header MUST be ignored.
- The payload MUST be a byte string of 1–16,384 bytes.
- The signature MUST be a byte string of exactly **96 bytes**, interpreted as `r || s` with 48-byte big-endian components. (The AWS document's CDDL sketch lists 32/48/64-byte sizes, but ES384 COSE signatures are 96 bytes in practice; this matches the NSM implementation and prior art.)
- Trailing bytes after the COSE structure MUST be rejected.

Violation: `InvalidFormat`.

### 3. Payload (attestation document) validation

The payload MUST decode as a CBOR map. Mandatory fields: `module_id`, `digest`, `timestamp`, `pcrs`, `certificate`, `cabundle`. Optional fields: `public_key`, `user_data`, `nonce`. No recognized field's value may be null; recognized fields MUST NOT appear more than once. Unrecognized keys MUST be skipped (they are covered by the AWS signature, so skipping is safe and preserves forward compatibility; this follows base/nitro-validator rather than a strict allowed-fields check).

| Field | Type | Constraint |
|---|---|---|
| `module_id` | text string | non-empty |
| `digest` | text string | exactly `"SHA384"` |
| `timestamp` | unsigned integer | `> 0`, `≤ 2^64 − 1` |
| `pcrs` | map | 1–32 entries; keys: integers in `[0, 32)`, no duplicates; values: byte strings of length 32, 48, or 64 |
| `certificate` | byte string | 1–1,024 bytes, DER X.509 |
| `cabundle` | array | 1–32 entries, each a byte string of 1–1,024 bytes, DER X.509 |
| `public_key` | byte string (optional) | 1–1,024 bytes |
| `user_data` | byte string (optional) | 0–512 bytes |
| `nonce` | byte string (optional) | 0–512 bytes |

Definite- and indefinite-length CBOR encodings MUST both be accepted, with collection size caps enforced before allocation and CBOR nesting depth bounded (implementation-defined, ≥ 16). Violation: `InvalidFormat`.

### 4. Certificate chain validation

The chain is assembled as specified by AWS: `cabundle` is ordered `[ROOT, INTERM_1, …, INTERM_N]` and the verification path is:

```
certificate (leaf) ← INTERM_N ← … ← INTERM_1 ← ROOT
```

- `cabundle[0]` MUST be byte-identical (DER) to the pinned AWS Nitro root certificate for the chain's configured trust root. Otherwise: `InvalidCertificate`.
- Every certificate MUST be strict DER, X.509 v3, with signature algorithm `ecdsa-with-SHA384` (OID `1.2.840.10045.4.3.3`) and a subject public key on NIST P-384 (`secp384r1`), encoded as an uncompressed point.
- Every certificate's validity window (`notBefore`, `notAfter`) MUST contain the current block timestamp.
- Issuer/subject linkage MUST hold at every step (child's issuer DN equals parent's subject DN, compared on DER bytes).
- Basic constraints and key usage, per the AWS specification:
  - Root: `CA=true`, `keyCertSign` asserted, `pathLenConstraint` (if present) ≥ chain length.
  - Intermediates: `CA=true`, `keyCertSign` asserted, and no certificate may appear in a path longer than its `pathLenConstraint` allows.
  - Leaf: `digitalSignature` asserted, MUST NOT be a CA.
- Certificates containing unrecognized *critical* extensions MUST be rejected.
- CRL and OCSP checking MUST NOT be performed (per AWS validator guidance).
- Each non-root certificate's signature MUST verify (ECDSA P-384 over the SHA-384 hash of its `tbsCertificate`) under its parent's public key. Failure: `InvalidSignature`; all other chain violations: `InvalidCertificate`.

### 5. Document signature verification

The COSE `Sig_structure` is constructed per RFC 9052:

```
[ "Signature1", protected, external_aad = h'', payload ]
```

The 96-byte signature MUST verify as ECDSA P-384 over the SHA-384 hash of the serialized `Sig_structure`, under the leaf certificate's public key. Both low-s and high-s signatures MUST be accepted (AWS does not guarantee canonical signatures). Failure: `InvalidSignature`.

### 6. Result

On success the precompile returns the ABI-encoded `NitroAttestation` struct. On any failure it reverts with the ABI-encoded error selector and MUST NOT return partial results.

## Trust Root Configuration

The pinned root certificate is part of the chain specification, not the precompile code:

- **Mainnet and testnet**: the AWS commercial-partition root, `CN=aws.nitro-enclaves`, SHA-256 `8cf60e2b2efca96c6a9e71e851d00c1b6991cc09eadbe64a6a1d1b1eb9faff7c`.
- **Devnets** MAY configure a different root, enabling locally generated fixture attestations in development and CI.

Adding or rotating roots on public networks requires a protocol upgrade.

## Gas Costs

All calls pay the standard Tempo precompile calldata cost of 6 gas per 32-byte word (rounded up) on the full ABI-encoded input, consistent with TIP-1020.

Verification gas is charged in two stages, each **before** the corresponding work is performed:

1. **Base cost** — `40,000` gas, charged before decoding. Covers COSE/CBOR parsing, payload validation, X.509 parsing, and SHA-384 hashing.
2. **Per-signature cost** — `35,000` gas per ECDSA P-384 verification, charged after parsing determines the chain length and before any signature verification. The number of verifications is `len(cabundle) − 1 + 1 (leaf) + 1 (document)` = `len(cabundle) + 1`.

A typical production chain (root + 3 intermediates + leaf) performs 5 verifications: `40,000 + 5 × 35,000 = 215,000` gas plus calldata cost.

If the available gas is insufficient at either stage, the precompile MUST fail with out-of-gas without performing the remaining work.

These numbers are provisional: before scheduling, the per-signature cost MUST be benchmarked against the node's P-384 implementation and calibrated to the compute-time ratio versus the existing P256 precompile pricing (8,000 gas, TIP-1020).

## Compatibility

This TIP is **additive**. It introduces a new precompile at a previously empty address, activated at the `protocolVersion` hardfork like other Tempo precompiles. It does not modify existing opcodes, precompiles, or transaction formats. Before activation, calls to the address behave as calls to an empty account.

Because certificate validity is checked against the block timestamp, verification of the same document can succeed in one block and fail in a later one (certificate expiry). This is intentional: contracts needing a durable record should verify once and persist the result (e.g., register the enclave's `publicKey` after checking PCRs).

# Tooling

- **Reference implementation guidance**: the node SHOULD reuse an existing audited stack — `aws-lc-rs` provides `ECDSA_P384_SHA384` (Tempo already uses AWS-LC for P256, see TIP-1020 implementation), with strict-DER X.509 and CBOR/COSE parsing on top. The parser MUST be treated as consensus-critical and fuzzed.
- **Test fixtures**: vendored sample attestation documents (valid, and mutated per validation rule) plus a fixture root CA and generator script, so tests do not depend on live AWS infrastructure or wall-clock time. The [base/nitro-validator test data](https://github.com/base/nitro-validator) and the `aws-nitro-enclaves-nsm-api` repository provide real-document references.
- **Devnet support**: a chain-spec field for the trust root (see above) so local networks can mint their own attestations.
- **SDK**: a helper in the Tempo TypeScript/Rust SDKs that takes raw NSM output bytes and produces the `verifyAttestation` calldata, plus decoding of the returned struct and errors.
- **Reference consumer contract**: a minimal example contract demonstrating correct policy enforcement (PCR allowlist, timestamp freshness window, nonce challenge, one-time key registration), to steer integrators away from the "verified = trusted" mistake.

# Observability

`N/A` for onchain events: the precompile is a stateless view function and emits no logs; outcomes are fully visible in transaction traces and call results.

Node implementations SHOULD export metrics for operational monitoring:

- `tempo_nitro_attestation_verify_total{result="ok"|"invalid_format"|"invalid_certificate"|"invalid_signature"}` — answers "are integrators' attestations failing, and why" and surfaces anomalous spikes (e.g., a certificate-expiry misconfiguration or an attack probing the parser).
- `tempo_nitro_attestation_verify_duration_seconds` — validates the gas schedule against real compute cost in production.

# Invariants

| ID | Invariant | Description |
|----|-----------|-------------|
| **NA1** | Determinism | For a fixed `(document, block timestamp, configured root)`, every node MUST produce an identical accept/reject result and identical output bytes. |
| **NA2** | Chain of trust | Verification MUST succeed only if the leaf certificate chains to the pinned root with every X.509 check (validity, linkage, constraints, key usage, critical extensions, signatures) passing. |
| **NA3** | Signature soundness | Verification MUST succeed only if the COSE document signature verifies under the leaf certificate's P-384 key over the exact `Sig_structure` bytes. |
| **NA4** | Field completeness | Every mandatory-field, type, and size rule from the AWS specification (§3 tables above) MUST be enforced; returned fields MUST be byte-identical to the signed payload contents. |
| **NA5** | Policy neutrality | Acceptance MUST be independent of PCR values, module ID contents, `timestamp` recency, `public_key`, `user_data`, and `nonce` contents (beyond structural constraints). Policy belongs to callers. |
| **NA6** | Revert on failure | Any validation failure MUST revert with the specified error; no partial results are ever returned. |
| **NA7** | Gas before work | Base gas is charged before parsing; per-signature gas is charged before any P-384 verification. Insufficient gas MUST fail without performing the work. |
| **NA8** | Bounded resources | Input ≤ 24,576 bytes; ≤ 32 PCRs; ≤ 32 cabundle entries; bounded CBOR nesting. Oversized or pathological inputs MUST be rejected before allocation-heavy work. |

The test suite MUST cover, at minimum: a known-good production attestation (frozen block timestamp); tagged and untagged COSE forms; definite- and indefinite-length CBOR collections; each field constraint violated individually; duplicate map keys and duplicate PCR indices; unknown non-critical payload fields (accepted) and unknown critical X.509 extensions (rejected); wrong root; broken issuer linkage; expired and not-yet-valid certificates at boundary timestamps; wrong-curve and compressed-point keys; corrupted certificate and document signatures; high-s signature acceptance; maximum-size inputs; and out-of-gas at both charging stages.
