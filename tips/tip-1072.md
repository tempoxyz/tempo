# TIP-1072: TIP-20 Backed Token Convertibility

## Abstract

This TIP adds native convertibility between TIP-20 tokens that share the same backing token. A TIP-20 token may declare a single backing token, and the protocol maintains the invariant that the backed token's total supply equals the backing token balance held by the backed token contract.

Conversion is executed by a native conversion precompile. It transfers the user's source token into the precompile, burns the source token, transfers backing from the source token contract to the destination token contract, and mints the destination token atomically.

## Motivation

Tempo stablecoins can represent issuer-specific wrappers over the same underlying reserve asset. Users and applications should be able to convert between those wrappers at exact par when both wrappers are backed by the same TIP-20 token, without relying on DEX liquidity, spreads, or issuer-posted orders.

Enshrining this as a token/backing primitive gives wallets, applications, and DEX routing a deterministic conversion edge. It also keeps issuer control explicit: the destination token's `DEFAULT_ADMIN_ROLE` controls who may convert into that token, and the shared backing token's existing TIP-403 transfer policy controls whether backing may move between wrapper contracts.

## Assumptions

- TIP-20 tokens use the same decimals. This TIP does not define decimal scaling or rate conversion.
- A backed token's backing token is another TIP-20 token.
- TIP-20 contract addresses can hold TIP-20 balances, but ordinary user transfers to TIP-20 contract addresses remain disallowed except for protocol-defined flows such as the backing transfers specified here.
- The destination token's `DEFAULT_ADMIN_ROLE` holder is responsible for configuring an appropriate conversion policy before allowing conversion into the token.
- The backing token's `DEFAULT_ADMIN_ROLE` holder can block unwanted wrapper participation through the backing token's TIP-403 transfer policy, because conversion moves backing token balances between wrapper contracts.

## Threat Model

- **Token admins.** A token's `DEFAULT_ADMIN_ROLE` holder is trusted to declare the token's backing token and configure conversion policy for conversion into that token. A malicious or mistaken admin can disable inbound conversion by clearing policy or can choose a backing token that users do not expect before supply exists.
- **Backing token admin.** The backing token's `DEFAULT_ADMIN_ROLE` holder controls the backing token's transfer policy and can allow or block backing movement between wrapper contracts.
- **Users.** Users may attempt to convert through paused tokens, unbacked tokens, blocked policies, or tokens that no longer share backing. These attempts MUST revert.
- **Conversion precompile.** The precompile is trusted protocol code. It can mint destination tokens for conversion without an explicit issuer grant, but only after all backing, pause, and policy checks pass.

# Specification

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119.

## TIP-20 Backing State

Each TIP-20 token has an optional singular backing token:

```solidity
address backingToken;
```

`backingToken == address(0)` means the token is not backed for purposes of this TIP.

A token MUST NOT set itself as its backing token. Backing relationships MUST NOT contain cycles. For example, if `A.backingToken = B` and `B.backingToken = C`, then `C.backingToken` MUST NOT be `A`.

### Backing Administration

Only a holder of the token's `DEFAULT_ADMIN_ROLE` MAY set or remove `backingToken`.

`addBackingToken(backingToken)` MUST fail unless `backingToken` is currently unset, the caller holds the token's `DEFAULT_ADMIN_ROLE`, `backingToken` is a nonzero TIP-20 token address, and setting it would not create a backing cycle.

`removeBackingToken()` MUST fail unless the caller holds the token's `DEFAULT_ADMIN_ROLE`, `backingToken` is currently set, and `totalSupply() == 0`.

If `totalSupply() > 0` when `backingToken` is initially set, `addBackingToken(backingToken)` MUST transfer exactly `totalSupply()` units of the backing token from the caller to the backed token contract. The operation MUST revert if that caller cannot provide exactly that backing amount or if the backing token's TIP-403 policy rejects the transfer.

TIP-20 exposes backing administration through add and remove functions only:

```solidity
interface ITIP20Backing {
    error BackingAlreadySet();
    error BackingCycle();
    error BackingNotSet();
    error LiveSupply();
    error UnauthorizedAdmin();

    event BackingTokenAdded(address indexed token, address indexed backingToken);
    event BackingTokenRemoved(address indexed token, address indexed backingToken);

    /// @notice Adds a backing token for this TIP-20 token.
    /// @dev Callable only by this token's DEFAULT_ADMIN_ROLE. Reverts if a backing token is
    /// already set, if `backingToken` is this token, or if the relationship
    /// would create a backing cycle.
    function addBackingToken(address backingToken) external;

    /// @notice Removes this token's backing token.
    /// @dev Callable only by this token's DEFAULT_ADMIN_ROLE. Reverts unless totalSupply() == 0.
    function removeBackingToken() external;
}
```

### Backed Supply Invariant

For any TIP-20 token `T` with `T.backingToken = R != address(0)`, the following invariant MUST hold:

```text
T.totalSupply() == R.balanceOf(address(T))
```

Implementations MUST preserve this invariant across mint, burn, backing setup, and conversion.

For backed tokens, `mint` and `mintWithMemo` MUST transfer exactly the minted amount of the backing token from the issuer calling the mint function to the backed token contract in the same atomic operation. This backing transfer MUST use the backing token's normal `transferFrom` path, including allowance and TIP-403 checks. Ordinary burning MUST decrease the token's backing balance by the burned amount in the same atomic operation, or MUST be disallowed unless another protocol rule maintains the invariant.

The conversion precompile is the exception to the backed mint transfer-from rule. When conversion mints destination tokens, the precompile first transfers backing from the source token contract to the destination token contract, then mints the destination token to `recipient`. It does not need an allowance from the DEX or swap user for the destination token's backing transfer.

More generally, for any TIP-20 token contract `T` and any other TIP-20 token `R`, the following invariant MUST hold:

```text
R.balanceOf(address(T)) == 0, unless T.backingToken == R
R.balanceOf(address(T)) == T.totalSupply(), if T.backingToken == R
```

This invariant is preserved because ordinary transfers to TIP-20 contract addresses are disallowed, and the protocol-defined exceptions only credit a TIP-20 contract address when that contract is backed by the credited token. Initial backing setup credits `address(T)` with exactly `T.totalSupply()` units of `R`. Conversion moves backing from one backed token contract to another backed token contract that shares the same immediate backing token, while burning the source wrapper and minting the destination wrapper by the same amount.

### Interface

```solidity
interface ITIP20Convert {
    error BackingNotSet();
    error BackingMismatch();
    error ConversionPolicyNotSet();
    error ConversionPolicyRejected();
    error TokenPaused();
    error ZeroAmount();
    error UnauthorizedAdmin();

    event ConversionPolicyUpdated(
        address indexed token,
        uint64 indexed policyId
    );

    event Convert(
        address indexed user,
        address indexed sourceToken,
        address indexed destinationToken,
        address backingToken,
        uint256 amount
    );

    /// @notice Sets the TIP-403 policy used to authorize conversion into `token`.
    /// @param token Destination TIP-20 token whose inbound conversion policy is being configured.
    /// @param policyId TIP-403 policy ID checked against the converting user.
    /// @dev Callable only by `token`'s DEFAULT_ADMIN_ROLE. A zero policy ID is valid only if
    /// policy ID zero is the canonical deny-all policy. Use `clearConversionPolicy`
    /// to make policy absence explicit.
    function setConversionPolicy(address token, uint64 policyId) external;

    /// @notice Clears the conversion policy for `token`.
    /// @param token Destination TIP-20 token whose inbound conversion policy is cleared.
    /// @dev Callable only by `token`'s DEFAULT_ADMIN_ROLE. After clearing, conversion into
    /// `token` MUST revert with `ConversionPolicyNotSet`.
    function clearConversionPolicy(address token) external;

    /// @notice Returns whether `token` has a configured conversion policy.
    function hasConversionPolicy(address token) external view returns (bool);

    /// @notice Returns the TIP-403 policy ID used for conversion into `token`.
    /// @dev Reverts with `ConversionPolicyNotSet` if no policy is configured.
    function conversionPolicy(address token) external view returns (uint64 policyId);

    /// @notice Converts `amount` of `sourceToken` into `destinationToken`.
    /// @param sourceToken TIP-20 token to burn from the caller.
    /// @param destinationToken TIP-20 token to mint.
    /// @param amount Amount to convert. Source, destination, and backing tokens
    /// all use the same decimals, so no scaling is applied.
    /// @param recipient Address that receives minted destination tokens.
    /// @param policySubject User address checked against the destination
    /// conversion policy.
    /// @dev Reverts unless both tokens have the same nonzero immediate backing
    /// token, all three immediate tokens are unpaused, and the destination
    /// conversion policy authorizes `policySubject`.
    function convert(
        address sourceToken,
        address destinationToken,
        uint256 amount,
        address recipient,
        address policySubject
    ) external;
}
```

### Conversion Policy

The conversion precompile stores conversion policy per destination token:

```solidity
mapping(address token => uint64 policyId) conversionPolicy;
mapping(address token => bool configured) hasConversionPolicy;
```

Only a holder of `token`'s `DEFAULT_ADMIN_ROLE` MAY set, update, or clear `conversionPolicy[token]`.

The conversion policy is a TIP-403 policy reference. During conversion into `destinationToken`, the precompile MUST check the policy against `policySubject`, not against the conversion precompile address, DEX, router, or `recipient`. The precompile MUST require both of the following checks to pass:

1. `isAuthorizedSender(policyId, policySubject)`.
2. `isAuthorizedRecipient(policyId, policySubject)`.

TIP-1015 compound policies MAY be used, and the sender and recipient sub-policies are evaluated according to TIP-1015.

If no policy is configured for `destinationToken`, conversion into `destinationToken` MUST revert.

## Conversion Eligibility

`convert(sourceToken, destinationToken, amount, recipient, policySubject)` MUST revert unless all of the following are true:

1. `amount > 0`.
2. `sourceToken != destinationToken`.
3. `sourceToken.backingToken != address(0)`.
4. `destinationToken.backingToken != address(0)`.
5. `sourceToken.backingToken == destinationToken.backingToken`.
6. `sourceToken`, `destinationToken`, and their shared immediate backing token are all unpaused.
7. `destinationToken` has a configured conversion policy.
8. The destination conversion policy authorizes `policySubject` as both sender and recipient.
9. The caller's source-token transfer into the conversion precompile is allowed by the source token's TIP-403 transfer policy.
10. The backing token transfer from the source token contract to the destination token contract is allowed by the backing token's TIP-403 transfer policy.

If the shared backing token is itself backed by another token, only the immediate shared backing token participates in conversion eligibility. The paused state, backing state, and policies of the backing token's own backing token are not checked for this conversion.

## Conversion State Transition

For `convert(A, B, amount, recipient, policySubject)` by caller `C`, where `A.backingToken == B.backingToken == R`, the precompile MUST execute the following atomically:

1. Transfer `amount` of `A` from `C` to the conversion precompile.
2. Burn `amount` of `A` from the conversion precompile.
3. Transfer `amount` of `R` from `address(A)` to `address(B)`.
4. Mint `amount` of `B` to `recipient`.
5. Emit `Convert(policySubject, A, B, R, amount)`.

If any step fails, the entire conversion MUST revert.

The mint event for step 4 MUST identify the conversion precompile as the issuer/minter. The precompile does not need to be explicitly granted a TIP-20 issuer role for this mint.

Burning the source token is allowed after the caller's transfer into the conversion precompile succeeds. There is no separate conversion-out policy on the source token.

If all conversion eligibility conditions are true and the caller has sufficient source-token balance or allowance for step 1, conversion MUST NOT revert.

## Stablecoin DEX Autorouting

The Stablecoin DEX MAY treat same-backing conversion as a deterministic fill before consuming order book liquidity for a hop.

For v1, the DEX MUST only check conversion between the hop token and that token's quote token. The DEX MUST NOT search arbitrary conversion paths or recursively inspect backing relationships.

For each hop, before iterating orders at a tick, the DEX SHOULD check conversion eligibility once if the tick price is no better than exact par for the taker. If the token and its quote token share the same immediate backing token and the destination conversion policy authorizes the swap user, then conversion at exact `1:1` is better than or equal to taking that tick and the DEX SHOULD execute the remaining hop amount through the conversion precompile instead of consuming orders at that tick or worse ticks.

DEX-initiated conversion MUST call the conversion precompile with:

```text
recipient = Stablecoin DEX
policySubject = swap user
```

The destination conversion policy is therefore checked against the user whose swap is being executed, not against the DEX. The converted destination tokens are minted to the DEX even on the final hop. The DEX then credits or transfers output according to the same settlement path used for ordinary fills.

For policy purposes, DEX autorouting through conversion MUST behave like a swap fill:

1. The source token movement from the user or the user's DEX internal balance into the conversion path MUST satisfy the same sender-side checks that the DEX applies to ordinary swaps for that hop.
2. The destination token, when later transferred or withdrawn from the DEX to the user, MUST satisfy the same recipient-side checks that ordinary DEX settlement applies.
3. Intermediate route tokens MUST satisfy the same DEX route validation applied to ordinary swaps, including pause checks for every token in the route.

The conversion amount MUST be accounted for as an exact `1:1` fill in swap accounting. It MUST count toward route fill amount, min-output checks, max-input checks, and any other slippage limits exactly as if the DEX had filled an order at par.

If the DEX precheck determines that conversion is ineligible because the tokens do not share immediate backing or the destination conversion policy does not authorize the swap user, the DEX MUST continue normal tick iteration. If the precheck determines that conversion is eligible, the conversion precompile call is expected not to revert. If it nevertheless reverts, the DEX MUST treat that as failed conversion eligibility for that hop and continue normal tick iteration rather than reverting the entire swap.

## Pause Behavior

Conversion MUST revert when any of the three immediate tokens is paused:

- source token;
- destination token;
- shared backing token.

If the shared backing token is itself backed by another token, that deeper backing token's paused state does not affect this conversion.

## Backing Token Changes

Changing a nonzero backing token requires removing the current backing token and then setting a new one. Removing a nonzero backing token is blocked while `totalSupply() > 0`. Therefore, existing same-backing conversion relationships can only disappear immediately because:

- source token supply is reduced to zero and its `DEFAULT_ADMIN_ROLE` holder removes or changes backing;
- destination token supply is reduced to zero and its `DEFAULT_ADMIN_ROLE` holder removes or changes backing;
- destination conversion policy is cleared or changed;
- one of the relevant TIP-403 policies begins rejecting the conversion path;
- one of the immediate tokens is paused.

No explicit conversion edge needs to be enabled or disabled beyond current backing metadata and destination conversion policy.

# Observability

The conversion precompile MUST emit:

```solidity
event ConversionPolicyUpdated(
    address indexed token,
    uint64 indexed policyId
);
```

This answers: which destination tokens are open to inbound conversion under which TIP-403 policy?

The conversion precompile MUST emit:

```solidity
event Convert(
    address indexed user,
    address indexed sourceToken,
    address indexed destinationToken,
    address backingToken,
    uint256 amount
);
```

This answers: who converted, between which wrappers, through which backing token, and for how much?

A successful conversion MUST also emit the existing TIP-20 events for:

1. transfer of source token from the caller to the conversion precompile;
2. burn of source token;
3. transfer of backing token from the source token contract to the destination token contract;
4. mint of destination token to the recipient.

# Invariants

| ID | Invariant | Description |
|----|-----------|-------------|
| **BC1** | Backed supply equality | For every backed token `T` with backing `R`, `T.totalSupply() == R.balanceOf(address(T))` MUST hold after every successful state transition. |
| **BC2** | Singular backing | A token has at most one backing token. |
| **BC3** | No backing removal with live supply | `removeBackingToken()` MUST fail while `totalSupply() > 0`; changing backing requires a successful remove followed by a successful add. |
| **BC4** | Exact initial backing | Setting backing on a token with existing supply MUST transfer exactly `totalSupply()` backing units into the token contract. |
| **BC5** | Same immediate backing only | Conversion MUST only occur when source and destination have the same nonzero immediate backing token. Recursive backing relationships MUST NOT expand eligibility in v1. |
| **BC6** | Destination controls inbound conversion | Conversion into a token MUST fail unless that destination token has a configured conversion policy that authorizes the policy subject as both sender and recipient. |
| **BC7** | Backing token policy controls wrapper movement | Conversion MUST perform an actual backing token transfer from the source token contract to the destination token contract, and that transfer MUST be subject to the backing token's TIP-403 policy. |
| **BC8** | Pause enforcement | Conversion MUST fail when the source, destination, or immediate backing token is paused. |
| **BC9** | Atomicity | A conversion either completes all source transfer, source burn, backing transfer, destination mint, and event emission steps, or reverts with no state changes. |
| **BC10** | No backing cycles | Backing relationships MUST NOT contain cycles. |
| **BC11** | No non-backing token balances | For any TIP-20 token contract `T`, another TIP-20 token `R` MUST have zero balance at `address(T)` unless `T.backingToken == R`, in which case that balance MUST equal `T.totalSupply()`. |
| **BC12** | Eligible conversion does not revert | If all conversion eligibility conditions are true and the caller has sufficient source-token balance or allowance, conversion MUST NOT revert. |
