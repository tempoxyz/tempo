# TIP-1031: Embed Consensus Context in the Block Header

## Abstract

Tempo blocks currently do not carry any consensus `Context` (round, leader, parent view+digest). Recovering any consensus information about a block requires the Notarization or Finalization certificates in the consensus layer. There is no verifiable linkage of consensus metadata in the execution layer.

Introducing consensus metadata into the blocks allows for stateless validation of this information prior to any produced notarization or finalization certificates. Commonware's new `Deferred` [verification](https://github.com/commonwarexyz/monorepo/pull/2604) which optimistically notarizes blocks relies on this upfront consensus metadata validation, resulting in a reduction in a finalization latency.

## Motivation

Enable `Deferred` verification in the consensus engine, via two traits:

1. **`CertifiableBlock`** â€” requires blocks to embed a `Context` and expose it via `fn context()`.
2. **`Deferred`** â€” a marshal wrapper that uses the embedded context to vote optimistically during `verify()` (checking only `block.context() == consensus_context`), then spawns full verification in the background and gates finalization on its result via `certify()`.

Tempo currently implements `Automaton` directly (lack of the `CertifiableBlock` implementation prevents the use of `Deferred`), meaning:

- Full EL execution (`new_payload`) must complete *before* a notarization vote can be cast.
- Crash recovery after notarization requires external state to reconstruct which round/view a block belonged to.

Embedding the context in the block header enables:

- **Faster consensus**: validators notarize after a cheap context check; execution runs in parallel.
- **Follower nodes**: can verify consensus chain linkage (parent view, round, leader) without needing certificates.

The reshare [example](https://github.com/commonwarexyz/monorepo/blob/240e0207ee10c3c37a42867ce4de97b581c06b32/examples/reshare/src/application/types.rs#L22) in the commonware monorepo demonstrates this pattern: the `Context` is the first field in the block and is always written/read as part of the block encoding.

# Specification

This TIP will be activated in a future hardfork (TBD).

When activated, a new field, `Option<Context>` is added as the **last** field of `TempoHeader`, which must be set on every subsequent block containing consensus metadata. The field follows the trailingâ€‘optional pattern used by Ethereum's forkâ€‘activated header fields (e.g. `base_fee_per_gas`, `blob_gas_used`): when `None`, it is omitted entirely from the RLP stream, preserving existing block hashes for preâ€‘activation headers.

```rust
#[derive(Debug, Clone, PartialEq, Eq, Hash, Default, RlpEncodable, RlpDecodable)]
#[cfg_attr(feature = "reth-codec", derive(reth_codecs::Compact))]
#[cfg_attr(feature = "serde", derive(serde::Serialize, serde::Deserialize))]
pub struct Context {
  pub epoch: u64,
  pub view: u64,
  pub leader: B256,
  pub parent_view: u64,
  pub parent_digest: B256,
}
```

The `Context` adds 5 fields to the header: 3 `u64` values and 2 `B256` values, totaling 88 bytes raw.

**RLP**: Each `u64` encodes to at most 9 bytes (1â€‘byte prefix + 8 bytes); each `B256` encodes to 33 bytes (1â€‘byte prefix + 32 bytes). With a list header, the worstâ€‘case overhead is **~97 bytes** per block. Preâ€‘activation headers incur zero overhead.

**Compact (DB)**: The `reth_codecs::Compact` representation is comparable, using bitflagâ€‘compressed integer widths.

## Block Production

When activated the block proposal adjustment has added requirements. The application actor:

1. Constructs the `Context` received from the consensus engine.
2. **MUST** set the context field on the header with provided data

## Block Verification

During `Automaton::verify()` step:

1. Compare the set context against information provided by the consensus engine.
2. If they do not match, reject the block immediately.
3. Continue with verification as-is.

Once `CertifiableBlock` is implemented, step 1-2 moves into the `Deferred` wrapper and the application's `verify` only handles step 3 and EL execution. The flow becomes:

```
verify():
  block.context() == consensus_context?  â†’  yes  â†’  vote notarize
                                                 â†’  spawn: full verification
  
certify():
  await background result  â†’  pass  â†’  vote finalize
                           â†’  fail  â†’  vote nullify
```

This migration to `Deferred` verification does not have to be coupled with this TIP. A validator that chooses to synchronize verification with their notarization will not affect consensus. The node simply does not contribute to any decrease in finalization latency.

## Genesis Block

If activated at genesis, genesis block encodes a sentinel context for consistency in the setting of this field.

```rust
let sentinel_context = Context {
  epoch: 0,
  view: 0,
  leader: B256::ZERO,
  parent_view: 0,
  parent_digest: B256::ZERO,
};
```

# Invariants

1. **Context presence**: Every block when activated has a set `Context`.

2. **Context correctness**: The encoded context MUST exactly match the `Context` provided by the consensus engine when the block was proposed. Validators MUST reject blocks where the embedded context does not match.

3. **Context commitment**: Because the context is a part of the header, and the header hash is the block's digest, the context is transitively committed to by any notarization or finalization certificate over that digest.

4. **Backward incompatibility**: This is a breaking change to block verification. All nodes must upgrade at the same protocol version. Blocks produced before the upgrade do not have the set context. Any blocks proposed by a non-upgraded node will have their proposals rejected, and will incorrectly notarize blocks with an invalid context.

5. **Encoding backward compatibility**: The `context` field MUST be the last field in `TempoHeader`. When `None`, it MUST be omitted entirely from the RLP stream so that the encoded representation of preâ€‘activation headers remains unchanged and existing block hashes are preserved.

6. **Round-trip fidelity**: Context serializes and de-serializes into the same values.

### Test Coverage

- `CertifiableBlock::context()` returns the correct context for a block built with known parameters.
- Block rejection when embedded context does not match the consensus engine's context.
- RLP round-trip for `TempoHeader` with `context: Some(...)` and `context: None`, and that a pre-activation header's hash is unchanged after the upgrade.
