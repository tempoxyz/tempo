# TIP-1031: Embed Consensus Context in Block Header

## Abstract

Tempo blocks currently do not carry any consensus `Context` (round, leader, parent view+digest) inside the block itself. Recovering any consensus information about a block requires the Notarization or Finalization certificates, completely separate from the execution layer. There is no verifiable linkage of consensus metadata in the execution layer.

Introducing consensus metadata into the blocks allows for stateless validation of this information prior to any notarization or finalization certificates. Commonware's new `CertifiableBlock` trait requires this in their new `Deferred` verification mechanism to support optimistic notarizations of blocks, allowing network latency to mask finalization latency with async execution verification.

## Motivation

Opt-into `Deferred` verification mechanism in the consensus engine. Commonware's consensus engine supports a deferred verification flow through two traits:

1. **`CertifiableBlock`** — requires blocks to embed a `Context` and expose it via `fn context()`.
2. **`Deferred`** — a marshal wrapper that uses the embedded context to vote optimistically during `verify()` (checking only `block.context() == consensus_context`), then spawns full verification in the background and gates finalization on its result via `certify()`.

Tempo currently implements `Automaton` directly (lack of the `CertifiableBlock` implementation prevents used of `Deferred`), meaning:

- Full EL execution (`new_payload`) must complete *before* a notarization vote can be cast.
- Follower nodes cannot extract the consensus context from a block to validate chain structure without re-executing.
- Crash recovery after notarization requires external state to reconstruct which round/view a block belonged to.

Embedding the context in the block header enables:

- **Faster consensus**: validators notarize after a cheap context check; execution runs in parallel.
- **Follower nodes**: can verify consensus chain linkage (parent view, round, leader) without needing certificates. Nodes can have gaps of stored notarizations and finalization certificates.

The reshare example in the commonware monorepo demonstrates this pattern: the `Context` is the first field in the block and is always written/read as part of the block encoding.

# Specification

## Extra Data Layout

The `extra_data` field of the Ethereum `Header` (accessed via `TempoHeader.inner.extra_data`) currently carries one of:

| Block type | Current `extra_data` contents |
|---|---|
| Epoch boundary (last block of epoch) | `OnchainDkgOutcome` encoded via `commonware_codec` |
| Regular block with DKG dealing | `SignedDealerLog` encoded via `commonware_codec` |
| Regular block (no dealing) | Empty |

This TIP prepends a fixed-size consensus context prefix to `extra_data`. The consensus context is encoded using the `commonware_codec::Write` as defined upstream:

```
[ varint(epoch) ][ varint(view) ][ pubkey_bytes(32) ][ varint(parent_view) ][ digest_bytes(32) ]
```

Depending on the epoch and view counts, this is roughly ~70B (For views/epoch in the order of billions it only grows to 80 bytes). Well within the limits of the max extra data size (10KiB).

Because the context has a known fixed-width trailer (32-byte pubkey + varint + 32-byte digest) but variable-width varint prefix, and the DKG payload that follows is also variable-length, a **1-byte discriminant** is prepended to distinguish the DKG payload type:

**NOTE**: The discriminant is not required as dkg payload type can be discerned from the block height and epoch length (boundary block).

| Discriminant | Meaning |
|---|---|
| `0x00` | No DKG payload follows |
| `0x01` | `SignedDealerLog` follows |
| `0x02` | `OnchainDkgOutcome` follows |

```
extra_data = context_bytes ++ discriminant(1 byte) ++ dkg_payload
```

Where:

- `context_bytes` is the `commonware_codec` encoding of `Context<Digest, PublicKey>`
- `dkg_payload` is the existing content (empty, `SignedDealerLog`, or `OnchainDkgOutcome`)

## Block Production

When proposing a block, the application actor constructs `extra_data` as:

1. Encode the `Context` received from the consensus engine (round, leader, parent view+digest) using `commonware_codec::Write`.
2. Append the discriminant byte.
3. Append the DKG payload (if any), using the same logic as today.

The `Context` used is the one provided by the consensus engine to `Automaton::propose(context)`.

## Block Verification

During `Automaton::verify()`:

1. Decode `context_bytes` from `extra_data`.
2. Compare the decoded context against the `Context` provided by the consensus engine.
3. If they do not match, reject the block immediately.
4. Decode the discriminant and DKG payload, and verify as today.

Once `CertifiableBlock` is implemented, step 1-2 moves into the `Deferred` wrapper and the application's `verify` only handles step 4 and EL execution. The flow becomes:

```
verify():
  block.context() == consensus_context?  →  yes  →  vote notarize
                                                     spawn background: EL execution + DKG check
  
certify():
  await background result  →  pass  →  vote finalize
                           →  fail  →  vote nullify
```

This migration to `Deferred` verification does not have to be coupled with this TIP. A validator that chooses to synchronize verification with their notarization will not affect consensus. The node simply does not contribute to any decrease in finalization latency.

## Genesis Block

The genesis block encodes a sentinel context (epoch 0, view 0, zero pubkey, zero parent) for consistency in the extra data wire format.

# Invariants

1. **Context presence**: Every block's `extra_data` MUST begin with a valid `commonware_codec`-encoded `Context<Digest, PublicKey>` followed by a 1-byte discriminant.

2. **Context correctness**: The encoded context MUST exactly match the `Context` provided by the consensus engine when the block was proposed. Validators MUST reject blocks where the embedded context does not match.

3. **Context commitment**: Because `extra_data` is part of the Ethereum header, and the header hash is the block's digest, the context is transitively committed to by any notarization or finalization certificate over that digest.

4. **Discriminant validity**: The discriminant byte MUST be one of `0x00`, `0x01`, or `0x02`. Any other value MUST cause block rejection.

5. **DKG payload consistency**: The DKG payload semantics are unchanged from the current protocol:
   - Epoch boundary blocks MUST have discriminant `0x02` with a valid `OnchainDkgOutcome`.
   - Regular blocks MAY have discriminant `0x01` (with a valid `SignedDealerLog`) or `0x00` (no DKG data).

6. **Backward incompatibility**: This is a breaking change to the block format. All nodes must upgrade at the same protocol version. Blocks produced before the upgrade do not contain the context prefix.

7. **Round-trip fidelity**: `Context::read(Context::write(ctx))` MUST produce a value equal to `ctx`.

### Test Coverage

- Round-trip encode/decode of `extra_data` with context + each discriminant variant.
- `CertifiableBlock::context()` returns the correct context for a block built with known parameters.
- Block rejection when embedded context does not match the consensus engine's context.
- Follower node can decode context and verify chain linkage without EL execution.
- Invalid discriminant byte causes block rejection.
- Genesis block encoding (sentinel context).
