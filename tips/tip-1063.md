# TIP-1063: Protocol Feature Flags

## Abstract

This TIP defines feature flags for Tempo protocol changes, decoupling individual feature activation from hardfork bundles.

Each feature can be tracked separately, signaled by validators, and activated onchain. A feature becomes active only after the Tempo multisig schedules it and enough active validators have signaled support.

## Motivation

Tempo currently ships protocol changes through hardfork bundles (e.g. T3, T4, T5). That means multiple protocol changes have to converge on the same activation boundary, even when they are ready on different timelines.

Feature flags separate code distribution from behavior activation. Validators can upgrade over time, and each feature can activate only after enough of the active validator set has signaled support. This means releases can continue on a regular cadence, while activation can be scheduled, delayed, or cancelled independently instead of requiring an emergency release to move an activation.

# Specification

## 1. Scope

This TIP defines the feature flag mechanism for Tempo protocol features.

A feature flag controls activation for a named protocol feature. A protocol feature is a unit of protocol behavior that can be shipped inactive and later activated by chain history.

A protocol feature is appropriate when the change affects consensus validation, state transition behavior, transaction validity, system contract behavior, precompile behavior, block building, RPC simulation, or other protocol-facing logic.

This TIP makes feature flags the normal replacement for shipping protocol changes through hardfork bundles. Hardforks can still be used when a change cannot be safely represented as a feature flag, including introduction of this mechanism, changes to feature signaling itself, or emergency changes that the feature mechanism cannot express.

This TIP does not define the behavior of any individual future feature. Each feature still needs its own TIP or approved feature record describing the behavior being activated, compatibility impact, tests, and rollout requirements.

## 2. Protocol Features

A protocol feature is a specific protocol behavior that can be tracked, signaled, and activated independently.

Protocol features are listed in a source-controlled feature manifest. The manifest is a single file that contains the feature metadata used to generate release bitmaps.

Each feature entry includes:

- `feature_id`: a globally unique, lowercase, dot-separated feature identifier
- `feature_index`: a stable numeric index used by the feature bitmap
- `description`: a short description of the behavior
- `tip`: the TIP that specifies the behavior
- `minimum_supported_version`: the first Tempo release expected to support the feature, for operator and tooling visibility
- `status`: the feature's manifest status, such as draft, approved, deprecated, or cancelled

Feature indexes are append-only. Once a `feature_index` has been assigned to a feature, that index is not reused.

Each Tempo release that signals protocol feature support has a source-controlled release manifest. The release manifest lists the full cumulative set of feature IDs supported by that release.

Release tooling generates the release bitmap from the release manifest and the feature manifest. For each feature ID listed in the release manifest, tooling resolves the feature's `feature_index` and sets the corresponding bit. Features omitted from the release manifest are not signaled by that release.

Because the release manifest is cumulative, the manifest for a release is sufficient to derive that release's full bitmap without replaying previous release manifests.

The generated bitmap is embedded in the release artifact as a hardcoded value. Nodes do not read the manifest at runtime to decide which features to signal.

## 3. Feature Bitmap

Supported protocol features are encoded as a bitmap. Each feature has one stable `feature_index`, and that index determines the word and bit used to represent support:

```text
word_index = feature_index / 256
bit_index  = feature_index % 256
bit_mask   = 1 << bit_index
```

Features `0..255` are encoded in word `0`, features `256..511` are encoded in word `1`, features `512..767` are encoded in word `2`, and so on.

A feature is supported by a release when the corresponding bit is set in that release's hardcoded bitmap:

```text
is_supported(feature_index):
  word_index = feature_index / 256
  bit_index  = feature_index % 256
  return (feature_bitmap[word_index] & (1 << bit_index)) != 0
```

The release bitmap must be hardcoded into each Tempo release artifact. Validators must not construct the bitmap dynamically, and operators must not be able to enable or disable individual feature bits through local runtime configuration. By running a release, a validator signals the protocol feature set encoded in that release.

A binary whose feature bitmap has been modified is not the corresponding Tempo release artifact. It is a custom build, and any release metadata, version-readiness claims, or support guarantees for the original release no longer apply to that binary.

An official Tempo release can stop signaling a feature that is not active. This is the expected rollback path for feature code that shipped inactive but is not ready to proceed toward activation. An official release must not silently drop support for an active feature unless another active protocol rule supersedes that feature or the binary fails closed before processing blocks that require it. The feature record status should be updated to reflect the decision, for example by marking the feature deprecated or cancelled.

The bitmap is a readiness signal, not a cryptographic proof of capability. A proposer can technically run a custom build that advertises a different bitmap. The protocol uses the bitmap to track declared support before activation; after activation, enforcement comes from deterministic protocol validation of the active feature's behavior.

## 4. Validator Signaling

Each block header includes a fixed number of feature bitmap words:

```text
HEADER_FEATURE_BITMAP_WORDS = 5
```

The number of feature indexes directly signalable by a block is:

```text
5 * 256 = 1280
```

The fixed header width keeps consensus encoding simple. The feature registry can store bitmap words beyond the header width for future expansion, but v1 block signaling applies only to feature indexes that fit within the fixed header bitmap.

When a validator proposes a block, the proposer includes the hardcoded feature bitmap words from its binary in the block header. If the binary supports fewer words than the header width, missing words are treated as zero.

Every block header includes the proposer's feature bitmap. During block processing, a protocol handler compares the header bitmap against the proposer's latest bitmap in the feature registry.

If the header bitmap differs from the proposer's latest committed bitmap, the handler commits the new bitmap to the feature registry. If the header bitmap is unchanged, registry state does not change for that proposer.

The handler commits the bitmap from the block header. No transaction supplies a separate bitmap payload.

Only protocol handler code can commit validator bitmaps. User transactions must not be able to spoof validator support.

Only committed registry state counts toward quorum. A bitmap present only in a block header does not affect quorum until the bitmap commit handler commits it to the registry.

The latest committed bitmap replaces the validator's previous bitmap. When a proposer's committed word changes from its previously stored word, the registry must update that validator's support for every changed feature bit in that word. Bits that change from `0` to `1` add support; bits that change from `1` to `0` remove support. The registry must maintain support counts as if it recomputed, for each feature, the number of active validators whose latest committed bitmap has that feature bit set.

A bitmap commit whose submitted words are identical to the proposer's stored bitmap does not change support counts and does not emit support-change events.

The registry must emit events when validator bitmap words are committed and when feature support changes. The feature-support event identifies the affected feature by index and ID; current support counts are read from registry state.

Validator set changes must be reflected in feature support counts. A validator entering the active set counts only after it is eligible under the validator set rules and has a committed bitmap. A validator leaving the active set no longer contributes to quorum.

## 5. Quorum

For v1, quorum is validator-count based and uses one global threshold for all features. The threshold is Tempo's consensus threshold, expressed as `n/m`, where `m` is the active validator set size and `n` is the required number of active validators with committed support for the feature.

Changing the global quorum threshold after deployment is outside the scope of this TIP and should be defined by a separate protocol change.

For a feature at block `B`:

```text
m = number of eligible active validators at B
n = required active-validator support threshold
support = number of eligible active validators whose latest bitmap supports the feature
```

The feature has quorum at block `B` if:

```text
support >= n
```

Quorum means current network readiness. It does not activate the feature by itself, and reaching quorum before activation does not permanently latch readiness. If support later falls below quorum before the feature activates, the feature remains inactive until current support reaches quorum again.

The bitmap signals readiness. Scheduling happens through a multisig transaction, and activation happens through the pre-execution activation handler once scheduling, timestamp, and quorum conditions are all satisfied.

## 6. Scheduling and Activation

Each feature has an activation signer configured in the feature registry. For v1, the activation signer is the Tempo-controlled multisig.

The activation signer can schedule a feature only if all of the following are true at transaction execution time:

- the feature is known to the registry
- the feature is not already active
- the feature is not already scheduled for activation
- `scheduled_activation_timestamp` is in the future

Scheduling does not require current quorum. The activation signer can schedule a feature before quorum, after quorum, or in the same period where quorum is first reached. 

The multisig can schedule before quorum, but the feature cannot activate until both the scheduled time has passed and current quorum is satisfied.

When the scheduling transaction is accepted, the registry sets:

```text
scheduled_at_block = block.number
scheduled_activation_timestamp = requested_activation_timestamp
```

The feature becomes active once both conditions are true:

```text
block.timestamp >= scheduled_activation_timestamp
feature_has_quorum(feature_index) == true
```

Activation is applied during the pre-execution phase at the start of block processing. Before user transactions execute, protocol handlers commit any changed proposer bitmap from the block header, update support counts, and check scheduled inactive features. 

If a feature's scheduled timestamp has been reached and current quorum is satisfied, the pre-execution phase marks the feature active, sets `activated_at_block = block.number`, sets `activated_at_timestamp = block.timestamp`, and emits the activation event. User transactions in that block then observe the feature as active.

`scheduled_activation_timestamp` records the earliest timestamp at which the feature is allowed to activate. `activated_at_block` and `activated_at_timestamp` record when the feature actually became active onchain. If quorum is reached after the scheduled timestamp has already passed, the actual activation block and timestamp are later than the scheduled timestamp.

Quorum is evaluated against the current eligible validator set during the pre-execution activation check. A feature whose scheduled timestamp has passed but whose quorum has not been reached remains scheduled but inactive until current quorum is reached. A feature that reaches quorum before scheduling remains inactive until the activation signer schedules it and the scheduled timestamp is reached.

Activation must be deterministic from chain history. Given the same chain, all nodes must compute the same active feature set for every block.

No force-activation path is defined by this TIP. Any path that allows activation without quorum requires a separate protocol change.

Before activation, the activation signer can cancel or reschedule a feature.

Cancellation is allowed only if the feature is scheduled and not active. Cancellation clears `scheduled_activation_timestamp` and leaves the feature inactive. The feature can be scheduled again later by the activation signer.

Rescheduling is allowed only if the feature is scheduled and not active. Rescheduling replaces `scheduled_activation_timestamp` with a new future timestamp.

Cancellation and rescheduling must be onchain, deterministic from chain history, and observable through feature registry events and read methods.

## 7. Feature Registry

Tempo introduces a feature registry precompile exposed through a normal contract ABI.

The registry is where features are registered and where activation state is tracked. Operators and tooling can inspect registry state to see which features are known, scheduled, active, or waiting on validator support.

The registry stores the latest committed bitmap for each active validator:

```text
validator_bitmap[validator][word_index] => uint256
```

The registry also stores feature identity, scheduling, activation, and quorum data:

```text
feature_config[feature_index] => {
  feature_id
}

activation_quorum => {
  quorum_numerator,
  quorum_denominator
}

feature_activation[feature_index] => {
  scheduled_at_block,
  scheduled_activation_timestamp,
  activated_at_block,
  activated_at_timestamp
}
```

The exact contract ABI and storage layout are implementation details, but the observable behavior must match this TIP.

Only the Tempo-controlled multisig can register features in the onchain registry. Feature registration must reject duplicate `feature_id` values, duplicate `feature_index` values, and feature IDs that do not match the canonical lowercase dot-separated format.

Registered feature metadata is immutable. After registration, the registry must not allow edits to registered feature fields, including `feature_id` or `feature_index`. Mutable registry state is limited to validator signaling state and activation lifecycle state, including scheduling, cancellation, rescheduling, and activation.

The registry must expose enough read methods for nodes, indexers, and operators to determine:

- whether a feature is known
- whether a validator's latest committed bitmap supports a feature
- the current validator support count for a feature
- the global activation quorum threshold
- whether a feature is scheduled
- whether a feature is active at a given block or timestamp
- the minimum Tempo version expected to support the feature, if exposed by the registry or accompanying release metadata

At minimum, the registry emits:

```solidity
event FeatureRegistered(uint256 indexed featureIndex, string featureId);
event FeatureScheduled(uint256 indexed featureIndex, string featureId, uint64 scheduledActivationTimestamp);
event FeatureScheduleCancelled(uint256 indexed featureIndex, string featureId);
event FeatureRescheduled(uint256 indexed featureIndex, string featureId, uint64 scheduledActivationTimestamp);
event FeatureSupportUpdated(uint256 indexed featureIndex, string featureId);
event FeatureActivated(uint256 indexed featureIndex, string featureId, uint64 activatedAtBlock, uint64 activatedAtTimestamp);
```

## 8. Runtime Integration

Nodes maintain an active feature set that backs protocol checks such as `feature.is_active(feature_id)`.

Nodes may cache the active feature set at the current head, but protocol checks must derive feature state for the specific block being validated, built, simulated, or replayed. If a feature is active in registry state at the node's current head, `feature.is_active(feature_id)` returns true before the node validates, builds, simulates, or serves protocol behavior at that head.

During block processing, the feature registry emits events when feature lifecycle state changes, including bitmap commits, scheduling, cancellation, rescheduling, and activation. Registry state is the source of truth. Feature state changes must happen in fixed steps that every node runs deterministically; events are only for observability and indexing.

Bitmap commits and activation are applied before user transaction execution, so protocol checks observe the updated active feature set during that block's execution. All nodes must update the active feature set at the same deterministic pre-execution point in block processing.

A scheduling transaction included in block `B` cannot activate the feature earlier than its scheduled timestamp. It records scheduling state and emits the corresponding registry event.

All protocol-facing paths that depend on feature state must use the same active feature set for the block they are evaluating. This includes:

- block validation
- block building
- transaction validation
- transaction execution
- mempool admission and eviction
- RPC simulation
- precompiles and system contracts
- protocol handlers
- tracing and debugging paths that replay protocol behavior

The first rollout preserves existing hardfork behavior while representative callsites move from hardfork-only checks toward feature checks.

## 9. Unsupported Active Features

Once a feature is active, every node that validates, builds, simulates, or serves protocol behavior at or after the activation block must understand that feature.

A node whose binary does not support an active feature must fail closed. It must not silently continue applying old protocol rules.

If a feature activates during the pre-execution phase and the node binary does not support it, the node fails closed before user transaction execution for that block. The failure must be explicit and observable.

## 10. Disabling and Rollback

Before activation, rollback is handled by shipping a new Tempo release that no longer signals the feature, leaving the feature unscheduled, cancelling the schedule, rescheduling to a later timestamp, or preventing the scheduled feature from satisfying quorum. If quorum and scheduling are never both satisfied, the feature cannot activate.

After quorum but before scheduling, the activation signer can choose not to schedule the feature. The feature remains inactive.

After a feature is scheduled but before it is active, the activation signer can cancel or reschedule the feature.

After activation, local disabling is not allowed. An operator must not be able to turn off an active feature through local configuration.

Post-activation disabling through local configuration or the feature registry is not supported in v1. An active feature that needs to be changed, disabled, or replaced must be handled by a new protocol feature, hardfork, or emergency upgrade that supersedes the old behavior. This is especially important for features that change state, deploy contracts, perform migrations, or alter assumptions that are unsafe to unwind locally.

## 11. Compatibility

Feature code can ship before activation. Running a Tempo binary that supports an inactive feature must not change protocol behavior.

Future behavior changes can activate as individual protocol features rather than only as T-version bundles. Operators, RPC providers, block builders, indexers, and partner systems must therefore track feature activation status, not only hardfork names.

Compatibility requirements for each individual feature must be documented by that feature's TIP. At minimum, each feature TIP states:

- what behavior changes after activation
- which releases support the feature
- what old nodes do after activation
- whether RPC, simulation, indexing, or partner behavior changes
- whether the feature is reversible, supersedable, or state-changing

## 12. Observability

Operators and developers need to understand why a feature is or is not active.

Nodes must expose enough information to answer:

- does this binary support the feature?
- if this node is a validator, is it signaling the feature?
- how close is the active validator set to quorum?
- which validators are currently signaling the feature?
- which features are active at the current head?
- when did each feature activate?
- which Tempo version is needed to support each feature?
- why did activation fail or remain unavailable?

This information can be exposed through metrics, logs, debug RPC, standard RPC, CLI output, or a combination of those interfaces. The exact interface is left to implementation, but the information must be available without requiring operators to inspect raw storage manually.

## 13. Security Considerations

The feature bitmap is not an operator-signed vote. It is a statement hardcoded into the validator's running binary. Running a release declares that the validator can safely validate, build, and simulate blocks with the features included in that release.

The activation signer is not a replacement for validator readiness. Scheduling alone cannot activate a feature; the feature must also reach quorum.

Feature activation must not rely on local node configuration. Any local override that changes active protocol behavior can split the network.

Irreversible features require extra review. A feature that performs a state migration, deploys a system contract, changes consensus encoding, changes account or token invariants, or affects transaction validity defines its rollback and supersession story before approval.
