# TIP-1035: Implicit Approval List

## Abstract

This TIP introduces a protocol-level Implicit Approval List: an explicit, enumerated set of contract addresses that may transfer TIP-20 tokens from `msg.sender` without requiring a prior `approve()` call. For listed spenders, `approve()` and `permit()` do not affect allowance state and do not emit an `Approval` event. For access keys, this shifts AccountKeychain spending limit enforcement from approve time to transfer time for listed contracts.

This TIP defines spender-specific exceptions to TIP-20 `approve` and `permit` semantics. It supersedes [TIP-1025](./tip-1025.md), which defined implicit approval behavior for the StablecoinDEX and TipFeeManager individually. TIP-1035 generalizes that into a named, extensible list, makes the behavioral specification self-contained, and adds the MPP Channels contract as the first non-precompile member.

The initial list consists of the TipFeeManager (FeeAMM), the StablecoinDEX, and the MPP Channels contract. See [Initial List](#initial-list) for addresses and rationale.

## Motivation

### Removing redundant approvals

Today, a user who wants to interact with the StablecoinDEX or MPP Channels must first submit a separate `approve` transaction for each token. This has three costs:

1. **User experience**: The first interaction with a given token requires two transactions (or a batch call). Wallets must prompt for approval before every new token interaction, adding friction.
2. **Gas cost**: Each `approve` call costs ~250,000 gas for the cold SSTORE to the allowance slot. Every subsequent operation pays extra gas to load, check, and update the allowance.
3. **State bloat**: Every approval writes a storage slot in the token's allowance mapping.

The approval check is redundant for contracts that only pull tokens from `msg.sender` — the caller is always the one authorizing the transaction, so the `approve` step provides no additional security.

Rather than adding implicit approval behavior ad-hoc to individual TIPs, the protocol should maintain an explicit list. This makes the set visible and auditable in one place, and simplifies future additions.

### Adding the MPP Channels contract

The MPP Channels contract (`0x33b901018174DDabE4841042ab76ba85D4e24f25`) uses `transferFrom` on the caller to escrow funds into payment channels. It only ever pulls from `msg.sender` and is a natural candidate for implicit approval.

# Specification

## Implicit Approval List

The protocol maintains an explicit set of addresses called the Implicit Approval List. For any spender address on this list:

1. The contract may transfer TIP-20 tokens from `msg.sender` without a prior `approve()` call. The mechanism depends on whether the contract is a precompile or a regular contract (see [Implementation](#implementation)).
2. `approve(spender, amount)` where `spender` is on the list: the call succeeds but does not affect allowance state, does not emit an `Approval` event, and does not deduct AccountKeychain spending limits. This applies regardless of `amount`.
3. `permit()` where `spender` is on the list: the call succeeds and increments the owner's nonce, but does not affect allowance state and does not emit an `Approval` event.
4. Token pulls through listed contracts still enforce balance checks, TIP-403 transfer policies, AccountKeychain spending limits, and `Transfer` event emission. The only check bypassed is the allowance check.

## Implementation

The mechanism for bypassing approval differs depending on whether the listed contract is a precompile or a regular deployed contract.

### Precompiles

Implicitly-approved precompiles (TipFeeManager, StablecoinDEX) MUST use `system_transfer_from()` to pull tokens. `system_transfer_from` is a privileged function on the TIP-20 precompile that performs a transfer without checking the allowance. It:

1. Checks that `msg.sender` (the direct caller) is on the Implicit Approval List. If not, the call reverts.
2. Enforces TIP-403 transfer policies via `ensure_transfer_authorized`.
3. Enforces AccountKeychain spending limits via `check_and_update_spending_limit`.
4. Debits `from` and credits `to`. Reverts with `InsufficientBalance` if `from` has insufficient balance.
5. Emits a standard TIP-20 `Transfer(from, to, amount)` event.

### Non-precompile contracts

Non-precompile contracts on the Implicit Approval List (such as MPP Channels) are already deployed and call the standard `transferFrom`. For these contracts, TIP-20's `transferFrom` itself checks whether `msg.sender` is on the Implicit Approval List. If so, the allowance check and allowance decrement are skipped, and a call to `check_and_update_spending_limit` is added. All other checks (balance, transfer policies, events) still apply.

Note that this is a behavioral change for access keys: the existing allowance-based `transferFrom` path does not call `check_and_update_spending_limit` (spending limits are instead enforced at `approve` time via `authorize_approve`). With implicit approval, there is no `approve` step, so spending limits are enforced at transfer time instead. The net effect is that access key spending limits are still enforced, but at a different point in the flow.

Non-precompile contracts do not need to be modified to benefit from implicit approval. The check is in the TIP-20 precompile's `transferFrom` implementation.

### Security model

In both paths, neither `transferFrom` nor `system_transfer_from` restricts the `from` parameter — a listed contract could pass any address, not just `msg.sender`. The caller-only-pulls property is enforced by the listed contract's own code, not by the implicit approval system. The security of this TIP depends on strict admission criteria and immutability of listed contracts. The implicit approval system itself provides no protection against a listed contract that passes a `from` address other than `msg.sender`.

## Pre-existing Approvals

Users who called `approve()` on a listed address before the fork activation may have stale allowance state in storage. While a spender remains on the Implicit Approval List, any pre-existing allowance for that spender is ignored and has no effect on token-pull behavior. Stale slots are inert while the spender remains listed.

For access keys that previously had spending limits deducted by a pre-fork `approve()` call to a listed address: there is no mechanism to refund that consumed spending limit. This is expected to be rare in practice, since access keys would not typically pre-approve these contracts without immediately spending.

## Eligibility Criteria

A contract MAY be added to the Implicit Approval List only if it satisfies all of the following:

1. **Caller-only pulls**: Every reachable code path that pulls tokens via `transferFrom` or `system_transfer_from` SHOULD use `from == msg.sender` for the current call, or transfer with a non-replayable signed authorization from the transferor. This is a security guideline — other patterns could theoretically be secure, but should be vetted with significantly more scrutiny before a contract is added to the list.
2. **No DELEGATECALL**: The contract MUST NOT contain any `DELEGATECALL` opcodes. If a listed contract were to delegatecall into external code, that code would run with the listed contract's address as `address(this)`. A subsequent call to `transferFrom` on a TIP-20 token would have `msg.sender` equal to the listed contract's address, inheriting the implicit approval privilege. This would allow the external code to call `transferFrom` with an arbitrary `from` address and bypass the allowance check.
3. **No proxy pattern**: The contract MUST NOT be an upgradeable proxy. An upgradeable proxy delegatecalls to its implementation, so a malicious upgrade would inherit the implicit approval privilege (same mechanism as above).
4. **Immutable code**: For non-precompile contracts, the deployed bytecode must be immutable. There must be no mechanism (e.g., `SELFDESTRUCT` + redeployment, metamorphic contracts) by which the code at the listed address could change.

For precompiles, criteria 2–4 are inherently satisfied: precompile code is part of the node implementation and is not executed via EVM opcodes. Precompiles can be upgraded through hardforks of the protocol; the security of such upgrades is the responsibility of the protocol.

## Integrator Guidance

Implicitly-approved precompiles (TipFeeManager, StablecoinDEX) are part of the protocol and integrators may treat these as stable unless superseded by a future TIP.

Implicitly-approved deployed contracts (such as MPP Channels) carry no such guarantee. The Implicit Approval List may be amended at any hardfork, and a deployed contract may be removed from the list — for example, if it is replaced by a new deployment or promoted to a precompile. Integrators MUST NOT assume that a deployed contract will remain implicitly approved.

Non-upgradeable integrations that interact with an implicitly-approved deployed contract SHOULD continue to call `approve()` before `transferFrom`. The `approve()` call does not grant allowance while the contract is on the list, but ensures the integration continues to work if the contract is removed in a future hardfork. Upgradeable integrations can omit the `approve()` call and adapt if the list changes.

## Future Amendments

Additional contracts may be added to the Implicit Approval List via future TIPs that reference and amend this one. Each addition must include a safety argument demonstrating that the contract satisfies the eligibility criteria above. Removal of a listed contract requires a separate TIP that defines how pre-existing allowance state for that spender is handled.

# Invariants

1. **Authorized pulls only**: Every contract on the Implicit Approval List SHOULD only pull tokens from the direct caller (`msg.sender`), or with a non-replayable signed authorization from the transferor. This invariant is enforced by the listed contract's code, not by the implicit approval system.
2. **Allowance bypass**: For listed spenders, `transferFrom` and `system_transfer_from` MUST skip the allowance check and allowance decrement. All other checks (balance, TIP-403 transfer policies, AccountKeychain spending limits, `Transfer` event) MUST still be enforced.
3. **List consistency**: The protocol-defined Implicit Approval List MUST match the set defined in this TIP (and any future amendments). No address may be implicitly approved without a corresponding TIP.
4. **Hardfork gating**: The behavior change MUST be gated behind the hardfork activation. Pre-fork transactions must continue to use the allowance-based path to preserve state root consistency.
