# TIP-1035: Implicit Approval List

## Abstract

This TIP introduces a protocol-level Implicit Approval List: an explicit, enumerated set of precompile addresses that may transfer TIP-20 tokens from `msg.sender` without requiring a prior `approve()` call. For listed spenders, `approve()` and `permit()` do not mutate allowance storage and do not emit an `Approval` event, but `allowance(owner, spender)` returns `type(uint256).max` while the spender remains listed. For access keys, this shifts AccountKeychain spending-limit enforcement from approve time to transfer time for listed precompiles.

This TIP defines spender-specific exceptions to TIP-20 `approve`, `permit`, and `allowance` semantics. It supersedes [TIP-1025](./tip-1025.md), which defined implicit approval behavior for the StablecoinDEX and TipFeeManager individually. TIP-1035 generalizes that into a named, extensible list, adds a canonical hardfork-aware discovery function on the AddressRegistry, and makes the behavioral specification self-contained.

The initial list consists of the TipFeeManager (FeeAMM) and the StablecoinDEX. See [Initial List](#initial-list) for addresses and rationale.

## Motivation

### Removing redundant approvals

Today, a user who wants to interact with the StablecoinDEX must first submit a separate `approve` transaction for each token. This has three costs:

1. **User experience**: The first interaction with a given token requires two transactions (or a batch call). Wallets must prompt for approval before every new token interaction, adding friction.
2. **Gas cost**: Each `approve` call costs ~250,000 gas for the cold SSTORE to the allowance slot. Every subsequent operation pays extra gas to load, check, and update the allowance.
3. **State bloat**: Every approval writes a storage slot in the token's allowance mapping.

The approval check is redundant for contracts that only pull tokens from `msg.sender` — the caller is always the one authorizing the transaction, so the `approve` step provides no additional security.

Rather than adding implicit approval behavior ad hoc to individual TIPs, the protocol should maintain an explicit list. This makes the set visible and auditable in one place, simplifies future additions and removals, and gives integrators a canonical hardfork-aware query surface.

# Specification

## Implicit Approval List

The protocol maintains an explicit set of addresses called the Implicit Approval List. For any spender address on this list:

1. The precompile may transfer TIP-20 tokens from `msg.sender` without a prior `approve()` call (see [Implementation](#implementation)).
2. `approve(spender, amount)` where `spender` is on the list: the call succeeds but does not affect allowance storage, does not emit an `Approval` event, and does not deduct AccountKeychain spending limits. This applies regardless of `amount`.
3. `permit()` where `spender` is on the list: the call MUST still enforce all normal validity checks, including deadline and signature validation. If validation fails, the call MUST revert and MUST NOT change the nonce or any other state. If validation succeeds, the owner's nonce increments, but allowance storage is unchanged and no `Approval` event is emitted.
4. `allowance(owner, spender)` where `spender` is on the list MUST return `type(uint256).max`, regardless of any historical stored allowance value.
5. Token pulls through listed precompiles still enforce balance checks, TIP-403 transfer policies, AccountKeychain spending limits, and `Transfer` event emission. The only check bypassed is the allowance check.

## Discoverability

The AddressRegistry precompile MUST expose a hardfork-aware helper:

```solidity
function isImplicitlyApproved(address addr) external view returns (bool);
```

This function returns `true` if and only if `addr` is on the Implicit Approval List for the active hardfork. Before TIP-1035 activates, it returns `false` for all addresses.

## Implementation

Implicitly approved precompiles MUST use `system_transfer_from()` or an equivalent privileged token-pull primitive that bypasses the allowance check while still enforcing transfer policies, AccountKeychain spending limits, balance checks, and standard `Transfer` event emission.

When `system_transfer_from()` is used, it:

1. Checks that `msg.sender` (the direct caller) is on the Implicit Approval List. If not, the call reverts.
2. Enforces TIP-403 transfer policies via `ensure_transfer_authorized`.
3. Enforces AccountKeychain spending limits via `check_and_update_spending_limit`.
4. Debits `from` and credits `to`. It reverts with `InsufficientBalance` if `from` has insufficient balance.
5. Emits a standard TIP-20 `Transfer(from, to, amount)` event.

### Security model

`system_transfer_from()` does not restrict the `from` parameter — a listed precompile could pass any address, not just `msg.sender`. The caller-only-pulls property is enforced by the listed precompile's own code, not by the implicit approval system. The security of this TIP therefore depends on strict admission criteria for the list and on review of each listed precompile's token-pull logic.

## Pre-existing Approvals

Users who called `approve()` on a listed address before TIP-1035 activation may have historical allowance values in storage. While a spender remains on the Implicit Approval List, those stored values are ignored for token-pull authorization, and `allowance(owner, spender)` MUST still return `type(uint256).max`.

Because `approve()` and `permit()` do not emit `Approval` events for listed spenders, event-only indexers cannot reconstruct effective authorization solely from logs. Wallets, indexers, and contracts MUST treat `allowance(owner, spender)` and `AddressRegistry.isImplicitlyApproved(spender)` as the canonical onchain truth while the spender remains listed.

For access keys that previously had spending limits deducted by a pre-fork `approve()` call to a listed address: there is no mechanism to refund that consumed spending limit. This is expected to be rare in practice, since access keys would not typically pre-approve these contracts without immediately spending.

## Eligibility Guidelines

The following are security guidelines for adding a precompile to the Implicit Approval List. Precompiles that do not satisfy these guidelines may still be added, but the amendment TIP MUST include an explicit safety argument explaining why the alternative pattern is secure.

1. **Caller-only pulls**: Every reachable code path that pulls tokens via `system_transfer_from()` SHOULD use `from == msg.sender` for the current call, or transfer with a non-replayable signed authorization from the transferor. Other patterns could theoretically be secure, but should be vetted with significantly more scrutiny.

Since precompile code is part of the node implementation and is not executed via EVM opcodes, concerns such as `DELEGATECALL`, proxy patterns, and code mutability do not apply. Precompiles can be upgraded through hardforks of the protocol; the security of such upgrades is the responsibility of the protocol.

## Integrator Guidance

Listed status is not a one-way ratchet. A future TIP MAY add or remove addresses from the Implicit Approval List.

Non-upgradeable contracts MUST NOT depend on an implicitly approved spender remaining implicitly approved forever. Contracts and offchain integrations that need compatibility across future list changes SHOULD include an explicit `approve()` step in their flow. While the spender remains listed, that `approve()` call is a no-op. If a future hardfork removes the spender from the list, the same flow continues to work because `approve()` regains its normal TIP-20 semantics after the change.

Contracts and integrations that need to branch on the current protocol state MAY query `AddressRegistry.isImplicitlyApproved(spender)`, but this helper is primarily for hardfork-aware UX and discovery. The compatibility-safe pattern is still to include an `approve()` step when continued operation across future list changes matters.

## Future Amendments

Additional precompiles may be added to or removed from the Implicit Approval List via future TIPs that reference and amend this one. Each addition should include a safety argument demonstrating that the precompile satisfies the eligibility guidelines above. Any amendment that removes a spender from the list MUST specify the post-removal interpretation of that spender's stored allowance slots.

# Alternatives Considered

## Default-to-Unlimited Stored Allowance

An alternative design would keep standard ERC-20 allowance execution semantics but default the allowance for listed spenders to `type(uint256).max` (unlimited). `approve()` would still write to storage and `transferFrom()` would still check the allowance. Users would simply never need to explicitly approve because the default is already unlimited, and they could opt out by calling `approve(spender, 0)`.

### Sentinel encoding

To distinguish "never set" (default unlimited) from "explicitly set to 0" (revoked) without an extra `SLOAD` on every read, a sentinel value would be used. The raw allowance slot encodes as:

- `0` (never written) → `allowance()` returns `type(uint256).max`
- `SENTINEL` (a hash-derived constant, e.g. `uint256(keccak256("IMPLICIT_APPROVAL_ZERO"))`) → `allowance()` returns `0`
- any other value `N` → `allowance()` returns `N`

When `approve(spender, 0)` is called, the contract writes `SENTINEL`. When `approve(spender, SENTINEL)` is called (astronomically unlikely in practice), a disambiguation flag in a secondary slot is set; the extra `SLOAD` to check this flag only occurs when the stored value happens to equal `SENTINEL`.

### Trade-offs

**Advantages:**

- `approve()`, `allowance()`, and `permit()` behave as standard ERC-20 operations.
- Users can revoke with `approve(spender, 0)`, providing an explicit opt-out mechanism.

**Disadvantages:**

- **Gas overhead on every transaction.** Every token pull by a listed precompile adds an allowance `SLOAD` (~2,100 gas cold). The current design skips this entirely. For the FeeManager, this cost would be paid on every transaction in the network.
- **Potential `SSTORE` on every transaction.** If a user sets a finite (non-max) allowance, every fee collection or swap decrements it, adding a warm `SSTORE` (~2,900 gas). Users who set finite allowances would also need to periodically re-approve, reintroducing the UX friction this TIP aims to eliminate.

The current design was chosen because the gas savings on hot token-pull paths are significant, while `allowance()` can still expose the effective authorization state to onchain callers and wallets.

# Invariants

1. **Allowance bypass**: For listed spenders, token-pull paths defined by this TIP MUST skip the allowance check and allowance decrement. All other checks — balance, TIP-403 transfer policies, AccountKeychain spending limits, and `Transfer` event emission — MUST still be enforced.
2. **No allowance storage mutation**: `approve(spender, amount)` and `permit()` where `spender` is on the list MUST NOT modify allowance storage and MUST NOT emit an `Approval` event. Successful `permit()` MUST still increment the nonce.
3. **Synthetic allowance reads**: While a spender is on the list, `allowance(owner, spender)` MUST return `type(uint256).max`.
4. **List discoverability**: `AddressRegistry.isImplicitlyApproved(addr)` MUST match the protocol-defined Implicit Approval List for the active hardfork.
5. **List consistency**: No address may be implicitly approved without a corresponding TIP that adds it to the list.
6. **Hardfork gating**: The behavior change MUST be gated behind the TIP-1035 activation hardfork. Before activation, TIP-20 approval semantics remain unchanged.
