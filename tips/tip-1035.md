# TIP-1035: Implicit Approval List

## Abstract

This TIP introduces a protocol-level Implicit Approval List: an explicit, enumerated set of precompile addresses that may transfer TIP-20 tokens from `msg.sender` without requiring a prior `approve()` call. For listed spenders, `approve()` and `permit()` do not affect allowance state and do not emit an `Approval` event. For access keys, this shifts AccountKeychain spending limit enforcement from approve time to transfer time for listed precompiles.

This TIP defines spender-specific exceptions to TIP-20 `approve` and `permit` semantics. It supersedes [TIP-1025](./tip-1025.md), which defined implicit approval behavior for the StablecoinDEX and TipFeeManager individually. TIP-1035 generalizes that into a named, extensible list and makes the behavioral specification self-contained.

The initial list consists of the TipFeeManager (FeeAMM) and the StablecoinDEX. See [Initial List](#initial-list) for addresses and rationale.

## Motivation

### Removing redundant approvals

Today, a user who wants to interact with the StablecoinDEX must first submit a separate `approve` transaction for each token. This has three costs:

1. **User experience**: The first interaction with a given token requires two transactions (or a batch call). Wallets must prompt for approval before every new token interaction, adding friction.
2. **Gas cost**: Each `approve` call costs ~250,000 gas for the cold SSTORE to the allowance slot. Every subsequent operation pays extra gas to load, check, and update the allowance.
3. **State bloat**: Every approval writes a storage slot in the token's allowance mapping.

The approval check is redundant for contracts that only pull tokens from `msg.sender` — the caller is always the one authorizing the transaction, so the `approve` step provides no additional security.

Rather than adding implicit approval behavior ad-hoc to individual TIPs, the protocol should maintain an explicit list. This makes the set visible and auditable in one place, and simplifies future additions.

# Specification

## Implicit Approval List

The protocol maintains an explicit set of addresses called the Implicit Approval List. For any spender address on this list:

1. The precompile may transfer TIP-20 tokens from `msg.sender` without a prior `approve()` call (see [Implementation](#implementation)).
2. `approve(spender, amount)` where `spender` is on the list: the call succeeds but does not affect allowance state, does not emit an `Approval` event, and does not deduct AccountKeychain spending limits. This applies regardless of `amount`.
3. `permit()` where `spender` is on the list: the call MUST still enforce all normal validity checks (including deadline and signature validation). If validation fails, the call MUST revert and MUST NOT change the nonce or any other state. If validation succeeds, the owner's nonce increments, but allowance state is unchanged and no `Approval` event is emitted.
4. Token pulls through listed precompiles still enforce balance checks, TIP-403 transfer policies, AccountKeychain spending limits, and `Transfer` event emission. The only check bypassed is the allowance check.

## Implementation

Implicitly-approved precompiles MUST use `system_transfer_from()` to pull tokens. The TipFeeManager already uses `system_transfer_from` today, so this TIP does not change its behavior. `system_transfer_from` is a privileged function on the TIP-20 precompile that performs a transfer without checking the allowance. It:

1. Checks that `msg.sender` (the direct caller) is on the Implicit Approval List. If not, the call reverts.
2. Enforces TIP-403 transfer policies via `ensure_transfer_authorized`.
3. Enforces AccountKeychain spending limits via `check_and_update_spending_limit`.
4. Debits `from` and credits `to`. Reverts with `InsufficientBalance` if `from` has insufficient balance.
5. Emits a standard TIP-20 `Transfer(from, to, amount)` event.

### Security model

`system_transfer_from` does not restrict the `from` parameter — a listed precompile could pass any address, not just `msg.sender`. The caller-only-pulls property is enforced by the listed precompile's own code, not by the implicit approval system. The security of this TIP depends on strict admission criteria for the list. The implicit approval system itself provides no protection against a listed precompile that passes a `from` address other than `msg.sender`. Since precompile code is part of the node implementation, the security of each listed precompile is the responsibility of the protocol.

## Pre-existing Approvals

Users who called `approve()` on a listed address before the fork activation may have stale allowance state in storage. While a spender remains on the Implicit Approval List, any pre-existing allowance for that spender is ignored and has no effect on token-pull behavior. Stale slots are inert while the spender remains listed.

`allowance(owner, spender)` continues to return the raw stored value and may therefore be stale for listed spenders. While a spender is listed, effective token-pull authorization is determined by list membership and the listed contract's behavior, not by the allowance mapping.

For access keys that previously had spending limits deducted by a pre-fork `approve()` call to a listed address: there is no mechanism to refund that consumed spending limit. This is expected to be rare in practice, since access keys would not typically pre-approve these contracts without immediately spending.

## Eligibility Guidelines

The following are security guidelines for adding a precompile to the Implicit Approval List. Precompiles that do not satisfy these guidelines may still be added, but the amendment TIP MUST include an explicit safety argument explaining why the alternative pattern is secure.

1. **Caller-only pulls**: Every reachable code path that pulls tokens via `system_transfer_from` SHOULD use `from == msg.sender` for the current call, or transfer with a non-replayable signed authorization from the transferor. Other patterns could theoretically be secure, but should be vetted with significantly more scrutiny.

Since precompile code is part of the node implementation and is not executed via EVM opcodes, concerns like DELEGATECALL, proxy patterns, and code mutability do not apply. Precompiles can be upgraded through hardforks of the protocol; the security of such upgrades is the responsibility of the protocol.

## Integrator Guidance

Precompiles on the Implicit Approval List MUST NOT be removed. Contracts MAY depend on implicitly-approved precompiles never requiring approvals.

## Future Amendments

Additional precompiles may be added to the Implicit Approval List via future TIPs that reference and amend this one. Each addition should include a safety argument demonstrating that the precompile satisfies the eligibility guidelines above.

# Invariants

1. **Allowance bypass**: For listed spenders, `transferFrom`, `transferFromWithMemo`, and `system_transfer_from` MUST skip the allowance check and allowance decrement. All other checks (balance, TIP-403 transfer policies, AccountKeychain spending limits, `Transfer` event) MUST still be enforced.
2. **No allowance mutation**: `approve(spender, amount)` where `spender` is on the list MUST NOT modify allowance state and MUST NOT emit an `Approval` event. `permit()` where `spender` is on the list MUST NOT modify allowance state and MUST NOT emit an `Approval` event (but MUST increment the nonce on success).
3. **List consistency**: The protocol-defined Implicit Approval List MUST match the set defined in this TIP (and any future amendments). No address may be implicitly approved without a corresponding TIP.
4. **List permanence**: Precompiles MUST NOT be removed from the Implicit Approval List once added.
5. **Hardfork gating**: The behavior change MUST be gated behind the hardfork activation. Pre-fork transactions must continue to use the allowance-based path to preserve state root consistency.
