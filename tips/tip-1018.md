# TIP-1018: Compliance Hook Policies

## Abstract

This TIP extends `TIP-403` with a new `VALIDATOR_CONTRACT` policy type. A validator-contract policy delegates authorization to an external EVM contract using a bounded `STATICCALL`, allowing issuers to apply custom compliance logic without introducing a new token standard or changing the TIP-20 user-facing ABI.

The design preserves the current `TIP-20` and `TIP-1015` model: tokens continue to point at a `transferPolicyId`, and the registry remains the single authorization surface. Existing issuers can launch with a simple whitelist today, then later migrate the token to a validator-backed policy or compound policy using the existing `changeTransferPolicyId` flow.

## Motivation

`TIP-20` already provides the core mechanics needed for regulated assets: minting, burning, transfer restrictions, gas-fee payment in the asset itself, and native integration with Tempo infrastructure. The missing piece is a way to enforce richer authorization logic than a pure whitelist or blacklist can express.

Examples include:

- country-specific transfer restrictions;
- accreditation or investor-class checks;
- issuer-specific compliance rules that must evolve independently of Tempo core code;
- asymmetric rules for senders, recipients, and mint recipients.

Creating a separate token standard for regulated assets would lose the native benefits of `TIP-20` and duplicate existing infrastructure. The better extension point is `TIP-403`: keep the token standard unchanged and make the policy engine pluggable.

This TIP therefore adds a new leaf policy type that can be used directly or as part of a `TIP-1015` compound policy.

This TIP intentionally does not standardize:

- an investor identity registry;
- a Tempo-to-Securitize event mapping layer;
- post-transfer notification hooks.

Those may be standardized separately. In particular, a read-only authorization hook is sufficient for KYC and jurisdiction checks, but not by itself sufficient to track stateful holding-period rules such as a 24-hour lockup that must be updated atomically on receipt.

# Specification

## Policy Type

`TIP-403` currently defines:

```solidity
enum PolicyType {
    WHITELIST,
    BLACKLIST,
    COMPOUND
}
```

This TIP adds a fourth policy type:

```solidity
enum PolicyType {
    WHITELIST,
    BLACKLIST,
    COMPOUND,
    VALIDATOR_CONTRACT
}
```

`VALIDATOR_CONTRACT` is a leaf policy type. It is not compound, and it MAY be referenced by `TIP-1015` compound policies anywhere a simple policy is accepted today.

## Validator Contract Interface

The external validator contract MUST implement the following interface:

```solidity
interface ITIP403Validator {
    enum AuthorizationRole {
        Transfer,
        Sender,
        Recipient,
        MintRecipient
    }

    /// @notice Returns whether `user` is authorized for the given role under `policyId`.
    /// @dev The caller is always the TIP403 registry precompile address.
    /// Implementations MUST treat this function as read-only and MUST NOT rely on side effects.
    function isAuthorized(
        uint64 policyId,
        address user,
        AuthorizationRole role
    ) external view returns (bool authorized);
}
```

### Rationale

- `policyId` lets a single validator contract serve multiple policies.
- `user` matches the current `TIP-403` authorization surface.
- `role` preserves the `TIP-1015` distinction between sender, recipient, and mint-recipient checks.

This interface deliberately does not pass `amount`, `from`, or `to`. That keeps the first version aligned with the current registry authorization model and avoids expanding the TIP-20 external interface. Stateful or amount-aware compliance rules can be standardized in a follow-up TIP.

## Storage Layout

`TIP-1015` stores policy state in a unified `PolicyRecord`:

```solidity
struct PolicyData {
    uint8 policyType;
    address admin;
}

struct CompoundPolicyData {
    uint64 senderPolicyId;
    uint64 recipientPolicyId;
    uint64 mintRecipientPolicyId;
}
```

This TIP extends the record with validator-contract data:

```solidity
struct ValidatorPolicyData {
    address validatorContract;
}

struct PolicyRecord {
    PolicyData base;
    CompoundPolicyData compound;
    ValidatorPolicyData validator;
}
```

The registry storage layout becomes:

| Slot | Field | Description |
|------|-------|-------------|
| 0 | `policyIdCounter` | Counter for generating unique policy IDs |
| 1 | `policyRecords` | `mapping(uint64 => PolicyRecord)` |
| 2 | `policySet` | `mapping(uint64 => mapping(address => bool))` for whitelist/blacklist membership |

For a given `policyId`, storage locations are:

- `PolicyData`: `keccak256(policyId, 1) + 0`
- `CompoundPolicyData`: `keccak256(policyId, 1) + 1`
- `ValidatorPolicyData`: `keccak256(policyId, 1) + 2`

For non-validator policies, `validator.validatorContract` is ignored.

## ITIP403Registry Additions

The registry interface is extended as follows:

```solidity
interface ITIP403Registry {
    // =========================================================================
    //                    Validator-Contract Policies
    // =========================================================================

    /// @notice Emitted when a validator-contract policy is created
    /// @param policyId The new policy ID
    /// @param updater The caller that created the policy
    /// @param validatorContract The validator contract bound to the policy
    event ValidatorPolicyCreated(
        uint64 indexed policyId,
        address indexed updater,
        address indexed validatorContract
    );

    /// @notice Error when validatorContract is zero or has no code
    error InvalidValidatorContract();

    /// @notice Error when the validator contract reverts, runs out of gas,
    /// or returns malformed data
    error ValidatorCallFailed();

    /// @notice Creates a new validator-contract policy
    /// @param admin The address recorded as the policy admin
    /// @param validatorContract The external validator contract
    /// @return newPolicyId The newly created policy ID
    function createValidatorContractPolicy(
        address admin,
        address validatorContract
    ) external returns (uint64 newPolicyId);

    /// @notice Returns the validator contract for a validator policy
    /// @param policyId The policy to query
    /// @return validatorContract The bound validator contract
    /// @dev Reverts with IncompatiblePolicyType if the policy is not VALIDATOR_CONTRACT
    function validatorPolicyData(uint64 policyId)
        external
        view
        returns (address validatorContract);
}
```

`setPolicyAdmin()` remains valid for validator-contract policies and updates only the recorded admin. This TIP does not include a mutable `setValidatorContract()` function. To change validation logic, issuers create a new validator policy and move the token to the new `transferPolicyId` using existing token administration flows.

## Policy Creation Rules

`createValidatorContractPolicy(admin, validatorContract)` MUST:

1. allocate a new policy ID from `policyIdCounter`;
2. set `policyType = VALIDATOR_CONTRACT`;
3. store `admin` in `PolicyData`;
4. store `validatorContract` in `ValidatorPolicyData`;
5. revert with `InvalidValidatorContract` if `validatorContract == address(0)`;
6. revert with `InvalidValidatorContract` if the target account has no code;
7. emit `PolicyCreated(policyId, updater, VALIDATOR_CONTRACT)`;
8. emit `PolicyAdminUpdated(policyId, updater, admin)`;
9. emit `ValidatorPolicyCreated(policyId, updater, validatorContract)`.

## validatorPolicyData Semantics

`validatorPolicyData(policyId)` MUST:

- revert with `PolicyNotFound` if `policyId` does not exist;
- revert with `IncompatiblePolicyType` if the policy is not `VALIDATOR_CONTRACT`;
- otherwise return the configured validator contract address.

## Authorization Semantics

### Existing Policies

For built-in, whitelist, blacklist, and compound policies, existing behavior is unchanged.

### Validator Policies

When `policyType == VALIDATOR_CONTRACT`, the registry MUST delegate authorization to the external validator contract:

```solidity
function isAuthorized(uint64 policyId, address user) external view returns (bool) {
    return _callValidator(policyId, user, AuthorizationRole.Transfer);
}

function isAuthorizedSender(uint64 policyId, address user) external view returns (bool) {
    return _callValidator(policyId, user, AuthorizationRole.Sender);
}

function isAuthorizedRecipient(uint64 policyId, address user) external view returns (bool) {
    return _callValidator(policyId, user, AuthorizationRole.Recipient);
}

function isAuthorizedMintRecipient(uint64 policyId, address user) external view returns (bool) {
    return _callValidator(policyId, user, AuthorizationRole.MintRecipient);
}
```

### Compound Policies

`TIP-1015` compound policies MAY reference validator-contract policies as constituent policies.

For example:

- sender policy: `VALIDATOR_CONTRACT`
- recipient policy: `WHITELIST`
- mint recipient policy: built-in allow policy `1`

No special compound-policy rules are added beyond treating validator policies as valid leaf policies.

## Precompile-To-EVM Callback Mechanism

The node implementation MUST support a read-only callback from the `TIP403Registry` precompile into an arbitrary EVM contract.

For each validator-policy authorization check, the registry performs a `STATICCALL` with the following properties:

- target address: `validatorContract` from `ValidatorPolicyData`;
- caller address: the canonical `TIP403_REGISTRY_ADDRESS`;
- calldata: `ITIP403Validator.isAuthorized(policyId, user, role)`;
- gas limit: `TIP403_VALIDATOR_GAS_LIMIT = 100_000`;
- value: `0`;
- state mutability: read-only.

The validator call MUST be fail-closed:

- if the call reverts, the registry reverts with `ValidatorCallFailed`;
- if the call runs out of gas, the registry reverts with `ValidatorCallFailed`;
- if the return data is not exactly one ABI-encoded `bool`, the registry reverts with `ValidatorCallFailed`.

The callback MUST NOT expose any additional privileges. The validator contract runs as ordinary EVM code with `msg.sender == TIP403_REGISTRY_ADDRESS` and can only read on-chain state available to any contract.

### Implementation Note

Today the precompile storage abstraction does not provide an external-call primitive. Implementing this TIP therefore requires extending the precompile/runtime boundary with a read-only system-call helper, analogous to the system-call execution path already exposed by the EVM runtime.

## TIP-20 Compatibility

This TIP does not change the `TIP-20` user-facing ABI.

Tokens continue to rely on a single `transferPolicyId`, and the existing authorization flow remains:

1. mint checks `MintRecipient` authorization;
2. transfer checks `Sender` and `Recipient` authorization;
3. burn-blocked logic checks `Sender` authorization.

The only behavioral change is that some leaf policies may now resolve authorization via an external validator contract rather than a local whitelist or blacklist mapping.

## Migration And Upgrade Path

An issuer can migrate in three steps:

1. launch with a simple whitelist or blacklist policy;
2. deploy a validator contract that implements issuer-specific compliance logic;
3. create a validator-contract policy, or a compound policy that references it, and update the token's `transferPolicyId` to the new policy.

This preserves the token address, balances, and existing `TIP-20` integrations.

## Known Limitation: Stateful Lockups

This TIP does not provide a post-transfer hook. As a result, it is not sufficient by itself to atomically maintain state that must be updated when tokens are received.

Examples include:

- a holding-period timer that starts when an account receives tokens;
- an investor-count limit that depends on whether a transfer creates a new holder;
- per-lot cost basis or issuance vintage tracking.

Those use cases require a companion mechanism that can observe successful transfers and persist new compliance state on-chain. Such a mechanism should be standardized separately rather than overloaded into this first validator-policy TIP.

# Invariants

The following invariants MUST hold:

1. Built-in policy IDs `0` and `1` retain their current behavior.
2. Whitelist, blacklist, and compound policies retain their current behavior when no validator-contract policy is involved.
3. A validator-contract policy always points to a non-zero address with code.
4. Validator-contract authorization is read-only and cannot mutate state during the callback.
5. Any validator callback failure is fail-closed and MUST NOT silently authorize a transfer.
6. A validator-contract policy MAY be used anywhere a compound policy expects a leaf policy.
7. Compound policies remain non-recursive: they cannot reference other compound policies.
8. `TIP-20` token administration and transfer-policy migration remain unchanged.

The test suite for this TIP should cover at minimum:

1. creation of validator policies with valid and invalid target addresses;
2. `validatorPolicyData()` success and failure cases;
3. direct authorization through a validator policy for all four authorization roles;
4. compound policies that reference validator policies;
5. revert, out-of-gas, and malformed-return behavior from validator contracts;
6. migration of an existing token from a simple policy to a validator-backed policy.
