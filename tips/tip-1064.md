# TIP-1064: StablecoinDEX User Storage Credits

## Abstract

This TIP adds user-level reusable storage accounting to the StablecoinDEX. When a user's DEX operation deletes reusable DEX storage, the DEX credits that user with storage credits. When a later operation by the same user creates reusable DEX storage, the DEX consumes that user's credits before allowing TIP-1060 precompile storage gas tokens to offset the TIP-1000 storage creation cost.

This gives users credit for the DEX state they previously freed, while still relying on TIP-1060 for the underlying precompile-level gas token balance.

## Motivation

TIP-1060 gives the StablecoinDEX a precompile-level storage gas token balance. That prevents repeated DEX storage churn from being charged as net-new state growth, but it does not decide which user should receive the benefit when one user deletes storage and another user later creates storage.

The StablecoinDEX exposes user-driven storage creation and deletion through order placement, order filling, order cancellation, internal balances, client order IDs, and orderbook maintenance. Without user-level accounting, a user who creates new DEX state could consume storage gas tokens minted by another user's deletion.

This TIP creates the user-level accounting suggested by TIP-1058 for the StablecoinDEX only.

# Specification

## Terminology

- **DEX storage credit**: A protocol-maintained `uint64` counter associated with one user address in the StablecoinDEX.
- **Reusable DEX storage**: StablecoinDEX-owned storage that can be attributed to a user for storage creation and deletion accounting.
- **Storage owner**: The user address charged or credited for a reusable DEX storage slot.
- **TIP-1060 token**: A precompile-level storage gas token minted or consumed by the StablecoinDEX under TIP-1060.

## State

The StablecoinDEX MUST maintain:

```text
dex_storage_credits: mapping(address user => uint64 credits)
```

The `dex_storage_credits` mapping is StablecoinDEX precompile state. Creating or updating a user's `dex_storage_credits` counter, including the first creation of that counter's backing storage slot, MUST NOT itself mint or consume TIP-1060 tokens and MUST NOT incur the 250,000 gas TIP-10-mandated storage creation cost.

If `dex_storage_credits[user] == type(uint64).max`, additional credits for that user MUST saturate at `type(uint64).max`.

## Reusable DEX Storage

The following StablecoinDEX storage is reusable DEX storage:

1. Active order records keyed by `orderId`.
2. Tick-level records for bid and ask lists.
3. Tick bitmap words for bid and ask liquidity.
4. Orderbook records keyed by `bookKey`.
5. User internal balance slots keyed by `(user, token)`.
6. Active client order ID mappings keyed by `(user, clientOrderId)`, if TIP-1003 is active.

Only storage owned by the StablecoinDEX is covered. TIP-20 token balances, allowances, policy state, EOA state, and ordinary smart-contract storage are outside this TIP's scope.

## Storage Owner Attribution

For each reusable DEX storage slot, the storage owner is:

| Storage | Storage owner |
|---------|---------------|
| Order record | `order.maker` |
| Client order ID mapping | the `user` key |
| User internal balance slot | the `user` key |
| Tick-level record | the user whose operation creates or deletes the last order at that tick |
| Tick bitmap word | the user whose operation creates or deletes the first or last liquidity bit in that word |
| Orderbook record | the user whose operation creates the orderbook |

When a single operation touches storage owned by multiple users, each storage creation or deletion MUST be attributed independently using this table.

## Crediting Deleted Storage

When a StablecoinDEX operation changes reusable DEX storage owned by user `U` from nonzero to zero:

1. Apply the normal DEX state transition.
2. Mint the StablecoinDEX TIP-1060 token according to TIP-1060.
3. Increment `dex_storage_credits[U]` by one, saturating at `type(uint64).max`.

Deleting a reusable slot credits exactly one DEX storage credit to the storage owner, regardless of the slot key, deleted value, transaction sender, or call path.

## Charging Created Storage

When a StablecoinDEX operation changes reusable DEX storage owned by user `U` from zero to nonzero:

1. If `dex_storage_credits[U] > 0`:
   - Decrement `dex_storage_credits[U]` by one.
   - Execute the storage creation in TIP-1060 consume-token mode so one available StablecoinDEX TIP-1060 token is consumed instead of charging the TIP-1000 storage creation component.
2. Otherwise:
   - Leave `dex_storage_credits[U]` unchanged.
   - Execute the storage creation in TIP-1060 preserve-token mode so the user pays the full TIP-1000-mandated storage creation cost and the StablecoinDEX TIP-1060 token balance is not consumed.

The DEX storage credit debit, TIP-1060 mode selection, TIP-1060 token debit if any, and DEX storage write MUST be atomic. If the operation reverts, all credit and token changes made in the same execution scope MUST revert according to normal EVM revert semantics.

## StablecoinDEX Operation Rules

Order placement and flip-order placement MUST attribute new order records, client order ID mappings, newly initialized tick-level records, newly set tick bitmap words, and newly created orderbook records to the order maker.

Order cancellation and full order fills MUST attribute deleted order records to the order maker. If the deletion empties a tick level or clears a tick bitmap word, the corresponding deleted storage is also attributed to the maker of the order whose removal caused the deletion.

Partial fills MUST NOT create or delete DEX storage credits unless they also cause a reusable DEX storage slot to transition between zero and nonzero.

Internal balance updates MUST attribute a newly nonzero or newly zero balance slot to the balance owner. Balance changes that remain nonzero do not create or delete DEX storage credits.

TIP-1056 flip-in-place behavior does not delete and recreate the order record when a flip succeeds. A successful in-place flip therefore MUST NOT credit or charge DEX storage credits for the order record itself, though it may credit or charge storage for tick-level or bitmap changes caused by moving the order.

## Query Interface

The StablecoinDEX interface MUST expose:

```solidity
/// @notice Returns the number of reusable DEX storage credits owned by a user.
/// @param user The user whose DEX storage credit balance is queried.
/// @return credits The number of storage credits available to that user.
function storageCredits(address user) external view returns (uint64 credits);
```

This function is read-only and does not expose any ability to transfer, mint, burn, or set storage credits.

## Compatibility

This TIP changes gas costs for StablecoinDEX calls that create reusable DEX storage. A user who has previously freed DEX storage may pay less for later DEX storage creation. A user without credits still pays the TIP-1000 storage creation cost for net-new DEX storage.

Existing DEX functional behavior, matching behavior, order priority, balances, and events are unchanged except for the new `storageCredits(address)` view.

# Invariants

1. A user's DEX storage credit balance increases only when reusable StablecoinDEX storage attributed to that user changes from nonzero to zero.
2. A user's DEX storage credit balance decreases only when reusable StablecoinDEX storage attributed to that user changes from zero to nonzero and a StablecoinDEX TIP-1060 token is consumed.
3. A DEX storage credit can offset at most one reusable DEX storage creation.
4. DEX storage credits are not transferable between users.
5. StablecoinDEX TIP-1060 tokens cannot be consumed for a user who has no DEX storage credits.
6. Storage credit accounting changes revert with the execution scope that caused them.
7. Creating or updating `dex_storage_credits` counters does not mint or consume TIP-1060 tokens and does not incur the 250,000 gas TIP-10-mandated storage creation cost.
8. Ordinary smart contracts, TIP-20 token state, and non-DEX precompile state do not mint or consume DEX storage credits under this TIP.
