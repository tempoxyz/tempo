# TIP-1039: Native Memory Gas for Dynamic Storage Types

## Abstract

Tempo precompiles are implemented in native Rust, bypassing the EVM's linear memory model. When a precompile reads a dynamic storage type (`Vec`, `String`, `Bytes`, `Set`), each storage slot read is individually metered (SLOAD gas), but the Rust heap allocation required to materialize the data is not. This TIP introduces a per-load gas charge on dynamic storage primitives using the EVM's quadratic memory expansion formula, with the first word free.

## Motivation

In standard EVM execution, contracts pay memory expansion gas (`3 * words + words² / 512`) when growing the return buffer. This cost is inherently incurred because all data passes through EVM linear memory. Tempo's native precompiles skip this entirely — they allocate Rust `Vec`s, `String`s, and `Set`s directly on the heap, with no gas charged for the allocation itself.

This creates an asymmetry: an attacker can force a node to perform large native memory allocations by calling view functions that return unbounded dynamic types (e.g., `getValidators()`, `getBookKeys()`), paying only SLOAD gas while the node bears the full cost of heap allocation, element deserialization, and ABI encoding.

The attack surface is particularly relevant for `eth_call` / RPC simulation, where calls are unpaid and operators may allow generous gas caps.

### Why charge at the storage primitive level

Several alternatives were considered and rejected:

1. **Output-word charging** (charge based on return data size): By the time the output size is known, the native memory is already allocated and the work is done. An attacker who OOGs at the output-charging step still forced the node to do all the allocation and compute.

2. **Pagination requirements**: Requiring all view functions to paginate adds API complexity, breaks existing interfaces, and pushes the mitigation burden onto every precompile implementer rather than solving it generically.

3. **Cumulative memory tracking** (a running word counter across all dynamic loads in a call): This penalizes precompiles that read several small dynamic types (e.g., `name()` + `symbol()` + small mapping lookups) as if they were one large allocation. In the EVM, cumulative tracking is correct because there is one contiguous memory buffer. In native Rust, each `Vec`/`String` is an independent heap allocation — the risk is per-allocation, not cumulative.

The correct chokepoint is the dynamic storage load functions themselves (`Vec::load`, `String::load`, `Bytes::load`). These are the exact points where native memory grows proportionally to stored data, and where the gas charge can be applied **before** the allocation occurs.

### Why quadratic

A linear per-word cost undercharges large allocations relative to the real resource cost. The EVM's quadratic memory formula (`words² / 512`) has a well-understood property: the marginal cost of each additional word increases, making unbounded growth superlinearly expensive. Applying the same formula per-load ensures that small reads (the common case) are nearly free, while genuinely large materializations become prohibitively expensive without requiring explicit caps or pagination.

## Assumptions

- The EVM memory expansion formula (`3 * words + words² / 512`) is an appropriate model for pricing native heap allocations. This is conservative: EVM memory expansion costs include contiguous allocation guarantees that the system allocator does not require.
- Precompile callers sending fixed, bounded dynamic types (e.g., an `Address[]` with 2 elements) are not meaningfully affected by the new cost.
- The first word being free means all short strings (≤31 bytes, stored inline) and single-element reads incur zero additional gas.

# Specification

## Native Memory Cost Function

A new gas cost function is introduced for native memory materialization:

```
native_mem_cost(words) =
    if words <= 1: 0
    else: words² / 512
```

Where `words = ceil(byte_length / 32)` is the number of 32-byte words required to hold the materialized data.

### Why quadratic-only (no linear component)

The EVM memory expansion formula is `3 * words + words² / 512`. The linear component (`3 * words`) prices the per-word cost of zeroing, addressing, and operating on contiguous EVM memory (MLOAD/MSTORE). In the precompile context, this per-word work is already fully covered by SLOAD gas — each word read from storage costs at minimum 100 gas (warm). Adding a linear component would double-charge for work that is already metered.

What is *not* metered is the superlinear resource cost of large native heap allocations: allocator pressure, ABI encoding buffer construction, and potential for memory exhaustion. The quadratic term alone (`words² / 512`) provides this — it is negligible for normal-sized reads but becomes dominant for large materializations, which is exactly the DoS surface we are closing.

### First-word exemption

When `words == 1`, the cost is 0. This ensures:
- Short strings (≤31 bytes, inline storage) cost 0 additional gas
- Single-element primitive reads cost 0 additional gas
- The charge activates only when multi-word dynamic data is materialized

## Application Points

The `native_mem_cost` charge is applied inside each dynamic `Storable::load` implementation, **after** reading the length from storage but **before** allocating native memory. If the charge causes an out-of-gas condition, the load aborts before any heap allocation occurs.

### `Vec<T>::load`

After reading the length slot and validating it, compute:

```
words = ceil(length * T::BYTES / 32)
deduct_gas(native_mem_cost(words))
```

Where `T::BYTES` is the byte size of each element and `length` is the number of elements.

### `String::load` / `Bytes::load`

After reading the base slot and computing the string length:

```
words = ceil(byte_length / 32)
deduct_gas(native_mem_cost(words))
```

For short strings (≤31 bytes), `words ≤ 1`, so `native_mem_cost = 0`.

### `Set<T>::load`

`Set<T>` delegates to `Vec<T>::load` internally, so it inherits the charge automatically.

## Cost Examples

| Operation | Data Size | Words | `native_mem_cost` | Notes |
|-----------|-----------|-------|-------------------|-------|
| `name()` → 6-byte string | 6 B | 1 | **0** | Single word, exempt |
| `symbol()` → 4-byte string | 4 B | 1 | **0** | Single word, exempt |
| `Vec<Address>` × 10 | 200 B | 7 | **0** | 49/512 rounds to 0 |
| `Vec<Address>` × 100 | 2,000 B | 63 | **7** | Negligible |
| `Vec<Validator>` × 100 | ~20 KB | 625 | **762** | Quadratic becoming visible |
| `Vec<Validator>` × 1,000 | ~200 KB | 6,250 | **76,293** | Quadratic dominant |
| `Vec<B256>` × 10,000 | ~312 KB | 10,000 | **195,312** | Prohibitively expensive |
| `Vec<u8>` × 1,000,000 | ~1 MB | 31,250 | **1,907,348** | Effectively DoS-proof |

For comparison, reading 10,000 warm storage slots costs 1,000,000 gas in SLOAD alone. The native memory charge adds ~19.5% overhead at this scale — proportional to the real resource cost, without overcharging on the linear component that SLOAD already covers.

## Gas Parameter

The quadratic divisor (512) matches the EVM memory expansion formula. This constant can be adjusted via `GasParams` in future hardforks if empirical data suggests different tuning.

## Scope

This charge applies **only** to the dynamic storage types defined in the precompile storage layer:
- `Vec<T>` (`crates/precompiles/src/storage/types/vec.rs`)
- `String` (`crates/precompiles/src/storage/types/bytes_like.rs`)
- `Bytes` (`crates/precompiles/src/storage/types/bytes_like.rs`)
- `Set<T>` (`crates/precompiles/src/storage/types/set.rs`, via `Vec<T>`)

Fixed-size types (`Mapping`, `Slot<T>`, `u64`, `Address`, etc.) are not affected.

# Invariants

1. **Charge-before-allocate**: The gas deduction MUST occur before any heap allocation for the dynamic data. If `deduct_gas` returns `OutOfGas`, no `Vec::new()`, `Vec::with_capacity()`, or equivalent allocation has been performed.

2. **Per-load independence**: Each dynamic load charges independently. Loading `name()` (1 word) followed by `get_validators()` (625 words) charges `0 + 762 = 762 gas`. There is no cumulative memory counter.

3. **First-word exemption**: Any dynamic load where `words == 1` MUST incur zero native memory gas. When `words > 1`, the full `words` value is used in the quadratic formula (i.e., `words² / 512`, not `(words - 1)² / 512`).

4. **SLOAD orthogonality**: The native memory charge is additive to, and independent of, SLOAD gas. A `Vec<T>::load` of `n` elements pays both the per-slot SLOAD gas (warm/cold) for each storage read AND the `native_mem_cost` for the materialized data.

5. **Write-path exclusion**: `Vec::store`, `String::store`, and `Set::write` do NOT incur native memory charges. The write path serializes from an already-allocated Rust value into storage slots — the SSTORE gas already covers this direction.

6. **Backward compatibility**: All existing precompile view functions continue to work without API changes. The only observable difference is increased gas consumption for calls that return large dynamic types.
