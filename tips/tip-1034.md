# TIP-1034: Implicit Approval List

## Abstract

This TIP introduces a protocol-level Implicit Approval List: an explicit, enumerated set of contract addresses that are allowed to use `system_transfer_from()` to transfer TIP-20 tokens from `msg.sender` without requiring a prior `approve()` call. For any address on this list, `approve()` and `permit()` with nonzero amounts do not affect allowance state and do not emit an `Approval` event. `approve()` and `permit()` with `amount = 0` delete any existing allowance slot, allowing users to clean up stale pre-fork approvals.

The initial list consists of:

| Address | Contract |
|---------|----------|
| `0xfeEC000000000000000000000000000000000000` | TipFeeManager (FeeAMM) |
| `0xDEc0000000000000000000000000000000000000` | StablecoinDEX |
| `0x33b901018174DDabE4841042ab76ba85D4e24f25` | MPP Channels |

This TIP supersedes [TIP-1025](./tip-1025.md), which defined implicit approval behavior for the StablecoinDEX and TipFeeManager individually. TIP-1034 generalizes that into a named, extensible list, makes the behavioral specification self-contained, and adds the MPP Channels contract as the first non-precompile member.

## Motivation

### Removing redundant approvals

Today, a user who wants to interact with the StablecoinDEX or MPP Channels must first submit a separate `approve` transaction for each token. This has three costs:

1. **User experience**: The first interaction with a given token requires two transactions (or a batch call). Wallets must prompt for approval before every new token interaction, adding friction.
2. **Gas cost**: Each `approve` call costs ~250,000 gas for the cold SSTORE to the allowance slot. Every subsequent operation pays extra gas to load, check, and update the allowance.
3. **State bloat**: Every approval writes a storage slot in the token's allowance mapping.

The approval check is redundant for contracts that only pull tokens from `msg.sender` — the caller is always the one authorizing the transaction, so the `approve` step provides no additional security.

Rather than adding implicit approval behavior ad-hoc to individual TIPs, the protocol should maintain an explicit list of approved addresses. This makes the set visible and auditable in one place, and simplifies future additions.

### Spending limit enforcement

Prior to this TIP, the `_transfer_from` path used by the allowance-based flow does not call `check_and_update_spending_limit`. This means access keys can bypass their spending limits by routing transfers through a contract the user has approved. This TIP intentionally closes that gap: `system_transfer_from` enforces AccountKeychain spending limits on every transfer, so all token pulls through contracts on the Implicit Approval List are subject to spending caps.

### Adding the MPP Channels contract

The MPP Channels contract (`0x33b901018174DDabE4841042ab76ba85D4e24f25`) uses `transferFrom` on the caller to escrow funds into payment channels. It only ever pulls from `msg.sender` and is a natural candidate for implicit approval.

## Assumptions

- Contracts on the Implicit Approval List SHOULD only call `transferFrom` (or the equivalent system call) on `msg.sender` for the current call. Contracts that pull tokens from addresses other than the direct caller MUST NOT be added to this list, as doing so would allow them to move tokens without the owner's transaction-level consent.

# Specification

## system_transfer_from

`system_transfer_from(from, to, amount)` is a privileged transfer function on the TIP-20 precompile. It behaves identically to a standard `transfer(from, to, amount)` — debiting the sender, crediting the recipient, emitting a `Transfer` event — but skips the allowance check. Specifically:

1. The TIP-20 precompile checks that `msg.sender` (the direct caller) is on the Implicit Approval List. If not, the call reverts.
2. The precompile checks that the caller was reached via `CALL`, not `DELEGATECALL`. If the calling context is a delegatecall, the call reverts. This prevents a listed contract's privilege from being inherited by untrusted code via delegatecall (see [Security: Non-precompile contracts](#non-precompile-contracts)).
3. TIP-403 transfer policies are enforced via `ensure_transfer_authorized`. If the token's transfer policy forbids the transfer, the call reverts.
4. AccountKeychain spending limits are enforced via `check_and_update_spending_limit`. If `from` is an access key and the transfer would exceed its spending cap, the call reverts.
5. The balance is debited from `from` and credited to `to`. If `from` has insufficient balance, the call reverts with `InsufficientBalance`.
6. A standard TIP-20 `Transfer(from, to, amount)` event is emitted.

## Implicit Approval List

The protocol maintains an explicit set of addresses called the Implicit Approval List. For any address on this list:

1. The contract MAY use `system_transfer_from()` to transfer TIP-20 tokens from `msg.sender` without a prior `approve()` call.
2. `approve(spender, amount)` where `spender` is on the list and `amount > 0`: the call succeeds but does not affect allowance state, does not emit an `Approval` event, and does not deduct AccountKeychain spending limits.
3. `approve(spender, 0)` where `spender` is on the list: the call succeeds and deletes any existing allowance slot for that owner/spender pair. This allows users to clean up stale pre-fork approvals.
4. `permit()` where `spender` is on the list and `amount > 0`: the call succeeds and increments the owner's nonce, but does not affect allowance state and does not emit an `Approval` event. The nonce increment keeps wallet nonce tracking in sync with the on-chain state.
5. `permit()` where `spender` is on the list and `amount = 0`: the call succeeds, increments the owner's nonce, and deletes any existing allowance slot for that owner/spender pair.
6. All other TIP-20 semantics (transfer policies via TIP-403, spending limit enforcement, `Transfer` event emission) remain unchanged.

## Pre-existing Approvals

Users who called `approve()` on a listed address before the fork activation may have stale allowance state in storage. Post-fork:

- These stale allowances are never consumed, since the contracts use `system_transfer_from` instead of the allowance-based `transferFrom`.
- Users can clean up stale slots by calling `approve(spender, 0)` or submitting a `permit()` with `amount = 0` (see rules 3 and 5 above).
- Stale slots that are not cleaned up are inert and have no effect on protocol behavior.

For access keys that previously had spending limits deducted by a pre-fork `approve()` call to a listed address: there is no mechanism to refund that consumed spending limit. This is expected to be rare in practice, since access keys would not typically pre-approve these contracts without immediately spending.

## Eligibility Criteria

A contract SHOULD only be added to the Implicit Approval List if it satisfies the following:

1. The contract only calls `transferFrom` (or `system_transfer_from`) on `msg.sender` for the current call. There MUST be no reachable code path where the contract pulls tokens from an address other than the direct caller.
2. The contract enforces direct-call-only semantics (no `DELEGATECALL`), so `msg.sender` cannot be spoofed by an intermediate contract. For precompiles, this is enforced by the runtime. For non-precompile contracts, the protocol enforces this via the delegatecall check in `system_transfer_from` (see below).
3. The contract MUST NOT be an upgradeable proxy. If the implementation behind a proxy is changed to malicious code, that code would inherit the `system_transfer_from` privilege. Non-precompile entries on the list must be immutable deployed contracts.

## Future Amendments

Additional contracts may be added to the Implicit Approval List via future TIPs that reference and amend this one. Each addition must include a safety argument demonstrating that the contract only pulls from `msg.sender`.

# Security

## Non-precompile contracts

The FeeAMM and StablecoinDEX are precompiles — their code is part of the node implementation and cannot be modified or delegatecalled into by other contracts. The MPP Channels contract is a regular deployed contract, which introduces two additional risk vectors:

### DELEGATECALL from a listed contract

If a listed contract delegatecalls into external code, that code runs with the listed contract's address and storage. A subsequent call to `system_transfer_from` would pass the Implicit Approval List check (since `msg.sender` is still the listed contract's address), allowing the external code to transfer tokens from arbitrary addresses.

This TIP mitigates this by requiring `system_transfer_from` to verify that the caller was reached via `CALL`, not `DELEGATECALL`. The TIP-20 precompile has access to the execution context in the node implementation and can inspect whether the calling frame is a delegatecall context. If it is, `system_transfer_from` MUST revert.

This is a belt-and-suspenders check: the eligibility criteria already require that listed contracts do not delegatecall to untrusted code, but the protocol-level check ensures that even a buggy or compromised contract cannot abuse the privilege via delegatecall.

### DELEGATECALL into a listed contract

If an unlisted contract B delegatecalls into the MPP Channels contract's code, the MPP Channels code runs in B's context (B's address, B's storage). Any call to `system_transfer_from` from that context would have `msg.sender = B`, which is not on the Implicit Approval List. The call would fail. This case is safe without additional checks.

### Upgradeability

If a listed contract were an upgradeable proxy, a malicious upgrade could replace the implementation with code that calls `system_transfer_from` with arbitrary `from` addresses. The delegatecall check mitigates this partially (the proxy would be delegatecalling to the new implementation), but to be safe, non-precompile entries MUST NOT be upgradeable proxies. The MPP Channels contract at `0x33b901018174DDabE4841042ab76ba85D4e24f25` is an immutable deployed contract.

# Invariants

1. **Caller-only pulls**: Every contract on the Implicit Approval List MUST only pull tokens from the direct caller (`msg.sender`). No external entry point may pass a different address to `system_transfer_from` for wallet-level token pulls.
2. **Implicit Approval List check**: `system_transfer_from` MUST revert if `msg.sender` is not on the Implicit Approval List.
3. **Delegatecall check**: `system_transfer_from` MUST revert if the calling frame is a delegatecall context.
4. **Approve does not grant allowance**: `approve(spender, amount)` where `spender` is on the Implicit Approval List and `amount > 0` MUST succeed, MUST NOT affect allowance state, MUST NOT emit an `Approval` event, and MUST NOT deduct access key spending limits.
5. **Zero-amount clears stale state**: `approve(spender, 0)` and `permit()` with `amount = 0` where `spender` is on the Implicit Approval List MUST delete any existing allowance slot for that owner/spender pair.
6. **Permit does not grant allowance**: `permit()` where `spender` is on the Implicit Approval List and `amount > 0` MUST succeed, MUST increment the owner's nonce, MUST NOT affect allowance state, and MUST NOT emit an `Approval` event.
7. **Policy enforcement**: All token pulls via `system_transfer_from` MUST enforce TIP-403 transfer policies via `ensure_transfer_authorized`.
8. **Spending limit enforcement**: All token pulls via `system_transfer_from` MUST enforce AccountKeychain spending limits via `check_and_update_spending_limit`.
9. **Event consistency**: All token pulls via `system_transfer_from` MUST emit the standard TIP-20 `Transfer` event.
10. **List consistency**: The on-chain Implicit Approval List MUST match the set defined in this TIP (and any future amendments). No address may be implicitly approved without a corresponding TIP.
11. **Hardfork gating**: The behavior change MUST be gated behind the hardfork activation. Pre-fork transactions must continue to use the allowance-based path to preserve state root consistency.
