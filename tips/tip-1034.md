# TIP-1034: Implicit Approval List

## Abstract

This TIP introduces a protocol-level Implicit Approval List: an explicit, enumerated set of contract addresses that are allowed to use `system_transfer_from()` to transfer TIP-20 tokens from `msg.sender` without a prior `approve()` call. For any address on this list, both `approve()` and `permit()` targeting that address are no-ops.

The initial list consists of:

| Address | Contract |
|---------|----------|
| `0xfeEC000000000000000000000000000000000000` | TipFeeManager (FeeAMM) |
| `0xDEc0000000000000000000000000000000000000` | StablecoinDEX |
| `0x33b901018174DDabE4841042ab76ba85D4e24f25` | MPP Channels |

This TIP supersedes [TIP-1025](./tip-1025.md), which defined implicit approval behavior for the StablecoinDEX and TipFeeManager individually. TIP-1034 generalizes that into a named, extensible list and adds the MPP Channels contract as the first non-precompile member.

## Motivation

### Removing redundant approvals

[TIP-1025](./tip-1025.md) demonstrated that system contracts which only pull tokens from `msg.sender` gain significant UX and gas improvements from bypassing the TIP-20 approval flow. The approval check is redundant for these contracts because the caller is always the one authorizing the transaction — the `approve` step provides no additional security.

Rather than adding implicit approval behavior ad-hoc to individual TIPs each time a new qualifying contract is introduced, the protocol should maintain an explicit list of approved addresses. This makes the set of implicitly approved contracts visible and auditable in one place, and simplifies future additions.

### Spending limit enforcement

Prior to this TIP, the `_transfer_from` path used by the allowance-based flow does not call `check_and_update_spending_limit`. This means access keys can bypass their spending limits by routing transfers through a contract the user has approved. This TIP intentionally closes that gap: `system_transfer_from` enforces AccountKeychain spending limits on every transfer, so all token pulls through contracts on the Implicit Approval List are subject to spending caps.

### Adding the MPP Channels contract

The MPP Channels contract (`0x33b901018174DDabE4841042ab76ba85D4e24f25`) uses `transferFrom` on the caller to escrow funds into payment channels. It only ever pulls from `msg.sender` and is a natural candidate for implicit approval.

## Assumptions

- The core behavior of `system_transfer_from` — skipping the allowance check while preserving transfer policy enforcement (TIP-403) and `Transfer` event emission — is as defined in [TIP-1025](./tip-1025.md).
- Contracts on the Implicit Approval List SHOULD only call `transferFrom` (or the equivalent system call) on `msg.sender` for the current call. Contracts that pull tokens from addresses other than the direct caller MUST NOT be added to this list, as doing so would allow them to move tokens without the owner's transaction-level consent.

# Specification

## Implicit Approval List

The protocol maintains an explicit set of addresses called the Implicit Approval List. For any address on this list:

1. The contract MAY use `system_transfer_from()` to transfer TIP-20 tokens from `msg.sender` without a prior `approve()` call.
2. `approve(spender, amount)` where `spender` is on the list is a no-op: the call succeeds, writes no allowance state, emits no `Approval` event, and does not deduct AccountKeychain spending limits. This applies regardless of `amount`, including `amount = 0`.
3. `permit()` targeting an address on the list is also a no-op: the call succeeds, writes no allowance or nonce state, and emits no `Approval` event.
4. All other TIP-20 semantics (transfer policies via TIP-403, spending limit enforcement, `Transfer` event emission) remain unchanged.

## Initial List

| Address | Contract | Rationale |
|---------|----------|-----------|
| `0xfeEC000000000000000000000000000000000000` | TipFeeManager (FeeAMM) | Already uses `system_transfer_from`; only pulls from `msg.sender` via `rebalanceSwap` and `mint` |
| `0xDEc0000000000000000000000000000000000000` | StablecoinDEX | Only pulls from `msg.sender` via `place`, `placeFlip`, `swapExactAmountIn`, `swapExactAmountOut` |
| `0x33b901018174DDabE4841042ab76ba85D4e24f25` | MPP Channels | Only pulls from `msg.sender` to escrow funds into payment channels |

## Pre-existing Approvals

Users who called `approve()` on a listed address before the fork activation may have stale allowance state in storage. Post-fork:

- These stale allowances are never consumed, since the contracts use `system_transfer_from` instead of the allowance-based `transferFrom`.
- Calling `approve(spender, 0)` to clear an existing allowance is a no-op (per rule 2 above) and does not delete the stale slot.
- The stale slots are inert and have no effect on protocol behavior. A future state cleanup TIP may address removal of these slots if state bloat is a concern.

For access keys that previously had spending limits deducted by a pre-fork `approve()` call to a listed address: there is no mechanism to refund that consumed spending limit. This is expected to be rare in practice, since access keys would not typically pre-approve these contracts without immediately spending.

## Eligibility Criteria

A contract SHOULD only be added to the Implicit Approval List if it satisfies the following:

1. The contract only calls `transferFrom` (or `system_transfer_from`) on `msg.sender` for the current call. There MUST be no reachable code path where the contract pulls tokens from an address other than the direct caller.
2. The contract enforces direct-call-only semantics (no `DELEGATECALL`), so `msg.sender` cannot be spoofed by an intermediate contract.

## Future Amendments

Additional contracts may be added to the Implicit Approval List via future TIPs that reference and amend this one. Each addition must include a safety argument demonstrating that the contract only pulls from `msg.sender`.

# Invariants

1. **Caller-only pulls**: Every contract on the Implicit Approval List MUST only pull tokens from the direct caller (`msg.sender`). No external entry point may pass a different address to `system_transfer_from` for wallet-level token pulls.
2. **No-op approve and permit**: `approve(spender, amount)` and `permit()` where `spender` is on the Implicit Approval List MUST succeed, write no allowance state, emit no `Approval` event, and not deduct access key spending limits.
3. **Policy enforcement**: All token pulls MUST continue to enforce TIP-403 transfer policies via `ensure_transfer_authorized`.
4. **Spending limit enforcement**: All token pulls via `system_transfer_from` MUST enforce AccountKeychain spending limits via `check_and_update_spending_limit`.
5. **Event consistency**: All token pulls MUST emit the standard TIP-20 `Transfer` event.
6. **List consistency**: The on-chain Implicit Approval List MUST match the set defined in this TIP (and any future amendments). No address may be implicitly approved without a corresponding TIP.
7. **Hardfork gating**: The behavior change MUST be gated behind the hardfork activation. Pre-fork transactions must continue to use the allowance-based path to preserve state root consistency.
