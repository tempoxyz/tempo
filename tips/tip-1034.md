# TIP-1034: Implicit Approval List

## Abstract

This TIP introduces a protocol-level Implicit Approval List: an explicit, enumerated set of contract addresses that may transfer TIP-20 tokens from `msg.sender` without requiring a prior `approve()` call. For any address on this list, `approve()` and `permit()` with nonzero amounts do not affect allowance state and do not emit an `Approval` event. `approve()` and `permit()` with `amount = 0` delete any existing allowance slot, allowing users to clean up stale pre-fork approvals.

The initial list consists of:

| Address | Contract |
|---------|----------|
| `0xfeEC000000000000000000000000000000000000` | TipFeeManager (FeeAMM) |
| `0xDEc0000000000000000000000000000000000000` | StablecoinDEX |
| `0x33b901018174DDabE4841042ab76ba85D4e24f25` | MPP Channels |

This TIP supersedes [TIP-1025](./tip-1025.md), which defined implicit approval behavior for the StablecoinDEX and TipFeeManager individually. TIP-1034 generalizes that into a named, extensible list, makes the behavioral specification self-contained, and adds the MPP Channels contract as the first non-precompile member.

## Motivation

### Removing redundant approvals

Today, a user who wants to interact with the StablecoinDEX or MPP Channels must first submit a separate `approve` transaction for each token. This has three costs:

1. **User experience**: The first interaction with a given token requires two transactions (or a batch call). Wallets must prompt for approval before every new token interaction, adding friction.
2. **Gas cost**: Each `approve` call costs ~250,000 gas for the cold SSTORE to the allowance slot. Every subsequent operation pays extra gas to load, check, and update the allowance.
3. **State bloat**: Every approval writes a storage slot in the token's allowance mapping.

The approval check is redundant for contracts that only pull tokens from `msg.sender` — the caller is always the one authorizing the transaction, so the `approve` step provides no additional security.

Rather than adding implicit approval behavior ad-hoc to individual TIPs, the protocol should maintain an explicit list of approved addresses. This makes the set visible and auditable in one place, and simplifies future additions.

### Spending limit enforcement

Prior to this TIP, the `_transfer_from` path used by the allowance-based flow does not call `check_and_update_spending_limit`. This means access keys can bypass their spending limits by routing transfers through a contract the user has approved. This TIP intentionally closes that gap: all token pulls through contracts on the Implicit Approval List enforce AccountKeychain spending limits.

### Adding the MPP Channels contract

The MPP Channels contract (`0x33b901018174DDabE4841042ab76ba85D4e24f25`) uses `transferFrom` on the caller to escrow funds into payment channels. It only ever pulls from `msg.sender` and is a natural candidate for implicit approval.

## Assumptions

- Contracts on the Implicit Approval List SHOULD only call `transferFrom` (or the equivalent system call) on `msg.sender` for the current call. Contracts that pull tokens from addresses other than the direct caller MUST NOT be added to this list, as doing so would allow them to move tokens without the owner's transaction-level consent.

# Specification

## Implicit Approval List

The protocol maintains an explicit set of addresses called the Implicit Approval List. For any address on this list:

1. The contract may transfer TIP-20 tokens from `msg.sender` without a prior `approve()` call. The mechanism depends on whether the contract is a precompile or a regular contract (see [Implementation](#implementation)).
2. `approve(spender, amount)` where `spender` is on the list and `amount > 0`: the call succeeds but does not affect allowance state, does not emit an `Approval` event, and does not deduct AccountKeychain spending limits.
3. `approve(spender, 0)` where `spender` is on the list: the call succeeds and deletes any existing allowance slot for that owner/spender pair. This allows users to clean up stale pre-fork approvals.
4. `permit()` where `spender` is on the list and `amount > 0`: the call succeeds and increments the owner's nonce, but does not affect allowance state and does not emit an `Approval` event. The nonce increment keeps wallet nonce tracking in sync with the on-chain state.
5. `permit()` where `spender` is on the list and `amount = 0`: the call succeeds, increments the owner's nonce, and deletes any existing allowance slot for that owner/spender pair.
6. All other TIP-20 semantics (transfer policies via TIP-403, spending limit enforcement, `Transfer` event emission) remain unchanged.

## Implementation

The mechanism for bypassing approval differs depending on whether the listed contract is a precompile or a regular deployed contract:

### Precompiles

Implicitly-approved precompiles (TipFeeManager, StablecoinDEX) MUST use `system_transfer_from()` to pull tokens. `system_transfer_from` is a privileged function on the TIP-20 precompile that performs a transfer without checking the allowance. It:

1. Checks that `msg.sender` (the direct caller) is on the Implicit Approval List. If not, the call reverts.
2. Enforces TIP-403 transfer policies via `ensure_transfer_authorized`.
3. Enforces AccountKeychain spending limits via `check_and_update_spending_limit`.
4. Debits `from` and credits `to`. Reverts with `InsufficientBalance` if `from` has insufficient balance.
5. Emits a standard TIP-20 `Transfer(from, to, amount)` event.

### Non-precompile contracts

Non-precompile contracts on the Implicit Approval List (such as MPP Channels) are already deployed and call the standard `transferFrom`. For these contracts, TIP-20's `transferFrom` itself checks whether `msg.sender` is on the Implicit Approval List. If so, the allowance check is skipped — the transfer proceeds as if the caller had unlimited approval. All other checks (balance, transfer policies, spending limits, events) still apply.

This means non-precompile contracts do not need to be modified to benefit from implicit approval. The check is in the TIP-20 precompile's `transferFrom` implementation.

### Security model

In both paths, `transferFrom` and `system_transfer_from` do not restrict the `from` parameter — a listed contract could pass any address, not just `msg.sender`. The caller-only-pulls invariant is enforced by the contract's own code, not by the implicit approval system. The security of this TIP depends entirely on the right contracts being on the list: contracts that have been verified to only pull from `msg.sender` and that satisfy the eligibility criteria below. The implicit approval system itself provides no protection against a listed contract that violates this invariant.

## Pre-existing Approvals

Users who called `approve()` on a listed address before the fork activation may have stale allowance state in storage. Post-fork:

- These stale allowances are never consumed, since the allowance check is bypassed for listed addresses.
- Users can clean up stale slots by calling `approve(spender, 0)` or submitting a `permit()` with `amount = 0` (see rules 3 and 5 above).
- Stale slots that are not cleaned up are inert and have no effect on protocol behavior.

For access keys that previously had spending limits deducted by a pre-fork `approve()` call to a listed address: there is no mechanism to refund that consumed spending limit. This is expected to be rare in practice, since access keys would not typically pre-approve these contracts without immediately spending.

## Eligibility Criteria

A contract SHOULD only be added to the Implicit Approval List if it satisfies the following:

1. **Caller-only pulls**: The contract only calls `transferFrom` on `msg.sender` for the current call. There MUST be no reachable code path where the contract pulls tokens from an address other than the direct caller.
2. **No DELEGATECALL**: The contract MUST NOT contain any `DELEGATECALL` opcodes. If a listed contract were to delegatecall into external code, that code would run with the listed contract's address as `address(this)`. A subsequent call to `transferFrom` on a TIP-20 token would have `msg.sender` equal to the listed contract's address, inheriting the implicit approval privilege. This would allow the external code to call `transferFrom` with an arbitrary `from` address and bypass the allowance check.
3. **No proxy pattern**: The contract MUST NOT be an upgradeable proxy. An upgradeable proxy delegatecalls to its implementation, so a malicious upgrade would inherit the implicit approval privilege (same mechanism as above).
4. **Immutable code**: For non-precompile contracts, the deployed bytecode must be immutable. There must be no mechanism (e.g., `SELFDESTRUCT` + redeployment, metamorphic contracts) by which the code at the listed address could change.

For precompiles, criteria 2–4 are inherently satisfied: precompile code is part of the node implementation and is not executed via EVM opcodes.

## Integrator Guidance

Implicitly-approved precompiles (TipFeeManager, StablecoinDEX) are part of the protocol and integrators MAY assume they will remain implicitly approved indefinitely.

Implicitly-approved deployed contracts (such as MPP Channels) carry no such guarantee. The Implicit Approval List may be amended at any hardfork, and a deployed contract may be removed from the list — for example, if it is replaced by a new deployment or promoted to a precompile. Integrators MUST NOT assume that a deployed contract will remain implicitly approved.

Non-upgradeable integrations that interact with an implicitly-approved deployed contract SHOULD continue to call `approve()` before `transferFrom`. The `approve()` call is a no-op while the contract is on the list, but ensures the integration continues to work if the contract is removed in a future hardfork. Upgradeable integrations can omit the `approve()` call and adapt if the list changes.

## Future Amendments

Additional contracts may be added to the Implicit Approval List via future TIPs that reference and amend this one. Each addition must include a safety argument demonstrating that the contract satisfies the eligibility criteria above.

# Invariants

1. **Caller-only pulls**: Every contract on the Implicit Approval List MUST only pull tokens from the direct caller (`msg.sender`). No external entry point may pass a different address as the `from` parameter for wallet-level token pulls.
2. **Approve does not grant allowance**: `approve(spender, amount)` where `spender` is on the Implicit Approval List and `amount > 0` MUST succeed, MUST NOT affect allowance state, MUST NOT emit an `Approval` event, and MUST NOT deduct access key spending limits.
3. **Zero-amount clears stale state**: `approve(spender, 0)` and `permit()` with `amount = 0` where `spender` is on the Implicit Approval List MUST delete any existing allowance slot for that owner/spender pair.
4. **Permit does not grant allowance**: `permit()` where `spender` is on the Implicit Approval List and `amount > 0` MUST succeed, MUST increment the owner's nonce, MUST NOT affect allowance state, and MUST NOT emit an `Approval` event.
5. **Policy enforcement**: All token pulls MUST enforce TIP-403 transfer policies via `ensure_transfer_authorized`.
6. **Spending limit enforcement**: All token pulls MUST enforce AccountKeychain spending limits via `check_and_update_spending_limit`.
7. **Event consistency**: All token pulls MUST emit the standard TIP-20 `Transfer` event.
8. **List consistency**: The on-chain Implicit Approval List MUST match the set defined in this TIP (and any future amendments). No address may be implicitly approved without a corresponding TIP.
9. **Hardfork gating**: The behavior change MUST be gated behind the hardfork activation. Pre-fork transactions must continue to use the allowance-based path to preserve state root consistency.
