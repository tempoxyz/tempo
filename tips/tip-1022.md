# TIP-1022: Virtual Addresses for TIP-20 Deposit Forwarding

## Abstract

This TIP introduces **virtual addresses**: a reserved address prefix that, when detected in TIP-20 recipient-bearing operations, causes the precompile to auto-credit a registered master wallet instead of the target address. This eliminates sweep transactions entirely for entities such as exchanges, ramps, and payment processors that generate per-user deposit addresses. Master registration is a one-time on-chain call; deposit address derivation is fully off-chain.

## Motivation

- **Eliminate sweep transactions.** Entities such as exchanges, ramps, and payment processors need to offer each customer a unique deposit address. Today, funds arriving at each address must be swept back to a central wallet in separate transactions, which is an enormous operational cost and burden at scale. Virtual addresses auto-credit the master wallet at the protocol level, making sweeps unnecessary.

- **Avoid the 250,000 gas new-account cost.** Tempo charges 250,000 gas to create state for a new address on first use. With virtual addresses, deposit addresses never create on-chain state, so the first transfer to a new deposit address costs the same as any other transfer.

- **Prevent state bloat.** Without virtual addresses, each customer deposit address creates a new account in the state trie. At enterprise scale (millions of deposit addresses), this is significant and permanent state growth. Virtual addresses avoid this entirely: no accounts are created, regardless of how many deposit addresses a business generates.

# Specification

## Address Layout

Virtual addresses are standard 20-byte EVM addresses with a reserved prefix: 

```
[10-byte PREFIX] [6-byte masterId] [4-byte userTag]
= 20 bytes total
```

| Field | Bytes | Description |
|-------|-------|-------------|
| **PREFIX** | 10 | Fixed prefix `0xFDFDFDFDFDFDFDFDFDFD`. Identifies virtual addresses. |
| **masterId** | 6 | Deterministic identifier derived from `(masterAddress, salt)` via registration PoW hash. Supports up to ~2.8×10¹⁴ masters. |
| **userTag** | 4 | Opaque per-user identifier, derived off-chain. 32-bit space supports >4 billion (2³²) unique deposit addresses per master. |

## Conformance and Scope

TIP-1022 applies only to TIP-20 precompile recipient resolution for the entrypoints listed in **Transfer Path Modification**.

TIP-1022 does **not** alter TIP-20 methods that do not carry a recipient (e.g., `approve`, `burn`, `permit`) and does not alter non-TIP-20 protocol behavior.

Before TIP-1022 activation (`protocolVersion` hardfork), all TIP-20 behavior remains unchanged.

Non-TIP-20 token transfers (e.g., ERC-20 contracts deployed on Tempo) to virtual addresses are **not** subject to TIP-1022 forwarding. Such transfers behave as standard EVM transfers to the literal address. Tokens sent this way may be irrecoverable — see **Risks and Limitations**.

### Prefix Selection: `0xFDFDFDFDFDFDFDFDFDFD`

A 10-byte prefix provides collision resistance of ~1/2⁸⁰. This is an explicit layout tradeoff that prioritizes address space safety over `masterId` and `userTag` space. The 6-byte `masterId` supports ~2.8×10¹⁴ masters, and the 4-byte `userTag` supports >4 billion (2³²) deposit addresses per master.

### Reserved Address Space

Any address whose first 10 bytes match `VIRTUAL_PREFIX` is treated as a virtual address by the TIP-20 precompile. If a user happens to control a private key whose corresponding address begins with `0xFDFDFDFDFDFDFDFDFDFD`, TIP-20 transfers to that address will be subject to TIP-1022 resolution: they will forward to the registered master (if the embedded `masterId` is registered) or revert (if it is not). The address will not accumulate a TIP-20 balance through normal transfers. Users who have such an address SHOULD not use it on Tempo — see **Risks and Limitations**.

### Master ID Derivation

The `masterId` is **deterministic**, derived from the registration hash computed during `registerVirtualMaster()`:

```
registrationHash = keccak256(abi.encodePacked(msg.sender, salt))
masterId = bytes6(registrationHash[4:10])
```

The first 4 bytes of `registrationHash` are consumed by the proof-of-work check (see **Registration Proof of Work**); the `masterId` is extracted from bytes `[4:10]` of the same hash.

The salt is a `bytes32` value chosen by the caller. Callers MUST grind the salt to satisfy the 32-bit proof-of-work requirement (see below). The `masterId` is permanently bound to the **registration address**.

In the event of a masterId collision (two `(address, salt)` pairs mapping to the same 6-byte masterID), the second registration reverts with `MasterIdCollision`. The caller can retry with a different salt.

### Registration Proof of Work

Registration requires a 32-bit proof of work. This makes it computationally expensive for an attacker to frontrun a registration transaction, without lengtheing the masterID. 

The registration hash is computed as:

```
registrationHash = keccak256(abi.encodePacked(msg.sender, salt))
```

The first 4 bytes of `registrationHash` MUST be zero:

```
require(bytes4(registrationHash[0:4]) == 0x00000000)  // 32-bit PoW
masterId = bytes6(registrationHash[4:10])
```

This requires the caller to grind ~2³² salt values to find a valid registration. If the first 4 bytes are not zero, the call reverts with `ProofOfWorkFailed`. This Proof-Of-Work renders certain attacks computationally infeasible while still being straightforward for a real user to register a masterId.

### User Tag Derivation (Off-Chain)

The `userTag` is an opaque 4-byte value generated off-chain by the operator. The protocol does not interpret or validate it — all values including `0x00000000` are valid. It exists solely so the operator can attribute deposits to specific users via the two-hop `Transfer` events (see **Event Emission**).

Operators maintain their own internal mapping `{internalUserId → virtualAddress}`. No on-chain transaction is needed to create a new deposit address.

### Worked Example

An exchange with master address `0xABCD...1234` registers with a salt that satisfies the 32-bit PoW:
- `registrationHash = keccak256(abi.encodePacked(0xABCD...1234, salt))`
- `registrationHash[0:4] == 0x00000000` (PoW satisfied)
- `masterId = bytes6(registrationHash[4:10])` → e.g., `0x07A3B1C2D4E5`
- For user #103048, the exchange derives a userTag → e.g., `0xa7c3f19e`

```
Virtual address = 0xFDFDFDFDFDFDFDFDFDFD  07A3B1C2D4E5  a7c3f19e
                  ^^^^^^^^^^^^^^^^^^^^    ^^^^^^^^^^^^  ^^^^^^^^
                  prefix (10)             masterId (6)  userTag (4)
```

## Registry Precompile

Virtual address resolution requires a registry that maps `masterId → masterAddress`. This is managed through a new precompile deployed at  `0xFDC0000000000000000000000000000000000000`.

The registry MUST maintain the following mapping constraints:

- each `masterId` maps to at most one registered master address (one-to-one from masterId)
- multiple `masterId`s MAY map to the same master address (many-to-one)

This many-to-one design allows a single underlying wallet to register multiple masterIds (e.g., with different salts).

A **valid master address** MUST satisfy TIP-20 recipient safety constraints:

- MUST NOT be `address(0)`
- MUST NOT match `VIRTUAL_PREFIX`
- MUST NOT be a TIP-20 token address (i.e., MUST pass TIP-20 recipient validation for transfer paths)

### Registry Storage Layout

Each `masterId` maps to a single 32-byte storage slot:

```
slot = keccak256(masterId, REGISTRY_BASE)
value = masterAddress | reserved   | masterType
        ^^^^^^^^^^^     ^^^^^^^^     ^^^^^^^^^^
        20 bytes        11 bytes     1 byte
```

| Field | Bytes | Description |
|-------|-------|-------------|
| `masterAddress` | 20 | The registered master address for this masterId. `address(0)` if unregistered. |
| `reserved` | 11 | Reserved for future use. MUST be zeroed. |
| `masterType` | 1 | Type discriminator for future extensibility. MUST be `0x00` in this version. |

This layout packs all metadata for a masterId into a single storage slot, enabling one-SLOAD resolution during transfer path execution.

### Interface

```solidity
interface IVirtualAddressRegistry {
    // ──────────────────── Events ────────────────────

    /// @notice Emitted when a new master is registered.
    event MasterRegistered(
        bytes6 indexed masterId,
        address indexed masterAddress
    );

    // ──────────────────── Errors ────────────────────

    /// @notice The computed masterId is already registered to a different address.
    error MasterIdCollision();

    /// @notice The caller/new master address is invalid for virtual forwarding.
    error InvalidMasterAddress();

    /// @notice The registration hash does not satisfy the 32-bit proof-of-work requirement.
    error ProofOfWorkFailed();

    /// @notice The virtual address has a valid prefix but its masterId is not registered.
    error VirtualAddressUnregistered();

    // ──────────────── Registration ──────────────────

    /// @notice Registers msg.sender as a virtual address master.
    /// @dev The registration hash is keccak256(abi.encodePacked(msg.sender, salt)).
    ///      The first 4 bytes of the hash MUST be zero (32-bit proof of work).
    ///      masterId is derived from bytes [4:10] of the registration hash.
    ///      Reverts with ProofOfWorkFailed if the first 4 bytes are not zero.
    ///      Reverts with InvalidMasterAddress if msg.sender is not a valid master address.
    ///      Reverts with MasterIdCollision if the derived masterId is already taken
    ///      by a different address. On collision, the caller can retry with a different salt.
    ///      The same address MAY register multiple masterIds using different salts.
    /// @param salt Caller-chosen salt for masterId derivation. Must satisfy 32-bit PoW.
    /// @return masterId The derived master identifier.
    function registerVirtualMaster(bytes32 salt) external returns (bytes6 masterId);

    // ────────────────── Queries ─────────────────────

    /// @notice Returns the registered master address for a given masterId, or address(0) if unregistered.
    function getMaster(bytes6 masterId) external view returns (address);

    /// @notice Resolves a transfer recipient using TIP-1022 execution semantics.
    ///         For non-virtual addresses, returns `to` unchanged.
    ///         For virtual addresses, returns the registered master or reverts with
    ///         VirtualAddressUnregistered.
    function resolveRecipient(address to) external view returns (address effectiveRecipient);

    /// @notice Resolves a virtual address to its registered master.
    ///         Returns address(0) if the address does not match VIRTUAL_PREFIX.
    ///         Returns address(0) if the masterId is not registered.
    function resolveVirtualAddress(address virtualAddr) external view returns (address master);

    /// @notice Returns true if the address matches the VIRTUAL_PREFIX.
    function isVirtualAddress(address addr) external pure returns (bool);

    /// @notice Decodes a virtual address into its components.
    /// @return isVirtual True if the address matches VIRTUAL_PREFIX.
    /// @return masterId The 6-byte master identifier (zero if not virtual).
    /// @return userTag The 4-byte user tag (zero if not virtual).
    function decodeVirtualAddress(address addr)
        external pure returns (bool isVirtual, bytes6 masterId, bytes4 userTag);
}
```

### Security Considerations for Master Addresses

The master address is permanently bound to its masterId at registration. If the master key is compromised, deposits to virtual addresses under that masterId cannot be redirected. Operators registering as virtual address masters SHOULD use appropriately hardened key management commensurate with the value flowing through their deposit addresses. A multisig or threshold signature scheme is strongly recommended.

| Name | Value | Description |
|------|-------|-------------|
| `VIRTUAL_PREFIX` | `0xFDFDFDFDFDFDFDFDFDFD` | 10-byte prefix identifying virtual addresses |
| `REGISTRY_ADDRESS` | `0xFDC0000000000000000000000000000000000000` | Precompile address for the virtual address registry |

## Transfer Path Modification

The following existing TIP-20 entrypoints are modified to resolve the `to` (recipient) address before crediting. The `from` address on `transferFrom`, `transferFromWithMemo`, and `systemTransferFrom` is **not** affected by TIP-1022 resolution.

- `transfer`
- `transferFrom`
- `transferWithMemo`
- `transferFromWithMemo`
- `mint`
- `mintWithMemo`
- `systemTransferFrom`

Additionally, TIP-1022 introduces the following **new** TIP-20 entrypoints:

- `transferThroughVirtual` (see **Master-Controlled Virtual Source Transfers**)
- `transferThroughVirtualWithMemo` (see **Master-Controlled Virtual Source Transfers**)

### Resolution Logic

```
function resolveRecipient(to: address) -> address:
    // Fast path: check first 10 bytes against VIRTUAL_PREFIX
    if to[0:10] != VIRTUAL_PREFIX:
        return to  // Not virtual, no overhead beyond a 10-byte comparison

    masterId = to[10:16]
    master = registry.getMaster(masterId)

    if master == address(0):
        // Unregistered masterId → revert
        revert VirtualAddressUnregistered()

    return master
```

#### Execution Flow for Standard Entrypoints

For `transfer`, `transferFrom`, `transferWithMemo`, `transferFromWithMemo`, `mint`, `mintWithMemo`, `systemTransferFrom`:

1. **Resolve recipient**: compute `effectiveRecipient = resolveRecipient(to)`. If `to` is not virtual, `effectiveRecipient = to` (fast path).
2. **TIP-403 sender check**: check sender authorization per standard TIP-403 rules.
3. **TIP-403 recipient check**: check `effectiveRecipient` per standard TIP-403 rules.
4. **Apply balance changes**: debit sender, credit `effectiveRecipient`.
5. **Emit events**: per **Event Emission** (two-hop `Transfer` if virtual, single `Transfer` otherwise).

If any step reverts, the enclosing TIP-20 operation MUST revert atomically with no balance changes and no events.

#### Execution Flow for `transferThroughVirtual` / `transferThroughVirtualWithMemo`

1. **Validate source**: `virtualFrom` MUST match `VIRTUAL_PREFIX` (`InvalidVirtualSender` on failure).
2. **Resolve source master**: resolve `virtualFrom` to its registered master; revert with `VirtualAddressUnregistered` if not registered.
3. **Verify ownership**: resolved master MUST equal `msg.sender` (`NotVirtualSenderOwner` otherwise).
4. **Resolve recipient**: compute `effectiveRecipient = resolveRecipient(to)`.
5. **TIP-403 checks**:
   - **Sender check**: check `msg.sender` (the resolved master) per standard TIP-403 sender rules.
   - **Recipient check**: check `effectiveRecipient` per standard TIP-403 rules.
6. **Apply balance changes**: debit `msg.sender` balance, credit `effectiveRecipient`. The `virtualFrom` address is never debited.
7. **Emit events**: per **Event Emission**. For `transferThroughVirtualWithMemo`, also emit `TransferWithMemo`.

### Composability: TIP-403 Virtual Address Resolution

TIP-1022 does not introduce new view functions on TIP-20. TIP-20 transfer authorization logic MUST resolve virtual recipient addresses before invoking existing TIP-403 sender/recipient authorization checks.
Concretely, for any TIP-20 entrypoint covered by TIP-1022:

1. Compute `effectiveRecipient = resolveRecipient(to)`.
2. Check sender authorization using the existing TIP-403 sender rule.
3. Check recipient authorization using the existing TIP-403 recipient rule, with `effectiveRecipient` (not the literal virtual address).

This preserves view/execution symmetry with the TIP-20 authorization path defined by TIP-1015: any internal TIP-20 helper such as `isTransferAuthorized(from, to)` MUST evaluate recipient authorization against the resolved master address when `to` is virtual.

`balanceOf(virtualAddress)` remains literal and MUST continue to return 0.

Contracts or integrators that need explicit resolution behavior outside the TIP-20 transfer path MAY call `resolveRecipient` on the registry.

### Master-Controlled Virtual Source Transfers (`transferThroughVirtual`)

TIP-1022 adds a TIP-20 entrypoint that lets a master spend from one of its own virtual addresses as a source alias (useful for refunds and customer-facing reconciliation), while preserving canonical balance accounting on the master.

```solidity
interface ITIP20VirtualSource {
    /// @notice `virtualFrom` does not match VIRTUAL_PREFIX.
    error InvalidVirtualSender();

    /// @notice `virtualFrom` resolves to a master different from msg.sender.
    error NotVirtualSenderOwner();

    /// @notice Transfers `amount` from the caller's master balance, using
    ///         `virtualFrom` as the source alias for attribution.
    /// @dev `virtualFrom` MUST:
    ///      - match VIRTUAL_PREFIX
    ///      - resolve to msg.sender's masterId
    ///      Recipient `to` is resolved with standard TIP-1022 recipient logic.
    function transferThroughVirtual(address virtualFrom, address to, uint256 amount)
        external
        returns (bool);

    /// @notice Like `transferThroughVirtual`, but with an attached memo for reconciliation.
    /// @dev Same validation as `transferThroughVirtual`. Emits additional `TransferWithMemo`.
    function transferThroughVirtualWithMemo(address virtualFrom, address to, uint256 amount, bytes32 memo)
        external
        returns (bool);
}
```

The execution flow for `transferThroughVirtual` and `transferThroughVirtualWithMemo` is specified in **Execution Flow for `transferThroughVirtual` / `transferThroughVirtualWithMemo`** above.

### Event Emission

TIP-1022 does **not** introduce new transfer-path events. The registry precompile emits `MasterRegistered` (see **Interface**), but forwarding itself is represented using **two-hop standard `Transfer` events**: one hop showing funds arriving at the virtual address, and a second hop showing funds moving from the virtual address to the resolved master (or vice versa for `transferThroughVirtual`). This two-hop pattern ensures that existing indexers, block explorers, and analytics tools that track standard ERC-20/TIP-20 `Transfer` events will correctly display the full flow without requiring custom event support. 

For transfers where the recipient is **not** a virtual address, event emission is unchanged from standard TIP-20 behavior — a single `Transfer(sender, to, amount)`. No two-hop pattern is used.

#### Deposit Forwarding (Inbound)

When a transfer targets a virtual address (`to` is virtual), the precompile MUST emit two `Transfer` events in sequence:

1. `Transfer(sender, virtualAddress, amount)` — shows funds arriving at the virtual address
2. `Transfer(virtualAddress, masterAddress, amount)` — shows funds forwarding to the master

The actual balance change is applied only to `masterAddress`. The virtual address never holds a balance; the first `Transfer` event is a logical representation of the deposit attribution, not a real balance credit.

Indexers that need deposit attribution SHOULD watch for pairs of `Transfer` events within the same transaction where the intermediate address matches `VIRTUAL_PREFIX`. The `userTag` can be extracted from the virtual address to identify the depositor.

#### Entrypoint-Specific Event Ordering

- `transfer`, `transferFrom`, `systemTransferFrom`:
  1. `Transfer(sender, virtualAddress, amount)`
  2. `Transfer(virtualAddress, masterAddress, amount)`

- `transferWithMemo`, `transferFromWithMemo`:
  1. `Transfer(sender, virtualAddress, amount)`
  2. `TransferWithMemo(sender, virtualAddress, amount, memo)`
  3. `Transfer(virtualAddress, masterAddress, amount)`

- `mint`:
  1. `Transfer(address(0), virtualAddress, amount)`
  2. `Mint(virtualAddress, amount)`
  3. `Transfer(virtualAddress, masterAddress, amount)`

- `mintWithMemo`:
  1. `Transfer(address(0), virtualAddress, amount)`
  2. `TransferWithMemo(address(0), virtualAddress, amount, memo)`
  3. `Mint(virtualAddress, amount)`
  4. `Transfer(virtualAddress, masterAddress, amount)`

- `transferThroughVirtual`:
  - when `virtualFrom == to`: self-transfer. No net balance change. Events:
    1. `Transfer(masterAddress, virtualFrom, amount)`
    2. `Transfer(virtualFrom, masterAddress, amount)`
  - when `to` is non-virtual:
    1. `Transfer(masterAddress, virtualFrom, amount)`
    2. `Transfer(virtualFrom, to, amount)`
  - when `to` is virtual (both source and destination are virtual):
    1. `Transfer(masterAddress, virtualFrom, amount)`
    2. `Transfer(virtualFrom, virtualTo, amount)`
    3. `Transfer(virtualTo, recipientMaster, amount)`

- `transferThroughVirtualWithMemo`:
  - when `virtualFrom == to`: self-transfer. No net balance change. Events:
    1. `Transfer(masterAddress, virtualFrom, amount)`
    2. `TransferWithMemo(masterAddress, virtualFrom, amount, memo)`
    3. `Transfer(virtualFrom, masterAddress, amount)`
  - when `to` is non-virtual:
    1. `Transfer(masterAddress, virtualFrom, amount)`
    2. `Transfer(virtualFrom, to, amount)`
    3. `TransferWithMemo(virtualFrom, to, amount, memo)`
  - when `to` is virtual:
    1. `Transfer(masterAddress, virtualFrom, amount)`
    2. `Transfer(virtualFrom, virtualTo, amount)`
    3. `TransferWithMemo(virtualFrom, virtualTo, amount, memo)`
    4. `Transfer(virtualTo, recipientMaster, amount)`

## Self-Forwarding

If the registered master sends tokens to one of its own virtual addresses, the transfer resolves back to the master, effectively a transfer to self. The standard TIP-20 self-transfer semantics apply (no net balance change). The two-hop `Transfer` events are still emitted per invariant #4: `Transfer(master, virtualAddress, amount)` followed by `Transfer(virtualAddress, master, amount)`. Indexers SHOULD NOT interpret this as net inflow when `from == masterAddress` in the first hop.

## Interaction with TIP-403 Policies

Virtual address resolution happens **before** TIP-403 transfer authorization checks. Policy evaluation uses the resolved `masterAddress` — not the virtual address.

- If the **master address** is not authorized to receive the token, transfers to any of its virtual addresses revert.
- If the **sender** is not authorized to send the token, the transfer reverts (existing TIP-403 behavior).
- TIP-403 policies set on individual virtual addresses are **ignored**. Virtual addresses have no independent on-chain identity.
- For `transferThroughVirtual` / `transferThroughVirtualWithMemo`, sender policy checks apply to `msg.sender` (the master), not `virtualFrom`.

## Interaction with Account-Level Features

- **`balanceOf(virtualAddress)`**: Always returns 0. Virtual addresses do not hold balances.
- **Nonce**: Virtual addresses are not expected to originate transactions. However, if someone were to control a private key whose address matches `VIRTUAL_PREFIX` (requiring ~2⁸⁰ work), EOA transactions sent *from* that address would execute normally — TIP-1022 resolution applies only to the `to` field in TIP-20 precompile calls, not to transaction senders.
- **`transferThroughVirtual` accounting**: Debits always come from `masterAddress`, never from `virtualFrom`.

## Risks and Limitations

### Address Poisoning

**Warning**: When a user or system encounters an address with the `0xFDFDFDFDFDFDFDFDFDFD` prefix, they MUST verify the **entire address**, i.e., that both the masterID and userTag match. 

Because all virtual addresses share the same 10-byte prefix, an attacker can generate virtual addresses that visually resemble a victim's deposit address when displayed in truncated form (e.g., `0xFDFD...f19e`). Standard address poisoning attacks — where an attacker sends small transfers from a lookalike address to trick users into copying the wrong address from transaction history — are amplified by virtual addresses because:

1. **Shared prefix**: All virtual addresses start with `0xFDFD...`, so the first 20 hex characters are always identical. Truncated displays that show "first N + last M" characters provide less differentiation than with random addresses.
2. **Low cost of generation**: userTags are generated off-chain with no on-chain cost. An attacker who registers a single masterId can generate 2³² virtual addresses instantly, grinding for suffixes that match a target.
3. **No on-chain trace of generation**: Unlike regular address generation (which requires transactions to become visible), virtual addresses appear in logs only when transfers occur, making prefix collisions easy to produce silently.

**Mitigations**:

- Wallets and UIs displaying virtual addresses SHOULD show the **full address** or at minimum the `masterId` and `userTag` components separately, rather than a truncated form.
- Users SHOULD always verify the full address through an independent channel (e.g., the exchange's deposit page) rather than copying from transaction history.

### Non-TIP-20 Token Loss

TIP-1022 forwarding applies exclusively to TIP-20 precompile operations. Non-TIP-20 tokens (e.g., ERC-20 contracts deployed on Tempo) transferred to a virtual address are credited to the virtual address by the ERC-20 contract and are **irrecoverable**: no private key controls the virtual address, and no recovery mechanism exists.

This risk is mitigated by the strong incentives for token issuers to use TIP-20 on Tempo (gas payment eligibility, access to the payment lane, and TIP-403 policy support). All stablecoins on Tempo are currently issued as TIP-20 tokens.

### Contracts and EOAs at virtual-prefixed addresses 

It is theoretically possible to deploy a contract or control an EOA whose address matches VIRTUAL_PREFIX, including by grinding CREATE2 salts or private keys. Such addresses can still exist and originate ordinary EVM transactions, but we consider this unlikely in practice because targeting the 10-byte prefix requires roughly 2^80 work, with additional cost for targeted collisions against registered virtual namespaces.

# Invariants

## Core Invariants

1. **No fund loss**: A TIP-20 transfer to a virtual address MUST either credit the registered master's balance by exactly the transfer amount, or revert. Funds MUST NOT be credited to the virtual address itself or lost.

2. **Revert on unregistered**: A transfer to an address matching `VIRTUAL_PREFIX` whose `masterId` is not registered MUST revert. It MUST NOT credit any account.

3. **Balance consistency**: After a successful virtual-forwarded transfer of amount `X`, `balanceOf(master)` MUST have increased by exactly `X`. `balanceOf(virtualAddress)` MUST remain 0.

4. **Event consistency**: For standard recipient-forwarding entrypoints, the precompile MUST emit two `Transfer` events: `Transfer(sender, virtualAddress, amount)` followed by `Transfer(virtualAddress, masterAddress, amount)`. For `transferThroughVirtual`, the precompile MUST emit `Transfer(masterAddress, virtualFrom, amount)` followed by one or two hops depending on whether `to` is virtual (see **Event Emission**). `TransferWithMemo` events MUST immediately follow their matching `Transfer` and MUST use `virtualAddress` as the recipient (preserving deposit attribution). `Mint` events MUST use `virtualAddress`.

5. **Non-virtual path unaffected**: Transfers to addresses that do not match `VIRTUAL_PREFIX` MUST behave identically to pre-TIP-1022 semantics, with no registry lookup.

6. **Deterministic masterId**: Given `registrationHash = keccak256(abi.encodePacked(registrationAddress, salt))`, the first 4 bytes of `registrationHash` MUST be zero (32-bit proof of work), and `masterId` MUST equal `bytes6(registrationHash[4:10])`, where `registrationAddress` is the address that originally called `registerVirtualMaster()` and `salt` is the caller-supplied salt. If the PoW check fails, registration MUST revert with `ProofOfWorkFailed`. `masterId` MUST NOT depend on registration order or transaction ordering.

7. **Master ID uniqueness**: Each `masterId` MUST map to at most one registered master address. Multiple `masterId`s MAY map to the same master address.

8. **Atomic revert behavior**: If virtual resolution fails (unregistered), the enclosing TIP-20 call MUST revert with no state changes and no events.

9. **Virtual-source accounting**: `transferThroughVirtual` MUST debit `masterAddress` balance and MUST NOT debit `virtualFrom` balance.

10. **View/execution symmetry**: TIP-20 transfer authorization logic MUST evaluate recipient authorization against the resolved master address when `to` is virtual, matching execution-time recipient resolution semantics.

11. **Policy on master**: TIP-403 authorization for virtual-forwarded transfers MUST check the resolved `masterAddress`. Policies set on individual virtual addresses MUST be ignored.
