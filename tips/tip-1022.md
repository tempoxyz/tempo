# TIP-1022: Virtual Addresses for TIP-20 Deposit Forwarding

## Abstract

This TIP introduces **virtual addresses**: a reserved address prefix that, when detected in TIP-20 recipient-bearing operations, causes the precompile to auto-credit a registered master wallet instead of the target address. This eliminates sweep transactions entirely for exchanges, ramps, and payment processors that generate per-user deposit addresses. Master registration is a one-time on-chain call; deposit address derivation is fully off-chain.

## Motivation

- **Eliminate sweep transactions.** Exchanges, ramps, and payment processors (e.g., Bridge/Stripe, Coinbase) need to offer each customer a unique deposit address. Today, funds arriving at each address must be swept back to a central wallet in separate transactions, which is an enormous operational cost and burden at scale. Virtual addresses auto-credit the master wallet at the protocol level, making sweeps unnecessary.

- **Avoid the 250,000 gas new-account cost.** Tempo charges 250,000 gas to create state for a new address on first use. With virtual addresses, deposit addresses never create on-chain state, so the first transfer to a new deposit address costs the same as any other transfer.

- **Prevent state bloat.** Without virtual addresses, each customer deposit address creates a new account in the state trie. At enterprise scale (millions of deposit addresses), this is significant and permanent state growth. Virtual addresses avoid this entirely: no accounts are created, regardless of how many deposit addresses an exchange generates.

# Specification

## Address Layout

Virtual addresses are standard 20-byte EVM addresses with a reserved prefix, interpreted as a byte array where byte `0` is the leftmost byte in the hex string:

```
[10-byte PREFIX] [6-byte masterId] [4-byte userTag]
= 20 bytes total
```

| Field | Bytes | Description |
|-------|-------|-------------|
| **PREFIX** | 10 | Fixed prefix `0xFDFDFDFDFDFDFDFDFDFD`. Identifies virtual addresses. |
| **masterId** | 6 | Deterministic identifier derived from the master address. Supports up to ~2.8×10¹⁴ masters. |
| **userTag** | 4 | Opaque per-user identifier, derived off-chain. 32-bit space supports >4 billion (2³²) unique deposit addresses per master. |

## Conformance and Scope

TIP-1022 applies only to TIP-20 precompile recipient resolution for the entrypoints listed in **Transfer Path Modification**.

TIP-1022 does **not** alter TIP-20 methods that do not carry a recipient (e.g., `approve`, `burn`, `permit`) and does not alter non-TIP-20 protocol behavior.

Before TIP-1022 activation (`protocolVersion` hardfork), all TIP-20 behavior remains unchanged.

Non-TIP-20 token transfers (including ERC-20 contracts deployed on Tempo) to virtual addresses are **not** subject to TIP-1022 forwarding. Such transfers behave as standard EVM transfers to the literal address. Tokens sent this way may be irrecoverable — see **Risks and Limitations**.

### Prefix Selection: `0xFDFDFDFDFDFDFDFDFDFD`

A 10-byte prefix provides collision resistance of ~1/2⁸⁰. This is an explicit layout tradeoff that prioritizes address space safety over `masterId` and `userTag` space. The 6-byte `masterId` supports ~2.8×10¹⁴ masters, and the 4-byte `userTag` supports >4 billion (2³²) deposit addresses per master.

### Reserved Address Space

Any address whose first 10 bytes match `VIRTUAL_PREFIX` is treated as a virtual address by the TIP-20 precompile. If a user happens to control a private key whose corresponding address begins with `0xFDFDFDFDFDFDFDFDFDFD`, TIP-20 transfers to that address will be subject to TIP-1022 resolution: they will forward to the registered master (if the embedded `masterId` is registered) or revert (if it is not). The address will not accumulate a TIP-20 balance through normal transfers. Users who have such an address SHOULD not use it on Tempo — see **Risks and Limitations**.

### Master ID Derivation

The `masterId` is **deterministic**, derived from the registration hash computed during `registerVirtualMaster()`:

```
registrationHash = keccak256(abi.encodePacked(msg.sender, salt))
masterId = bytes6(registrationHash[4:10])
```

The first 4 bytes of `registrationHash` are consumed by the proof-of-work check (see **Registration Proof of Work**); the `masterId` is extracted from bytes `[4:10]` of the same hash.

The salt is a `bytes32` value chosen by the caller. Callers MUST grind the salt to satisfy the 32-bit proof-of-work requirement (see below). The `masterId` is permanently bound to the **registration address** and does not change if the master is later rotated to a new address via `updateMaster`. This is what makes deposit addresses stable: they embed `masterId`, so they continue to work regardless of master rotation.

In the event of a masterId collision (two `(address, salt)` pairs mapping to the same 6-byte masterID), the second registration reverts with `MasterIdCollision`. The caller can retry with a different salt.

### Registration Proof of Work

Registration requires a 32-bit proof of work to make it computationally expensive to register masterIds that collide with specific CREATE2-deployed contract addresses.

The registration hash is computed as:

```
registrationHash = keccak256(abi.encodePacked(msg.sender, salt))
```

The first 4 bytes of `registrationHash` MUST be zero:

```
require(bytes4(registrationHash[0:4]) == 0x00000000)  // 32-bit PoW
masterId = bytes6(registrationHash[4:10])
```

This requires the caller to grind ~2³² salt values to find a valid registration. If the first 4 bytes are not zero, the call reverts with `ProofOfWorkFailed`. This Proof-Of-Work certain attacks computationally infeasible while still being straighforward for a real user to register a masterId.

### User Tag Derivation (Off-Chain)

The `userTag` is an opaque 4-byte value generated off-chain by the exchange/processor. The protocol does not interpret or validate it. It exists solely so the exchange can attribute deposits to specific users via the two-hop `Transfer` events (see **Event Emission**).

Exchanges maintain their own internal mapping `{internalUserId → virtualAddress}`. No on-chain transaction is needed to create a new deposit address.

Operators SHOULD assign userTags randomly with collision checks to reduce third-party enumeration of deposit addresses.

### Worked Example

An exchange with master address `0xABCD...1234` registers with a salt that satisfies the 32-bit PoW:
- `registrationHash = keccak256(abi.encodePacked(0xABCD...1234, salt))`
- `registrationHash[0:4] == 0x00000000` (PoW satisfied)
- `masterId = bytes6(registrationHash[4:10])` → e.g., `0x07A3B1C2D4E5`
- For user #103048, the exchange derives a userTag → e.g., `0xa7c3f19e`

```
Virtual address = 0xFDFDFDFDFDFDFDFDFDFD  07A3B1C2D4E5  a7c3f19e
                  ^^^^^^^^^^^^^^^^^^^^    ^^^^^^^^^^^^  ^^^^^^^^
                  prefix (10)             masterId (6)  userTag (4)
```

## Registry Precompile

Virtual address resolution requires a registry that maps `masterId → masterAddress`. This is managed through a new precompile.

The registry MUST maintain the following mapping constraints:

- each `masterId` maps to at most one active master address (one-to-one from masterId)
- multiple `masterId`s MAY map to the same master address (many-to-one)

This many-to-one design allows a single underlying wallet to register multiple masterIds (e.g., with different salts), each potentially subject to different TIP-403 policies. This supports use cases where an exchange needs separate policy configurations for different user segments (e.g., US vs. non-US users) while consolidating funds into one wallet.

A **valid master address** MUST satisfy TIP-20 recipient safety constraints:

- MUST NOT be `address(0)`
- MUST NOT match `VIRTUAL_PREFIX`
- MUST NOT be a TIP-20 token address (i.e., MUST pass TIP-20 recipient validation for transfer paths)

### Registry Storage Layout

Each `masterId` maps to a single 32-byte storage slot:

```
slot = keccak256(masterId, REGISTRY_BASE)
value = masterAddress | masterType | reserved
        ^^^^^^^^^^^     ^^^^^^^^^^   ^^^^^^^^
        20 bytes        1 byte       11 bytes (zeroed)
```

| Field | Bytes | Description |
|-------|-------|-------------|
| `masterAddress` | 20 | The current master address for this masterId. `address(0)` if unregistered. |
| `masterType` | 1 | Type discriminator for the master for future extensibility. MUST be `0x00` in this version. |
| `reserved` | 11 | Reserved for future use. MUST be zeroed. |

This layout packs all metadata for a masterId into a single storage slot, enabling one-SLOAD resolution during transfer path execution.

### Interface

```solidity
interface IVirtualAddressRegistry {
    // ──────────────────── Events ────────────────────

    /// @notice Emitted when a new master is registered.
    event MasterRegistered(
        bytes6 indexed masterId,
        address indexed masterAddress
    );

    /// @notice Emitted when a master address is updated.
    event MasterUpdated(
        bytes6 indexed masterId,
        address indexed oldMaster,
        address indexed newMaster
    );

    // ──────────────────── Errors ────────────────────

    /// @notice The computed masterId is already registered to a different address.
    error MasterIdCollision();

    /// @notice Caller is not the current master for this masterId.
    error NotMaster();

    /// @notice The caller/new master address is invalid for virtual forwarding.
    error InvalidMasterAddress();

    /// @notice The provided new-master consent signature is invalid.
    error InvalidMasterConsent();

    /// @notice The provided new-master consent signature has expired.
    error MasterConsentExpired();

    /// @notice The registration hash does not satisfy the 32-bit proof-of-work requirement.
    error ProofOfWorkFailed();

    /// @notice The virtual address has a valid prefix but its masterId is not registered.
    error VirtualAddressUnregistered();

    // ──────────────── Registration ──────────────────

    /// @notice Registers msg.sender as a virtual address master.
    /// @dev The registration hash is keccak256(abi.encodePacked(msg.sender, salt)).
    ///      The first 4 bytes of the hash MUST be zero (32-bit proof of work).
    ///      masterId is derived from bytes [4:10] of the registration hash.
    ///      Reverts with ProofOfWorkFailed if the first 4 bytes are not zero.
    ///      Reverts with InvalidMasterAddress if msg.sender is not a valid master address.
    ///      Reverts with MasterIdCollision if the derived masterId is already taken
    ///      by a different address. On collision, the caller can retry with a different salt.
    ///      The same address MAY register multiple masterIds using different salts.
    /// @param salt Caller-chosen salt for masterId derivation. Must satisfy 32-bit PoW.
    /// @return masterId The derived master identifier.
    function registerVirtualMaster(bytes32 salt) external returns (bytes6 masterId);

    // ────────────────── Queries ─────────────────────

    /// @notice Returns the current master address for a given masterId, or address(0) if unregistered.
    function getMaster(bytes6 masterId) external view returns (address);

    /// @notice Returns all active masterIds for a given master address.
    /// @dev A single address may have multiple masterIds (registered with different salts).
    /// @return masterIds Array of active masterIds for this address (empty if none).
    function getMasterIds(address masterAddress)
        external view returns (bytes6[] memory masterIds);

    /// @notice Resolves a transfer recipient using TIP-1022 execution semantics.
    ///         For non-virtual addresses, returns `to` unchanged.
    ///         For virtual addresses, returns the current master or reverts with
    ///         VirtualAddressUnregistered.
    function resolveRecipient(address to) external view returns (address effectiveRecipient);

    /// @notice Resolves a virtual address to its current master.
    ///         Returns address(0) if the address does not match VIRTUAL_PREFIX.
    ///         Returns address(0) if the masterId is not registered.
    function resolveVirtualAddress(address virtualAddr) external view returns (address master);

    /// @notice Returns true if the address matches the VIRTUAL_PREFIX.
    function isVirtualAddress(address addr) external pure returns (bool);

    /// @notice Decodes a virtual address into its components.
    /// @return isVirtual True if the address matches VIRTUAL_PREFIX.
    /// @return masterId The 6-byte master identifier (zero if not virtual).
    /// @return userTag The 4-byte user tag (zero if not virtual).
    function decodeVirtualAddress(address addr)
        external pure returns (bool isVirtual, bytes6 masterId, bytes4 userTag);

    // ──────────── Master Rotation ────────────────────

    /// @notice Proposed new master is equal to current master (no-op).
    error InvalidNewMaster();

    /// @notice Updates the master address for a specific masterId. Takes effect immediately.
    /// @dev Only callable by the current master for this masterId.
    ///      Reverts with NotMaster if msg.sender is not the current master for this masterId.
    ///      Reverts with InvalidMasterAddress if newMaster is not a valid master address.
    ///      Reverts with InvalidNewMaster if newMaster equals current master (no-op).
    ///      Reverts with InvalidMasterConsent / MasterConsentExpired if consent is invalid.
    ///      newMaster MAY already be the active master for other masterIds (many-to-one).
    ///      Consent MUST be EIP-712 over (chainId, registryAddress, masterId, currentMaster,
    ///      newMaster, deadline), signed by newMaster.
    /// @param masterId The master identifier to rotate.
    /// @param newMaster The new master address.
    /// @param deadline Signature expiration timestamp.
    /// @param newMasterSig EIP-712 signature by newMaster consenting to this rotation.
    function updateMaster(bytes6 masterId, address newMaster, uint256 deadline, bytes calldata newMasterSig)
        external;
}
```

### Security Considerations for Master Addresses

The master address has full control over forwarding: it can rotate the master to a new address immediately. If the master key is compromised, an attacker can redirect future deposits instantly. TIP-1022 intentionally does not impose protocol-level timelocks or pause mechanisms — these concerns are best managed by the master address itself (e.g., via multisig wallets, hardware security modules, or governance contracts that enforce their own cooldown periods).

Operators registering as virtual address masters SHOULD use appropriately hardened key management commensurate with the value flowing through their deposit addresses. A multisig or threshold signature scheme is strongly recommended, as it provides both access control and implicit cooldown via multi-party coordination.

### Master Rotation Consent

The proposed new master MUST explicitly consent to a rotation via EIP-712 signature. This ensures that a master address cannot be rotated to an unwilling recipient, and that the new master has affirmatively acknowledged responsibility for the forwarded deposit flow associated with the masterId.

The signed payload MUST include:

- `chainId`
- `registryAddress` (verifying contract)
- `masterId`
- `currentMaster`
- `newMaster`
- `deadline`
- `nonce`

This binds consent to one chain, one registry instance, and one specific master rotation tuple.

## Transfer Path Modification

The following existing TIP-20 entrypoints are modified to detect virtual addresses and resolve them before crediting:

- `transfer`
- `transferFrom`
- `transferWithMemo`
- `transferFromWithMemo`
- `mint`
- `mintWithMemo`
- `systemTransferFrom`

Additionally, TIP-1022 introduces the following **new** TIP-20 entrypoints:

- `transferFromVirtual` (see **Master-Controlled Virtual Source Transfers**)
- `transferFromVirtualWithMemo` (see **Master-Controlled Virtual Source Transfers**)

### Resolution Logic

```
function resolveRecipient(to: address, sender: address) -> address:
    // Fast path: check first 10 bytes against VIRTUAL_PREFIX
    if to[0:10] != VIRTUAL_PREFIX:
        return to  // Not virtual, no overhead beyond a 10-byte comparison

    masterId = to[10:16]
    master = registry.getMaster(masterId)

    if master == address(0):
        // Unregistered masterId → revert
        revert VirtualAddressUnregistered()

    return master
```

#### Execution Flow for Standard Entrypoints

For `transfer`, `transferFrom`, `transferWithMemo`, `transferFromWithMemo`, `mint`, `mintWithMemo`, `systemTransferFrom`:

1. **Resolve recipient**: compute `effectiveRecipient = resolveRecipient(to)`. If `to` is not virtual, `effectiveRecipient = to` (fast path).
2. **TIP-403 sender check**: check sender authorization per standard TIP-403 rules.
3. **TIP-403 recipient check (two-level if virtual)**:
   - if `to` is virtual: check the masterId policy address (`VIRTUAL_PREFIX | masterId | 00..0`), then check `effectiveRecipient`. Both MUST pass (logical AND).
   - if `to` is non-virtual: check `effectiveRecipient` only.
4. **Apply balance changes**: debit sender, credit `effectiveRecipient`.
5. **Emit events**: per **Event Emission** (two-hop `Transfer` if virtual, single `Transfer` otherwise).

If any step reverts, the enclosing TIP-20 operation MUST revert atomically with no balance changes and no events.

#### Execution Flow for `transferFromVirtual` / `transferFromVirtualWithMemo`

1. **Validate source**: `virtualFrom` MUST match `VIRTUAL_PREFIX` (`InvalidVirtualSender` on failure).
2. **Resolve source master**: resolve `virtualFrom` to its current master; revert with `VirtualAddressUnregistered` if not registered.
3. **Verify ownership**: resolved master MUST equal `msg.sender` (`NotVirtualSenderOwner` otherwise).
4. **Resolve recipient**: compute `effectiveRecipient = resolveRecipient(to)`.
5. **TIP-403 checks**:
   - **Sender check**: check `msg.sender` (the resolved master) per standard TIP-403 sender rules.
   - **Recipient check (two-level if virtual)**: if `to` is virtual, check the destination masterId policy address, then the destination master account. If `to` is non-virtual, check `effectiveRecipient` only.
6. **Apply balance changes**: debit `msg.sender` balance, credit `effectiveRecipient`. The `virtualFrom` address is never debited.
7. **Emit events**: per **Event Emission**. For `transferFromVirtualWithMemo`, also emit `TransferWithMemo`.

### View/Execution Symmetry (Composability Helpers)

To avoid view/execution mismatches for routers, vaults, and integrators, TIP-20 implementations adopting TIP-1022 MUST expose helper views that mirror execution-time recipient resolution:

```solidity
interface ITIP20VirtualViews {
    /// @notice Returns execution-equivalent recipient resolution for `to`.
    /// @dev Returns `to` for non-virtual addresses; resolves/reverts exactly like transfer paths for virtual addresses.
    function effectiveRecipient(address to) external view returns (address);

    /// @notice Returns balance using execution-equivalent recipient resolution.
    /// @dev Equivalent to balanceOf(effectiveRecipient(account)).
    function balanceOfEffective(address account) external view returns (uint256);

    /// @notice Returns TIP-403 transfer authorization using execution-equivalent recipient resolution.
    /// @dev Equivalent to isTransferAuthorized(from, effectiveRecipient(to)).
    function isTransferAuthorizedEffective(address from, address to) external view returns (bool);
}
```

`balanceOf(virtualAddress)` remains literal and MUST continue to return 0.

### Master-Controlled Virtual Source Transfers (`transferFromVirtual`)

TIP-1022 adds a TIP-20 entrypoint that lets a master spend from one of its own virtual addresses as a source alias (useful for refunds and customer-facing reconciliation), while preserving canonical balance accounting on the master.

```solidity
interface ITIP20VirtualSource {
    /// @notice `virtualFrom` does not match VIRTUAL_PREFIX.
    error InvalidVirtualSender();

    /// @notice `virtualFrom` resolves to a master different from msg.sender.
    error NotVirtualSenderOwner();

    /// @notice Transfers `amount` from the caller's master balance, using
    ///         `virtualFrom` as the source alias for attribution.
    /// @dev `virtualFrom` MUST:
    ///      - match VIRTUAL_PREFIX
    ///      - resolve to msg.sender's masterId
    ///      Recipient `to` is resolved with standard TIP-1022 recipient logic.
    function transferFromVirtual(address virtualFrom, address to, uint256 amount)
        external
        returns (bool);

    /// @notice Like `transferFromVirtual`, but with an attached memo for reconciliation.
    /// @dev Same validation as `transferFromVirtual`. Emits additional `TransferWithMemo`.
    function transferFromVirtualWithMemo(address virtualFrom, address to, uint256 amount, bytes calldata memo)
        external
        returns (bool);
}
```

The execution flow for `transferFromVirtual` and `transferFromVirtualWithMemo` is specified in **Execution Flow for `transferFromVirtual` / `transferFromVirtualWithMemo`** above.

### Gas Overhead

| Path | Additional cost |
|------|----------------|
| Non-virtual transfer | One 10-byte prefix comparison; no registry lookup |
| Virtual transfer (cold) | ~2,200 gas (one cold registry metadata lookup) |
| Virtual transfer (warm) | ~100 gas (one warm registry metadata lookup) |

Precompile storage reads are metered using the same warm/cold SLOAD gas parameters as EVM storage. These estimates assume the registry metadata needed for resolution (`masterAddress`) is retrievable in one storage lookup. Implementations that split this metadata across multiple slots MUST meter accordingly.

A standard TIP-20 `transfer` already performs ~4 SLOADs (policy lookup, sender balance, recipient balance, etc.) plus SSTOREs and log emissions, so one additional registry lookup is a modest increment.

The gas overhead for virtual transfers is borne by the sender.

### Event Emission

TIP-1022 does **not** introduce any new events. Instead, virtual address forwarding is represented using **two-hop standard `Transfer` events**: one hop showing funds arriving at the virtual address, and a second hop showing funds moving from the virtual address to the resolved master (or vice versa for `transferFromVirtual`). This two-hop pattern ensures that existing indexers, block explorers, and analytics tools that track standard ERC-20/TIP-20 `Transfer` events will correctly display the full flow without requiring custom event support.

#### Deposit Forwarding (Inbound)

When a transfer targets a virtual address (`to` is virtual), the precompile MUST emit two `Transfer` events in sequence:

1. `Transfer(sender, virtualAddress, amount)` — shows funds arriving at the virtual address
2. `Transfer(virtualAddress, masterAddress, amount)` — shows funds forwarding to the master

The actual balance change is applied only to `masterAddress`. The virtual address never holds a balance; the first `Transfer` event is a logical representation of the deposit attribution, not a real balance credit.

#### Entrypoint-Specific Event Ordering

- `transfer`, `transferFrom`, `systemTransferFrom`:
  1. `Transfer(sender, virtualAddress, amount)`
  2. `Transfer(virtualAddress, masterAddress, amount)`

- `transferWithMemo`, `transferFromWithMemo`:
  1. `Transfer(sender, virtualAddress, amount)`
  2. `Transfer(virtualAddress, masterAddress, amount)`
  3. `TransferWithMemo(sender, masterAddress, amount, memo)`

- `mint`:
  1. `Transfer(address(0), virtualAddress, amount)`
  2. `Transfer(virtualAddress, masterAddress, amount)`
  3. `Mint(masterAddress, amount)`

- `mintWithMemo` (matches TIP-20 ordering: Transfer → TransferWithMemo → Mint):
  1. `Transfer(address(0), virtualAddress, amount)`
  2. `Transfer(virtualAddress, masterAddress, amount)`
  3. `TransferWithMemo(address(0), masterAddress, amount, memo)`
  4. `Mint(masterAddress, amount)`

- `transferFromVirtual`:
  - when `to` is non-virtual:
    1. `Transfer(masterAddress, virtualFrom, amount)`
    2. `Transfer(virtualFrom, to, amount)`
  - when `to` is virtual (both source and destination are virtual):
    1. `Transfer(masterAddress, virtualFrom, amount)`
    2. `Transfer(virtualFrom, virtualTo, amount)`
    3. `Transfer(virtualTo, recipientMaster, amount)`

- `transferFromVirtualWithMemo`:
  - when `to` is non-virtual:
    1. `Transfer(masterAddress, virtualFrom, amount)`
    2. `Transfer(virtualFrom, to, amount)`
    3. `TransferWithMemo(masterAddress, to, amount, memo)`
  - when `to` is virtual:
    1. `Transfer(masterAddress, virtualFrom, amount)`
    2. `Transfer(virtualFrom, virtualTo, amount)`
    3. `Transfer(virtualTo, recipientMaster, amount)`
    4. `TransferWithMemo(masterAddress, recipientMaster, amount, memo)`

#### Non-Virtual Transfers

For transfers where the recipient is **not** a virtual address, event emission is unchanged from standard TIP-20 behavior — a single `Transfer(sender, to, amount)`. No two-hop pattern is used.

#### Rationale

The two-hop `Transfer` approach was chosen over custom events (`VirtualForward`, `VirtualTransferFrom`) because:

- **Indexer compatibility**: Existing indexers that track `Transfer` events will see virtual address activity without code changes. The virtual address appears as both a recipient and a sender in the event log, making the forwarding path visible.
- **Block explorer support**: Explorers display `Transfer` events natively. Two-hop transfers show the full deposit path (sender → virtual → master) in standard transaction views.
- **Wallet integration**: Wallets that watch for incoming `Transfer` events to the master address will see the second hop and can attribute deposits by inspecting the first hop's `from` field (the virtual address).

Indexers that need deposit attribution SHOULD watch for pairs of `Transfer` events within the same transaction where the intermediate address matches `VIRTUAL_PREFIX`. The `userTag` can be extracted from the virtual address to identify the depositor.

#### Revert Behavior

If an operation reverts at any point, no `Transfer` events MUST be emitted.

## Self-Forwarding

If the current master sends tokens to one of its own virtual addresses, the transfer resolves back to the master, effectively a transfer to self. The standard TIP-20 self-transfer semantics apply (no net balance change). The two-hop `Transfer` events are still emitted per invariant #4: `Transfer(master, virtualAddress, amount)` followed by `Transfer(virtualAddress, master, amount)`. Indexers SHOULD NOT interpret this as net inflow when `from == masterAddress` in the first hop.

## Interaction with TIP-403 Policies

Virtual address resolution happens **before** TIP-403 transfer authorization checks. Policy evaluation uses a two-level model combining the master account's policy with an optional per-masterId policy.

### Per-MasterId Policies (requires TIP-1023)

A master MAY set a TIP-403 account-level policy scoped to a specific `masterId` by setting a policy on the **masterId policy address**, a synthetic address constructed as:

```
VIRTUAL_PREFIX | masterId | 00000000
^^^^^^^^^^^^^^   ^^^^^^^^   ^^^^^^^^
prefix (10)      masterId (6)  userTag (4)
```

This is the virtual address with `userTag = 0x00000000`. By convention, this address serves as a policy anchor for TIP-403. The protocol does not enforce any special restriction on the zero userTag — it is a valid virtual address like any other — but it is the canonical address where per-masterId policies are looked up. The current master for a `masterId` MAY call TIP-1023 account-level policy functions using this address to set token-level restrictions (e.g., "virtual addresses under this masterId may only receive USDC") or other TIP-403 constraints.

This enables a single wallet holding multiple masterIds to enforce different policies per masterId — for example, one masterId restricted to USDC for US users and another masterId allowing both USDC and USDT for non-US users — while all funds consolidate into the same master wallet.

### Policy Evaluation Order

For a transfer to a virtual address, TIP-403 authorization MUST check **both** layers (see step 3 in **Execution Flow for Standard Entrypoints** and step 5 in **Execution Flow for `transferFromVirtual`**):

1. **MasterId policy**: Check the policy set on the masterId policy address (`VIRTUAL_PREFIX | masterId | 00..0`). If a policy is set and it rejects the transfer, the operation MUST revert.
2. **Master account policy**: Check the policy set on the resolved `masterAddress`. If a policy is set and it rejects the transfer, the operation MUST revert.

A transfer succeeds only if it passes both checks (logical AND). If neither the masterId nor the master account has a policy set, standard TIP-403 defaults apply.

### Policy on Individual Virtual Addresses

TIP-403 policies set directly on an individual virtual address (i.e., with a non-zero `userTag`) MUST be **ignored** by the protocol. Virtual addresses have no independent on-chain identity — they do not hold balances, cannot sign transactions, and cannot set policies. Only the masterId policy address (zero `userTag`) and the master account address are checked.

### Summary

- If the **master account** is sanctioned/blocked, transfers to any of its virtual addresses revert.
- If a **masterId policy** restricts a token, transfers of that token to virtual addresses under that masterId revert — even if the master account's own policy would allow it.
- If the **sender** is sanctioned/blocked, the transfer reverts (existing TIP-403 behavior).
- Policies on individual virtual addresses (non-zero `userTag`) are ignored.
- For `transferFromVirtual` / `transferFromVirtualWithMemo`, sender policy checks apply to the resolved master (`msg.sender`), not `virtualFrom`. Recipient-side checks follow the same two-level model.

## Interaction with Account-Level Features

- **`balanceOf(virtualAddress)`**: Always returns 0. Virtual addresses do not hold balances.
- **Nonce**: Virtual addresses cannot sign transactions (no private key controls the forwarding). They have no nonce.
- **Composability helpers**: Integrators SHOULD use `effectiveRecipient`, `balanceOfEffective`, and `isTransferAuthorizedEffective` for preflight checks that match execution semantics.
- **`transferFromVirtual` accounting**: Debits always come from `masterAddress`, never from `virtualFrom`.

## Risks and Limitations

### Address Poisoning

> ⚠️ **Warning**: When a user or system encounters an address with the `0xFDFDFDFDFDFDFDFDFDFD` prefix, they MUST verify the **entire 20-byte address**, not just the prefix or a truncated display form.

Because all virtual addresses share the same 10-byte prefix, an attacker can trivially generate virtual addresses that visually resemble a victim's deposit address when displayed in truncated form (e.g., `0xFDFD...f19e`). Standard address poisoning attacks — where an attacker sends small transfers from a lookalike address to trick users into copying the wrong address from transaction history — are amplified by virtual addresses because:

1. **Shared prefix**: All virtual addresses start with `0xFDFD...`, so the first 20 hex characters are always identical. Truncated displays that show "first N + last M" characters provide less differentiation than with random addresses.
2. **Low cost of generation**: userTags are generated off-chain with no on-chain cost. An attacker who registers a single masterId can generate 2³² virtual addresses instantly, grinding for suffixes that match a target.
3. **No on-chain trace of generation**: Unlike regular address generation (which requires transactions to become visible), virtual addresses appear in logs only when transfers occur, making prefix collisions easy to produce silently.

**Mitigations**:

- Wallets and UIs displaying virtual addresses SHOULD show the **full address** or at minimum the `masterId` and `userTag` components separately, rather than a truncated form.
- Exchanges and payment processors SHOULD validate the full address before crediting deposits and SHOULD NOT rely on transaction-history-based address selection for virtual addresses.
- Users SHOULD always verify the full address through an independent channel (e.g., the exchange's deposit page) rather than copying from transaction history.

### Non-TIP-20 Token Loss

TIP-1022 forwarding applies exclusively to TIP-20 precompile operations. Non-TIP-20 tokens (e.g., ERC-20 contracts deployed on Tempo) transferred to a virtual address are credited to the virtual address by the ERC-20 contract and are **irrecoverable**: no private key controls the virtual address, and no recovery mechanism exists in this version of TIP-1022.

This risk is mitigated by the strong incentives for token issuers to use TIP-20 on Tempo (gas payment eligibility, access to the payment pipeline, and TIP-403 policy support). All stablecoins on Tempo are currently issued as TIP-20 tokens.

### Design Alternatives Considered

An alternative approach — embedding memos in a new Tempo address format (`tempom1...`) with no consensus changes — was considered. This approach avoids protocol-level modifications but requires universal wallet and integrator adoption of the new format, which is a higher coordination burden. Virtual addresses were chosen because they work with existing wallet infrastructure: any wallet that can send to a 20-byte address can send to a virtual address without modification.

# Invariants

## Core Invariants

1. **No fund loss**: A TIP-20 transfer to a virtual address MUST either credit the registered master's balance by exactly the transfer amount, or revert. Funds MUST NOT be credited to the virtual address itself or lost.

2. **Revert on unregistered**: A transfer to an address matching `VIRTUAL_PREFIX` whose `masterId` is not registered MUST revert. It MUST NOT credit any account.

3. **Balance consistency**: After a successful virtual-forwarded transfer of amount `X`, `balanceOf(master)` MUST have increased by exactly `X`. `balanceOf(virtualAddress)` MUST remain 0.

4. **Event consistency**: Every virtual-forwarded operation MUST emit exactly two `Transfer` events: `Transfer(sender, virtualAddress, amount)` followed by `Transfer(virtualAddress, masterAddress, amount)`. Any additional TIP-20 events for that entrypoint (e.g., `TransferWithMemo`, `Mint`) MUST use `masterAddress` in recipient fields.

5. **Non-virtual path unaffected**: Transfers to addresses that do not match `VIRTUAL_PREFIX` MUST behave identically to pre-TIP-1022 semantics, with no registry lookup.

6. **Deterministic masterId**: Given `registrationHash = keccak256(abi.encodePacked(registrationAddress, salt))`, the first 4 bytes of `registrationHash` MUST be zero (32-bit proof of work), and `masterId` MUST equal `bytes6(registrationHash[4:10])`, where `registrationAddress` is the address that originally called `registerVirtualMaster()` and `salt` is the caller-supplied salt. If the PoW check fails, registration MUST revert with `ProofOfWorkFailed`. `masterId` MUST NOT change on master rotation, and MUST NOT depend on registration order or transaction ordering.

7. **Master ID uniqueness**: Each `masterId` MUST map to at most one active master address. Multiple `masterId`s MAY map to the same master address.

8. **Atomic revert behavior**: If virtual resolution fails (unregistered), the enclosing TIP-20 call MUST revert with no state changes and no events.

9. **`getMasterIds` completeness**: `getMasterIds` MUST return all active masterIds for a given address. An empty array indicates no registrations.

10. **Virtual-source accounting**: `transferFromVirtual` MUST debit `masterAddress` balance and MUST NOT debit `virtualFrom` balance.

11. **Rotation consent binding**: `updateMaster` MUST require valid new-master consent bound to `(chainId, registryAddress, masterId, currentMaster, newMaster, deadline)`.

12. **View/execution symmetry**: `effectiveRecipient`, `balanceOfEffective`, and `isTransferAuthorizedEffective` MUST match execution-time recipient resolution semantics.

13. **Two-level policy enforcement**: For virtual-forwarded transfers, TIP-403 authorization MUST check both the masterId policy address (`VIRTUAL_PREFIX | masterId | 00..0`) and the resolved master account. The transfer MUST be rejected if either policy disallows it.

14. **Virtual address policy ignorance**: TIP-403 policies set on individual virtual addresses (non-zero `userTag`) MUST be ignored during transfer authorization. Only the masterId policy address and the master account address are checked.

## Critical Test Cases

- Transfer to virtual address with registered master → credits master, emits two-hop `Transfer` events: `Transfer(sender, virtualAddress, amount)` then `Transfer(virtualAddress, masterAddress, amount)` (plus any method-specific TIP-20 events, if applicable).
- Transfer to virtual address with unregistered masterId → reverts with `VirtualAddressUnregistered`.
- Transfer to non-virtual address starting with `0xFDFDFDFDFDFDFDFDFD` but not matching full 10-byte prefix → normal transfer (no forwarding).
- Byte parsing conformance: `decodeVirtualAddress` MUST treat address bytes as `[0:10]=prefix,[10:16]=masterId,[16:20]=userTag` independent of host endianness.
- `masterId` collision: two different `(address, salt)` pairs that hash to the same `bytes6` → second registration reverts with `MasterIdCollision`. Retrying with a different salt succeeds.
- Registration with salt whose hash does not have zero first 4 bytes → reverts with `ProofOfWorkFailed`.
- Registration with valid PoW salt → succeeds, masterId equals `bytes6(registrationHash[4:10])`.
- Register from an invalid master address (virtual-prefix or TIP-20 token address) → reverts with `InvalidMasterAddress`.
- Master rotation: `updateMaster` succeeds when called by current master with valid new-master consent. New master receives forwards immediately; old master loses all privileges.
- `updateMaster` with invalid signature → reverts with `InvalidMasterConsent`.
- `updateMaster` with expired signature (`block.timestamp > deadline`) → reverts with `MasterConsentExpired`.
- Rotation consent replay across chain or different registry address fails (domain mismatch).
- `updateMaster(address(0))` → reverts with `InvalidMasterAddress`.
- `updateMaster(currentMaster)` → reverts with `InvalidNewMaster`.
- `updateMaster(virtualPrefixAddress)` → reverts with `InvalidMasterAddress`.
- `updateMaster(tip20TokenAddress)` → reverts with `InvalidMasterAddress`.
- `updateMaster` to an address that already has other masterIds → succeeds (many-to-one allowed).
- `updateMaster` called by non-master → reverts with `NotMaster`.
- `getMasterIds(unregisteredAddress)` returns empty array; `getMasterIds(registeredMaster)` returns all active masterIds.
- Many-to-one: same address registers with salt `0` and salt `1` → two different masterIds, both forwarding to the same address. Transfers to virtual addresses under either masterId credit the same wallet.
- Many-to-one: rotating one masterId to a new address does not affect the other masterIds still pointing to the original address.
- Reserved address space: TIP-20 transfers to an address beginning with `0xFDFDFDFDFDFDFDFDFDFD` are subject to TIP-1022 resolution — they revert with `VirtualAddressUnregistered` (if no masterId is registered for those bytes) or forward to the registered master (if one is).
- Reorg safety: registration + transfer in same block, block reorged → transfer reverts (masterId no longer registered).
- Self-forwarding: master sends to own virtual address → credits master (no net change), emits two-hop `Transfer` events.
- `transferFrom`, `transferWithMemo`, `transferFromWithMemo`, `mint`, `mintWithMemo`, and `systemTransferFrom` all respect virtual address resolution.
- `transferFromVirtual` succeeds only when `virtualFrom` resolves to `msg.sender`; otherwise reverts with `NotVirtualSenderOwner`.
- `transferFromVirtual` with non-virtual `virtualFrom` reverts with `InvalidVirtualSender`.
- `transferFromVirtual` debits master balance and emits two-hop `Transfer` events: `Transfer(masterAddress, virtualFrom, amount)` then `Transfer(virtualFrom, to, amount)`; `balanceOf(virtualFrom)` remains 0.
- Event ordering for forwarded `transferWithMemo`, `transferFromWithMemo`, `mint`, and `mintWithMemo` matches the two-hop rules in **Event Emission**.
- Event ordering for `transferFromVirtual` and `transferFromVirtualWithMemo` matches the two-hop rules in **Event Emission**.
- `transferFromVirtualWithMemo` emits two-hop `Transfer` events plus `TransferWithMemo`; when `to` is virtual, also emits a third `Transfer` hop for the recipient-side forwarding.
- TIP-403 two-level policy: masterId policy allows USDC only, master account has no policy → transfer of USDC succeeds, transfer of USDT reverts.
- TIP-403 two-level policy: masterId policy has no restrictions, master account blocks a sender → transfer reverts.
- TIP-403 two-level policy: both masterId policy and master account policy are set → transfer must pass both (logical AND).
- TIP-403 two-level policy: neither masterId policy nor master account policy is set → standard TIP-403 defaults apply.
- TIP-403 policy set on individual virtual address (non-zero userTag) → ignored; transfer proceeds based on masterId policy address and master account only.
- TIP-403 masterId policy address is `VIRTUAL_PREFIX | masterId | 0x00000000` — setting a policy on this address restricts all virtual addresses under that masterId.
- Many-to-one with policies: same master wallet, two masterIds with different policies → transfers to virtual addresses under each masterId are subject to their respective masterId policy, not each other's.
- `balanceOf(virtualAddress)` returns 0 before and after forwarded transfer.
- `effectiveRecipient(nonVirtual)` returns input unchanged.
- `effectiveRecipient(virtual)` equals resolved master for registered masterIds and reverts on unregistered exactly like execution paths.
- `balanceOfEffective(virtualAddress)` equals `balanceOf(resolvedMaster)`.
- `isTransferAuthorizedEffective(from, to)` equals execution-time authorization outcome for transfer paths.
- On forwarded path revert (`VirtualAddressUnregistered`), no `Transfer`, `TransferWithMemo`, or `Mint` events are emitted.
