# TIP-1071: FX support for StablecoinDEX

## Abstract

This TIP lets the StablecoinDEX support non-USD TIP-20 tokens whose `quoteToken` is a USD-denominated TIP-20 token. The DEX prices these FX pairs by multiplying the token's current PriceOracle quote by the order's tick multiplier. A limit order at tick `0` trades at the oracle price. Positive and negative ticks trade at a premium or discount to that oracle price.

FX support keeps the existing DEX shape: one order book per `(base, quote)` pair, price-time priority within each tick, and the same swap entry points. The main new behavior is for bids and flip orders. FX bids escrow a USD quote-token budget so price movement cannot make them insolvent. FX flip orders preserve their configured size, but the active liquidity that rests after a flip can be smaller when the maker's internal DEX balance cannot fund the full size.

## Motivation

Tempo's StablecoinDEX is useful for trading different issuers' USD tokens. The same mechanism should support stablecoins in other currencies, such as EUR or GBP stablecoins trading against a USD quote token.

Market makers should not need to cancel and replace every order when the EUR/USD oracle moves. They should be able to express a spread around the current oracle price: buy EUR at 10 bps below the feed and sell EUR at 10 bps above the feed. The tick then describes the maker's spread, not an absolute EUR/USD price that becomes stale as soon as the oracle changes.

Flip orders matter for this use case. They let a maker run a standing two-sided market with one logical order ID. With oracle-relative prices, the amount that can be funded after a flip can change over time. This TIP keeps the flip order's configured size stable, but lets the active resting liquidity be smaller when the maker has less available balance than that size requires.

## Assumptions

- TIP-1054 defines the shared `PriceOracle`, `priceOracleFeedId`, `PriceQuote`, `QuotePolicy`, and `latestQuote` surfaces.
- A non-USD token that uses this TIP has a USD-denominated `quoteToken`. Direct FX/FX pairs are out of scope. They can route through USD tokens.
- TIP-1056 flip-in-place behavior is active. A flip order keeps the same `orderId` when it flips.
- TIP-1062 versioned DEX order storage is available. FX orders need additional state, so they cannot use the legacy order layout unchanged.
- TIP-1060 and TIP-1064 are active for DEX order-storage accounting.
- TIP-20 token amounts and USD quote-token amounts use the existing 6-decimal unit convention.
- Existing USD/USD StablecoinDEX behavior remains unchanged.

If the oracle feed is missing, unauthorized, stale for DEX use, or otherwise invalid, FX DEX operations for that token cannot trade. Existing orders remain cancellable and cleanable even if the token later loses its valid oracle configuration.

## Threat Model

- **Feed admin and updater.** The DEX relies on them to publish accurate USD prices. A bad, stale, or compromised feed can move the effective DEX price. Quote-age and quote-change limits reduce this risk but do not remove it.
- **Token issuer and token admin.** The DEX relies on token metadata, `quoteToken`, oracle feed selection, pause state, and transfer-policy state. Issuer or admin changes can make a pair temporarily unusable or reprice resting orders.
- **Makers.** Makers choose ticks, sizes, and whether to use flip orders. They cannot receive more than a taker paid, and an FX bid cannot spend more quote tokens than it escrowed.
- **Takers.** Takers choose slippage bounds through the existing exact-in and exact-out swap functions. They may get a different result from a prior view quote if the oracle updates or order-book state changes before execution.
- **The DEX.** The DEX is trusted protocol code. It must use one captured oracle quote for each FX pair during a state-changing operation so a fill and the automatic flip that follows it are priced consistently.

# Specification

## Pair eligibility

`createPair(base)` continues to accept the existing USD/USD pairs.

Starting with this TIP, `createPair(base)` also accepts an FX/USD pair when all of the following are true:

1. `base` is a valid non-USD TIP-20 token.
2. `quote = base.quoteToken()` is a valid USD-denominated TIP-20 token.
3. `base.priceOracleFeedId()` is nonzero.
4. The selected PriceOracle feed exists and authorizes `base`.

The pair key remains `computeBookKey(base, quote)`. The base token is the non-USD token, and the quote token is the USD token. Reverse trading through the same book is supported by the existing swap functions.

A USD token with a nonzero `priceOracleFeedId` is invalid under TIP-1054 and cannot be used to create an FX pair.

The DEX does not snapshot the feed id in the order. Existing FX orders use the base token's current `priceOracleFeedId()` whenever they are quoted, filled, flipped, cancelled, or cleaned up. If the token admin changes the feed id, the existing book is repriced against the new feed as soon as the new feed is DEX-valid. This is an issuer-controlled market-structure change and should be visible through token and PriceOracle events.

## DEX quote validity

The DEX uses the newest accepted PriceOracle quote for the base token. It does not use TIP-1054's one-block-delayed `usableQuote` rule for trading.

A state-changing FX DEX operation MUST read the feed's `latestQuote(feedId)` before pricing an FX pair. If an oracle update and a DEX trade occur in the same block, transaction ordering determines the quote used by the trade. A trade before the oracle update uses the previous latest quote. A trade after the oracle update uses the new latest quote. This is intentional. FX liquidity providers bear the race risk from trading against the newest quote visible at their transaction's position in the block.

A quote is DEX-valid only if all of the following hold:

1. `base.priceOracleFeedId()` is nonzero.
2. The feed authorizes the base token.
3. The feed has a valid `QuotePolicy`.
4. `latestQuote(feedId)` has a nonzero `quoteId` and nonzero `usdPerTokenE18`.
5. `block.number - latestQuoteBlock <= min(policy.maxQuoteAgeBlocks, PROTOCOL_MAX_FX_DEX_QUOTE_AGE_BLOCKS)`.

`PROTOCOL_MAX_FX_DEX_QUOTE_AGE_BLOCKS` SHOULD be lower than the maximum quote age allowed for fee payment. Fee payment can tolerate a somewhat older quote because gas charges are small and bounded. DEX trades can move materially more value, so they need fresher prices.

View quote functions use the same `latestQuote` rule. A view quote and a later state-changing swap can still disagree if an oracle update or order-book mutation occurs between the two calls.

## Constants

The DEX uses these constants. The values below are proposed activation values and should be fixed before this TIP moves out of Draft.

```text
ORACLE_SCALE                         = 1e18
MIN_ORDER_USD_E6                     = 100_000_000   // $100.000000
FX_RESTING_DUST_USD_E6               = 1_000_000     // $1.000000
PROTOCOL_MAX_FX_DEX_QUOTE_AGE_BLOCKS = 60
```

`MIN_ORDER_USD_E6` matches the existing `MIN_ORDER_AMOUNT` value, interpreted as USD quote-token units. `FX_RESTING_DUST_USD_E6` is lower because it applies only to existing logical orders that would otherwise rest with a tiny active amount after a flip, stale cleanup, or oracle movement.

## Oracle-relative ticks

For USD/USD pairs, ticks keep their current meaning.

For FX/USD pairs, the tick is a multiplier on the oracle price:

```text
multiplier(tick) = tickToPrice(tick) / PRICE_SCALE
limitPrice       = usdPerTokenE18 * multiplier(tick)
```

If the oracle says one EUR token is worth `1.100000 USD` and `tickToPrice(tick) = 100500`, the order price is `1.105500 USD` per EUR token.

This keeps price priority simple. All orders in an FX book share the same oracle quote at execution time, so higher bid ticks are still better bids and lower ask ticks are still better asks.

## Arithmetic

The DEX adds FX-aware base/quote conversion helpers. `baseAmount` is denominated in the non-USD base token. `quoteAmount` is denominated in the USD quote token. `priceE18` is `usdPerTokenE18` from the captured PriceOracle quote.

```text
ceilDiv(a, b) = 0 if a == 0, otherwise floor((a - 1) / b) + 1

fxBaseToQuote(baseAmount, tick, priceE18, rounding)
    = round(baseAmount * priceE18 * tickToPrice(tick)
            / (ORACLE_SCALE * PRICE_SCALE), rounding)

fxQuoteToBase(quoteAmount, tick, priceE18, rounding)
    = round(quoteAmount * ORACLE_SCALE * PRICE_SCALE
            / (priceE18 * tickToPrice(tick)), rounding)
```

All arithmetic uses checked unsigned 256-bit integers. Overflow, underflow, division by zero, or a value that cannot fit in storage MUST revert.

Rounding follows the current DEX rule of favoring the maker:

- Bid escrow and bid quote budget calculations round up.
- Bid fills round quote paid to the taker down.
- Ask fills round quote paid by the taker and credited to the maker up.
- Exact-output calculations round input requirements up.
- Exact-input calculations round output down when needed to avoid spending more input than supplied.

A fill that would move one token leg by zero MUST NOT execute. The route traversal must either continue to another fillable order or terminate with the existing insufficient-liquidity, insufficient-output, or max-input error.

## Minimum sizes and dust

FX order placement uses USD notional at the order's limit price. A new FX order MUST revert if:

```text
fxBaseToQuote(amount, tick, priceE18, Down) < MIN_ORDER_USD_E6
```

Automatic flips use the lower active dust threshold:

```text
fxBaseToQuote(activeBase, tick, priceE18, Down) < FX_RESTING_DUST_USD_E6
```

A user-submitted order must be large enough to justify creating new order state. An automatic flip may continue with less active liquidity because it is preserving an existing logical order. If the active liquidity after a flip is below `FX_RESTING_DUST_USD_E6`, the DEX drops the flip instead of resting dust in the book.

## FX order state

FX orders use a new order storage version introduced after TIP-1062. The exact packed layout is specified by the implementation TIP, but it MUST represent the following logical fields:

```text
orderId          uint128   // mapping key remains canonical
maker            address
bookKey          bytes32
isBid            bool
isFlip           bool
tick             int16
flipTick         int16
amount           uint128   // configured cycle size, denominated in base token
openBase         uint128   // unfilled base cap for the current resting side
quoteBudget      uint128   // remaining quote escrow for FX bids, zero for asks
prev             uint128
next             uint128
```

This TIP uses three concepts that should not be collapsed:

- `amount` is the configured order size. For a flip order, it is preserved across flips.
- `openBase` is the amount of the current side that has not been actually or synthetically filled.
- `remaining` means the base amount that can actually fill now under the DEX-valid quote used by the current call.

For FX asks, `remaining = openBase` whenever the order has a DEX-valid quote. The ask has already escrowed that amount of base token.

For FX bids, `quoteBudget` is the escrowed USD quote-token budget. The current active base liquidity is:

```text
currentRemaining(order, priceE18) = min(
    order.openBase,
    fxQuoteToBase(order.quoteBudget, order.tick, priceE18, Down)
)
```

Oracle movement alone does not change `openBase` or `quoteBudget`. If the oracle price rises, an FX bid's current `remaining` can fall below `openBase`. If the oracle price later falls, `remaining` can rise again, but never above `openBase`. Actual fills and synthetic fills reduce `openBase`; oracle movement does not.

This distinction avoids path dependence. A bid that becomes temporarily underfunded because the oracle moved against it is not permanently shrunk unless it is filled, flipped, cancelled, or cleaned up.

### Synthetic fills

A synthetic fill is an accounting reduction of `openBase` that does not move tokens and does not emit `OrderFilled`. It is used only when the DEX is preserving a flip order's configured `amount` but the current side cannot rest the full amount.

Synthetic fills occur in two cases:

1. A flip can fund less than `amount` on the new side. The new `openBase` is the fundable amount, and `amount - openBase` is treated as already filled for that side.
2. An FX bid's entire current `remaining` is filled while `openBase` was larger than `remaining` because the oracle price moved up. The unfunded part of `openBase` is treated as already filled before the order is deleted or rewritten as a flip.

Synthetic fills do not pay either party. They only keep order accounting consistent: `amount` remains the configured cycle size, while `openBase` records how much of the current cycle is still able to rest.

### Tick-level liquidity

For FX asks, tick-level `totalLiquidity` is the sum of ask `openBase` values at that tick and is equal to active base liquidity while the quote is valid.

For FX bids, tick-level `totalLiquidity` is the sum of bid `openBase` values at that tick. It is an upper bound, not an authoritative fillable amount, because the current oracle quote and each order's `quoteBudget` determine how much base is actually fundable. FX quote and fill paths MUST inspect bid orders one by one and compute `currentRemaining` before relying on their liquidity.

The existing `getTickLevel(address base, int16 tick, bool isBid)` remains ABI-compatible. For FX bids, its `totalLiquidity` is the stored `openBase` upper bound. FX-aware clients MUST NOT treat that value as currently fillable liquidity.

To support bounded inspection, the DEX adds a paginated FX liquidity view:

```solidity
function getFXTickLiquidity(
    address base,
    int16 tick,
    bool isBid,
    uint128 startOrderId,
    uint16 limit
)
    external
    view
    returns (
        uint128 openBaseLiquidity,
        uint128 currentLiquidity,
        uint128 nextOrderId
    );
```

The view walks at most `limit` orders starting at `startOrderId`, or at the tick head if `startOrderId == 0`. `limit == 0` MUST revert. `startOrderId` MUST be zero or an active order at the requested tick. `nextOrderId == 0` means the page reached the end of the tick. For invalid or stale FX quotes, `currentLiquidity` is zero.

### Order views

The existing `getOrder(uint128)` remains ABI-compatible. For FX orders, it returns the existing shape with `amount = amount` and `remaining = current remaining` when a DEX-valid quote exists. If no DEX-valid quote exists, `remaining` is zero because the order has no fillable DEX liquidity until the quote becomes valid again.

FX-aware clients should use the new view:

```solidity
struct FXOrderInfo {
    uint128 orderId;
    address maker;
    bytes32 bookKey;
    bool isBid;
    bool isFlip;
    int16 tick;
    int16 flipTick;
    uint128 amount;
    uint128 openBase;
    uint128 remaining;
    uint128 quoteBudget;
    bool quoteValid;
    bytes32 feedId;
    uint64 quoteId;
    uint192 usdPerTokenE18;
}

function getFXOrder(uint128 orderId) external view returns (FXOrderInfo memory order);
```

`getFXOrder` MUST revert for non-FX orders and inactive orders. For active FX orders with no DEX-valid quote, it returns `quoteValid = false`, `remaining = 0`, `quoteId = 0`, and `usdPerTokenE18 = 0`, while still returning `openBase` and `quoteBudget`.

## Placing FX orders

When placing an FX order, the DEX captures the base token's DEX-valid PriceOracle quote before computing escrow.

For an ask:

1. Validate the order amount against `MIN_ORDER_USD_E6`.
2. Escrow `amount` base tokens from the maker's internal DEX balance and external wallet using the existing order-placement funding path.
3. Store `amount = amount`, `openBase = amount`, and `quoteBudget = 0`.

For a bid:

1. Validate the order amount against `MIN_ORDER_USD_E6`.
2. Compute `quoteBudget = fxBaseToQuote(amount, tick, priceE18, Up)`.
3. Escrow `quoteBudget` quote tokens from the maker's internal DEX balance and external wallet using the existing order-placement funding path.
4. Store `amount = amount`, `openBase = amount`, and `quoteBudget = quoteBudget`.

Policy and pause checks follow the existing DEX rules. The DEX checks the token that would be received by the maker at placement time because fills may credit internal balances without calling TIP-20 transfer functions.

## Filling FX orders

A state-changing swap captures one DEX-valid latest quote for each FX pair in the route before filling orders for that pair. The same captured quote MUST be used for every fill on that pair during the operation.

For every FX fill, the DEX emits the existing `OrderFilled` event and then the FX-specific `FXOrderFilled` event defined below.

### Filling asks

For an ask fill of `fillBase <= openBase`:

```text
quotePaid = fxBaseToQuote(fillBase, tick, priceE18, Up)
```

The taker receives `fillBase` base tokens. The maker receives `quotePaid` quote tokens in internal DEX balance. The order's `openBase` decreases by `fillBase`.

An FX ask is terminal when `openBase == 0`. A terminal ask is deleted or, if it is a flip order, rewritten in place under the flip rules below.

### Filling bids

Before filling an FX bid, the DEX computes `remaining = currentRemaining(order, priceE18)`. The fill amount MUST satisfy `fillBase <= remaining`.

For a bid fill:

```text
quotePaid = fxBaseToQuote(fillBase, tick, priceE18, Down)
```

The taker receives `quotePaid` quote tokens. The maker receives `fillBase` base tokens in internal DEX balance. The order's `openBase` decreases by `fillBase`, and `quoteBudget` decreases by `quotePaid`.

An FX bid's active side is exhausted when `currentRemaining(order, priceE18) == 0` after the fill. At that point, any remaining `openBase` is treated as a synthetic fill, `openBase` is set to zero, unused `quoteBudget` is credited back to the maker's internal DEX balance, and the order is deleted or rewritten as a flip order.

If a state-changing swap encounters an FX order whose current active notional is below `FX_RESTING_DUST_USD_E6` before it can make a fill, the DEX MAY drop that order through the cleanup path and continue traversal. Exact algorithms are specified in the appendix.

## Flip orders

FX flip orders preserve their configured `amount` across flips. `amount` is the target cycle size. `openBase` is the unfilled cap for the current side, and `remaining` is the active liquidity that can fill under the captured quote.

When an FX flip order is fully filled, the DEX attempts to rewrite the same `orderId` in place. For an FX bid, "fully filled" means its active `remaining` was exhausted under the captured quote, even if `openBase` was larger because part of the order was not fundable at that quote. The unfunded part is handled as a synthetic fill before the flip.

The DEX MUST use the same captured PriceOracle quote that priced the fill. This avoids a race where a fill is priced with one quote but the automatic flip is funded with another.

Automatic flips MUST NOT pull from the maker's external wallet. They may use only the maker's internal DEX balance after the fill and any refund from the filled order.

### Bid to ask

A filled bid has delivered base tokens to the maker. The flipped ask sells base tokens at `flipTick`.

Let `A = order.amount`. Let `B` be the maker's internal DEX balance of the base token after the bid fill and quote-budget refund.

```text
newOpenBase = min(A, B)
```

If the USD notional of `newOpenBase` at `flipTick` and the captured quote is below `FX_RESTING_DUST_USD_E6`, the DEX drops the flip. Otherwise, the DEX escrows `newOpenBase` base tokens from the maker's internal DEX balance and rewrites the order as an ask with:

```text
amount      = A
openBase    = newOpenBase
quoteBudget = 0
```

If `newOpenBase < A`, the order state is equivalent to a full-size ask that was immediately partially filled by `A - newOpenBase`. No trade occurred for that synthetic fill, and no `OrderFilled` event is emitted for it.

### Ask to bid

A filled ask has delivered quote tokens to the maker. The flipped bid buys base tokens at `flipTick`.

Let `A = order.amount`. Let `Q` be the maker's internal DEX balance of the quote token after the ask fill.

```text
fundableBase = min(A, fxQuoteToBase(Q, flipTick, priceE18, Down))
```

If the USD notional of `fundableBase` at `flipTick` and the captured quote is below `FX_RESTING_DUST_USD_E6`, the DEX drops the flip. Otherwise, the DEX computes:

```text
newQuoteBudget = fxBaseToQuote(fundableBase, flipTick, priceE18, Up)
```

The DEX escrows `newQuoteBudget` quote tokens from the maker's internal DEX balance and rewrites the order as a bid with:

```text
amount      = A
openBase    = fundableBase
quoteBudget = newQuoteBudget
```

If `fundableBase < A`, the order state is equivalent to a full-size bid that was immediately partially filled by `A - fundableBase`. No trade occurred for that synthetic fill, and no `OrderFilled` event is emitted for it.

### Same-tick flips

TIP-1030 and TIP-1056 behavior still applies. Same-tick flips are allowed when the active hardfork allows them. For FX pairs, a same-tick flip uses the same multiplier on the captured oracle quote.

### Failed and dropped flips

If a flip fails because of policy, pause state, missing authorization, arithmetic failure, or another existing non-system DEX error, existing `FlipFailed` behavior applies.

If a flip has valid authorization but the maker's available internal balance can fund less than the active dust threshold, the DEX treats the flip as dropped rather than failed. The filled order is deleted, no new resting order is created, and the DEX emits the FX drop event defined below.

## Permissionless cleanup

Anyone MAY call:

```solidity
function cleanupFXOrder(uint128 orderId) external;
```

`cleanupFXOrder` removes an active FX order when at least one of these conditions holds:

1. The pair no longer has a DEX-valid latest quote.
2. The feed no longer authorizes the base token.
3. The maker is no longer authorized under the transfer-policy checks that would be required for the order to remain open.
4. The order's current active notional is below `FX_RESTING_DUST_USD_E6` under the latest DEX-valid quote.

If none of these conditions holds, `cleanupFXOrder` MUST revert.

Cleanup refunds escrow exactly as cancellation would: FX asks refund `openBase` base tokens, and FX bids refund `quoteBudget` quote tokens. The difference between `amount` and `openBase` is not refunded because it represents actual fills or synthetic fills.

Cleanup is a normal order deletion for storage-accounting purposes. Deleting the order record participates in TIP-1060 and TIP-1064 exactly like cancellation: the StablecoinDEX earns the underlying precompile storage gas token, and the reusable DEX order-storage credit is attributed to the order's storage owner under TIP-1064. This TIP does not add a separate bounty for `msg.sender`. A third-party cleaner can remove stale or dust orders, but maker-attributed DEX storage credits remain maker-attributed unless TIP-1064 changes.

## Cancellation

Cancelling an FX ask refunds its `openBase` base tokens.

Cancelling an FX bid refunds its `quoteBudget` quote tokens.

No refund is owed for the difference between `amount` and `openBase`. That difference represents liquidity that has already been filled or synthetic liquidity that did not rest because the maker lacked enough internal balance at a prior flip or because active FX bid liquidity was exhausted under the captured quote.

Cancellation does not require a DEX-valid quote. Ordinary cancellation emits the existing cancellation event and does not emit an FX quote event.

## Interactions with routing

The existing path-finding model remains based on `quoteToken` chains. An FX token whose quote token is a USD token can participate in direct swaps against that quote token and in multi-hop swaps through the USD token graph.

A route MAY contain more than one FX pair. The DEX captures and validates a quote per FX base token. If the same feed appears more than once in a route, the DEX MUST use a single captured quote for that feed within the operation.

## Interfaces and events

The existing order events remain in place. For FX orders, the DEX emits additional events that include the captured quote and the active liquidity.

Event ordering is consensus behavior:

1. User-submitted FX placement emits `OrderPlaced`, then `FXOrderPlaced`.
2. Every FX fill emits `OrderFilled`, then `FXOrderFilled`.
3. A successful automatic FX flip emits `OrderFilled`, then `FXOrderFilled`, then `OrderFlipped`, then `FXOrderFlipped`.
4. A filled FX flip that is dropped because the next side is below dust emits `OrderFilled`, then `FXOrderFilled`, then `FXOrderDropped`. It does not emit `OrderFlipped` or `FlipFailed`.
5. A resting FX order dropped by `cleanupFXOrder`, or by a swap that encounters the order after it has become stale or dust, emits `OrderCancelled`, then `FXOrderDropped`.
6. A non-dust flip failure follows existing TIP-1056 behavior and emits `OrderFilled`, then `FXOrderFilled`, then `FlipFailed`.

```solidity
event FXOrderPlaced(
    uint128 indexed orderId,
    address indexed maker,
    address indexed token,
    uint128 amount,
    uint128 openBase,
    uint128 remaining,
    bool isBid,
    int16 tick,
    bool isFlipOrder,
    int16 flipTick,
    uint128 quoteBudget,
    bytes32 feedId,
    uint64 quoteId,
    uint192 usdPerTokenE18
);
```

`amount` is the configured order size. `openBase` is the current cycle's unfilled cap. `remaining` is the active base liquidity under the captured quote. `quoteBudget` is nonzero only for FX bids.

```solidity
event FXOrderFilled(
    uint128 indexed orderId,
    address indexed maker,
    address indexed taker,
    bool isBid,
    int16 tick,
    uint128 baseAmount,
    uint128 quoteAmount,
    uint128 openBase,
    uint128 remaining,
    uint128 quoteBudget,
    bytes32 feedId,
    uint64 quoteId,
    uint192 usdPerTokenE18
);
```

`baseAmount` is the base amount filled. `quoteAmount` is the quote amount paid. `openBase`, `remaining`, and `quoteBudget` are the post-fill values. For an FX bid whose active side was exhausted, `openBase` and `remaining` are zero and `quoteBudget` is the post-refund value, normally zero.

```solidity
event FXOrderFlipped(
    uint128 indexed orderId,
    address indexed maker,
    address indexed token,
    uint128 amount,
    uint128 openBase,
    uint128 remaining,
    bool isBid,
    int16 tick,
    int16 flipTick,
    uint128 quoteBudget,
    bytes32 feedId,
    uint64 quoteId,
    uint192 usdPerTokenE18
);
```

For FX flips, the DEX also emits the existing `OrderFlipped` event for compatibility. In `OrderFlipped`, `amount` is the preserved configured size. Consumers that need active liquidity should read `FXOrderFlipped.remaining` or call `getFXOrder`.

```solidity
event FXOrderDropped(
    uint128 indexed orderId,
    address indexed maker,
    address indexed token,
    bool isBid,
    int16 tick,
    uint128 amount,
    uint128 openBase,
    uint128 remaining,
    uint128 quoteBudget,
    bytes32 feedId,
    uint64 quoteId,
    uint192 usdPerTokenE18,
    bytes4 reason
);
```

`FXOrderDropped` is emitted when an FX order is removed because it is stale, no longer authorized, or below the active dust threshold. `reason` is the selector for the DEX error that explains the drop, such as `StaleFXQuote()`, `FXFeedUnauthorized()`, `FXMakerUnauthorized()`, or `BelowFXRestingDust()`. If no DEX-valid quote is available, `quoteId = 0`, `usdPerTokenE18 = 0`, and `remaining = 0`.

The implementation should add explicit errors for FX oracle and cleanup failures. At minimum, it needs errors for invalid quotes, stale quotes, unauthorized feeds, below-dust active liquidity, non-FX orders passed to FX-only views, and FX orders that are not cleanable.

# Appendix: exact FX fill algorithms

This appendix defines the per-order fill rules used by the existing route traversal. `priceE18` is the captured latest quote for the pair. All helpers use the rounding rules in the specification above.

```text
currentRemaining(order, priceE18):
    if order.isBid == false:
        return order.openBase

    funded = fxQuoteToBase(order.quoteBudget, order.tick, priceE18, Down)
    return min(order.openBase, funded)

activeNotional(order, priceE18):
    return fxBaseToQuote(currentRemaining(order, priceE18),
                         order.tick,
                         priceE18,
                         Down)

maybeDropBeforeFill(order, priceE18):
    r = currentRemaining(order, priceE18)
    if r == 0:
        drop order through cleanup path
        return dropped

    if fxBaseToQuote(r, order.tick, priceE18, Down)
         < FX_RESTING_DUST_USD_E6:
        drop order through cleanup path
        return dropped

    return live

finishBidIfActiveSideExhausted(order, priceE18):
    if currentRemaining(order, priceE18) != 0:
        return stillResting

    order.openBase = 0
    refund order.quoteBudget to maker internal quote balance
    order.quoteBudget = 0
    delete order or rewrite it under the flip rules
    return terminal
```

A route traversal that drops an order continues with the next order at the same tick. A route traversal that cannot make progress on a live order MUST NOT loop. It must stop that leg and let the existing exact-in or exact-out slippage checks decide whether the swap succeeds.

## Exact input against asks

The taker spends quote tokens and receives base tokens.

```text
fillAskExactIn(order, quoteRemaining, priceE18):
    if maybeDropBeforeFill(order, priceE18) == dropped:
        return dropped

    baseOut = fxQuoteToBase(quoteRemaining, order.tick, priceE18, Down)
    fillBase = min(order.openBase, baseOut)

    if fillBase == 0:
        return noProgress

    quoteIn = fxBaseToQuote(fillBase, order.tick, priceE18, Up)

    if quoteIn == 0 or quoteIn > quoteRemaining:
        return noProgress

    order.openBase -= fillBase
    maker quote balance += quoteIn
    taker receives fillBase base

    if order.openBase == 0:
        delete order or rewrite it under the flip rules

    return (quoteIn, fillBase)
```

## Exact output against asks

The taker receives an exact amount of base tokens and pays quote tokens.

```text
fillAskExactOut(order, baseRemainingOut, priceE18):

    fillBase = min(order.openBase, baseRemainingOut)

        return noProgress

    order.openBase -= fillBase

    taker receives fillBase base

    if order.openBase == 0:
        delete order or rewrite it under the flip rules

```

## Exact input against bids

The taker spends base tokens and receives quote tokens.

```text
fillBidExactIn(order, baseRemainingIn, priceE18):
    if maybeDropBeforeFill(order, priceE18) == dropped:
        return dropped

    remainingNow = currentRemaining(order, priceE18)
    fillBase = min(remainingNow, baseRemainingIn)

    if fillBase == 0:
        return noProgress

    quoteOut = fxBaseToQuote(fillBase, order.tick, priceE18, Down)

    if quoteOut == 0:
        return noProgress

    order.openBase -= fillBase
    order.quoteBudget -= quoteOut
    maker base balance += fillBase
    taker receives quoteOut quote

    finishBidIfActiveSideExhausted(order, priceE18)

    return (fillBase, quoteOut)
```

## Exact output against bids

The taker receives an exact amount of quote tokens and pays base tokens.

```text
fillBidExactOut(order, quoteRemainingOut, priceE18):
    if maybeDropBeforeFill(order, priceE18) == dropped:
        return dropped

    remainingNow = currentRemaining(order, priceE18)
    maxQuoteOut = min(
        order.quoteBudget,
        fxBaseToQuote(remainingNow, order.tick, priceE18, Down)
    )
    quoteOut = min(quoteRemainingOut, maxQuoteOut)

    if quoteOut == 0:
        return noProgress

    baseIn = fxQuoteToBase(quoteOut, order.tick, priceE18, Up)

    if baseIn > remainingNow:
        baseIn = remainingNow
        quoteOut = fxBaseToQuote(baseIn, order.tick, priceE18, Down)

    if baseIn == 0 or quoteOut == 0:
        return noProgress

    order.openBase -= baseIn
    order.quoteBudget -= quoteOut
    maker base balance += baseIn
    taker receives quoteOut quote

    finishBidIfActiveSideExhausted(order, priceE18)

    return (baseIn, quoteOut)
```

For exact-output swaps, the route continues until the requested output amount has been delivered. If the book is exhausted first, the swap reverts with the existing insufficient-liquidity error. If a bid becomes terminal, unused `quoteBudget` is refunded before deletion or flip rewrite.

# Observability

The FX events add details that existing DEX events cannot carry:

- `FXOrderPlaced` records the oracle quote that priced the order and the active liquidity it created.
- `FXOrderFilled` records the oracle quote, base amount, quote amount, and post-fill FX state for each fill.
- `FXOrderFlipped` records whether the flip rested with the full configured size or with less liquidity because the maker had less available balance.
- `FXOrderDropped` records when an FX order was removed because it was stale, unauthorized, or below the dust threshold.

Existing `OrderFilled`, `OrderCancelled`, `OrderFlipped`, and `FlipFailed` events continue to describe the common order lifecycle. FX indexers must consume the FX events to track quote ids, quote prices, quote budgets, `openBase`, and active liquidity.

Oracle updates can change an FX bid's current `remaining` without any DEX event. Indexers that display current FX liquidity must combine DEX order state with PriceOracle events or call the FX views.

# Risks and non-goals

This TIP does not turn the StablecoinDEX into an independent FX price-discovery venue. The oracle sets the reference FX rate, and ticks express spreads around that rate. A maker who wants to quote an absolute EUR/USD price must translate that view into a tick against the current oracle.

The DEX also does not price issuer-specific depeg risk for non-USD tokens that share a feed. If two EUR tokens use the same EUR/USD feed, their DEX prices differ only by maker ticks and available liquidity. Issuer risk still matters, but it is not represented by the oracle unless the issuer selects a different feed.

Using `latestQuote` instead of the delayed `usableQuote` makes DEX trading responsive, but it gives oracle updates same-block market impact. Makers accept the risk that an updater transaction immediately before a swap changes the effective price of their resting orders.

During volatility, FX bid liquidity can disappear because quote budgets no longer fund as much base as they did when the orders were placed. This is intended. It prevents insolvent bids, but takers and routers must quote against current liquidity rather than stored tick totals.

# Invariants

1. USD/USD DEX behavior is unchanged.
2. Every state-changing FX DEX operation uses the DEX-valid latest PriceOracle quote.
3. A fill and the automatic flip caused by that fill use the same captured quote.
4. An FX bid never pays more quote tokens than its escrowed `quoteBudget`.
5. An FX bid's current `remaining` is `min(openBase, quoteBudget converted to base at the captured quote)`.
6. Oracle movement alone does not change `openBase` or `quoteBudget`.
7. Actual fills and synthetic fills reduce `openBase`.
8. FX flip orders preserve `amount` across flips.
9. FX flip orders may reduce `openBase` when the maker has less internal balance than the preserved `amount` requires.
10. If the active liquidity after an FX flip is below `FX_RESTING_DUST_USD_E6`, no new order rests.
11. Cancelling an FX ask refunds exactly its `openBase` base tokens.
12. Cancelling an FX bid refunds exactly its `quoteBudget` quote tokens.
13. `OrderFlipped.amount` for an FX order is the preserved configured size, not necessarily the active liquidity.
14. FX order storage reads and writes are version-aware and remain compatible with TIP-1062 mixed-version linked lists.
15. `getOrder`, `getFXOrder`, FX events, and fill paths report `remaining` as active base liquidity under the quote used by that path, or zero if no DEX-valid quote exists.
16. `getTickLevel.totalLiquidity` is only an upper bound for FX bid liquidity.
17. View quotes use the same `latestQuote` rule as state-changing swaps, assuming no intervening state change.
18. Permissionless FX cleanup follows TIP-1060 and TIP-1064 storage-accounting rules and does not create a separate caller bounty.
