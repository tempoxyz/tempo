# TIP-1071: FX support for StablecoinDEX

## Abstract

This TIP lets the StablecoinDEX trade non-USD TIP-20 tokens whose `quoteToken` is a USD token. An FX pair is priced by multiplying the base token's current PriceOracle quote by the order's tick multiplier: an order at tick `0` trades at the oracle price, and positive or negative ticks trade at a premium or discount to it. A tick is therefore a maker's spread around the feed, not an absolute price that goes stale when the feed moves.

The DEX shape is unchanged: one order book per `(base, quote)` pair, price-time priority within each tick, and the same swap entry points. Two behaviors are new. FX bids escrow a fixed USD budget, so oracle movement cannot make them insolvent. And FX trading requires a feed that is fresh, has a continuous quote history, and has had a stable configuration — feed repoints and policy changes freeze the pair briefly so makers can react before the book reprices.

## Motivation

Tempo's StablecoinDEX trades different issuers' USD tokens. The same mechanism should serve EUR or GBP stablecoins trading against a USD quote token.

Market makers should not have to cancel and replace orders every time the EUR/USD feed moves. With oracle-relative ticks they express a standing spread — buy EUR 10 bps below the feed, sell 10 bps above — and the oracle carries the level. Flip orders complete the picture: one logical order runs a two-sided market. Because oracle movement changes how much of a bid its escrowed budget can fund, this TIP keeps a flip order's configured size stable and lets the actively resting liquidity be smaller when the maker's balance cannot fund the full size.

## Assumptions

- TIP-1054 defines the PriceOracle (`latestQuote`, `previousQuote`, `QuotePolicy`, token authorizations) and the TIP-20 `priceOracleFeedId`. This TIP additionally requires the two configuration-block surfaces defined below, which should be folded back into TIP-1054.
- TIP-1056 flip-in-place is active: a flip order keeps its `orderId` when it flips.
- TIP-1062 versioned DEX order storage is active. FX orders need new fields and use a new storage version.
- TIP-1060 (Storage Credits) and TIP-1064 (Order Storage Credits) are active: deleting a nonzero order-record slot credits the order's maker; shared accounting storage is non-creditable.
- Token amounts and USD amounts use the existing 6-decimal convention.
- A non-USD token using this TIP has a USD-denominated `quoteToken`. FX/FX pairs are out of scope; they route through USD tokens.
- Existing USD/USD DEX behavior is unchanged.
- This TIP activates no earlier than the fork that activates both TIP-1054 and TIP-1062.

If a pair's feed becomes invalid for DEX use, the pair cannot trade, but existing orders remain cancellable and cleanable.

## Threat Model

- **Feed admin and updater.** The DEX prices trades off their feed. The validity rules below bound how fast a quote can move and forbid trading across configuration changes, but a plausibly wrong price within those bounds still reprices the book.
- **Token issuer and admin.** They control token metadata, `quoteToken`, feed selection, pause state, and transfer policy. Feed and policy changes freeze the pair for a cooldown instead of repricing it atomically; the residual power is to freeze or grief, not to extract resting escrow in one transaction.
- **Makers.** They choose ticks, sizes, and flip behavior. A maker cannot receive more than a taker paid, and an FX bid cannot spend more than its escrowed budget.
- **Takers.** They are protected by the existing exact-in and exact-out slippage bounds. A view quote and a later swap can disagree if the oracle or the book changes in between.
- **The DEX.** Trusted protocol code. It must use one captured quote per feed within a state-changing operation so fills and the flips they trigger are priced consistently.

# Specification

## Pair eligibility

`createPair(base)` continues to accept USD/USD pairs. It additionally accepts an FX/USD pair when:

1. `base` is a valid non-USD TIP-20 token,
2. `quote = base.quoteToken()` is a valid USD-denominated TIP-20 token, and
3. `base.priceOracleFeedId()` is nonzero and the selected feed authorizes `base`.

The pair key remains `computeBookKey(base, quote)`. A USD token with a nonzero `priceOracleFeedId` is invalid under TIP-1054 and cannot create an FX pair.

The DEX does not snapshot the feed id per order. Orders are always priced against the base token's current `priceOracleFeedId()`, so an issuer can migrate the whole book to a new feed. The stability rule below makes that migration a scheduled event rather than an instantaneous repricing. If the issuer later changes `quoteToken`, the book stops matching `base.quoteToken()`, becomes unroutable, and its orders become cleanable.

## Required oracle surfaces

The validity rules below need to know when a feed's configuration last changed and when a token last changed feeds. Neither is exposed by TIP-1054 today, so this TIP requires two additions:

```solidity
// PriceOracle: block of the feed's creation or most recent
// updater or policy change. The oracle already clears quote
// tuples on those changes; this records when.
function configChangedBlock(bytes32 feedId) external view returns (uint64);

// TIP-20: block of the most recent successful
// setPriceOracleFeedId, including the first.
function priceOracleFeedChangedBlock() external view returns (uint64);
```

## DEX quote validity

The DEX trades against the feed's `latestQuote`, not the one-block-delayed quote that fee settlement uses. This makes resting orders track the newest published price: within a block, transaction order decides whether a trade sees the previous or the new quote, and FX makers bear that race. The delayed quote would instead make every resting order stale by one block, which external arbitrageurs harvest continuously; a race against the updater is the cheaper risk, and the change and stability rules below bound it.

A quote is **DEX-valid** for a base token when all of the following hold, with `W = min(policy.maxQuoteAgeBlocks, PROTOCOL_MAX_FX_DEX_QUOTE_AGE_BLOCKS)`:

1. **Configured.** `base.priceOracleFeedId()` is nonzero, the feed authorizes `base`, and the feed has a valid `QuotePolicy` with `maxQuoteChangeBps <= PROTOCOL_MAX_FX_DEX_QUOTE_CHANGE_BPS`.
2. **Fresh.** `latestQuote(feedId)` has a nonzero `quoteId` and `usdPerTokenE18`, and `block.number - latestQuoteBlock <= W`.
3. **Continuous.** `previousQuoteBlock != 0` and `latestQuoteBlock - previousQuoteBlock <= W`.
4. **Stable.** `block.number - configChangedBlock(feedId) >= FX_FEED_COOLDOWN_BLOCKS` and `block.number - base.priceOracleFeedChangedBlock() >= FX_FEED_COOLDOWN_BLOCKS`.

Each rule closes a specific hole:

- **Freshness** uses a DEX-specific age cap that should be lower than the fee-payment cap. Fees are small and bounded; DEX trades move more value and need fresher prices.
- **Continuity** requires the last two quotes to sit within one freshness window of each other. A lone quote after a feed outage or a policy reset cannot price trades: TIP-1054 trusts the first quote after a reset at any level, and the quote-change limit only binds between consecutive quotes. Requiring two in-window quotes keeps the change-limit chain intact and adds one update of hysteresis after outages.
- **Stability** answers the atomic-repricing attack. Without it, a token admin could repoint `priceOracleFeedId` to a feed they control — or a feed admin could reset the policy and push a bootstrap quote — and trade against the mispriced book in the same transaction. The cooldown turns any configuration change into a visible freeze: `PriceOracleFeedSet`, `PriceOracleUpdaterSet`, and `PriceOraclePolicySet` announce it, and makers have the full cooldown to cancel, which never requires a valid quote. New feeds and pairs wait out the cooldown once; honest policy changes freeze the pair for one cooldown. Both are intended.

Every state-changing FX operation MUST read and validate the quote before pricing the pair, and MUST use one captured quote per feed for the whole operation. View functions apply the same rule; a view quote and a later swap can still disagree if state changes in between.

## Constants

Values are proposed activation values, assume roughly one-second blocks, and must be fixed before this TIP leaves Draft.

```text
ORACLE_SCALE                          = 1e18
MIN_ORDER_USD_E6                      = 100_000_000  // $100, same value as MIN_ORDER_AMOUNT
FX_RESTING_DUST_USD_E6                = 1_000_000    // $1
PROTOCOL_MAX_FX_DEX_QUOTE_AGE_BLOCKS  = 60           // ~1 minute
PROTOCOL_MAX_FX_DEX_QUOTE_CHANGE_BPS  = 500
FX_FEED_COOLDOWN_BLOCKS               = 300          // ~5 minutes
FX_CLEANUP_STALE_BLOCKS               = 600          // ~10 minutes
FX_MAX_DROPS_PER_OPERATION            = 16
```

`MIN_ORDER_USD_E6` gates new orders. `FX_RESTING_DUST_USD_E6` is lower because it applies only to existing logical orders that would otherwise rest with a tiny active amount.

## Oracle-relative ticks

For USD/USD pairs, ticks keep their current meaning. For FX/USD pairs, the tick is a multiplier on the oracle price:

```text
multiplier(tick) = tickToPrice(tick) / PRICE_SCALE
limitPrice       = usdPerTokenE18 * multiplier(tick)
```

If one EUR token is worth `1.100000 USD` and `tickToPrice(tick) = 100500`, the order price is `1.105500 USD` per EUR token. All orders in a book share the same captured quote at execution time, so price priority is unchanged: higher bid ticks are better bids, lower ask ticks are better asks.

FX pairs keep the existing tick geometry: ticks in `[MIN_TICK, MAX_TICK] = [-2000, 2000]`, aligned to `TICK_SPACING = 10`, with `tickToPrice(tick) = PRICE_SCALE + tick`. Spreads are quotable in 1 bp steps within ±2% of the oracle price. The band bounds maker spreads, not market prices — the oracle carries the level. A token whose feed cannot track the market within ±2% per freshness window is a poor fit for this design.

## Arithmetic

The DEX adds FX conversion helpers. `baseAmount` is in the non-USD base token, `quoteAmount` in the USD quote token, and `priceE18` is `usdPerTokenE18` from the captured quote.

```text
ceilDiv(a, b) = 0 if a == 0, otherwise floor((a - 1) / b) + 1

fxBaseToQuote(baseAmount, tick, priceE18, rounding)
    = round(baseAmount * priceE18 * tickToPrice(tick)
            / (ORACLE_SCALE * PRICE_SCALE), rounding)

fxQuoteToBase(quoteAmount, tick, priceE18, rounding)
    = round(quoteAmount * ORACLE_SCALE * PRICE_SCALE
            / (priceE18 * tickToPrice(tick)), rounding)
```

All arithmetic uses checked unsigned 256-bit integers; overflow, underflow, division by zero, or a value that does not fit in storage MUST revert. These products cannot overflow in practice: TIP-1054's `updateLatestQuote` rejects any price whose smallest token unit exceeds one cent, so `usdPerTokenE18 <= 1e22`, and with `u128` amounts and `tickToPrice <= 102000` every intermediate fits in 256 bits. The same cap means a token worth more than $10,000 per whole token cannot configure a feed; such assets are outside this TIP's scope.

Rounding favors the maker, matching the current DEX:

- Bid escrow and bid budget calculations round up.
- Bid fills round the quote paid to the taker down.
- Ask fills round the quote paid by the taker and credited to the maker up.
- Exact-output input requirements round up; exact-input outputs round down where needed to avoid overspending input.

A fill that would move either token leg by zero MUST NOT execute. Traversal either continues to another fillable order or terminates with the existing insufficient-liquidity, insufficient-output, or max-input error.

## Minimum sizes and dust

A new FX order MUST revert if its USD notional at the limit price is below the minimum:

```text
fxBaseToQuote(amount, tick, priceE18, Down) < MIN_ORDER_USD_E6
```

Automatic flips instead use the lower dust threshold, because they preserve an existing logical order rather than creating state:

```text
fxBaseToQuote(activeBase, tick, priceE18, Down) < FX_RESTING_DUST_USD_E6
```

If the active liquidity after a flip is below the dust threshold, the DEX drops the flip instead of resting dust.

## FX order state

FX orders track three quantities that must not be collapsed:

- `amount` — the configured cycle size, preserved across flips.
- `openBase` — the unfilled base cap for the current side, reduced only by actual and synthetic fills.
- `remaining` — the base amount that can fill right now under the captured quote.

For asks, `remaining = openBase`: the base tokens are already escrowed. For bids, `quoteBudget` holds the escrowed USD budget and:

```text
currentRemaining(order, priceE18) = min(
    order.openBase,
    fxQuoteToBase(order.quoteBudget, order.tick, priceE18, Down)
)
```

Oracle movement alone never changes `openBase` or `quoteBudget`. If the price rises, a bid's `remaining` can fall below `openBase`; if the price falls back, `remaining` recovers, but never above `openBase`. This keeps orders path-independent: a temporarily underfunded bid is not permanently shrunk unless it is filled, flipped, cancelled, or cleaned up.

The logical fields are:

```text
orderId          uint128   // mapping key remains canonical
maker            address
bookKey          bytes32
isBid            bool
isFlip           bool
tick             int16
flipTick         int16
amount           uint128
openBase         uint128
quoteBudget      uint128   // zero for asks
prev             uint128
next             uint128
```

### Storage layout

FX orders use a new order-storage version after TIP-1062, decoded by the slot-0 version discriminator, and participate in mixed-version linked lists. Two requirements are consensus-relevant:

1. FX order records MUST be decoded by their stored version, never by re-reading token state, which is mutable.
2. The set of occupied storage slots in an FX order record MUST NOT change between placement and deletion. In particular, flips MUST NOT zero or create slots: `quoteBudget` is zero while the order rests as an ask, so it MUST share a slot with a field that is nonzero for the order's lifetime, such as `amount`.

The second rule exists because of TIP-1064: deleting an order-record slot credits the maker, and creating one during a taker-triggered flip is charged to the taker's transaction. If flips changed slot occupancy, every flip cycle would mint the maker a storage credit at a taker's expense.

### Synthetic fills

A synthetic fill reduces `openBase` without moving tokens and without emitting `OrderFilled`. It keeps flip accounting consistent when the current side cannot rest the full configured `amount`:

1. A flip funds less than `amount` on the new side: `openBase` is set to the fundable amount, and the shortfall is treated as already filled.
2. An FX bid's entire `remaining` is filled while `openBase` was larger because the oracle had moved up: the unfunded part is synthetically filled before the order is deleted or rewritten as a flip.

Synthetic fills pay no one. They record that part of the current cycle could not rest.

### Tick-level liquidity

For FX asks, tick-level `totalLiquidity` is the sum of ask `openBase` values and equals active liquidity while the quote is valid. For FX bids it is the sum of bid `openBase` values — an upper bound only, because each order's budget and the current quote decide what is actually fundable. Fill and quote paths MUST inspect bid orders individually via `currentRemaining`.

`getTickLevel` remains ABI-compatible; FX-aware clients MUST NOT treat its `totalLiquidity` as fillable bid liquidity. For bounded inspection the DEX adds a paginated view:

```solidity
function getFXTickLiquidity(
    address base,
    int16 tick,
    bool isBid,
    uint128 startOrderId,
    uint16 limit
)
    external
    view
    returns (
        uint128 openBaseLiquidity,
        uint128 currentLiquidity,
        uint128 nextOrderId
    );
```

The view walks at most `limit` orders from `startOrderId`, or from the tick head if `startOrderId == 0`. `limit == 0` MUST revert; `startOrderId` MUST be zero or an active order at the tick. `nextOrderId == 0` means the page reached the end. Without a DEX-valid quote, `currentLiquidity` is zero.

### Order views

`getOrder(uint128)` remains ABI-compatible. For FX orders it returns `amount = amount` and `remaining = currentRemaining` under a DEX-valid quote, or `remaining = 0` without one.

FX-aware clients use:

```solidity
struct FXOrderInfo {
    uint128 orderId;
    address maker;
    bytes32 bookKey;
    bool isBid;
    bool isFlip;
    int16 tick;
    int16 flipTick;
    uint128 amount;
    uint128 openBase;
    uint128 remaining;
    uint128 quoteBudget;
    bool quoteValid;
    bytes32 feedId;
    uint64 quoteId;
    uint64 quoteBlock;
    uint192 usdPerTokenE18;
}

function getFXOrder(uint128 orderId) external view returns (FXOrderInfo memory order);
```

`getFXOrder` MUST revert for non-FX and inactive orders. Without a DEX-valid quote it returns `quoteValid = false`, `remaining = 0`, and zeroed quote fields, while still returning `openBase` and `quoteBudget`.

## Placing FX orders

Placement captures a DEX-valid quote before computing escrow; a pair in feed cooldown cannot accept orders. Funding uses the existing placement path (internal DEX balance, then external wallet), and the existing policy and pause checks apply to both tokens, including the token the maker would receive on fills.

For an ask: validate against `MIN_ORDER_USD_E6`, escrow `amount` base tokens, store `amount = amount`, `openBase = amount`, `quoteBudget = 0`.

For a bid: validate against `MIN_ORDER_USD_E6`, compute `quoteBudget = fxBaseToQuote(amount, tick, priceE18, Up)`, escrow that many quote tokens, store `amount = amount`, `openBase = amount`, and `quoteBudget`.

## Filling FX orders

A state-changing swap captures one DEX-valid quote per FX pair in the route before filling that pair, and uses it for every fill on the pair. Every FX fill emits `OrderFilled`, then `FXOrderFilled`.

### Filling asks

For an ask fill of `fillBase <= openBase`:

```text
quotePaid = fxBaseToQuote(fillBase, tick, priceE18, Up)
```

The taker receives `fillBase` base tokens; the maker's internal balance receives `quotePaid` quote tokens; `openBase` decreases by `fillBase`. An ask is terminal at `openBase == 0` and is then deleted or rewritten under the flip rules.

### Filling bids

The DEX computes `remaining = currentRemaining(order, priceE18)` first; the fill MUST satisfy `fillBase <= remaining`.

```text
quotePaid = fxBaseToQuote(fillBase, tick, priceE18, Down)
```

The taker receives `quotePaid` quote tokens; the maker's internal balance receives `fillBase` base tokens; `openBase` decreases by `fillBase` and `quoteBudget` by `quotePaid`.

A bid is terminal when `currentRemaining == 0` after the fill. Any leftover `openBase` is synthetically filled, unused `quoteBudget` is refunded to the maker's internal balance, and the order is deleted or rewritten as a flip. The `OrderFilled` event for the fill that exhausts the bid reports `partialFill = false` even when `openBase` was still nonzero before the synthetic fill; the synthetic fill itself emits no event.

### Drop budget

If traversal encounters an FX order whose active notional is below `FX_RESTING_DUST_USD_E6`, the DEX MAY drop it through the cleanup path and continue. A single state-changing operation drops at most `FX_MAX_DROPS_PER_OPERATION` orders; at the cap, traversal treats the rest of that leg as having no liquidity. This bounds the cleanup gas a swap can be made to pay. Exact algorithms are in the appendix.

## Flip orders

FX flip orders preserve `amount` across flips; `openBase` and `remaining` describe what the current cycle could rest and can fill. When a flip order's active side is exhausted — for a bid, when `remaining` is exhausted under the captured quote, even if `openBase` was larger — the DEX rewrites the same `orderId` in place, using the same captured quote that priced the fill.

Automatic flips MUST NOT pull from the maker's external wallet. They use only the maker's internal DEX balance after the fill and any budget refund.

### Bid to ask

Let `A = order.amount` and `B` the maker's internal base-token balance after the fill and refund:

```text
newOpenBase = min(A, B)
```

If the USD notional of `newOpenBase` at `flipTick` is below `FX_RESTING_DUST_USD_E6`, the flip is dropped. Otherwise the DEX escrows `newOpenBase` base tokens and rewrites the order as an ask with `amount = A`, `openBase = newOpenBase`, `quoteBudget = 0`. A shortfall `A - newOpenBase` is a synthetic fill: no trade, no `OrderFilled`.

### Ask to bid

Let `A = order.amount` and `Q` the maker's internal quote-token balance after the fill:

```text
fundableBase   = min(A, fxQuoteToBase(Q, flipTick, priceE18, Down))
newQuoteBudget = fxBaseToQuote(fundableBase, flipTick, priceE18, Up)
```

If the USD notional of `fundableBase` at `flipTick` is below `FX_RESTING_DUST_USD_E6`, the flip is dropped. Otherwise the DEX escrows `newQuoteBudget` quote tokens and rewrites the order as a bid with `amount = A`, `openBase = fundableBase`, and `quoteBudget = newQuoteBudget`. A shortfall is a synthetic fill as above.

### Same-tick flips

TIP-1030 and TIP-1056 behavior applies unchanged; a same-tick FX flip uses the same multiplier on the captured quote.

### Failed and dropped flips

A flip that fails for policy, pause, authorization, or arithmetic reasons follows existing `FlipFailed` behavior. A flip with valid authorization whose fundable liquidity is below the dust threshold is *dropped*: the filled order is deleted, nothing new rests, and the DEX emits `FXOrderDropped`.

## Permissionless cleanup

Anyone MAY call:

```solidity
function cleanupFXOrder(uint128 orderId) external;
```

It removes an active FX order when at least one condition holds, and MUST revert otherwise:

1. **Stale beyond grace.** The pair has had no DEX-valid quote for a prolonged period: `block.number - max(latestQuoteBlock, configChangedBlock(feedId)) > FX_CLEANUP_STALE_BLOCKS`. Ordinary staleness and feed cooldowns are not enough — a book frozen by a transient outage stays intact, since stale orders cannot trade anyway. Only an abandoned feed exposes orders to cleanup.
2. **De-configured.** `base.priceOracleFeedId()` is zero, the feed no longer authorizes `base`, or the feed's policy is invalid for DEX use. These are durable states and cleanable immediately.
3. **Maker unauthorized.** The maker no longer passes the transfer-policy checks the order needs to stay open.
4. **Dust.** The order's active notional is below `FX_RESTING_DUST_USD_E6` under the latest DEX-valid quote.
5. **Unroutable.** `base.quoteToken()` no longer matches the book's quote token.

Cleanup refunds exactly what cancellation would: `openBase` base tokens for asks, `quoteBudget` quote tokens for bids. Because refunds equal remaining escrow, cleanup can grief queue position but never extract funds.

Cleanup is a normal order deletion for storage accounting: under TIP-1060 and TIP-1064, deleted order-record slots credit the order's maker, shared accounting storage credits no one, and there is no separate bounty for the caller.

## Cancellation

Cancelling an FX ask refunds its `openBase` base tokens; cancelling an FX bid refunds its `quoteBudget` quote tokens. The difference between `amount` and `openBase` represents actual or synthetic fills and is not owed. Cancellation never requires a DEX-valid quote and emits the existing cancellation event.

The existing entry points MUST be version-aware. `cancel(orderId)` on an FX order applies the refund rules above and MUST NOT recompute a bid refund from the tick, which is how USD/USD refunds are derived. `cancelStaleOrder(orderId)` on an FX order MUST behave exactly as `cleanupFXOrder(orderId)`: same conditions, same refunds, same events.

## Interactions with routing

Path finding still follows `quoteToken` chains, so an FX token trades directly against its USD quote token and through the USD graph beyond it. A route MAY contain multiple FX pairs; the DEX captures and validates one quote per FX base token, and if one feed serves several tokens in a route, all of them MUST use the same captured quote.

## Interfaces and events

FX orders emit FX events after the existing ones, carrying the captured quote and active liquidity. Ordering is consensus behavior:

1. Placement: `OrderPlaced`, then `FXOrderPlaced`.
2. Fill: `OrderFilled`, then `FXOrderFilled`.
3. Successful flip: `OrderFilled`, `FXOrderFilled`, `OrderFlipped`, `FXOrderFlipped`.
4. Flip dropped below dust: `OrderFilled`, `FXOrderFilled`, `FXOrderDropped`. No `OrderFlipped`, no `FlipFailed`.
5. Resting order dropped by cleanup, or by a swap that finds it stale or dust: `OrderCancelled`, then `FXOrderDropped`.
6. Non-dust flip failure: `OrderFilled`, `FXOrderFilled`, `FlipFailed` (existing TIP-1056 behavior).

```solidity
event FXOrderPlaced(
    uint128 indexed orderId,
    address indexed maker,
    address indexed token,
    uint128 amount,
    uint128 openBase,
    uint128 remaining,
    bool isBid,
    int16 tick,
    bool isFlipOrder,
    int16 flipTick,
    uint128 quoteBudget,
    bytes32 feedId,
    uint64 quoteId,
    uint64 quoteBlock,
    uint192 usdPerTokenE18
);

event FXOrderFilled(
    uint128 indexed orderId,
    address indexed maker,
    address indexed taker,
    bool isBid,
    int16 tick,
    uint128 baseAmount,
    uint128 quoteAmount,
    uint128 openBase,
    uint128 remaining,
    uint128 quoteBudget,
    bytes32 feedId,
    uint64 quoteId,
    uint64 quoteBlock,
    uint192 usdPerTokenE18
);

event FXOrderFlipped(
    uint128 indexed orderId,
    address indexed maker,
    address indexed token,
    uint128 amount,
    uint128 openBase,
    uint128 remaining,
    bool isBid,
    int16 tick,
    int16 flipTick,
    uint128 quoteBudget,
    bytes32 feedId,
    uint64 quoteId,
    uint64 quoteBlock,
    uint192 usdPerTokenE18
);

event FXOrderDropped(
    uint128 indexed orderId,
    address indexed maker,
    address indexed token,
    bool isBid,
    int16 tick,
    uint128 amount,
    uint128 openBase,
    uint128 remaining,
    uint128 quoteBudget,
    bytes32 feedId,
    uint64 quoteId,
    uint64 quoteBlock,
    uint192 usdPerTokenE18,

);
```

In all four events, `amount` is the configured size, `openBase` the current cycle's unfilled cap, `remaining` the active liquidity under the captured quote, and `quoteBudget` nonzero only for bids. `FXOrderFilled` reports post-fill values; for a bid whose active side was exhausted, `openBase` and `remaining` are zero and `quoteBudget` is post-refund. In `OrderFlipped`, `amount` remains the preserved configured size; consumers needing active liquidity read `FXOrderFlipped.remaining` or `getFXOrder`. When no DEX-valid quote exists, `quoteId`, `quoteBlock`, `usdPerTokenE18`, and `remaining` are zero.

The DEX defines these errors; `FXOrderDropped.reason` MUST be the selector of the condition that caused the drop, drawn from the starred entries:

```solidity
error NotFXOrder();
error FXOrderNotCleanable();
error FXPairNotEligible();      // *
error InvalidFXQuote();         // * validity rule 1 failed
error StaleFXQuote();           // * validity rule 2 or 3 failed
error FXFeedCooldown();         //   validity rule 4 failed
error FXMakerUnauthorized();    // *
error BelowFXRestingDust();     // *
```

# Appendix: exact FX fill algorithms

This appendix defines the per-order fill rules used by the existing route traversal. `priceE18` is the captured quote for the pair; rounding follows the specification. Drops through the cleanup path count against `FX_MAX_DROPS_PER_OPERATION`.

```text
currentRemaining(order, priceE18):
    if order.isBid == false:
        return order.openBase

    funded = fxQuoteToBase(order.quoteBudget, order.tick, priceE18, Down)
    return min(order.openBase, funded)

maybeDropBeforeFill(order, priceE18):
    r = currentRemaining(order, priceE18)
    if r == 0
       or fxBaseToQuote(r, order.tick, priceE18, Down) < FX_RESTING_DUST_USD_E6:
        drop order through cleanup path
        return dropped
    return live

finishBidIfActiveSideExhausted(order, priceE18):
    if currentRemaining(order, priceE18) != 0:
        return stillResting

    order.openBase = 0
    refund order.quoteBudget to maker internal quote balance
    order.quoteBudget = 0
    delete order or rewrite it under the flip rules
    return terminal
```

A traversal that drops an order continues with the next order at the same tick. A traversal that cannot make progress on a live order MUST NOT loop; it stops that leg and lets the existing exact-in or exact-out checks decide whether the swap succeeds.

## Exact input against asks

The taker spends quote tokens and receives base tokens.

```text
fillAskExactIn(order, quoteRemaining, priceE18):
    if maybeDropBeforeFill(order, priceE18) == dropped:
        return dropped

    baseOut = fxQuoteToBase(quoteRemaining, order.tick, priceE18, Down)
    fillBase = min(order.openBase, baseOut)

    if fillBase == 0:
        return noProgress

    quoteIn = fxBaseToQuote(fillBase, order.tick, priceE18, Up)

    if quoteIn == 0 or quoteIn > quoteRemaining:
        return noProgress

    order.openBase -= fillBase
    maker quote balance += quoteIn
    taker receives fillBase base

    if order.openBase == 0:
        delete order or rewrite it under the flip rules

    return (quoteIn, fillBase)
```

## Exact input against bids

The taker spends base tokens and receives quote tokens.

```text
fillBidExactIn(order, baseRemainingIn, priceE18):
    if maybeDropBeforeFill(order, priceE18) == dropped:
        return dropped

    remainingNow = currentRemaining(order, priceE18)
    fillBase = min(remainingNow, baseRemainingIn)

    if fillBase == 0:
        return noProgress

    quoteOut = fxBaseToQuote(fillBase, order.tick, priceE18, Down)

    if quoteOut == 0:
        return noProgress

    order.openBase -= fillBase
    order.quoteBudget -= quoteOut
    maker base balance += fillBase
    taker receives quoteOut quote

    finishBidIfActiveSideExhausted(order, priceE18)

    return (fillBase, quoteOut)
```

## Exact output against bids

The taker receives an exact amount of quote tokens and pays base tokens.

```text
fillBidExactOut(order, quoteRemainingOut, priceE18):
    if maybeDropBeforeFill(order, priceE18) == dropped:
        return dropped

    remainingNow = currentRemaining(order, priceE18)
    maxQuoteOut = min(
        order.quoteBudget,
        fxBaseToQuote(remainingNow, order.tick, priceE18, Down)
    )
    quoteOut = min(quoteRemainingOut, maxQuoteOut)

    if quoteOut == 0:
        return noProgress

    baseIn = fxQuoteToBase(quoteOut, order.tick, priceE18, Up)

    if baseIn > remainingNow:
        baseIn = remainingNow
        quoteOut = fxBaseToQuote(baseIn, order.tick, priceE18, Down)

    if baseIn == 0 or quoteOut == 0:
        return noProgress

    order.openBase -= baseIn
    order.quoteBudget -= quoteOut
    maker base balance += baseIn
    taker receives quoteOut quote

    finishBidIfActiveSideExhausted(order, priceE18)

    return (baseIn, quoteOut)
```

For exact-output swaps, traversal continues until the requested output is delivered, or reverts with the existing insufficient-liquidity error if the book is exhausted. When a bid becomes terminal, unused `quoteBudget` is refunded before deletion or flip rewrite.

# Observability

The FX events carry what the existing events cannot: the feed id, quote id, quote block, and price behind every placement, fill, flip, and drop, plus post-operation `openBase`, `remaining`, and `quoteBudget`. Existing events continue to describe the common lifecycle; FX indexers must consume the FX events to track quote and budget state.

Oracle updates change FX bid `remaining` without any DEX event. Indexers displaying current liquidity must combine order state with PriceOracle events or call the FX views.

# Risks and non-goals

**No independent price discovery.** The oracle sets the level; ticks express spreads around it. A maker with an absolute price view must translate it into a tick against the current quote. The DEX also does not price issuer-specific risk: two EUR tokens sharing one feed differ only by ticks and depth.

**Updater exposure.** Trading against `latestQuote` gives oracle updates same-block market impact. A maker's spread protects them only up to the feed's `maxQuoteChangeBps` per block — a compromised updater can extract roughly the difference, each block, until keys are rotated. Makers should quote pairs whose feed policy is tighter than their spread; `PROTOCOL_MAX_FX_DEX_QUOTE_CHANGE_BPS` bounds what any DEX-eligible feed can allow.

**Recovery walks.** After a feed outage during a market move, the quote-change limit forces the feed to walk back to the market at `maxQuoteChangeBps` per block. Once the continuity rule passes, the book trades at lagging prices until the walk completes. Makers should cancel during outages; cancellation never needs a valid quote.

**Vanishing bid liquidity.** During volatility, bid budgets fund less base than at placement, and displayed liquidity shrinks. This is intended — it is what prevents insolvent bids — but routers must quote against current liquidity, not stored tick totals.

**Cleanup economics.** Swaps that drop stale or dust orders pay the deletion gas; under TIP-1064 the storage credits go to the order's maker, and shared-slot deletions credit no one. `FX_MAX_DROPS_PER_OPERATION` bounds the per-swap exposure.

**Blocks, not wall-clock.** All freshness and cooldown windows are measured in blocks. If block production stalls, a quote can stay technically fresh while old in real time.

# Invariants

1. USD/USD DEX behavior is unchanged.
2. Every state-changing FX operation prices against a captured DEX-valid `latestQuote`; a fill and the flip it triggers share the quote; a feed appearing more than once in a route is captured once.
3. DEX validity requires the feed to be configured, fresh, continuous, and stable, as defined above.
4. An FX bid never pays more quote tokens than its escrowed `quoteBudget`.
5. An FX bid's `remaining` is `min(openBase, quoteBudget converted at the captured quote)`.
6. Oracle movement alone never changes `openBase` or `quoteBudget`; only actual and synthetic fills reduce `openBase`.
7. FX flip orders preserve `amount` across flips; a flip may rest less than `amount`, and rests nothing below `FX_RESTING_DUST_USD_E6`.
8. The set of occupied storage slots in an FX order record never changes while the order is live; flips neither create nor delete slots.
9. Cancellation and cleanup, through every entry point, refund exactly `openBase` (asks) or `quoteBudget` (bids).
10. `OrderFlipped.amount` is the preserved configured size, not the active liquidity.
11. FX order storage is version-aware and coexists with legacy and TIP-1062 orders in mixed-version linked lists.
12. Views, events, and fill paths report `remaining` as active liquidity under that path's quote, or zero without a DEX-valid quote.
13. `getTickLevel.totalLiquidity` is only an upper bound for FX bid liquidity.
14. A single state-changing operation drops at most `FX_MAX_DROPS_PER_OPERATION` orders.
15. Cleanup follows TIP-1060 and TIP-1064 storage accounting and grants no caller bounty.
