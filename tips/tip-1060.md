# TIP-1060: Precompile Storage Gas Tokens

## Abstract

This TIP introduces storage gas tokens for selected Tempo precompiles. Whenever the StablecoinDEX or payment channels precompile deletes a storage element, the protocol increments that precompile's gas token balance. Whenever the same precompile creates a new storage element, it may either pay the TIP-1000 storage creation cost or consume one available gas token instead.

For now, this mechanism applies only to the StablecoinDEX and payment channels precompiles. Ordinary smart contracts and other precompiles are out of scope.

## Motivation

TIP-1000 makes net-new storage creation expensive to protect Tempo from state growth attacks. That cost is appropriate when a contract increases its long-term state footprint, but it overcharges contracts that repeatedly delete and recreate storage slots while staying within a previously paid state footprint.

TIP-1058 applies reusable storage accounting to selected precompile-owned state with owner attribution. This TIP defines the underlying precompile-level storage token mechanism only for the StablecoinDEX and payment channels precompiles, leaving any broader smart-contract generalization to a future TIP.

# Specification

## Terminology

- **Eligible precompile**: The StablecoinDEX precompile or payment channels precompile.
- **Precompile gas token balance**: A protocol-maintained `uint64` counter associated with one eligible precompile.
- **Storage creation**: An eligible precompile storage slot transition from zero to nonzero.
- **Storage deletion**: An eligible precompile storage slot transition from nonzero to zero.
- **Consume-token mode**: A storage creation mode in which storage creation consumes one available gas token before charging the TIP-1000 creation cost.
- **Preserve-token mode**: A storage creation mode in which storage creation always pays gas and leaves any gas token balance unchanged.

## State

The protocol MUST maintain the following state for each eligible precompile:

```text
precompile_storage_gas_token_state: mapping(address precompile => uint256 packed_state)
```

The packed state word uses the following layout:

```text
bits   0..63: gas_token_balance (uint64)
bit       64: preserve_tokens flag
bits 65..255: reserved for future hardfork-gated extensions
```

If `preserve_tokens == 0`, the precompile is in consume-token mode. If `preserve_tokens == 1`, the precompile is in preserve-token mode. The default packed state for every eligible precompile is zero, which means a zero balance and `ConsumeTokens` mode. Implementations MAY omit explicit storage for eligible precompiles with the default packed state.

Reserved bits MUST be written as zero by this TIP. Future TIPs MAY assign reserved bits, but clients MUST NOT infer semantics from nonzero reserved bits unless those semantics have been activated by a hardfork.

The storage gas token accounting state MUST be ordinary protocol state. Creating or updating the storage gas token counter, including the first creation of that counter's backing storage slot, MUST NOT itself mint or consume gas tokens and MUST NOT incur the 250,000 gas TIP-10-mandated storage creation cost.

## Gas Token Minting

When eligible precompile `P` changes one of `P`'s storage slots from nonzero to zero:

1. Apply the normal storage deletion gas behavior.
2. Increment the `gas_token_balance` field for `P` by one.

Deleting a slot mints exactly one gas token regardless of the deleted value, the slot key, the transaction sender, or the call path that caused the deletion.

If `gas_token_balance[P] == type(uint64).max`, deleting another slot MUST NOT overflow the counter. The balance remains saturated at `type(uint64).max`, and the deletion otherwise proceeds normally.

## Gas Token Consumption

When eligible precompile `P` changes one of `P`'s storage slots from zero to nonzero:

1. If `preserve_tokens[P] == 0` and `gas_token_balance[P] > 0`:
   - Decrement the `gas_token_balance` field for `P` by one.
   - Charge the normal zero-to-nonzero storage write cost except for the TIP-1000 storage creation component.
2. Otherwise:
   - Leave `gas_token_balance[P]` unchanged.
   - Charge the full TIP-1000-mandated storage creation cost.

The token debit and storage write MUST be atomic. If the storage write reverts, the gas token balance and mode changes made in the same execution scope MUST revert according to normal EVM revert semantics.

## Other Storage Transitions

- `nonzero -> nonzero`: does not mint or consume gas tokens and pays normal update gas.
- `zero -> zero`: does not mint or consume gas tokens and pays normal no-op gas.
- Same-transaction transitions follow normal journal and revert behavior. A write that consumes a gas token MUST NOT later receive a refund for a storage creation cost that it did not pay.

## Scope

This TIP applies only to storage owned by the StablecoinDEX and payment channels precompiles. It does not change SSTORE gas behavior for ordinary smart contracts, EOAs, TIP-20 token state, or any other precompile-owned state.

Each eligible precompile MAY choose consume-token mode or preserve-token mode internally for each operation. This TIP does not introduce a public precompile for arbitrary contracts to query balances or set a default mode.

## Default Behavior

Eligible precompiles default to `ConsumeTokens`. This means an eligible precompile that deletes a storage slot and later creates a storage slot will automatically consume one gas token for the later creation if one is available, unless that precompile's operation explicitly selects preserve-token mode.

The default behavior for ordinary smart contracts, EOAs, and other precompiles is unchanged because they are outside this TIP's scope.

## Relationship to TIP-1058

This TIP can serve as the underlying mechanism for TIP-1058 for the StablecoinDEX and payment channels precompiles. An eligible precompile that wants owner-level reusable storage can keep its own owner accounting in precompile state, select preserve-token mode when it wants a user to pay the TIP-1000 creation cost, and select consume-token mode when the user is entitled to reuse deleted storage capacity.

Alternatively, an eligible precompile can use the default mode and receive precompile-level reuse without attributing savings to specific users. In that mode, cheaper storage creation benefits the precompile as a whole and may not correspond to the user who originally caused the deletion.

TIP-1064 defines the StablecoinDEX-specific owner accounting layer on top of this mechanism.

## Abuse and Scope

Gas tokens are not transferable and cannot be created without deleting eligible precompile storage. This preserves TIP-1000's protection against net state growth: an eligible precompile that increases its live storage footprint beyond its deleted-slot balance still pays the full storage creation cost for each net-new slot.

The mechanism intentionally uses precompile-level accounting rather than protocol-level user attribution. This keeps the core rule simple and lets the eligible precompiles decide whether they need stricter attribution. Precompile operations that expose arbitrary storage writes to users remain responsible for ensuring their own economics are not exploitable.

## Compatibility

Existing contracts are unaffected. Only the StablecoinDEX and payment channels precompiles receive storage gas token accounting under this TIP.

Gas estimation for eligible precompile calls becomes somewhat less predictable because the same zero-to-nonzero write may cost less when the precompile has an available gas token. Traces and debug tooling SHOULD expose token minting, token consumption, and storage creation mode reads or writes so developers can explain gas differences.

Smart contracts that call the eligible precompiles may observe lower costs after this TIP. Ordinary smart contract SSTORE behavior remains unchanged.

# Invariants

1. A precompile gas token balance increases only when that eligible precompile deletes one of its own nonzero storage slots.
2. A precompile gas token balance decreases only when that eligible precompile creates a storage slot while in `ConsumeTokens` mode and the balance is nonzero.
3. A gas token can offset at most one storage creation.
4. Gas tokens cannot be transferred between precompiles.
5. Gas token minting, consumption, and mode changes revert with the execution scope that caused them.
6. The storage gas token accounting writes do not mint or consume gas tokens.
7. A storage creation that consumed a gas token cannot receive a refund for a storage creation cost it did not pay.
8. The default mode for every eligible precompile is `ConsumeTokens`.
9. The creation of a storage gas token counter's backing storage slot does not incur the 250,000 gas TIP-10-mandated storage creation cost.
10. Ordinary smart contracts and non-eligible precompiles do not mint or consume gas tokens under this TIP.
11. The packed state word's reserved bits remain zero until assigned by a future TIP.
