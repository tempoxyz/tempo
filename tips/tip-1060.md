# TIP-1060: Storage Gas Tokens

## Abstract

This TIP introduces *storage gas tokens*: a per-account counter that increments whenever the account deletes one of its own storage slots and that can later offset the TIP-1000 storage creation cost (250,000 gas per zero-to-nonzero SSTORE) when the same account creates a new slot. Each account chooses, via a new precompile, one of three storage creation modes: `RefundTokens` (pay the full TIP-1000 cost upfront; at end-of-transaction, each refund-eligible creation is matched against the account's then-current token balance to grant an equivalent refund), `PreserveTokens` (always pay the full cost, never consume tokens), or `DirectTokens` (when a token is available, consume it synchronously so the TIP-1000 creation cost is not charged at all).

Because `RefundTokens` settles at the end of the transaction, a transaction that creates storage and later deletes its own storage in the same transaction can use the just-minted token to refund the earlier creation. This is the main reason `RefundTokens` is the default: it is at least as cheap as `DirectTokens` in practice and avoids the simulation-vs-inclusion failure mode that synchronous consumption would expose.

The mechanism applies uniformly to every account on Tempo, including ordinary smart contracts, precompile-owned state, and EOAs. EOAs do not normally hold storage, so in practice they will neither accumulate nor consume tokens, but the rule is account-keyed for uniformity.

## Motivation

TIP-1000 makes net-new storage creation expensive to protect Tempo from state growth attacks. That cost is appropriate when an account increases its long-term state footprint, but it overcharges accounts that repeatedly delete and recreate storage slots while staying within a previously paid state footprint. Examples include AMM orderbooks that constantly add and cancel orders, channel-style precompiles that open and close escrow slots, and ordinary contracts implementing transient data structures.

TIP-1058 applies reusable storage accounting to selected precompile-owned state with owner attribution. This TIP provides the underlying account-level storage token mechanism that TIP-1058 and similar precompile-specific designs can build on, and makes the same primitive available to ordinary smart contracts through a single protocol-managed precompile.

# Specification

## Terminology

- **Account**: Any address that can own EVM storage, including ordinary smart contracts, precompiles, and externally owned accounts.
- **Storage gas token balance**: A protocol-maintained `uint64` counter associated with one account, holding the number of unspent storage gas tokens for that account.
- **Storage creation**: A storage slot transition from zero to nonzero in storage owned by some account.
- **Storage deletion**: A storage slot transition from nonzero to zero in storage owned by some account.
- **TIP-1000 storage creation cost**: The 250,000-gas state-creation component charged by TIP-1000 on every zero-to-nonzero SSTORE, in addition to the EIP-2929 access cost (2,100 gas cold, 100 gas warm) and any other base SSTORE gas. References elsewhere in this TIP to "the TIP-1000 storage creation cost", "the TIP-1000 creation cost", or "the TIP-1000 storage creation component" all denote this same 250,000-gas component. Storage gas tokens offset only this component; the EIP-2929 and base SSTORE gas remain payable in full.
- **Refund-token mode**: The default storage creation mode. Each storage creation pays the TIP-1000 creation cost upfront and increments a transaction-local *pending refund-eligible creation* counter for the storage-owning account. At end-of-transaction, the protocol matches each pending refund-eligible creation against the account's gas token balance (as it stands at end-of-transaction, after all in-transaction deletions and mode changes) and credits a refund of the TIP-1000 creation cost per token consumed.
- **Pending refund-eligible creation**: A `RefundTokens`-mode storage creation that has not yet been redeemed for a refund. Counted per account, transaction-local, journaled with EVM scopes, and reset to zero at the start of each transaction.
- **Preserve-token mode**: A storage creation mode in which storage creation always pays the TIP-1000 creation cost and leaves the gas token balance unchanged.
- **Direct-token mode**: A storage creation mode in which, if a gas token is available, one token is consumed and the TIP-1000 storage creation cost is not charged at all.
- **Storage Gas Tokens precompile**: The protocol precompile, defined below, through which accounts read their balance and set their storage creation mode.

## State

The protocol MUST maintain the following state:

```text
storage_gas_token_state: mapping(address account => uint256 packed_state)
```

The packed state word uses the following layout:

```text
bits   0..63: gas_token_balance (uint64)
bits  64..65: storage_creation_mode
bits 66..255: reserved for future hardfork-gated extensions
```

`storage_creation_mode` has the following values:

```text
0: RefundTokens
1: PreserveTokens
2: DirectTokens
3: reserved
```

The default packed state for every account is zero, which means a zero balance and `RefundTokens` mode. Implementations MAY omit explicit storage for accounts with the default packed state.

If `storage_creation_mode == 3`, clients MUST treat the state as invalid before any future hardfork assigns semantics to that value.

Reserved bits MUST be written as zero by this TIP. Future TIPs MAY assign reserved bits, but clients MUST NOT infer semantics from nonzero reserved bits unless those semantics have been activated by a hardfork.

The storage gas token accounting state MUST be ordinary protocol state. Creating or updating the storage gas token counter, including the first creation of that counter's backing storage slot, MUST NOT itself mint or consume gas tokens and MUST NOT incur the 250,000 gas TIP-1000-mandated storage creation cost.

In addition to this persistent state, the protocol MUST maintain transaction-local accounting:

```text
pending_refund_eligible_creations: mapping(address account => uint64)
```

`pending_refund_eligible_creations` is not persisted across transactions. It is initialized to zero at the start of every transaction, journaled with EVM scopes (so it reverts with the call frame that incremented it, just like gas token balance and mode changes), and consulted only by the end-of-transaction settlement procedure defined below.

## Gas Token Minting

When account `A` changes one of `A`'s storage slots from nonzero to zero:

1. Apply the normal storage deletion gas behavior.
2. Increment the `gas_token_balance` field for `A` by one.

The increment in step 2 MUST be atomic with the storage write that caused it and MUST be journaled with the same EVM execution scope. If the storage write reverts under normal EVM revert semantics — because the call frame containing it reverts, runs out of gas, or is unwound by a parent frame's revert — the gas token mint MUST also revert, restoring the prior `gas_token_balance[A]` value. An account therefore cannot accrue a storage gas token without leaving a corresponding persisted nonzero-to-zero transition at end-of-transaction: a token and its underlying deletion either both persist or both unwind. Same-transaction execution-scope nesting (including `try/catch`, repeated reverted sub-calls, and `CALL`/`DELEGATECALL`/`STATICCALL`) follows the normal journal and gives no path to mint a token without persisting the deletion.

Deleting a slot mints exactly one gas token regardless of the deleted value, the slot key, the transaction sender, or the call path that caused the deletion. The token is credited to the account whose storage was modified. Under `DELEGATECALL` and `CALLCODE`, this is the calling account (the account whose storage the SSTORE actually wrote), per normal EVM semantics.

If `gas_token_balance[A] == type(uint64).max`, deleting another slot MUST NOT overflow the counter. The balance remains saturated at `type(uint64).max`, and the deletion otherwise proceeds normally.

## Gas Token Consumption

When account `A` changes one of `A`'s storage slots from zero to nonzero, the protocol selects exactly one of the following branches based on `storage_creation_mode[A]`:

1. `storage_creation_mode[A] == RefundTokens`:
   - Charge the full TIP-1000-mandated storage creation cost.
   - Increment `pending_refund_eligible_creations[A]` by one (transaction-local, journaled with the current EVM scope).
   - Do NOT decrement `gas_token_balance[A]` and do NOT credit a gas refund at this point. Token consumption and the resulting refund happen during end-of-transaction settlement.
2. `storage_creation_mode[A] == PreserveTokens`:
   - Leave `gas_token_balance[A]` unchanged.
   - Charge the full TIP-1000-mandated storage creation cost.
3. `storage_creation_mode[A] == DirectTokens` and `gas_token_balance[A] > 0`:
   - Decrement `gas_token_balance[A]` by one immediately.
   - Charge the normal zero-to-nonzero storage write cost except for the TIP-1000 storage creation component.
4. `storage_creation_mode[A] == DirectTokens` and `gas_token_balance[A] == 0`:
   - Leave `gas_token_balance[A]` unchanged.
   - Charge the full TIP-1000-mandated storage creation cost.

The synchronous parts (storage write, `DirectTokens`-mode token debit, `RefundTokens`-mode pending counter increment, and any mode changes) MUST be atomic with the storage write that caused them. If the storage write reverts, all of the above MUST revert according to normal EVM revert semantics.

### End-of-Transaction Settlement

After all EVM execution for the transaction has completed (after the top-level call frame returns) and before the EVM refund cap is applied to the transaction's total refund, the protocol MUST settle `RefundTokens`-mode creations for every account `A` with `pending_refund_eligible_creations[A] > 0`:

1. Let `settled := min(pending_refund_eligible_creations[A], gas_token_balance[A])`.
2. Decrement `gas_token_balance[A]` by `settled`.
3. Add `settled * 250_000` gas to the transaction's gas refund counter (250,000 gas per redeemed creation, matching the TIP-1000 storage creation cost).

`pending_refund_eligible_creations[A]` is then discarded with the transaction. The order in which accounts are settled MUST NOT affect any account's outcome, because settlement is per-account and gas tokens cannot move between accounts.

Settlement uses `gas_token_balance[A]` and `pending_refund_eligible_creations[A]` as they stand at end-of-transaction, after all journaling has resolved. Tokens minted by storage deletions earlier in the same transaction are therefore available to fund refunds for earlier `RefundTokens`-mode creations in the same transaction. A transaction that creates storage and later (within the same transaction) deletes one of its own storage slots can receive a refund for the earlier creation against the just-minted token, even if the account's pre-transaction balance was zero.

Refunds granted by settlement are then subject to the standard EVM refund cap together with any other refund accumulated in the transaction. A transaction whose total refund (storage gas token settlement plus any other refunds) exceeds the protocol-wide cap will have its effective refund clamped per existing rules.

If a `RefundTokens`-mode creation occurred in a call frame that reverted, the increment to `pending_refund_eligible_creations[A]` is unwound by the journal before settlement and therefore does not produce a refund. Symmetrically, deletions in reverted scopes do not contribute to `gas_token_balance[A]` at settlement time.

### Rationale for the default mode

`DirectTokens` mode charges less gas upfront when a token is available, but it creates a simulation-vs-inclusion failure mode: a transaction simulated against a state where `gas_token_balance[A] > 0` may be included against a state where another transaction has already drained the balance, in which case the included transaction can run out of gas and revert even though the user's intent and signed gas limit were unchanged.

`RefundTokens` mode avoids that by always charging the full TIP-1000 creation cost upfront. When tokens are available at the end-of-transaction settlement, the user receives an equivalent refund per redeemed creation and pays the same net amount as `DirectTokens` would (subject to the EVM refund cap), but the worst-case upfront gas is bounded by the always-charged amount.

End-of-transaction settlement also lets a single transaction self-finance its own creations: a transaction that creates one or more storage slots and later deletes any of its own storage slots in `RefundTokens` mode can use the just-minted tokens to refund the earlier creations, regardless of whether the account had a positive token balance when the transaction started.

`DirectTokens` mode remains available for accounts whose callers explicitly accept the simulation-vs-inclusion risk, for example because they reserve their own per-user storage credits inside the contract and only select `DirectTokens` mode when their accounting guarantees that a token is reserved for the specific operation about to execute.

## Other Storage Transitions

- `nonzero -> nonzero`: does not mint or consume gas tokens and pays normal update gas.
- `zero -> zero`: does not mint or consume gas tokens and pays normal no-op gas.
- Same-transaction transitions follow normal journal and revert behavior. A `DirectTokens`-mode write that consumed a gas token (and therefore did not pay the TIP-1000 storage creation cost) MUST NOT later receive any additional refund for the omitted cost. `RefundTokens`-mode writes always pay the TIP-1000 cost upfront, so the refund granted at end-of-transaction settlement is for a cost that was actually paid earlier in the same transaction.
- A `DirectTokens`-mode storage creation in the same transaction as a `RefundTokens`-mode creation consumes its own token synchronously and does NOT increment `pending_refund_eligible_creations`. The two modes' settlement is independent.

## Storage Gas Tokens Precompile

The protocol MUST deploy a precompile at the reserved address below (the *Storage Gas Tokens precompile*) that exposes the storage gas token state to EVM callers and lets each account choose its storage creation mode.

### Precompile Address

```solidity
address constant STORAGE_GAS_TOKENS_ADDRESS = 0x1060000000000000000000000000000000000000;
```

Implementations MUST use this address consistently across consensus, RPC, and tooling. The address is reserved by this TIP; the final byte pattern is subject to confirmation during implementation review.

### Interface

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.13;

/// @title IStorageGasTokens - TIP-1060 precompile interface
/// @notice Per-account storage gas token accounting. Each account earns a
///         token when it deletes one of its own storage slots and can use
///         it to offset a later storage creation. Each account selects how
///         tokens are applied to its own storage creations via `setMode`.
interface IStorageGasTokens {
    /// @notice Storage creation mode for an account.
    /// - `RefundTokens`   (default): full TIP-1000 creation cost is charged
    ///                               upfront; if a token is available it is
    ///                               consumed and an equal refund is credited.
    /// - `PreserveTokens`:           full TIP-1000 creation cost is always
    ///                               charged; tokens are never consumed.
    /// - `DirectTokens`:             if a token is available it is consumed
    ///                               and the TIP-1000 storage creation
    ///                               component is not charged.
    enum Mode {
        RefundTokens,
        PreserveTokens,
        DirectTokens
    }

    /// @notice Reverts if `setMode` is called with an undefined `Mode` value
    ///         (i.e. anything other than `RefundTokens`, `PreserveTokens`,
    ///         or `DirectTokens`).
    error InvalidMode();

    /// @notice Emitted when an account's storage creation mode changes.
    /// @param account The account whose mode changed. This is the immediate
    ///                caller of `setMode`, which under DELEGATECALL/CALLCODE
    ///                is the account that owns the storage being affected.
    /// @param newMode The new storage creation mode.
    event ModeUpdated(address indexed account, Mode newMode);

    /// @notice Read the calling account's storage gas token balance.
    /// @return The number of unspent tokens for `msg.sender`. Saturates at
    ///         `type(uint64).max`.
    function balance() external view returns (uint64);

    /// @notice Read the storage gas token balance for `account`.
    function balanceOf(address account) external view returns (uint64);

    /// @notice Read the calling account's storage creation mode.
    function mode() external view returns (Mode);

    /// @notice Read the storage creation mode for `account`.
    function modeOf(address account) external view returns (Mode);

    /// @notice Set the storage creation mode for the calling account.
    /// @dev Applies to every subsequent zero-to-nonzero write to
    ///      `msg.sender`'s storage until another `setMode` call from the
    ///      same account. The change is journaled with the calling
    ///      execution scope and reverts with it under normal EVM revert
    ///      semantics.
    /// @param newMode New storage creation mode. MUST be a defined `Mode`.
    function setMode(Mode newMode) external;
}
```

### Behavior

- `balance()` and `mode()` are equivalent to `balanceOf(msg.sender)` and `modeOf(msg.sender)`. All four read functions MUST NOT modify state.
- `balanceOf(account)` returns the `gas_token_balance` field of `storage_gas_token_state[account]`.
- `modeOf(account)` returns the `storage_creation_mode` field of `storage_gas_token_state[account]`.
- `setMode(newMode)` MUST revert with `InvalidMode` if `newMode` is the reserved value `3` (or any encoding outside `{0, 1, 2}`). Otherwise it MUST set `storage_creation_mode[msg.sender] := newMode`, emit `ModeUpdated(msg.sender, newMode)`, and consume only the ordinary protocol gas for the underlying state write per the *State* section. `setMode` MUST NOT incur a TIP-1000 creation charge and MUST NOT mint or consume gas tokens.
- Under `DELEGATECALL` or `CALLCODE`, `msg.sender` is propagated per normal EVM semantics, so the account whose mode is written is the same account whose storage gas tokens would be earned and consumed by subsequent SSTOREs in the same scope.
- A mode write performed inside an execution scope that later reverts MUST itself revert, restoring the prior `storage_creation_mode` value for the same account.

## Scope

This TIP applies to every account on Tempo. Storage gas token minting, consumption, and mode selection follow the same rules whether the account is an ordinary smart contract, a precompile, or an EOA. The effective SSTORE cost for accounts that never delete and re-create storage slots is unchanged, because their `gas_token_balance` stays at zero and `RefundTokens` mode degenerates to charging the full TIP-1000 cost with no refund.

## Default Behavior

Every account defaults to `RefundTokens` mode. This means an account that deletes a storage slot and later creates a storage slot will automatically pay the full storage creation cost upfront and receive an offsetting refund if a gas token is available, unless that account has explicitly called `setMode(PreserveTokens)` or `setMode(DirectTokens)`.

The default mode is intentionally refund-based rather than direct consumption: a caller must always provide enough gas for the full storage creation cost, so transactions do not unexpectedly fail when the gas token balance changes between simulation and inclusion.

## Relationship to TIP-1058

TIP-1058 applies reusable storage accounting to specific precompile-owned state with owner attribution. This TIP provides the underlying account-level storage token mechanism that TIP-1058 (and similar designs) can build on. A contract that wants owner-level reusable storage can keep its own owner accounting in contract state and call `setMode` at runtime to select the appropriate mode for each operation:

- `PreserveTokens` when it wants the user to fully pay the TIP-1000 creation cost.
- `RefundTokens` when the user may be entitled to reuse deleted storage capacity but the contract cannot pre-confirm at the worst-case gas-limit boundary that a token will be available.
- `DirectTokens` when the contract's per-user accounting guarantees that a token is available for this specific user at execution time.

Alternatively, an account can leave the mode at the default `RefundTokens` and receive account-level reuse without attributing savings to specific users. In that mode, cheaper storage creation benefits the account as a whole and may not correspond to the user who originally caused the deletion.

TIP-1064 defines the StablecoinDEX-specific owner accounting layer on top of this mechanism.

## Abuse and Scope

Gas tokens are not transferable and cannot be created without deleting an account's own storage. This preserves TIP-1000's protection against net state growth: an account that increases its live storage footprint beyond its deleted-slot balance still pays the full storage creation cost for each net-new slot.

The mechanism intentionally uses account-level accounting rather than protocol-level user attribution. This keeps the core rule simple and lets contracts that need stricter attribution layer their own per-user accounting on top of the account-level token balance and mode. Contracts that expose arbitrary storage writes to users while in `DirectTokens` mode remain responsible for ensuring their own economics are not exploitable.

`setMode` is unauthenticated: any account can set its own mode at any time, but only its own. A contract that wants `setMode` to be invoked only under specific conditions MUST gate its own calls to the precompile accordingly. Calls from one account that attempt to alter another account's mode are not possible under this interface.

## Compatibility

Existing contracts are unaffected unless they choose to interact with the Storage Gas Tokens precompile. Accounts that do not interact with the precompile keep the default `RefundTokens` mode and a zero balance until they delete one of their own existing storage slots.

Gas estimation for storage creations in the default `RefundTokens` mode is unchanged in upfront cost: the same zero-to-nonzero write always charges the full TIP-1000 storage creation cost and the required gas limit does not depend on the account's current token balance. The effective cost is determined at end-of-transaction settlement and may be lower via a refund, including via tokens minted by deletions earlier in the same transaction. Storage creations performed under `DirectTokens` mode depend on whether a gas token is available at the moment of the creation, so callers using `DirectTokens` SHOULD assume the worst case (no token available) when sizing gas limits or accept that the transaction may fail if the balance changes between simulation and inclusion.

Traces and debug tooling SHOULD expose token minting, synchronous `DirectTokens` token consumption, `RefundTokens` pending counter increments, end-of-transaction settlement (per-account tokens consumed and refund credited), and storage creation mode reads or writes so developers can explain gas differences. Smart contracts that perform storage churn may observe lower effective costs after this TIP, including for transactions that net to zero new storage. SSTORE semantics for accounts that do not churn storage remain unchanged.

# Invariants

1. An account's storage gas token balance increases only when that account deletes one of its own nonzero storage slots, and only if that deletion persists past the end of the enclosing transaction (i.e., is not unwound by any subsequent revert).
2. An account's storage gas token balance decreases only via: (a) a `DirectTokens`-mode storage creation when `gas_token_balance[A] > 0` (decrement of one per creation, synchronous); or (b) end-of-transaction settlement of `RefundTokens`-mode pending refund-eligible creations. In `PreserveTokens` mode the balance never decreases.
3. A gas token can offset at most one storage creation, either by funding one redeemed `RefundTokens`-mode pending creation at settlement or by removing the TIP-1000 component of a single `DirectTokens`-mode creation.
4. Gas tokens cannot be transferred between accounts.
5. Gas token minting, `DirectTokens`-mode consumption, `RefundTokens`-mode pending counter increments, mode changes, and any synchronous refund credits revert with the execution scope that caused them. End-of-transaction settlement reads only the journaled, non-reverted state.
6. The storage gas token accounting writes do not mint or consume gas tokens and do not incur the TIP-1000 storage creation cost.
7. A storage creation in `DirectTokens` mode that consumed a gas token cannot also receive a refund for the omitted storage creation cost.
8. The default mode for every account is `RefundTokens`.
9. The packed state word's reserved bits and the reserved `storage_creation_mode` value (`3`) remain unwritten until assigned by a future TIP.
10. The gas charged upfront for a `RefundTokens`-mode zero-to-nonzero write does not depend on `gas_token_balance[A]`, so a transaction's required gas limit is independent of the account's current token balance. End-of-transaction settlement can only add to the transaction's refund counter and cannot cause the transaction to run out of gas.
11. At end-of-transaction settlement, for each account `A` the number of tokens consumed equals `min(pending_refund_eligible_creations[A], gas_token_balance[A])` evaluated against the journaled, non-reverted end-of-transaction state, and exactly that many TIP-1000 storage creation refunds are credited to the transaction's refund counter (subject to the standard EVM refund cap when computing the effective refund).
12. `pending_refund_eligible_creations` is transaction-local: it is zero at the start of every transaction and is not visible to or readable by EVM code in any subsequent transaction.
13. `setMode(newMode)` from account `A` MUST set `storage_creation_mode[A]` to `newMode` if `newMode ∈ {RefundTokens, PreserveTokens, DirectTokens}` and MUST revert with `InvalidMode` otherwise. It MUST NOT change `storage_creation_mode` for any account other than `A`.
14. `setMode` MUST NOT change any account's `gas_token_balance` and MUST NOT change `pending_refund_eligible_creations` for any account.
