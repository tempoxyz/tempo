# TIP-1051: EIP-712 Signing for Tempo Transactions and Key Authorizations

## Abstract

This TIP adds EIP-712 signing as a new signature type for both
`TempoTransaction` and `KeyAuthorization`. The canonical wire encodings of
transactions and key authorizations remain unchanged; instead, EIP-712
signatures are carried in a small envelope that wraps the existing signature
bytes.

## Motivation

Many wallet APIs support `eth_signTypedData_v4` but do not support signing an
arbitrary `keccak256(rlp(...))` digest. This blocks integrations for both Tempo
transactions and key provisioning.

`KeyAuthorization` can grant a delegated key the ability to spend tokens and
execute calls on behalf of the root account. Presenting this as a blind hash is
poor UX for a high-stakes authorization. EIP-712 gives wallets a standard way
to show a structured authorization to the signer.

# Specification

## Overview

`TempoTransaction` remains the existing Tempo transaction type with type byte
`0x76`, and `KeyAuthorization` remains the existing authorization payload.
Unwrapped signatures continue to use the current Tempo signing path. Wrapped
signatures use EIP-712. Verification is deterministic: implementations select
exactly one digest based on whether the signature is wrapped with `0x7f` and
MUST NOT try multiple digests for the same signature.

## EIP-712 Signature Type

This TIP defines a new single-byte EIP-712 signature type marker:

```text
EIP712_SIGNATURE_TYPE = 0x7f
```

Unwrapped signatures remain the canonical Tempo-native signing path. Wrapped
signatures indicate that the inner signature was produced over the EIP-712 typed
data defined in this TIP.

## Signature Envelope Encoding

When a signature uses the EIP-712 signature type, the encoded signature bytes
are:

```text
0x7f || inner_signature_bytes
```

`inner_signature_bytes` is the existing encoded signature payload:

1. For `KeyAuthorization`, the existing `PrimitiveSignature` byte encoding.
2. For `TempoTransaction`, the existing `TempoSignature` byte encoding.

The current unwrapped signature encoding remains the canonical Tempo-native
signing path. Implementations MUST NOT wrap a non-EIP-712 signature with `0x7f`.

Implementations MUST reject all of the following:

1. An empty signature byte string.
2. A byte string beginning with `0x7f` that is shorter than 2 bytes.
3. A nested envelope, where `inner_signature_bytes` itself begins with `0x7f`.
4. Any inner signature bytes that fail to decode under the existing
   `PrimitiveSignature` or `TempoSignature` rules.

The outer RLP shape of the signed objects does not change. The signature field
continues to be encoded as a single RLP byte string.

## Activation and Compatibility

This TIP activates at protocol version `T5`.

At `T5`, the protocol adds support for the `0x7f || inner_signature_bytes`
EIP-712 signature type defined in this TIP. The protocol continues to support
existing unwrapped signatures. Implementations MUST verify signatures against
exactly one digest determined by whether the signature is wrapped with `0x7f`.

## Current Tempo Signing Path

This TIP does not change the current digests or verification rules for
unwrapped signatures.

## EIP-712 Domain Definitions

Wallets and SDKs that produce a wrapped `0x7f` signature MUST use the following
domains.

### `KeyAuthorization` Domain

| Field | Value |
|-------|-------|
| name | `"Tempo Key Authorization"` |
| version | `"1"` |
| chainId | `authorization.chain_id` |

### `TempoTransaction` Domain

| Field | Value |
|-------|-------|
| name | `"Tempo Transaction"` |
| version | `"1"` |
| chainId | `tx.chain_id` |

The chain ID is present in both the domain and the message payload. This is
intentional. The domain binds the EIP-712 replay domain, while the message
payload preserves the existing transaction and authorization fields.

Implementations MUST apply standard EIP-712 encoding rules.

## `KeyAuthorization` EIP-712 Signing

When a `KeyAuthorization` signature is wrapped with `0x7f`, the signed digest is
an EIP-712 digest under the `Tempo Key Authorization` domain.

The typed data MUST commit to the same semantic fields as the current unsigned
`KeyAuthorization`, including:

1. `chain_id`
2. `key_type`
3. `key_id`
4. `expiry`
5. `limits`
6. `allowed_calls`

The typed data MUST preserve the current protocol distinctions between omitted
and explicit empty optional fields.

## `TempoTransaction` EIP-712 Signing

When a `TempoTransaction` signature is wrapped with `0x7f`, the signed digest is
an EIP-712 digest under the `Tempo Transaction` domain.

The typed data MUST commit to the same semantic fields as the current unsigned
`TempoTransaction`, including:

1. `chain_id`
2. fee fields
3. `gas_limit`
4. `calls`
5. `access_list`
6. `nonce_key`
7. `nonce`
8. `valid_before`
9. `valid_after`
10. `tempo_authorization_list`
11. `key_authorization`

The typed data MUST preserve the current sender-commitment semantics of
`encode_for_signing()`. In particular:

1. The sender commits to whether a fee payer signature is present, but not to
   the fee payer signature bytes themselves.
2. The sender does not commit to a specific `fee_token` when a fee payer
   signature is present.
3. The sender commits to attached authorization sidecars.

## Verification Rules

When recovering or verifying the signer of a `SignedKeyAuthorization`:

1. If the signature is unwrapped, the current `KeyAuthorization` digest MUST be
   used.
2. If the signature is enveloped with `0x7f`, the EIP-712 `KeyAuthorization`
   digest MUST be used.
3. The implementation MUST recover the signer against exactly that digest.
4. The implementation MUST NOT attempt fallback verification against any other
   digest.

When recovering or verifying the signer of a signed Tempo transaction:

1. If the signature is unwrapped, the current `TempoTransaction` digest MUST be
   used.
2. If the signature is enveloped with `0x7f`, the EIP-712 `TempoTransaction`
   digest MUST be used.
3. The implementation MUST recover the signer against exactly that digest.
4. The implementation MUST NOT attempt fallback verification against any other
   digest.

## Interaction with Keychain Signatures

This TIP does not change the semantics of keychain signatures. It changes only
the outer transaction digest that keychain signatures are applied to.

When the outer transaction signature is a keychain signature:

1. The outer signature type first determines the transaction digest.
2. Keychain V1 or V2 processing then applies to that digest exactly as it does
   today.

No new keychain version is introduced by this TIP.

## Replay Protection and Expiring Nonce Behavior

The choice of signing scheme MUST NOT change semantic replay protection for the
same Tempo transaction.

For purposes of expiring nonce replay protection and any other deduplication that
is intended to be independent of signature type, implementations MUST use the
existing canonical replay payload derived from the current Tempo signing
payload, not the EIP-712 digest.

Specifically, the expiring nonce hash remains:

```text
keccak256(current_tempo_tx_signing_payload || sender)
```

where `current_tempo_tx_signing_payload` is the existing scheme-independent
Tempo signing payload produced by the current `encode_for_signing()` rules.

This ensures that the same semantic transaction signed once with the current
Tempo signing path and once with EIP-712 does not gain two distinct replay
identities.

## Transaction Hashing

The transaction hash (`tx_hash`) remains the hash of the final signed wire
transaction. The same unsigned transaction signed once with the current
Tempo-native path and once with EIP-712 therefore yields different `tx_hash`
values.

# Invariants

The following invariants MUST always hold.

1. The canonical RLP encoding of `TempoTransaction` and `KeyAuthorization` does
   not change under this TIP.
2. Existing unwrapped signatures remain valid after `T5`.
3. Nested signature envelopes are invalid.
4. EIP-712 verification uses exactly one digest and never falls back to the
   current Tempo-native digest.
5. Current Tempo-native verification uses exactly one digest and never falls
   back to the EIP-712 digest.
6. Keychain V1 and V2 semantics are unchanged except that they operate on the
   digest selected by the outer signature type.
7. Expiring nonce replay protection remains independent of signature type for
   the same semantic Tempo transaction.

## Test Cases

The test suite must cover at least the following cases.

1. Unwrapped `KeyAuthorization` signatures continue to decode and verify.
2. Unwrapped Tempo transaction signatures continue to decode and verify.
3. `0x7f`-wrapped key-authorization signatures decode and verify.
4. `0x7f`-wrapped Tempo transaction signatures decode and verify.
5. The protocol continues to accept existing unwrapped signatures after `T5`.
6. Nested signature envelopes are rejected.
7. A transaction with keychain V2 signature verifies correctly under the
   EIP-712 signature type.
8. The same semantic expiring-nonce transaction signed once with the current
   Tempo signing path and once with EIP-712 yields the same replay-protection
   hash.
9. The same signed payload encoded once with the current Tempo signing path and
   once with EIP-712 yields different `tx_hash` values because the signed wire
   bytes differ.
