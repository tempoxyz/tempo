# TIP-1011: Enhanced Access Key Permissions

## Abstract

This TIP extends Access Keys with two permission features:

1. **Periodic spending limits** that reset on fixed intervals.
2. **Call scoping** that limits where a key can call and which selectors it can use.


## Motivation

Current Access Keys support per-token limits and expiry, but miss two practical controls.

### Periodic Spending Limits

One-time limits cannot express recurring allowances.

**Use cases:**

1. Subscription billing (`10 USDC / month`).
2. Payroll schedules (monthly budgeted payouts).
3. Rate-limited agent/API budgets.

### Call Scoping (Target + Selector Set)

Users need finer controls than "any call". They want keys like:

1. "Only call `swap()` and `exactInput()` on DEX X."
2. "Only call gameplay methods on contract Y."
3. "Only perform plain transfers, not token extension methods."
4. "Only vote() on governance contracts."

**Current workaround**: Deploy a proxy contract that enforces destination/function restrictions, adding gas overhead and complexity.

# Specification

## Extended Data Structures

### TokenLimit

**Current:**

```solidity
struct TokenLimit {
    address token;
    uint256 limit;
}
```

**Proposed:**

```solidity
struct TokenLimit {
    address token;
    uint256 limit;              // Per-period cap when period > 0, one-time cap otherwise
    uint256 remainingInPeriod;  // Remaining allowance in current period
    uint64 period;              // Period duration in seconds (0 = one-time limit)
    uint64 periodEnd;           // Timestamp when current period ends (0 for one-time)
}
```

### CallScope

Call scoping is defined with optional fields (protocol-level / RLP model):

```text
CallScope {
    target: address,
    selectors: Option<Vec<bytes4>>,   // None => any selector on this target
}
```

`selectors` behavior intentionally mirrors `limits` tri-state semantics:

1. `None` => allow any selector.
2. `Some([])` => allow no selectors (matches nothing for that scope).
3. `Some([s1, s2, ...])` => allow exactly the listed selectors.

Examples:

1. `{ target: 0x123, selectors: Some([0xaabbccdd, 0xeeff0011]) }` => allow two selectors on one target.
2. `{ target: 0x123, selectors: None }` => address-only scoping (any selector on `0x123`).
3. `{ target: 0x123, selectors: Some([]) }` => matches no selector on `0x123`.
4. `allowedCalls = None` => unrestricted key.
5. `allowedCalls = Some([])` => key is authorized but cannot make scoped calls.

### KeyAuthorization

Existing fields remain, with a trailing optional call-scope field:

```text
KeyAuthorization {
    chain_id: u64,
    key_type: SignatureType,
    key_id: address,
    expiry: Option<u64>,
    limits: Option<Vec<TokenLimit>>,
    allowed_calls: Option<Vec<CallScope>>,  // New trailing field
}
```

## Interface Changes

### Events

```solidity
/// @notice Emitted when an access key spends tokens against a spending limit
/// @param account The account whose key was used
/// @param publicKey The public key (address) that initiated the spend
/// @param token The token address being spent
/// @param amount The amount spent in this transaction
/// @param remainingLimit The remaining spending limit after this spend
event AccessKeySpend(
    address indexed account,
    address indexed publicKey,
    address indexed token,
    uint256 amount,
    uint256 remainingLimit
);
```

This event MUST be emitted whenever an access-key transaction deducts from a spending limit (one-time or periodic).

### IAccountKeychain.sol

```solidity
/// @notice Authorizes a key with enhanced permissions
/// @param key The public key to authorize
/// @param expiry Block timestamp when key expires
/// @param spendingLimits Token spending limits (may include periodic limits)
/// @param allowedCalls Call scopes. Omitted at tx auth layer => unrestricted.
function authorizeKey(
    bytes calldata key,
    uint64 expiry,
    TokenLimit[] calldata spendingLimits,
    CallScope[] calldata allowedCalls
) external;

/// @notice Creates or replaces one target scope for a key
/// @dev Root key only. If `target` does not exist, creates a new scope; otherwise replaces it atomically.
/// @dev `allowAnySelector = true` sets `selectors = None` semantics for this target.
/// @dev `allowAnySelector = false` and `selectors = []` disables this target scope (delete-equivalent).
function setAllowedCalls(
    bytes calldata key,
    address target,
    bool allowAnySelector,
    bytes4[] calldata selectors
) external;

/// @notice Returns allowed call scopes for an account+key
function getAllowedCalls(
    address account,
    bytes calldata key
) external view returns (CallScope[] memory calls);

/// @notice Returns remaining limit for a token, accounting for period resets
function getRemainingLimit(
    address account,
    bytes calldata key,
    address token
) external view returns (uint256 remaining, uint64 periodEnd);
```

## Semantic Behavior

### Periodic Limit Reset Logic

On spend:

```text
function verifyAndUpdateSpending(account, key, token, amount):
    limit = getTokenLimit(account, key, token)

    if limit.period > 0:
        if block.timestamp >= limit.periodEnd:
            // Reset period — advance periodEnd by whole multiples of period
            // to prevent drift when transactions land late
            elapsed = block.timestamp - limit.periodEnd
            periodsElapsed = elapsed / limit.period + 1
            limit.periodEnd += periodsElapsed * limit.period
            limit.remainingInPeriod = limit.limit

    if amount > limit.remainingInPeriod:
        revert SpendingLimitExceeded()

    limit.remainingInPeriod -= amount
```

### Call Validation Logic

#### Selector Extraction

```text
function extractSelector(calldata):
    if calldata.length < 4:
        return None
    return Some(calldata[0:4])
```

Calldata smaller than 4 bytes is treated as selectorless (`None`) and MUST NOT panic.

#### Access-Key Contract Creation Ban

If a transaction is signed with an access key (`key != Address::ZERO`), contract creation MUST be rejected in all configurations.

This ban applies regardless of:

1. Whether `allowed_calls` is `None` or `Some(...)`.
2. Whether any target scope has `selectors = None` (allow-any-selector).
3. Whether the creation call appears in a batch.

Only the root key (`key == Address::ZERO`) may submit contract-creation calls. This is done to simplify the implementation and remove edge cases.

#### Single Call Validation

```text
function validateCall(account, key, destination, calldata, isContractCreation):
    allowed = getAllowedCalls(account, key)  // Option<Vec<CallScope>>

    if key != Address::ZERO && isContractCreation:
        revert CallNotAllowed()  // global contract-creation ban for access keys

    if allowed is None:
        return true  // unrestricted

    if isContractCreation:
        revert CallNotAllowed()  // root-key create cannot be whitelisted when call scopes are set

    scopes = allowed.unwrap()
    if scopes.length == 0:
        revert CallNotAllowed()  // explicit deny-all

    selector = extractSelector(calldata)  // Option<bytes4>

    for scope in scopes:
        if scope.target != destination:
            continue

        selectorMatch = match scope.selectors:
            None => true
            Some(list) => selector is Some(s) && list.contains(s)

        if selectorMatch:
            return true

    revert CallNotAllowed()
```

#### Batch Validation

For AA transactions with multiple calls, each call MUST be validated independently. If any call fails scope validation, the entire batch MUST revert.

### Root-Controlled Scope Updates

`setAllowedCalls(key, target, allowAnySelector, selectors)` provides per-target updates without rewriting every scope:

1. It is root-key-only (same authorization model as `updateSpendingLimit`).
2. If `target` has no scope yet, the call creates one; otherwise it atomically replaces the existing scope.
3. `allowAnySelector = true` sets `selectors = None` semantics (any selector allowed on `target`).
4. `allowAnySelector = false` with `selectors = []` disables that target scope and is delete-equivalent.
5. A disabled target scope cannot authorize any call by itself.
6. Implementations MUST enforce at most one scope per target for each `(account, key)`.

### Interaction Rules

1. Keys may mix one-time and periodic token limits.
2. Spending limits and call scopes are independent checks; both must pass.
3. `updateSpendingLimit()` resets `remainingInPeriod` and updates periodic metadata.
4. `allowed_calls = None` is unrestricted for non-create calls; `Some([])` is explicit deny-all.
5. Calls with calldata length `0..=3` have no selector and only match scopes with `selectors = None`.
6. Every scope has an explicit target address, so there is no wildcard-target precedence ambiguity.
7. Per-target updates are create-or-replace and duplicate target scopes are not allowed.

## Gas And Complexity Bounds

To prevent unbounded validation/storage growth, implementations MUST enforce:

1. `MAX_CALL_SCOPES` (recommended: 64).
2. `MAX_SELECTORS_PER_SCOPE` (recommended: 64).
3. `MAX_TOTAL_SELECTORS` (recommended: 256).

Gas model:

| Operation | Additional Gas |
|-----------|----------------|
| `authorizeKey` with `S` scopes and `K` total selectors | ~20,000 + 7,000 × S + 300 × K |
| `setAllowedCalls` (single target) with `K` selectors | ~5,000 + 300 × K |
| `authorizeKey` with periodic limit | ~3,000 per periodic token |
| Scope validation (per runtime call) | ~2,100 + 300 × K' |
| Period reset (when triggered) | ~5,000 |

`K'` is selectors examined for that destination during matching.

## Encoding

### Signing Format

Authorization digest format:

```text
key_auth_digest = keccak256(rlp([
  chain_id,
  key_type,
  key_id,
  expiry?,
  limits?,
  allowed_calls?
]))

limits = rlp([token, limit, remainingInPeriod, period, periodEnd])
```

### Transaction Authorization RLP

```text
KeyAuthorization := RLP([
    chain_id: u64,
    key_type: u8,
    key_id: address,
    expiry?: uint64,
    limits?: [TokenLimit, ...],
    allowed_calls?: [CallScope, ...]
])

TokenLimit := RLP([
    token: address,
    limit: uint256,
    remainingInPeriod: uint256,
    period: uint64,
    periodEnd: uint64
])

CallScope := RLP([
    target: address,
    selectors?: [bytes4, ...]
])
```

Optional encoding rules:

1. Optional scalar fields (`expiry`) use `None => 0x80`.
2. Optional list fields (`limits`, `allowed_calls`, `selectors`) use `None => 0x80`.
3. `Some([])` list values encode as RLP empty list (`0xc0`).
4. `Some([x, ...])` list values encode as normal lists.

# Backward Compatibility

This TIP requires a hardfork due to transaction encoding and execution semantic changes.

## RLP Changes

### TokenLimit (2 fields -> 5 fields)

`TokenLimit` expands from `[token, limit]` to `[token, limit, remainingInPeriod, period, periodEnd]`.

Version-tolerant decode:

```text
if list.len() == 2:
  remainingInPeriod = limit
  period = 0
  periodEnd = 0
else if list.len() == 5:
  decode all fields
else:
  error
```

### KeyAuthorization Trailing Field

`KeyAuthorization` uses trailing optional fields. `allowed_calls` is appended at the end to preserve decode for old encodings:

1. Old tx encodings decode as `allowed_calls = None` (unrestricted).
2. New encodings are rejected by old nodes (expected hardfork behavior).

## Compact/Database Encoding

`SignedKeyAuthorization` uses custom Compact encoding backed by RLP. `TokenLimit` is therefore stored as RLP in production paths; version-tolerant decode avoids DB rebuild requirements.

## Precompile Storage Changes

Current layout:

1. `keys[account][keyId] -> AuthorizedKey`
2. `spending_limits[(account,keyId)][token] -> U256`

Additive periodic-limit layout:

| Mapping | Type | Description |
|---------|------|-------------|
| `spending_limits[account_key][token]` | `U256` | Reinterpreted as `remainingInPeriod` |
| `spending_limit_max[account_key][token]` | `U256` | Per-period cap |
| `spending_limit_period[account_key][token]` | `u64` | Period duration |
| `spending_limit_period_end[account_key][token]` | `u64` | Current period end |

Call-scope storage MUST be account-scoped:

| Mapping | Type | Description |
|---------|------|-------------|
| `allowed_calls_len[account_key]` | `u64` | Number of scopes |
| `allowed_calls_target_index[account_key][target]` | `u64` | Scope index + 1 for O(1) replace/update |
| `allowed_calls_target[account_key][scope]` | `Address` | Target address for scope |
| `allowed_calls_selectors_is_any[account_key][scope]` | `bool` | `selectors=None` flag |
| `allowed_calls_selectors_len[account_key][scope]` | `u64` | Number of selectors in scope |
| `allowed_calls_selector[account_key][scope][idx]` | `bytes4` | Selector entries |

`account_key = keccak256(account || key_id)` to avoid cross-account collisions for shared key IDs.

Implementations MAY flatten grouped selectors to internal `(target, selector)` rows for execution efficiency, but external semantics MUST remain identical.

## Hardfork-Gated Features

The following MUST be fork-gated:

1. New `TokenLimit` decode/encode behavior.
2. `allowed_calls` decode/encode behavior.
3. Periodic reset logic.
4. Call-scope validation logic.
5. New precompile storage writes/reads for periodic + call-scope data.
6. Updated precompile read APIs (`getAllowedCalls(account,key)`, richer `getRemainingLimit`).
7. New root-control mutator (`setAllowedCalls`) semantics.
8. Global ban on contract creation when using access keys.

Pre-fork blocks MUST replay under old semantics to preserve state roots.

# Invariants

1. `periodEnd` is monotonic and never set to the past.
2. `remainingInPeriod <= limit` after any operation.
3. If `allowed_calls` is `Some`, calls not matching any scope MUST revert.
4. `allowed_calls = None` preserves unrestricted behavior for non-create calls.
5. If `key != Address::ZERO`, any contract-creation call MUST revert regardless of `allowed_calls`.
6. Expiry check runs before spending/call-scope checks.
7. For each `(account, key)`, there is at most one scope per target.
8. `setAllowedCalls(..., allowAnySelector = false, selectors = [])` disables that target scope.

## Test Cases

1. Periodic reset after elapsed period.
2. No rollover of unused periodic allowance.
3. Address + multi-selector scope allow.
4. Address-only allow (`selectors=None`).
5. Deny when no scope matches.
6. `allowed_calls=None` allows all non-create calls.
7. `allowed_calls=Some([])` denies all calls.
8. Mixed one-time and periodic token limits.
9. Existing pre-fork keys still function post-fork.
10. Batch validation reverts entire batch on one invalid call.
11. Calldata lengths `0..=3` never panic and follow selectorless semantics.
12. Shared key IDs across accounts cannot overwrite each other’s scopes.
13. Enforced bounds reject too many scopes/selectors.
14. `setAllowedCalls` creates a new target scope when none exists, and replaces it atomically when present.
15. `setAllowedCalls(..., allowAnySelector = true, selectors = [])` allows any selector on that target.
16. `setAllowedCalls(..., allowAnySelector = false, selectors = [])` blocks all calls to that target (unless unrestricted mode is active).
17. Access-key transactions with CREATE as first call are rejected.
18. Access-key transactions with any CREATE in a batch are rejected.
