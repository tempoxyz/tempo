# TIP-1044: Discount 1,500,000-gas storage initialization cost for DEX flip order placement

## Abstract

When a StablecoinDEX flip order is filled, the precompile writes a new order to 6 fresh slots and then deletes the filled order's 6 slots. Under TIP-1000's gas schedule the new order pays the full 250,000-gas state creation cost per slot (1,500,000 gas total), even though an equal number of slots are cleared in the same call. This TIP treats the create-then-delete pair as a single storage-neutral operation: the state creation cost for the new order is not charged, and the clearing refund for the old order is not granted. The net effect is that the swapper pays only the computational cost of clearing the old slots and writing the new ones, not the storage initialization cost.

## Motivation

Flip orders are designed as perpetual two-sided liquidity: when one side fills, a new order is automatically placed on the opposite side. Because the old order is deleted and the new order is created in the same `fill_order` call, the net state growth is zero — the same number of storage slots are cleared as are initialized.

However, the SSTORE gas schedule prices each slot independently. Per TIP-1000, Tempo charges 250,000 gas for writing a new state element (zero → non-zero), while clearing a slot (non-zero → zero) earns only a 15,000 gas refund capped at 20% of total transaction gas by EIP-3529. The refund is negligible compared to the initialization cost, so swappers pay the full state creation price even though no net storage is consumed.

This makes filling flip orders substantially more expensive than filling regular orders of the same size, increasing costs for swappers who interact with them. Since the StablecoinDEX is a precompile with full control over its own gas accounting, it can apply a targeted discount without modifying the general EVM gas schedule.

## Assumptions

- The `Order` struct occupies 6 storage slots (determined by the `Storable` derive macro). This does not change between the filled order and the newly placed flip order.
- The new order is always placed (`place_flip()`) and the old order is always deleted (`orders[old_id].delete()`) within the same `fill_order` call. This is guaranteed by the `fill_order` control flow.
- This discount applies only within the StablecoinDEX precompile's `fill_order` path and does not affect general SSTORE gas accounting elsewhere in the EVM.
- If the `Order` struct layout changes in a future TIP (adding or removing fields), the discount must be re-evaluated to ensure slot counts still match.

# Specification

## Discounted flip storage

When `fill_order` processes a flip order (i.e., `order.is_flip() == true`), the precompile treats the create-then-delete pair as a single storage-neutral operation. Each of the 6 slots undergoes two storage operations: one write (new order, zero → non-zero) charged at `STORAGE_MOVE` (20,000 gas), and one clear (old order, non-zero → zero) charged at `WARM_SSTORE_RESET` (2,900 gas) with no clearing refund. Per-slot cost: 20,000 + 2,900 = 22,900 gas. Total: 6 × 22,900 = 137,400 gas. This replaces the standard cost of 6 × 250,000 = 1,500,000 gas for the new order's state creation, saving 1,362,600 gas.

`STORAGE_MOVE` is a new constant defined by this TIP, set to 20,000 gas. This value matches the regular (computational) gas component of a cold SSTORE as defined in TIP-1016's regular/state gas split, and reflects the actual cost of inserting a new trie node — writing and hashing — without the 230,000-gas state gas component that pays for permanent state growth. Since a flip order's storage is net-zero (every slot created is paired with a slot cleared in the same call), the permanent storage burden does not apply, but the trie computation cost does. Under TIP-1016, `STORAGE_MOVE` is classified entirely as **regular gas** with zero state gas, since the operation does not contribute to permanent state growth.

Specifically:

1. **Charge `STORAGE_MOVE` on the new order's 6 writes**: When `place_flip()` writes the new order's 6 slots (zero → non-zero), the TIP-1000 state creation cost is **not charged**. Instead, each slot is charged at `STORAGE_MOVE` (20,000 gas, regular gas only), covering the computational cost of the trie insertion. This reflects the fact that the write is a logical replacement of the deleted order, not new state creation. Total: 6 × 20,000 = 120,000 gas.
2. **Charge `WARM_SSTORE_RESET` on the old order's 6 clears**: When `orders[old_id].delete()` clears the old order's 6 slots (non-zero → zero), each slot is charged at the standard `WARM_SSTORE_RESET` (2,900 gas) — the normal cost for modifying an existing slot — but the 15,000-gas clearing refund is **not granted**. Total: 6 × 2,900 = 17,400 gas.
3. **Net effect**: The flip's 12 storage operations cost 137,400 gas total, compared to 1,500,000 gas under standard accounting. The state growth penalty is discounted since the net state change is zero.

The discount applies to the `fill_order` call as a whole — the create (`place_flip`) and delete (`orders[old_id].delete()`) are treated as a single logical operation regardless of their execution order within the call.

This is not a post-hoc refund — the state creation cost is never charged and the clearing refund is never granted. This means the gas is never counted against the transaction's gas limit, avoiding situations where a transaction would fail due to insufficient gas even though the net state change is zero.

### Precedent

TIP-1016 splits every cold SSTORE (zero → non-zero) into 20,000 regular gas (computational: writing, hashing, trie insertion) and 230,000 state gas (permanent storage burden). `STORAGE_MOVE` adopts the regular gas component directly: since a flip order's create-then-delete is net-zero for state growth, only the computational cost applies.

The expiring nonce system (TIP-1009) also discounts SSTORE for ephemeral data, but uses `WARM_SSTORE_RESET` (2,900 gas) rather than `STORAGE_MOVE` (20,000 gas) for its writes. This is because nonce writes are updates to existing ring buffer slots (non-zero → non-zero), not insertions into fresh slots — the trie node already exists, so no trie insertion work is needed. Flip order writes are genuine zero → non-zero transitions that create new trie nodes, so the 20,000-gas computational cost of trie insertion applies even though the state growth penalty does not.

`STORAGE_MOVE` may be adjusted in future TIPs based on real trie benchmarks, and could be reused for other precompile operations that move storage without net growth.

## Scope

- **Applies to**: The `fill_order` method in `StablecoinDEX` when processing a fully filled flip order that successfully places a new order via `place_flip()`.
- **Does not apply to**: Initial `placeFlip()` calls, regular order placement, order cancellation, or any non-DEX storage operations.
- **Does not apply if flip fails**: If `place_flip()` returns a business logic error (e.g., insufficient balance, policy rejection), the `STORAGE_MOVE` discount is not applied since no new order was created. The old order's slot clears earn the standard 15,000-gas clearing refund as they normally would — gas accounting is identical to filling a non-flip order. Any `STORAGE_MOVE` gas consumed by the reverted `place_flip` writes is not refunded, consistent with standard EVM gas-on-revert semantics.

## Tick-level storage

The discount covers only the `Order` struct slots. Tick-level storage changes (clearing an exhausted tick level, initializing a new tick level) are **not** discounted, because:

1. The old and new orders may be at different ticks, so tick-level storage changes are not guaranteed to be symmetric.
2. Tick-level slots may already be warm from other orders at the same tick.

## Hardfork gating

This change is gated behind a hardfork flag. The discount is only applied when the active spec satisfies the gating check.

# Invariants

- **Net state neutrality**: A flip order fill always deletes exactly 6 `Order` storage slots and creates 6 new ones. The number of slots discounted from state creation cost must equal the number of slots where the clearing refund is withheld.
- **No discount on flip failure**: If `place_flip()` fails, standard gas accounting applies — the old order's slot clears earn the normal 15,000-gas clearing refund and no `STORAGE_MOVE` discount is applied. Gas behavior is identical to filling a non-flip order.
- **Gas never counted against limit**: The discounted state creation cost must never be charged, even transiently. The gas must not count against the transaction's gas limit at any point during execution.
- **No clearing refund on flip**: The old order's slot clears must not grant the 15,000-gas refund when the flip succeeds, since the paired state creation cost is also not charged.
- **Isolated to precompile**: The gas-neutral treatment must not affect SSTORE gas accounting for any contract or precompile outside of `StablecoinDEX.fill_order`.
