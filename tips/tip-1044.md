# TIP-1044: Waive 1,500,000-gas storage initialization cost for DEX flip order placement

## Abstract

When a StablecoinDEX flip order is filled, the precompile deletes the old order's 6 storage slots and writes a new order to 6 fresh slots. Under TIP-1000's gas schedule the new order pays the full 250,000-gas state creation cost per slot (1,500,000 gas total), even though an equal number of slots are simultaneously cleared. This TIP makes the flip's storage operations gas-neutral: the state creation cost for the new order is not charged, and the clearing refund for the old order is not granted. The net effect is that the swap is priced as though the order slots were overwritten in place.

## Motivation

Flip orders are designed as perpetual two-sided liquidity: when one side fills, a new order is automatically placed on the opposite side. Because the old order is deleted and the new order is created in the same `fill_order` call, the net state growth is zero — the same number of storage slots are cleared as are initialized.

However, the SSTORE gas schedule prices each slot independently. Per TIP-1000, Tempo charges 250,000 gas for writing a new state element (zero → non-zero), while clearing a slot (non-zero → zero) earns only a 15,000 gas refund capped at 20% of total transaction gas by EIP-3529. The refund is negligible compared to the initialization cost, so swappers pay the full state creation price even though no net storage is consumed.

This makes filling flip orders substantially more expensive than filling regular orders of the same size, increasing costs for swappers who interact with them. Since the StablecoinDEX is a precompile with full control over its own gas accounting, it can apply a targeted discount without modifying the general EVM gas schedule.

## Assumptions

- The `Order` struct occupies 6 storage slots (determined by the `Storable` derive macro). This does not change between the filled order and the newly placed flip order.
- The old order is always fully deleted (`orders[old_id].delete()`) in the same call that places the new order (`place_flip()`). This is guaranteed by the `fill_order` control flow.
- This discount applies only within the StablecoinDEX precompile's `fill_order` path and does not affect general SSTORE gas accounting elsewhere in the EVM.
- If the `Order` struct layout changes in a future TIP (adding or removing fields), the discount must be re-evaluated to ensure slot counts still match.

# Specification

## Gas-neutral flip storage

When `fill_order` processes a flip order (i.e., `order.is_flip() == true`), the precompile treats the delete-then-create pair as a single storage-neutral operation. The new order's 6 slots are charged at `STORAGE_MOVE` (20,000 gas) instead of the TIP-1000 state creation cost (250,000 gas), saving 6 × 230,000 = 1,380,000 gas, and the old order's 6 slot clears do not earn the 15,000-gas clearing refund.

`STORAGE_MOVE` is a new constant defined by this TIP, set to 20,000 gas. This value matches the regular (computational) gas component of a cold SSTORE as defined in TIP-1016's regular/state gas split, and reflects the actual cost of inserting a new trie node — writing and hashing — without the 230,000-gas state gas component that pays for permanent state growth. Since a flip order's storage is net-zero (every slot created is paired with a slot cleared in the same call), the permanent storage burden does not apply, but the trie computation cost does.

Specifically:

1. **Charge `STORAGE_MOVE` on the new order**: When `place_flip()` writes the new order's slots (zero → non-zero), the TIP-1000 state creation cost is **not charged**. Instead, each slot is charged at `STORAGE_MOVE` (20,000 gas), covering the computational cost of the trie insertion. This reflects the fact that the write is a logical replacement of the deleted order, not new state creation.
2. **Skip clearing refund on the old order**: When `orders[old_id].delete()` clears the old order's slots (non-zero → zero), the 15,000-gas refund is **not granted**. The clearing is treated as the other half of the same logical replacement.
3. **Net effect**: The flip's storage operations are priced at their computational cost only, waiving the state growth penalty since the net state change is zero.

This is not a post-hoc refund — the state creation cost is never charged and the clearing refund is never granted. This means the gas is never counted against the transaction's gas limit, avoiding situations where a transaction would fail due to insufficient gas even though the net state change is zero.

### Precedent

TIP-1016 splits every cold SSTORE (zero → non-zero) into 20,000 regular gas (computational: writing, hashing, trie insertion) and 230,000 state gas (permanent storage burden). `STORAGE_MOVE` adopts the regular gas component directly: since a flip order's delete-then-create is net-zero for state growth, only the computational cost applies. This value may be adjusted in future TIPs based on real trie benchmarks, and could be reused for other precompile operations that move storage without net growth.

## Scope

- **Applies to**: The `fill_order` method in `StablecoinDEX` when processing a fully filled flip order that successfully places a new order via `place_flip()`.
- **Does not apply to**: Initial `placeFlip()` calls, regular order placement, order cancellation, or any non-DEX storage operations.
- **Does not apply if flip fails**: If `place_flip()` returns a business logic error (e.g., insufficient balance, policy rejection), the `STORAGE_MOVE` discount is not applied since no new order was created. The old order's slot clears earn the standard 15,000-gas clearing refund as they normally would — gas accounting is identical to filling a non-flip order.

## Tick-level storage

The discount covers only the `Order` struct slots. Tick-level storage changes (clearing an exhausted tick level, initializing a new tick level) are **not** discounted, because:

1. The old and new orders may be at different ticks, so tick-level storage changes are not guaranteed to be symmetric.
2. Tick-level slots may already be warm from other orders at the same tick.

## Hardfork gating

This change is gated behind a hardfork flag. The discount is only applied when the active spec satisfies the gating check.

# Invariants

- **Net state neutrality**: A flip order fill always deletes exactly 6 `Order` storage slots and creates 6 new ones. The number of slots exempted from state creation cost must equal the number of slots exempted from clearing refund.
- **No exemption on flip failure**: If `place_flip()` fails, standard gas accounting applies — the old order's slot clears earn the normal 15,000-gas clearing refund and no `STORAGE_MOVE` discount is applied. Gas behavior is identical to filling a non-flip order.
- **Gas never counted against limit**: The exempted state creation cost must never be charged, even transiently. The gas must not count against the transaction's gas limit at any point during execution.
- **No clearing refund on flip**: The old order's slot clears must not grant the 15,000-gas refund when the flip succeeds, since the paired state creation cost is also not charged.
- **Isolated to precompile**: The gas-neutral treatment must not affect SSTORE gas accounting for any contract or precompile outside of `StablecoinDEX.fill_order`.
